# AI Slice 项目架构解读

> 目标:从用户操作一路到 AI 出片,把"前端 → 后端 → AI"全链路串清楚。
> 不深入代码细节,只讲架构关系、模块职责、数据流。

---

## 1. 一句话理解

**这是一个"视频/音频 → AI 自动找精彩片段 → 输出短视频切片"的端到端 Agent。**

用户传一段长视频(直播回放、面试、讲座……),AI 自己听完、自己挑出 10 个左右最有传播价值的片段,自动配标题、文案,前端再用浏览器内的 FFmpeg.wasm 一键裁切打包 ZIP 下载。

---

## 2. 系统三层架构

```
┌──────────────────────────────────────────────────────────────┐
│              前端 Frontend (浏览器内)                          │
│   React 19 + Vite + Tailwind + FFmpeg.wasm                   │
│   职责:UI、浏览器内音频提取、上传、SSE 进度监听、ZIP 切片导出  │
└────────────────┬─────────────────────────────────────────────┘
                 │  HTTP / SSE
                 ▼
┌──────────────────────────────────────────────────────────────┐
│              后端 Backend (FastAPI)                            │
│   ├ REST API:上传/任务 CRUD/进度 SSE                          │
│   ├ Task Runner:后台轮询 pending 任务并串行执行                │
│   └ Pipeline:转录 → AI 分析 → (可选)FFmpeg 切片              │
└────────────┬─────────────────────────────┬──────────────────┘
             │                             │
             ▼                             ▼
   ┌─────────────────┐         ┌───────────────────────┐
   │  PostgreSQL      │         │  外部 AI 服务          │
   │  Task / Clip 表  │         │  • Groq Whisper (ASR)│
   └─────────────────┘         │  • DeepSeek (LLM)    │
   ┌─────────────────┐         └───────────────────────┘
   │  本地文件存储     │
   │  ./storage       │
   └─────────────────┘
```

| 层 | 关键技术 | 核心职责 |
|---|---|---|
| **前端** | React 19 + TypeScript + Vite + Tailwind + FFmpeg.wasm | 用户界面、浏览器内提取音频、上传、SSE 实时进度、浏览器内切片导出 ZIP |
| **后端** | FastAPI(async)+ SQLAlchemy 2.0 + Alembic | REST API、任务编排、Pipeline 执行、SSE 推送 |
| **存储** | PostgreSQL + 本地文件系统 | 任务/切片元数据、上传音频/视频、切片输出 |
| **外部 AI** | Groq Whisper、DeepSeek Chat | 语音转录、内容理解与精彩片段筛选 |

## 3. 核心流程总览

### 全链路时序

```
[用户]
  │ 1. 拖入视频
  ▼
[浏览器/FFmpeg.wasm]
  │ 2. 本地提取 16kHz mono MP3 音频   ← 2.5GB → 30MB (-99%)
  │ 3. POST /api/upload/audio          ← 只传 30MB
  ▼
[FastAPI /api/upload/audio]
  │ 4. 流式落盘到 ./storage/uploads/
  │ 5. POST /api/tasks 写入 Task(status=pending)
  ▼
[Task Runner 轮询线程]
  │ 6. 每秒扫一次 status=pending 任务
  │ 7. 抢占成功后:status → downloading,开始执行 Pipeline
  ▼
[Pipeline]
  │ 8. Step 1:音频已就绪(若是视频则 FFmpeg 提取音频)→ status=transcribing,10%
  │ 9. Step 2:Groq Whisper 转录(长音频自动分段)→ 60%
  │10. Step 3:DeepSeek 分批分析转录文本 → 输出 N 个切片建议 → 80%
  │11. Step 4:(视频模式)FFmpeg 批量裁切 mp4 → 90%
  │12. Step 5:把切片记录写入 clips 表 → status=done,100%
  ▼
[前端 SSE 长连接]
  │ 同步推送 8~12 步骤的进度,UI 实时刷新百分比和文案
  ▼
[用户进入任务详情页]
  │13. 看到切片清单(标题、时间区间、爆款分、推荐文案)
  │14. 一键「✂️ 切片」→ FFmpeg.wasm 在浏览器内按时间段批量裁
  │15. 打包 ZIP 下载本地
```

## 4. 数据流详解

### 4.1 上传阶段

**关键设计:浏览器内提取音频再上传(默认推荐)**

| 步骤 | 在哪里 | 做什么 | 数据形态 |
|------|--------|--------|---------|
| 拖入视频 | 浏览器 | `FileUploader` 拿到 `File` 对象 | 2.5 GB MP4 |
| FFmpeg.wasm 提取音频 | 浏览器 | WORKERFS 零拷贝挂载源文件,转码为 16kHz mono MP3 64kbps | ~30 MB MP3 |
| 上传 | 浏览器 → 后端 | `POST /api/upload/audio` multipart | ~30 MB |
| 落盘 | 后端 | 流式分块写到 `./storage/uploads/<timestamp>_<filename>` | 本地文件 |
| 创建任务 | 后端 | `POST /api/tasks`,把 `audio_path` 写入 `Task.source_path` | DB 一行 |

> 💡 **为什么先在浏览器提音频?**
> 直播回放动辄 2~10 GB,直传后端慢且占带宽。先在浏览器内用 FFmpeg.wasm 提音频(AI 只需要音频),上传体积缩到 1%,极大缩短"等上传"的时间。
>
> 不愿意走这条路的也支持完整视频直传(`/api/upload/video`),后端 Pipeline 会自动用本地 FFmpeg 提音频。

### 4.2 任务调度

| 角色 | 实现 | 说明 |
|------|------|------|
| **入队** | API 创建 Task,`status=pending` | 不阻塞 HTTP 请求,立即返回 task_id |
| **出队** | `task_runner._task_runner_loop` 每秒扫一次 `pending` 任务 | 单机进程内串行执行,避免抢占 |
| **领取** | `_claim_next_pending_task` 取最早一条,改 `status=downloading` | 数据库当作消息队列 |
| **执行** | `run_video_pipeline(task_id)` | 真正干活的 Pipeline |
| **崩溃恢复** | `_reset_interrupted_tasks` 启动时把残留的"进行中"任务重置回 pending | 进程被杀也能自动续上 |

> 💡 **没用 Celery / RQ 等专门的队列?**
> 项目主打"零云依赖、单机可跑",直接用 PostgreSQL 当队列足够,部署/运维大幅简化。

### 4.3 Pipeline 内部(核心)

```
                         ┌────────────────┐
   音频 / 视频   ──────▶ │  Step 1        │
                         │  音频准备       │
                         │ (FFmpeg 提音频)│
                         └────────┬───────┘
                                  │ MP3 16kHz mono
                                  ▼
                         ┌────────────────┐
                         │  Step 2 ASR    │
                         │ Groq Whisper   │
                         │ 长音频自动分段  │
                         └────────┬───────┘
                                  │ transcript:[{start,end,text}, ...]
                                  ▼
                         ┌────────────────┐
                         │  Step 3 LLM    │
                         │ DeepSeek 分析   │
                         │ 按 6000 token  │
                         │ 分批 + 去重     │
                         └────────┬───────┘
                                  │ clips_plan:[{start,end,title,score,...}]
                                  ▼
                  ┌───────────────┴───────────────┐
                  │                               │
        ┌─────────▼─────────┐         ┌──────────▼──────────┐
        │ 音频直传模式        │         │ 视频模式             │
        │ 仅入库时间点清单    │         │ FFmpeg 批量裁 MP4   │
        │ 用户去剪映/前端剪   │         │ 复制到 ./storage    │
        └─────────┬─────────┘         └──────────┬──────────┘
                  └─────────────┬─────────────────┘
                                ▼
                       Task.status = done
                       Clip 记录入库
                       SSE 推送 100%
```

**几个关键设计点:**

1. **双模式自动切换**
   `pipeline.py` 根据 `Task.source_path` 后缀判定:音频文件(`.mp3/.wav/...`)走"音频直传模式"(跳过 FFmpeg 提音频和后端切片);视频文件走完整流程。

2. **LLM 分批策略**
   `analyzer.py` 把转录按 ~6000 token 切批,每批独立调一次 DeepSeek,最后合并 + 去重(重叠率 > 30% 取高分项)。这样既能处理 6 小时长视频,又避免单次上下文爆炸。

3. **Prompt 场景化**
   `SceneConfig` 内置 3 个模式:`livestream`(直播) / `interview`(面试) / `lecture`(讲座),Prompt 不同、最小切片时长不同、去重阈值不同。用户上传时选场景。

4. **进度统一写库**
   每个 Step 都通过 `update_task_progress(db, task, percent, message, status)` 把进度写到 `tasks` 表,前端 SSE 从库里读。这样**进度源唯一**,即使有多个客户端订阅也一致,不会丢。

### 4.4 实时进度推送(SSE)

```
[前端]                              [后端]                       [DB]
  │ GET /api/tasks/{id}/progress      │                          │
  │ ────────────────────────────────▶ │ event_stream() 循环开始  │
  │                                   │ ──── SELECT progress ──▶ │
  │ ◀── data:{"progress":15,...} ─── │ ◀── 15%, "提取音频..." ─ │
  │                                   │ sleep(1s)                │
  │ ◀── data:{"progress":60,...} ─── │ ◀── 60%, "转录完成"  ─── │
  │                                   │  ...                     │
  │ ◀── data:{"progress":100,...} ── │ status=done 时 break    │
```

每 1 秒推一次,前端 `EventSource` 接收并刷新进度条。这个机制简单稳定,不需要 WebSocket。

## 5. AI 模块详解

### 5.1 ASR(语音转录)— Groq Whisper

| 项 | 内容 |
|----|------|
| 服务 | [Groq](https://groq.com/) 的 `whisper-large-v3-turbo` |
| 调用方 | `transcriber.py` 内 `GroqASRTranscriber` |
| 输入 | 16kHz mono MP3 |
| 输出 | 时间戳化的文本段数组:`[{start: 12.3, end: 14.7, text: "..."}, ...]` |
| 关键依赖 | 后端本地的 **FFprobe**(读时长)+ **FFmpeg**(切片) |

**长音频自动分段策略**

`_split_audio()` 会按 `GROQ_ASR_CHUNK_MINUTES`(默认 **25 分钟**)阈值决定是否分段:

| 音频时长 | 处理方式 |
|---------|---------|
| ≤ 25 分钟 | **不分段**,整段直接发给 Groq |
| > 25 分钟 | 用本地 `ffmpeg -c copy`(无损流复制)按 25 分钟切成 N 段,**串行**发给 Groq(逐段拿结果),每段返回的相对时间戳加上 `offset` 还原成绝对时间,最后合并排序 |

> 例:**1 小时音频** → 切成 3 段(25min + 25min + 10min)→ 依次调用 3 次 Groq → 拼成完整 transcript。
> 切片用 `-c copy` 不重编码,几乎不消耗 CPU、几乎瞬间完成;真正耗时的是 3 次网络往返到 Groq。

> 为什么要分段:Groq 单次上传文件大小、单次推理时长都有上限,25 分钟是经验上既能避免超限、又能让单次响应控制在合理时间的折中值,可通过 `.env` 的 `GROQ_ASR_CHUNK_MINUTES` 调整。

### 5.2 LLM(内容理解)— DeepSeek Chat

| 项 | 内容 |
|----|------|
| 服务 | [DeepSeek](https://www.deepseek.com/) 的 `deepseek-chat`(走 OpenAI 兼容协议) |
| 调用方 | `analyzer.py` 内 `ClipAnalyzer` |
| 输入 | 带时间轴的转录文本(纯文本拼接,见 `_format_transcript`) |
| Prompt | 内置 3 套场景 Prompt(直播/面试/讲座) |
| 输出契约 | JSON 数组:每项含 `clip_id / title / start_time / end_time / type / summary / virality_score(1-10) / suggested_caption` |
| 健壮性 | 单批最多重试 3 次;markdown code fence 自动剥;非数组兜底用正则提取 |
| 后处理 | ① 按分数降序 ② 双向重叠去重 ③ 越界过滤 |

### 5.3 增值生成器(按需调用)

| 模块 | 触发 | 说明 |
|------|------|------|
| `viral_title_generator.py` | 前端点「生成爆款标题」 | 调用 DeepSeek 为单个切片生成 5 个抖音风格标题(悬念/反常识/情绪化/数字化/反差) |
| `editing_guide_generator.py` | 前端点「AI 剪辑思路」 | 调用 DeepSeek 输出结构化剪辑指导:特效时间点、配乐推荐、字幕贴纸、节奏控制、封面截取 |

这两块独立于主 Pipeline,**用户按需触发**,失败也不影响主流程。

## 6. 存储模型

### 6.1 数据库表(简化)

```
tasks
├─ id (UUID, PK)
├─ status:enum(pending/downloading/transcribing/analyzing/clipping/uploading/done/failed)
├─ scene_mode:enum(livestream/interview/lecture)
├─ source_path:本地路径(音频或视频)
├─ video_filename / video_duration / video_start_offset
├─ progress (0-100) + progress_message       ← SSE 数据源
├─ transcript_json:转录结果(JSON)             ← 留存便于重试/调试
├─ error_message
└─ created_at / updated_at

clips
├─ id (UUID, PK)
├─ task_id (FK → tasks.id)
├─ clip_index / title / summary / clip_type
├─ start_time / end_time / duration
├─ virality_score (1-10)
├─ suggested_caption
├─ viral_titles_json:🆕 5 个爆款标题数组
├─ editing_guide_json:🆕 AI 剪辑思路结构体
└─ file_key:切片 MP4 的相对路径(音频直传模式为 NULL)
```

> 任务**状态机的中间状态**(transcribing / analyzing / clipping...)同时承担"进度阶段标识"的功能,前端按状态切换 UI 文案。

### 6.2 文件系统

```
storage/
├─ uploads/                # 用户上传的原始音频/视频
│   └─ 1715567890_xxx.mp3
└─ clips/                  # 切片输出
    └─ <task_id>/
        ├─ clip_001.mp4
        ├─ clip_002.mp4
        └─ ...
```

零云依赖,删任务时清这两块的文件即可。

## 7. 前端核心交互

| 页面 | 路径 | 核心动作 |
|------|------|--------|
| 上传页 | `/` | 拖入视频 → 浏览器内提音频 → 上传 → 创建任务 → 跳详情页 |
| 任务列表 | `/tasks` | 看板风格,每个任务卡显示状态/进度/缩略信息;支持重命名/重试/删除 |
| 任务详情 | `/tasks/:id` | 上半部分是任务状态 + SSE 实时进度条;下半部分是切片清单(`ClipCard`),每张卡可看时间区间、爆款分、复制时间点、生成爆款标题/AI 剪辑思路;顶部按钮「✂️ 一键切片」用 FFmpeg.wasm 浏览器内裁切打包 ZIP |

**FFmpeg.wasm 的两次出场**:
1. **上传前**——提取音频(把 2GB 视频压成 30MB MP3)
2. **导出时**——按 AI 给出的时间段批量裁视频,打包 ZIP

两次都让用户的浏览器干活,后端不需要处理重量级转码,部署成本极低。

## 9. 几个值得记住的设计取舍

| 取舍 | 原因 |
|------|------|
| **浏览器内提音频而不是直传视频** | 上传体积缩到 1%,用户等待时间从分钟级 → 秒级,后端带宽和磁盘压力都骤减 |
| **PostgreSQL 当任务队列** | 单机部署,无需引入 Redis/RabbitMQ/Celery,运维成本最低 |
| **进度统一写库 + SSE 读库** | 进度数据源唯一,断线重连可继续,多终端同步天然一致 |
| **AI 调用全部走 HTTP 云服务**(Groq + DeepSeek) | 本机不跑大模型,部署轻量;Groq Whisper 处理 6 小时音频也只要几分钟 |
| **切片导出放浏览器(FFmpeg.wasm)** | 不占后端 CPU,用户机器算力闲置反而被利用上 |
| **场景化 Prompt** | 直播/面试/讲座的"精彩"定义完全不同,一套 Prompt 解决不了所有问题 |
| **LLM 分批 + 去重** | 长视频上下文超限的工程兜底;同时让"重叠的相似切片"合并为最佳一条 |

---

## 10. 一句话总结

**AI Slice = 浏览器内 FFmpeg.wasm 把视频"轻量化" + 后端 Pipeline 把"听觉信息"变成"语义片段建议" + 数据库当队列让长任务可观测可恢复 + 前端再用一次 FFmpeg.wasm 把建议变成可下载的成片。**

整个项目把"用户等"这件事的成本压到了极致,把"AI 算"这件事的复杂度拆成了可观测、可重试的 Pipeline。

## 11. 常见疑问 FAQ

### Q1:本地下载的 `ffmpeg.exe` 是给前端 FFmpeg.wasm 用的吗?

**不是。两个完全独立、互不依赖。**

| | 本地 `ffmpeg.exe` / `ffprobe.exe` | 前端 FFmpeg.wasm |
|---|---|---|
| **运行位置** | 后端服务器(Windows 上的 exe 进程) | 用户浏览器内(WebAssembly) |
| **谁加载** | `services/transcriber.py`、`services/clipper.py` 通过 `subprocess` 调用 | `frontend/src/services/ffmpegRuntime.ts` 通过 npm 包 `@ffmpeg/ffmpeg` + `@ffmpeg/core` 加载 |
| **来源** | 用户从 <https://ffmpeg.org/download.html#build-windows> 下载,路径配在 `.env` 的 `FFMPEG_BIN_DIR` | 跟随 `npm install` 安装的 wasm 文件,Vite 构建时打包 |
| **用途** | ① 后端从视频提音频 <br/>② 长音频按 25 分钟切段(给 ASR)<br/>③ 视频模式下批量裁切 MP4 | ① 上传前在浏览器内从视频提取 MP3 <br/>② 一键导出时按 AI 时间段批量裁视频打 ZIP |
| **依赖关系** | 跟 wasm 无关 | 跟本地 exe 无关 |

> **直观记忆**:本地 `ffmpeg.exe` 是**后端的工具**,前端 FFmpeg.wasm 是**浏览器自带的"小型 ffmpeg"**。
> 即使你没装本地 ffmpeg,前端 wasm 也能跑;反过来,即使浏览器不支持 wasm,后端 ffmpeg 也照常工作(只是要走"视频直传"模式而非"音频预提取")。

**那为什么后端还需要本地 ffmpeg?**
因为 ASR 长音频分段、视频模式的批量裁切要在服务端做,wasm 跑在用户浏览器里,后端访问不到。两边各干各的。

### Q2:浏览器提取出 1 小时音频,Groq ASR 那边是一次性处理,还是要分段?

**会自动分段,1 小时音频 → 切成 3 段串行转录。**

具体逻辑在后端 `transcriber.py` 的 `_split_audio()` + `GroqASRTranscriber.transcribe()`:

```
[前端上传 1h MP3]
        │
        ▼
[后端 transcribe() 被调用]
        │
        │ 读取 GROQ_ASR_CHUNK_MINUTES = 25 (分钟)
        │
        ▼
[_split_audio]
        │ 1h > 25min,需要切段
        │ 用本地 ffmpeg -c copy 无损流复制
        │
        ├─▶ chunk_000.mp3 (0 ~ 25min,    offset=0s)
        ├─▶ chunk_001.mp3 (25 ~ 50min,   offset=1500s)
        └─▶ chunk_002.mp3 (50 ~ 60min,   offset=3000s)
        │
        ▼
[串行循环:for chunk in chunks]
        │
        │ chunk_000 → POST Groq → segments → 时间戳加 0s
        │ chunk_001 → POST Groq → segments → 时间戳加 1500s
        │ chunk_002 → POST Groq → segments → 时间戳加 3000s
        │
        │ 删除临时 chunk 文件
        │
        ▼
[合并 + 按 start 升序排序]
        │
        ▼
[返回完整 transcript,时间戳是 0~3600s 的绝对值]
```

**几个细节:**

1. **阈值可配**:`.env` 里改 `GROQ_ASR_CHUNK_MINUTES` 可以调大调小(默认 25 分钟,Groq 单次请求最稳定的区间)
2. **切片几乎零开销**:`-c copy` 是流复制(只重写容器、不重编码),1 小时音频切 3 段大概几秒钟搞定,主要耗时在 3 次 Groq 网络往返
3. **是串行不是并发**:代码里是普通 `for` 循环 + `await`,一段一段处理。**没并发是有意为之**——并发的话容易触发 Groq 速率限制,而且 Whisper 转录本身就快(Groq 的 turbo 模型大概 1 秒处理几十秒音频),串行也只多花几十秒
4. **短音频不切**:`≤ 25 分钟` 直接走 `_transcribe_single`,不进 `_split_audio` 的分段分支
5. **时间戳还原**:每段 Groq 返回的时间戳都是从 0 开始的"相对时间",代码里加 `offset` 还原成"绝对时间"(`seg["start"] += offset`),保证下游 LLM 拿到的转录是连续的、正确的全局时间轴

**那"前端浏览器内 FFmpeg.wasm 提的音频"和"后端 ffmpeg 给 ASR 分的段"有什么关系?**

没有关系,是**两段独立的处理**:

```
[浏览器 wasm 提音频]    [后端 ffmpeg 给 ASR 分段]
   2GB 视频              1h 完整 MP3
      ↓                       ↓
   1h MP3 ──── 上传 ────▶  切成 3 段
                              ↓
                          逐段调 Groq
                              ↓
                          合并成完整 transcript
```

浏览器提音频是"为了少传文件给后端",后端 ASR 分段是"为了避开 Groq 单次请求上限",目的完全不同。

### Q3:为啥前端能做音频提取和视频裁切,后端还要保留 FFmpeg?这不是浪费服务器资源吗?

**首先澄清:项目的默认链路就是你想的那样——前端干重活、后端不碰视频。** 后端 ffmpeg 只在两个绕不开的场景里出现。

#### 默认链路实际是什么

```
[浏览器]                                        [后端]
 提音频 (wasm)  ────── 上传 MP3 ───────▶  存盘
                                                │
                                                ▼
                                          (后端 ffmpeg 用了一下: 切音频段)
                                                │
                                                ▼
                                          调 Groq + 调 DeepSeek
                                                │
                                                ▼
                                          切片清单入库,推送 100% 给前端
 裁切视频 (wasm) ◀──── 拿清单 ──────────  /api/tasks/{id}
 打包 ZIP 下载
```

整条默认链路里,**后端只摸了一次音频**(切段给 ASR),完全没碰过视频文件。

#### 后端 FFmpeg 用在哪里:三个场景

| 场景 | 触发条件 | 操作 | 开销 | 是否可去掉 |
|------|---------|------|------|----------|
| **① ASR 长音频分段** | 音频 > 25 分钟 | `ffmpeg -c copy` 流复制切段 | 极低(几秒) | **不能** |
| **② 视频直传模式提音频** | 用户走 `/api/upload/video` 直传完整视频 | `ffmpeg` 转码出 16kHz mono MP3 | 中等 | **不能**(这条路径用户选了就要做) |
| **③ 视频直传模式批量切片** | 同上 | `ffmpeg` 按时间段裁 MP4 | 较高 | **不能**(同上) |

**默认路径只触发 ①,② 和 ③ 是备用路径。** 用户不上传视频就完全不会跑。

#### 为什么不把 ① 也丢给浏览器?

理论上能做,但实际上不划算,三个原因:

1. **文件已经在后端了** — 用户传完音频,文件躺在 `./storage/uploads/` 里。要让浏览器切的话,得"后端 → 前端 → 后端"来回两次,纯浪费带宽和时间。

2. **Groq API Key 必须后端持有** — 调 ASR 这一步**铁定**是后端干的(密钥不能暴露给浏览器)。既然必须后端调,那紧挨着的"切段"也顺手后端做,代码最简单。

3. **`-c copy` 几乎不耗资源** — 这不是重编码,是"流复制":只重写文件容器、不解码不重压。1 小时音频切 3 段大概几秒钟,后端 CPU 占用几乎可以忽略。

#### 为什么保留 ② ③(视频直传)路径?

这是**给特殊场景留的后门**:

| 用户场景 | 选音频直传 | 选视频直传 |
|----------|----------|-----------|
| 普通用户在浏览器里完成 | ✅ 推荐 | ✗ |
| 浏览器不支持 SharedArrayBuffer / wasm | ✗ | ✅ 唯一选择 |
| 手机端浏览器 wasm 太慢或崩溃 | ✗ | ✅ |
| 想把切片**持久化到服务器**供长期下载/分享 URL | ✗(切完只能本地下载,关浏览器就没了) | ✅(切片存在 `./storage/clips/`,有持久下载 API) |
| 大型团队需要在多个客户端访问同一批切片 | ✗ | ✅ |

**两条路径各有定位:**
- **音频直传 + 浏览器裁切**:轻量、快速,**"看一次出片"** 的体验最佳
- **视频直传 + 后端裁切**:重型、持久化,**"产物长期保存"** 的能力托底

去掉视频直传的话,上面这些场景就没解。

#### 真的会"浪费服务器资源"吗?

| 默认路径(音频直传) | 后端开销 |
|-------------------|---------|
| 用户上传 30 MB MP3 | 几秒磁盘写入 |
| ffmpeg `-c copy` 切 3 段 | 几秒,CPU 几乎 0% |
| HTTP 调 Groq × 3 | 几乎不占 CPU(等网络) |
| HTTP 调 DeepSeek × N | 几乎不占 CPU(等网络) |
| 写库 + SSE 推送 | 微秒级 |

**实测下来,后端基本是 I/O 等待型负载,没什么 CPU 压力。** 真正的 CPU 重活(视频提音频、视频裁切)默认都没跑。

#### 结论

> 项目的架构原则就是 **"能在浏览器干的就在浏览器干,后端只做浏览器干不了的事"**:
> - 调 AI 服务(API Key 保护)→ 后端
> - 任务编排和持久化 → 后端
> - 长音频分段(顺手做)→ 后端
> - **视频转码、切片(默认)→ 浏览器**
>
> 后端保留 FFmpeg 不是"重复造轮子",而是 ① 给 ASR 分段当工具用 ② 给不能用 wasm 的客户端留兜底路径。
> 默认链路下,后端 FFmpeg 只跑很轻量的 `-c copy`,没有浪费。

### Q4:浏览器(FFmpeg.wasm)真的能扛住音频提取和视频切片吗?

**结论:能扛,但有明确的边界。** 项目对参数和文件加载方式做了几处关键优化,让 wasm 在桌面浏览器上是可行的。

#### 实际跑的参数

**音频提取**(`frontend/src/services/audioExtractor.ts`):

```
ffmpeg -i /input/video.mp4 -vn \
       -acodec libmp3lame -ar 16000 -ac 1 -b:a 64k \
       /output.mp3
```

- `-vn`:丢掉视频流(只要音轨)
- `-acodec libmp3lame -ar 16000 -ac 1 -b:a 64k`:**必须重编码**(改采样率/通道/编码器/码率),wasm 实际在做 CPU 密集工作
- 文件挂载用 **WORKERFS 零拷贝**:不把视频读进 wasm 内存,只在需要时按字节区间访问

**视频切片导出**(`frontend/src/services/videoClipExporter.ts`):

```
ffmpeg -ss <start> -i /input/video.mp4 -t <duration> \
       -c copy -avoid_negative_ts make_zero -y /output/clip.mp4
```

- `-ss` 在 `-i` 之前:**先定位再读取**,快速跳到最近的关键帧
- `-c copy`:**流复制,不重编码**,只重写容器
- 几乎是磁盘 IO 速度

#### 实测性能(经验值,因机器而异)

| 任务 | 普通桌面 Chrome/Edge | 性能瓶颈 |
|------|---------------------|---------|
| 1h 视频 → 16kHz MP3 | **6-12 分钟** | wasm 的 libmp3lame 编码,大约 5-10x 实时速度 |
| 4h 直播视频 → 16kHz MP3 | **25-50 分钟** | 同上,基本线性 |
| 10 个 1 分钟切片(`-c copy`) | **20-60 秒** | 主要是 ZIP 打包,FFmpeg 本身极快 |
| 2GB 视频文件加载 | **几秒挂载** | WORKERFS 零拷贝,不预读 |

#### 能扛住的关键设计

| 优化点 | 解决的问题 |
|--------|----------|
| **WORKERFS 零拷贝挂载** | 不把 2GB 视频整个塞进 wasm 内存(否则 32 位 wasm 4GB 寻址直接 OOM) |
| **音频参数极致压缩**(16kHz/单声道/64kbps) | 输出文件小(30 MB)+ 编码计算量小 + 后端 ASR 不需要更高质量 |
| **切片用 `-c copy`** | 不需要重编码,wasm 在这一步几乎是零负担 |
| **COOP/COEP 安全头**(Vite 配好) | 启用 `SharedArrayBuffer`,让 wasm 多线程跑(否则慢 5-10 倍) |
| **AbortSignal 全链路传递** | 用户取消任务时能立即终止 wasm 实例,不卡住浏览器 |

#### 能力边界(什么时候顶不住)

| 场景 | 能不能扛 | 原因 |
|------|---------|------|
| **6h+ 超长视频提音频** | ⚠️ 勉强 | 单次提取要 30-60 分钟,用户得保持 tab 活动;**建议:让用户用本地 ffmpeg 提取后走音频直传**(项目支持) |
| **手机浏览器** | ❌ 不行 | 移动端 wasm 性能差 + 内存限制严 + iOS Safari 之前对 SharedArrayBuffer 兼容差 |
| **iOS Safari < 17 / 老版本浏览器** | ❌ 不行 | 不支持 SharedArrayBuffer 或者支持不完整,wasm 退化到单线程,慢到不可用 |
| **需要帧级精确裁切** | ⚠️ 有偏差 | `-c copy` 只能切到关键帧,误差通常 0~5 秒;对短视频"AI 找的精彩片段"来说够用,但精剪不行 |
| **裁切后还要转码/调码率/加字幕** | ❌ 这套不支持 | 项目定位是"出粗剪",精修请丢剪映 |
| **企业内网/低配机器/Atom 类 CPU** | ⚠️ 慢但能跑 | 4GB 内存的老笔记本能跑,但音频提取可能要 30 分钟级 |

#### 项目对边界场景的兜底

| 边界场景 | 兜底方案 |
|---------|--------|
| 浏览器扛不住提音频 | 用户自己用本地 ffmpeg 提音频后**直接拖 MP3 上传**,跳过 wasm |
| 浏览器不能用 wasm | 走 **`/api/upload/video` 视频直传**,后端用本地 ffmpeg 干活(就是 Q3 说的备用路径) |
| 切片精度不够 | 任务详情页只输出**时间点清单**,用户拷贝到剪映精切 |

#### 一句话回答

> **桌面 Chromium 内核浏览器 + 中等以上配置 + 视频时长在 1-3 小时以内,wasm 完全胜任**。
> 这正是项目的目标用户场景。
> 边缘场景(超长视频、手机、老 Safari、要帧精度)有兜底路径,但不是 wasm 的舒适区——这种情况下走视频直传让后端干,或者干脆让用户本地 ffmpeg 预处理。

**所以这套架构本质上是用"目标用户的浏览器算力"换"服务器算力",在大多数场景下这笔买卖很划算;少数边缘场景留了后门。**

### Q5:Task Runner Loop 到底是干啥的?SSE 又是怎么和它配合的?

这是项目最核心的异步处理模型,先放结论:

> **Task Runner 解决"长任务"问题:用户点上传秒回,后台慢慢跑,跑的过程中把进度写到数据库,SSE 每秒读数据库推给前端。**

用户和"干活的人"完全不直接交流,中间隔着一张数据库表 —— 这就是这个架构的全部秘密。

#### 先搞清楚 SSE 是啥

**SSE = Server-Sent Events,本质是一个"半双工长连接"。**

```
        浏览器                               服务器
          │                                   │
          │  GET /api/tasks/xxx/progress      │
          │ ──────────────────────────────▶   │  连接建立(普通 HTTP)
          │                                   │
          │  ◀── data: {"progress": 15} ───   │  ← 连接不关闭
          │                                   │     服务器主动推
          │  ◀── data: {"progress": 60} ───   │
          │                                   │
          │  ◀── data: {"progress": 100} ──   │  推送最后一条
          │ ──────────── 关闭 ──────────────▶ │
```

| | SSE | WebSocket |
|---|---|---|
| 方向 | **单向**(server → client) | 双向 |
| 协议 | 普通 HTTP | 升级到 ws:// |
| 浏览器实现 | `new EventSource(url)`,一行代码 | 复杂 |
| 用途 | 推进度/通知 | 聊天/协作 |

> 项目只需要"服务器告诉浏览器进度",前端不需要往后端发消息,所以选 SSE 而不是 WebSocket。

#### 为什么不能"直接在 HTTP 请求里跑 Pipeline"?

来对比三种方案,你就明白 Task Runner 为什么必须存在。

**方案 1:同步处理(直接在 POST handler 里跑 Pipeline)**

```
[浏览器]                              [后端]
   │  POST /api/upload                  │
   │ ─────────────────────────────────▶ │ ← 开始处理
   │                                    │   提音频...转录...AI 分析...切片...
   │           (20 分钟)                 │
   │ ◀────────── 200 OK ───────────────  │
```

| 问题 | 后果 |
|------|------|
| 浏览器/Nginx 默认 60s 超时 | 1 分钟报 504 Gateway Timeout |
| 用户关浏览器 | 连接断开,前功尽弃 |
| 同时 5 个用户 | 5 个 HTTP 连接挂 20 分钟,worker 全占满 |
| 服务器重启 | 内存里的任务全没 |

❌ **同步死路。**

**方案 2:HTTP 秒回,后台开个线程慢慢跑(初级异步)**

```
POST 接口 ──▶ 启动后台线程 ──▶ 立刻返回 task_id
              (线程跑 20 分钟)
```

看着可以,但有 3 个致命问题:

| 问题 | 后果 |
|------|------|
| 服务器重启 = 线程全死 | 任务消失,DB 没记录,找不回 |
| 用户怎么看进度? | 前端只能 setInterval 轮询,服务器压力大 |
| 同时来 50 个用户 = 50 个线程并发跑 | 内存爆 / API 限流 |

**方案 3:Task Runner(项目采用)**

```
POST /api/tasks ──▶ INSERT task=pending ──▶ 50ms 返回 task_id
                        ↓
                 ┌──────────────────────────────────┐
                 │  Task Runner 后台常驻协程         │
                 │  每秒 SELECT 一次 pending 任务    │
                 │  抢到一个就跑 Pipeline,期间      │
                 │  每完成一步 UPDATE progress      │
                 └──────────────────────────────────┘
                        ↓
                 SSE 接口每秒 SELECT progress 推给前端
```

✅ **这就是项目的方案。**

#### 用"工厂车间"隐喻一句话理解

| 工厂车间 | 项目代码 |
|---------|----------|
| 流水线工人(只管干活,不管报告) | **Task Runner Loop** |
| 工人手边的进度卡片(干一步打一个钩) | DB `tasks.progress` 字段 |
| 巡视员(每分钟过来看卡片打了几个钩) | **SSE 接口** |
| 监控大屏(显示进度) | 前端的进度条 |

> 工人不需要喊话告诉所有人"我做完一步了",他**只管打钩**。
> 巡视员不需要打扰工人,他**只管看卡片**。
> 所有人通过那张卡片间接同步 —— 这就是用"共享状态"做异步通信的经典思路。

#### 完整时序图

```
时间    [浏览器]         [POST API]       [Task Runner]      [SSE API]         [DB]
─────  ──────────       ──────────       ──────────────     ──────────       ─────────
0:00s  POST /api/tasks ─▶
                         INSERT ──────────────────────────────────────────▶   task=pending,0%
0:01s  ◀── task_id ────

0:02s  GET .../progress
       (SSE 建立) ═════════════════════════════════════════════════════════▶
                                          扫表 ──────────────────────────▶   SELECT pending
                                          ◀── 抢到了 ─────────────────────   UPDATE→downloading
                                          开始 Step1:提音频
0:03s                                    写进度 10% ────────────────────▶   progress=10
0:04s                                                         ◀── 读 ─────   progress=10
       ◀══════ progress:10 ═══════════════════════════════════
                                          Step2:转录
```

#### 4 个关键事实

| 事实 | 为什么重要 |
|------|----------|
| **POST 接口和 Task Runner 不直接说话** | 接口只写一行 `pending` 就返回,跟干活的人解耦 |
| **Task Runner 和 SSE 接口不直接说话** | Runner 只写 DB,SSE 只读 DB,各干各的 |
| **数据库是唯一的"中间人"** | 状态/进度/任务队列全靠这一张 `tasks` 表 |
| **进程崩了不丢任务** | DB 是持久的,重启时把卡在中间态的任务捡回来重跑 |

#### 把 Task Runner 砍掉会怎样(反向验证)

| 场景 | 没有 Runner 的后果 |
|------|-----------------|
| 用户上传 1h 视频 | HTTP 卡 10 分钟 → 浏览器/Nginx 504 → 上传失败 |
| 用户中途关浏览器 | 连接断开 → asyncio 任务被取消 → 处理中断 |
| 同时 3 个用户上传 | 3 个 ffmpeg + 3 个 LLM 并发 → 内存爆 |
| `Ctrl+C` 重启 uvicorn | 内存里 3 个任务全没,DB 也没记录 |

**Runner 一招全解:把任务变成数据库里的一条记录,让一个独立的后台协程慢慢取出来串行处理。**

#### 一句话总结

> **Task Runner 解决"用户秒回 + 服务器可控 + 重启不丢任务"。**
> **SSE 解决"用户能看到实时进度"。**
> 两者通过 PostgreSQL 的 `progress` 字段间接联动:**Runner 写,SSE 读**,从不直接交流。
