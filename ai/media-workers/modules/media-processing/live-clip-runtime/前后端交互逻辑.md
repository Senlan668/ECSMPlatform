# 前后端交互逻辑详解

> 项目：AI Slice（直播切片自动剪辑）
> 前端：React + Vite + TypeScript + axios + EventSource + FFmpeg.wasm
> 后端：FastAPI + SQLAlchemy(async) + PostgreSQL + Whisper + DeepSeek
> 通信：HTTP REST（JSON / multipart）+ Server-Sent Events（SSE）
> 代理：Vite dev server 把 `/api` 代理到 `http://localhost:8001`

---

## 0. 全局通信约定

| 项 | 配置 |
|----|------|
| 前端 axios 实例 | `frontend/src/services/api.ts` |
| baseURL | `/api`（经 Vite 代理到 FastAPI） |
| 默认 timeout | 10 秒（上传、AI 接口单独覆盖） |
| FastAPI 路由前缀 | `/api/upload`、`/api/tasks`、`/api`（clips）|
| CORS | 后端开 `allow_origins=["*"]`，开发环境 |
| 一致性约束 | 浏览器端 FFmpeg 抽音频参数与后端 `extract_audio()` 严格对齐：16 kHz / 单声道 / MP3 64 kbps |

## 1. 整体数据流概览

```
┌──────────────────────────────┐                          ┌──────────────────────────────┐
│           前端                │                          │           后端                │
│ (浏览器：React + FFmpeg.wasm) │                          │  (FastAPI + Worker + Whisper) │
└──────────────┬───────────────┘                          └──────────────┬───────────────┘
               │                                                          │
   ① 选择/拖入视频文件 (本地 File)                                          │
   ② 浏览器端 WORKERFS 挂载 + FFmpeg 抽 MP3 音频 (~30MB)                    │
               │                                                          │
   ③ POST /api/upload/audio  ────── multipart 上传 MP3 ──────────────────► │
               │                                                          │ 写入 storage/uploads/，返回本地路径
               │ ◄──────── { audio_path, original_filename, size_bytes }──┤
               │                                                          │
   ④ POST /api/tasks  ──── JSON: { video_path, video_filename, ... } ────► │
               │                                                          │ 写入 tasks 表，status=pending
               │ ◄────────────────── TaskResponse (含 id) ────────────────┤
               │                                                          │
   ⑤ 跳转 /tasks/{id}                                                      │
               │                                                          │
   ⑥ GET /api/tasks/{id}  ── 拉取详情(首次加载)──────────────────────────► │
               │ ◄──── TaskResponse (含 clips 列表 + download_url) ───────┤
               │                                                          │
   ⑦ GET /api/tasks/{id}/progress  ── EventSource (SSE) ────────────────► │
               │                                                          │ 后台 task_runner 每秒推进度
               │ ◄════════════ data: {progress, message, status} ═════════│
               │       (SSE 断开则降级为每 5 秒 GET /api/tasks/{id})        │
               │                                                          │
   ⑧ 完成后渲染切片卡片                                                     │
       ├── 点击下载: GET /api/clips/{clip_id}/download ──────────────────► │ FileResponse mp4
       ├── 生成爆款标题: POST /api/clips/{clip_id}/viral-titles ─────────► │ DeepSeek 推理后落库
       ├── 生成剪辑思路: POST /api/clips/{clip_id}/editing-guide ────────► │ DeepSeek 推理后落库
       └── 一键切片(本地): 浏览器端 FFmpeg.wasm 切原视频 → 打 ZIP 下载        │
```

关键点：**完整视频不上传服务器**，前端先在浏览器本地用 FFmpeg.wasm 抽出 ~30 MB 的 MP3 再上传。原视频从头到尾留在用户电脑上，最后导出切片时也是在浏览器本地用 FFmpeg.wasm 切的。

| 项 | 内容 |
|----|------|
| 触发位置 | 调试/容器健康检查（前端代码无固定调用点） |
| 前端发送 | 无 |
| 后端返回 | `{"status":"ok","service":"ai-slice","storage":"local"}` |

---

### 2.2 上传 MP3 音频 — `POST /api/upload/audio` ⭐ 主流程

**前端调用点**：`FileUploader.processFile` → `services/api.ts` 的 `uploadAudio()`

| 项 | 内容 |
|----|------|
| Content-Type | `multipart/form-data` |
| 表单字段 | `file: Blob`（浏览器 FFmpeg.wasm 输出的 MP3） |
| 文件名 | `原视频名.mp3`（前端 `extractAudio()` 拼出来） |
| timeout | `0`（关闭超时） |
| 支持取消 | 是（`AbortSignal`） |
| 进度上报 | `onUploadProgress` 把 0–100% 映射到全局 70%–95% |

**后端返回**：
```json
{
  "audio_path": "D:\\...\\storage\\uploads\\1715680000000_xxx.mp3",
  "original_filename": "xxx.mp4",
  "size_bytes": 31415926
}
```

**后端行为**：
- 校验后缀 ∈ {mp3/wav/m4a/ogg}
- 加时间戳前缀 + 安全化文件名
- 1 MB 分块流式落盘到 `storage/uploads/`
- 失败清理半成品文件

**前端调用点**：`services/api.ts::uploadVideo()`（保留 API，主流程不再使用）

| 项 | 内容 |
|----|------|
| Content-Type | `multipart/form-data` |
| 表单字段 | `file: File`（原始视频） |
| 后端返回 | `{ video_path, video_filename, size_bytes }` |

行为与音频上传一致，区别是不校验后缀。

### 2.4 创建任务 — `POST /api/tasks` ⭐

**前端调用点**：`UploadPage.handleUploadSuccess` → `api.ts::createTask()`

**前端发送（JSON）**：
```ts
interface TaskCreate {
  video_path: string;          // 后端返回的 audio_path
  video_filename: string;      // 原视频文件名（用于显示标题）
  video_start_offset?: number; // OBS 分段录制的 PTS 偏移(秒)
  video_duration?: number;     // 原视频时长(秒)
  scene_mode?: 'livestream' | 'interview' | 'lecture';
}
```

> 字段含糊点：`video_path` 实际传的是音频路径。后端 `Task.source_path` 字段统一接收。

**后端返回**（`TaskResponse`）：
```jsonc
{
  "id": "uuid",
  "status": "pending",
  "video_filename": "...",
  "source_path": "",
  "video_duration": 7200,
  "scene_mode": "livestream",
  "video_start_offset": 0,
  "progress": 0,
  "progress_message": "等待处理",
  "error_message": null,
  "created_at": "2026-05-14T...",
  "updated_at": "2026-05-14T...",
  "clips": []
}
```

**后端行为**：
- 插入 `tasks` 表，状态 `pending`
- 后台 `task_runner` 启动时会轮询扫描 pending 任务并执行 ASR → 分析 → 切片

前端拿到 `task.id` 后立刻 `navigate(/tasks/:id)`。

### 2.5 任务列表 — `GET /api/tasks`

**前端调用点**：`TaskListPage` 进入页面、轮询（每 3 秒，前提：列表里存在非终态任务）

**后端返回**：`TaskListItem[]`
```ts
interface TaskListItem {
  id: string;
  status: string;          // pending/transcribing/analyzing/clipping/done/failed
  video_filename: string;
  video_duration?: number | null;
  scene_mode: SceneMode;
  progress: number;        // 0-100
  progress_message: string;
  created_at: string;
}
```

后端附加行为：列表查询时若发现部分任务缺 `video_duration`，会调用 `hydrate_missing_task_durations()` 用 ffprobe 补全后再 commit。

### 2.6 任务详情 — `GET /api/tasks/{task_id}` ⭐

**前端调用点**：
- `TaskDetailPage` 首次加载
- SSE 推送 `status=done` 时再拉一次（拿到切片完整字段）
- SSE 断开后的轮询兜底（5 秒一次）

**后端返回**：`TaskResponse` + `clips: ClipResponse[]`

```ts
interface TaskClip {
  id: string;
  clip_index: number;
  title: string;
  summary: string;
  clip_type: string;
  start_time: number;          // 相对音频时间(秒)
  end_time: number;
  duration: number;
  virality_score: number;      // 1-10
  suggested_caption: string;
  oss_key?: string | null;     // 实际是本地相对路径 clips/{task_id}/xxx.mp4
  download_url?: string | null;// /api/clips/{clip_id}/download
  viral_titles?: string[] | null;
  editing_guide?: EditingGuide | null;
}
```

后端会逐个 clip 填 `download_url = /api/clips/{clip_id}/download`（前端因此能直接 `<a href>` 触发下载）。

> 字段对齐：后端 `ClipResponse.file_key`，前端 `TaskClip.oss_key`，目前由 `normalizeTaskDetailResponse` 兼容并把缺省字段补成空值。

---

### 2.7 SSE 实时进度 — `GET /api/tasks/{task_id}/progress` ⭐

**前端调用点**：`TaskDetailPage` 用 `new EventSource(url)` 订阅

**响应**：`text/event-stream`，每条消息：
```
data: {"progress": 42, "message": "ASR 转写中...", "status": "transcribing"}

```

**前端处理逻辑**（`TaskDetailPage` 的 useEffect）：
1. SSE 优先，`onopen` 时关闭轮询定时器
2. 每条 `onmessage` 更新本地 `task` 状态
3. `status === 'done'` → 关闭 SSE，并再发一次 `GET /api/tasks/{id}` 拿到完整 clips
4. `onerror` → 关闭 SSE，启动 5 秒一次的轮询兜底
5. 页面切到后台且 `status` 终态变化时，触发 `Notification` 浏览器通知

**后端行为**（`tasks.py::task_progress_sse`）：
- 每秒查一次 DB，把 `progress / progress_message / status` 序列化推送
- 任务终态（done/failed）后跳出循环，主动关闭流

> 字段细节：后端目前推送字段是 `progress / message / status`，前端代码里还读取了 `data.started_at` —— 这是为 ProgressBar 计时预留的扩展点，目前后端没有，永远走默认值（0）。

---

### 2.8 重试任务 — `POST /api/tasks/{task_id}/retry`

**前端调用点**：`TaskDetailPage.handleRetry`

**前端发送**：无请求体

**后端行为**：
- 校验当前状态 ∈ {pending, failed, done}（否则 400）
- 调 `reset_task_for_retry`：清空 progress、status 改回 pending、删除旧 clips（cascade）
- 调 `cleanup_task_outputs`：删本地切片文件
- commit 后由 `task_runner` 重新调度

**后端返回**：完整 `TaskResponse`（clips 为空，等待重跑）

前端如果碰到 400（任务已经在跑），会主动重新 `getTask()` 刷新状态。

## 3. 前端核心数据流转细节

### 3.1 上传流（UploadPage）

```
用户拖入视频 File
   │
   ├─ FileUploader.processFile
   │    │
   │    ├─ extractAudio(File)               // services/audioExtractor.ts
   │    │    ├─ WORKERFS 挂载 File（零拷贝，不进 WASM 堆）
   │    │    ├─ FFmpeg.wasm 探测 start_time / duration（先 -t 0.01 跑一遍读 log）
   │    │    └─ FFmpeg.wasm 编码 16kHz mono MP3 64kbps
   │    │    → { blob: MP3 Blob, filename, startOffset, videoDuration }
   │    │
   │    │   全局进度映射：抽音频占 0–70%
   │    │
   │    └─ uploadAudio(blob, filename, onProgress, signal)
   │         → POST /api/upload/audio
   │         → { audio_path, original_filename, size_bytes }
   │
   │   全局进度映射：上传占 70–95%
   │
   └─ handleUploadSuccess
        └─ createTask({ video_path: audio_path, video_filename, video_start_offset, video_duration, scene_mode })
             → POST /api/tasks
             → TaskResponse
        └─ navigate(`/tasks/${task.id}`)
```

要点：
- 整段视频从头到尾留在浏览器，`AbortController` 全程贯穿 (能在任何阶段取消)。
- `video_start_offset` 用来矫正 OBS 分段录制的 PTS 偏移，最终展示切片时间时会减回去。
- 上传体积缩减 99%（GB → MB）。

---

### 3.2 任务详情流（TaskDetailPage）

```
进入页面 (taskId from URL)
   │
   ├─ getTask(taskId)  ── GET /api/tasks/{id}  ── 首屏渲染
   │
   ├─ 若状态非终态：new EventSource(/api/tasks/{id}/progress)
   │     ├─ onmessage: 更新 task.progress / message / status
   │     │   └─ status=done 时再拉一次 getTask 拿完整 clips
   │     │   └─ 页面隐藏时发浏览器通知
   │     │
   │     └─ onerror: 关闭 SSE，启动 5s 轮询兜底
   │           └─ 每次轮询 getTask(taskId)，终态时停止
   │
   ├─ 卡片操作
   │     ├─ 重命名: renameTask  →  PATCH /api/tasks/{id}/rename
   │     ├─ 重试:   retryTask   →  POST /api/tasks/{id}/retry
   │     │            400 时主动 getTask 刷新状态
   │     └─ 下载:   <a href={clip.download_url}>  → 浏览器原生下载
   │
   ├─ AI 二次生成（按 clip 维度）
   │     ├─ generateViralTitles(clipId)
   │     └─ generateEditingGuide(clipId)
   │
   └─ 一键切片（本地 FFmpeg.wasm，无后端参与）
         exportVideoClipsLocally({ videoFile, clips, videoStartOffset })
            ├─ WORKERFS 挂载原视频 File
            ├─ 遍历 clips: ffmpeg -ss ... -t ... -c copy 切片（不重编码）
            ├─ fflate.zipSync 打包成 ZIP
            └─ 触发 a[download] 下载本地 ZIP
```

---

### 3.3 任务列表流（TaskListPage）

```
进入页面
   │
   ├─ getTasks() → GET /api/tasks
   │
   ├─ 若存在非终态任务: setInterval 每 3s 再 fetchTasks()
   │
   └─ 删除任务: deleteTask → DELETE /api/tasks/{id}

```

`shouldPollTaskList` 判断是否需要继续轮询（任意 task 状态非 done/failed）。

## 4. 异步任务的后端编排（前端无感知，但影响 SSE 内容）

后端 `task_runner` 启动循环：
1. 从 DB 拉一个 `status=pending` 的 Task
2. 状态变化（每一步都会同步写 progress/progress_message）：
   - `pending` → `transcribing`（Whisper ASR）
   - `transcribing` → `analyzing`（DeepSeek 切片决策）
   - `analyzing` → `clipping`（FFmpeg 在服务器切音频对应的视频时间段...等等 注：本地切已交给前端，服务端 `clipper.py` 主要处理切片元数据）
   - `clipping` → `done` 或 `failed`
3. 期间所有进度变化由 `task_progress.py` 统一更新 `tasks.progress / progress_message`，SSE 读 DB 推送。

这意味着前端 SSE 看到的 `status` 取值集合（参考 `TaskStatus` enum）：
`pending / downloading / transcribing / analyzing / clipping / uploading / done / failed`

## 5. 字段映射对照表（前端字段 ↔ 后端字段）

| 前端 (`types/task.ts`)              | 后端 (`schemas.py`)              | 备注 |
|------------------------------------|---------------------------------|------|
| `TaskCreate.video_path`            | `TaskCreate.video_path`         | 实际是音频本地路径 |
| `TaskCreate.video_start_offset`    | `TaskCreate.video_start_offset` | OBS PTS 偏移 |

| `TaskClip.download_url`            | `ClipResponse.download_url`     | 后端组装后下发：`/api/clips/{id}/download` |
| SSE `data.started_at`              | （后端目前不下发）                | 前端预留字段，默认 0 |

## 6. 错误处理与降级策略

| 场景 | 前端策略 |
|------|---------|
| 上传被取消 | `AbortController.abort()` → `classifyUploadFailure` 区分 `cancelled` / `error`，前者展示 "已停止当前处理" |
| 任务创建失败 | UploadPage 切 `status=error`，展示错误信息 + "重新尝试" |
| 任务详情 404 | 显示 "未找到该任务" + 返回列表入口 |
| SSE 断开 | 自动降级为 5 秒轮询 `GET /api/tasks/{id}` |
| 列表任务有进行中 | 3 秒轮询 `GET /api/tasks` 直到全部终态 |
| 重试时任务已在跑 (400) | 主动 `getTask` 拉最新状态显示 |
| 一键切片中途失败 | 单 clip 失败不阻塞整体，最后汇总 `failed: LocalClipExportFailure[]` |
| 浏览器通知 | 进入详情页静默 `requestPermission`，终态切换 + 页面 hidden 时弹通知 |

## 7. 一句话总结前端发什么、服务端返什么

| 前端动作 | HTTP | 前端发送 | 服务端返回 |
|---------|------|---------|-----------|
| 上传音频 | POST `/api/upload/audio` | multipart MP3 Blob | 本地音频路径 |
| 创建任务 | POST `/api/tasks` | JSON（音频路径+原文件名+偏移+场景）| 完整任务 (status=pending) |
| 拉任务列表 | GET `/api/tasks` | — | TaskListItem 数组 |
| 拉任务详情 | GET `/api/tasks/{id}` | — | 任务详情 + clips（含 download_url） |
| 订阅进度 | GET `/api/tasks/{id}/progress` (SSE) | — | 持续推送 `{progress, message, status}` |
| 重试 | POST `/api/tasks/{id}/retry` | — | 重置后的任务 |
| 重命名 | PATCH `/api/tasks/{id}/rename` | `{video_filename}` | 更新后的任务 |
| 删除 | DELETE `/api/tasks/{id}` | — | `{message, id}` |
| 下载切片 | GET `/api/clips/{id}/download` | — | mp4 文件流 |
| 生成爆款标题 | POST `/api/clips/{id}/viral-titles` | — | `{clip_id, viral_titles[]}` |
| 生成剪辑思路 | POST `/api/clips/{id}/editing-guide` | — | `{clip_id, editing_guide{}}` |

**核心特征**：前端只把"音频"和"元数据"交给服务端，原始视频和最终切片成片都在浏览器本地完成。服务端只负责 ASR + AI 切片决策 + 状态机推进 + 二次 AI 文本生成。
