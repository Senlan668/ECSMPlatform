# 阶段 E2：从串行 Worker 升级到并发 Worker —— 方案选型

> 目标场景：**单机部署，~10 个用户同时提交任务，要求并发处理而不是排队**。
> 约束：轻量、简单、好维护、不引入新中间件（不上 Redis / Celery / RabbitMQ）。
> 现状：单协程 Worker 串行 FIFO，10 个并发用户最差排队 10 分钟。

---

## 一、先看清楚瓶颈在哪（拍脑袋之前先量）

把 `run_video_pipeline` 拆成 6 步看资源占用：

| Step | 操作 | 资源类型 | 单任务耗时（估） | 并发友好度 |
| --- | --- | --- | --- | --- |
| 1 | 读本地视频 / 获取时长 | 磁盘 I/O | < 1s | ★★★★★ 完全无关 |
| 2 | FFmpeg 提取音频 | **CPU + 磁盘** | 5~30s | ★★ CPU 抢核 |
| 3 | Groq ASR 转录 | **网络 I/O（第三方 API）** | 30s ~ 3min | ★★★★★ 本机几乎 0 负载 |
| 4 | DeepSeek 分析 | **网络 I/O（第三方 API）** | 10~30s | ★★★★★ 本机几乎 0 负载 |
| 5 | FFmpeg 批量切片 | **CPU + 磁盘** | 5~30s | ★★ CPU 抢核 |
| 6 | 拷贝切片 + 写 DB | 磁盘 I/O | < 2s | ★★★★ |

**关键观察**：
- **Step 3 + 4 大概占整条流水线 70%~85% 的时间**，而这段时间**本机 CPU 是闲着的**（只在等网络）。
- 真正会抢资源的是 Step 2 和 Step 5 的 **FFmpeg**——但它们短，加起来 10~60 秒。
- 所以"并行 5 个任务"在 95% 的时刻其实是"5 个 asyncio 协程同时挂在第三方 API 响应上"，本机毫无压力。

**结论**：你的判断是对的。**这条流水线天然适合 asyncio 并发**，不并发就是浪费机器。

---

## 二、三个候选方案

### 方案 A：多 Worker 协程池（**推荐**）

**思路**：把现在的 1 个 `_task_runner_loop` 改成 N 个并行的 loop，每个 loop 独立从 DB 抢任务，抢到就跑。

```
启动时：
  for i in range(WORKER_CONCURRENCY):
      asyncio.create_task(_task_runner_loop(i, stop_event))

每个 loop 做的事跟现在一模一样：
  while True:
      task = claim_next_pending_task()   # ← 关键：要防止 N 个 loop 抢同一行
      if task:
          await run_video_pipeline(task)
      else:
          sleep(1)
```

**抢任务防冲突**——两种实现方式：

| 方式 | 代码 | 适用 | 备注 |
| --- | --- | --- | --- |
| **A1：asyncio.Lock**（推荐你用这个） | 在 `_claim_next_pending_task` 外包一层 `async with claim_lock:` | 单进程足够 | 0 依赖、5 行代码、绝对安全 |
| **A2：PostgreSQL 行锁** | `SELECT ... FOR UPDATE SKIP LOCKED` | 多进程也安全 | 你用的就是 Postgres，可平滑升级 |

**优点**：
- 改动小（< 30 行代码）
- 复用现有的"启动恢复 / 失败回写 / SSE 进度推送"逻辑，**零回归风险**
- 并发度一行配置可调（环境变量 `WORKER_CONCURRENCY=5`）
- 数据库依然是队列，**部署还是一个进程**，没有任何新中间件

**缺点**：
- 任务积压时仍然要等 1 秒"轮询间隔"才发现（可忽略，相比几十秒的任务）
- 单进程承载，进程崩了全停（但你已经有 `_reset_interrupted_tasks` 兜底）

---

### 方案 B：POST 时直接 spawn + 信号量

**思路**：彻底放弃"Worker 轮询 DB"模式。`POST /api/tasks` 来了直接 `asyncio.create_task(run_video_pipeline(...))`，用一个全局 `asyncio.Semaphore(N)` 卡并发度。

```python
sem = asyncio.Semaphore(5)

@app.post("/api/tasks")
async def create_task(...):
    # 1. 插入 DB
    task = Task(status="pending", ...)
    db.add(task); await db.commit()

    # 2. 立即 spawn，不等 Worker 轮询
    async def runner():
        async with sem:  # 超过 5 个就在这里挂着等
            await run_video_pipeline(task.id)

    asyncio.create_task(runner())
    return {task_id: ...}
```

**优点**：
- **延迟最低**——任务一创建就开跑，省掉那 1 秒轮询
- 没有"轮询数据库"的常驻开销

**缺点（注意，这些是真问题）**：
- **进程一重启，所有"在 semaphore 排队"的任务全丢**——它们在内存里，不在 DB 队列里
- `_reset_interrupted_tasks` 的逻辑就废了，必须重写恢复路径（重启时还得扫 DB 重新 spawn）
- 任务积压时的状态不在 DB 里查不到（无法显示队列位置）
- 失败重试机制要重写

**结论**：对你这种"宕机不能丢任务"的场景**不推荐**。延迟那 1 秒不值这些代价。

---

### 方案 C：上 Celery + Redis（**明确不推荐**）

**思路**：把"任务队列"从 DB 搬到 Redis，用 Celery 起 N 个 worker 进程。

**优点**：
- 横向扩展能力强（可以多机器）
- 生态成熟，重试 / 死信队列 / 监控（Flower）现成的

**缺点**：
- **多一个 Redis 进程 + Celery worker 进程**，部署复杂度直接翻倍
- 进度推送链路要改（pipeline 在另一个进程里跑，SSE 怎么知道？要么走 Redis pub/sub，要么继续靠 DB 轮询）
- 序列化、上下文传递、错误堆栈都要适配
- 对单机 10 用户场景**完全是杀鸡用牛刀**

**结论**：不要上。除非将来变 SaaS 要跨机器，否则没必要。

## 三、方案对比一图流

| 维度 | A 多 Worker 协程池 | B 直接 spawn + 信号量 | C Celery + Redis |
| --- | --- | --- | --- |
| 代码改动量 | **~30 行** | ~60 行（要重写恢复）| **~500+ 行 + 部署改造** |
| 新依赖 | **无** | 无 | Redis + Celery + Kombu |
| 部署复杂度 | **跟现在一样** | 跟现在一样 | 多 2 个常驻进程 |
| 宕机后任务安全 | **✅ 完全继承现有恢复逻辑** | ❌ 内存里的会丢，要重写 | ✅ 但要重做 |
| 队列位置可查询 | **✅ DB 一查就有** | ❌ 内存里查不到 | 需要额外接 Flower |
| 进度 SSE 兼容 | **✅ 零改动** | ✅ 零改动 | ❌ 要走 Redis pub/sub |
| 横向扩展能力 | 单机（足够你用）| 单机 | 多机 |
| 学习 & 维护成本 | **★ 几乎为零** | ★★ | ★★★★ |

## 四、推荐方案 A 的落地细节

### 4.1 关键参数

新增 1 个环境变量：

```env
# .env
WORKER_CONCURRENCY=5    # 同时跑的任务上限，建议 3~8
```

**怎么选这个数？**

| 并发度 | 适合场景 | 风险 |
| --- | --- | --- |
| 3 | 保守起步，机器配置一般 | 高峰时仍有排队 |
| **5（推荐起步值）** | 10 用户日常使用 | 平衡的甜点 |
| 8 | 机器 8 核以上，API 配额充足 | 注意 Groq/DeepSeek 速率限制 |
| 10+ | 用户量持续增长后 | **大概率撞第三方 API 限流** |

> 💡 **不是越大越好的根本原因**：
> - **Groq 免费版 ASR 有 RPM/TPM 限制**（请求/分钟、Token/分钟），并发太高会被 429
> - **DeepSeek 也有并发限制**
> - 万一 5 个 FFmpeg 同时切片，机器 CPU 会瞬时打满（虽然每次只持续几十秒）

### 4.2 代码改动概览（只列要点，先评审再写）

只动 1 个文件：`backend/app/services/task_runner.py`

1. 把 `_runner_task: asyncio.Task | None` 改成 `_runner_tasks: list[asyncio.Task]`
2. `start_task_runner()` 里循环 `WORKER_CONCURRENCY` 次创建协程
3. 在模块顶部加 `_claim_lock = asyncio.Lock()`
4. `_claim_next_pending_task()` 整个外面包 `async with _claim_lock:`
5. `stop_task_runner()` 改成 `await asyncio.gather(*_runner_tasks)`
6. 给 `_task_runner_loop` 加个 `worker_id` 参数仅用于日志

**配置侧**：`config.py` 加一行 `worker_concurrency: int = 5`。

**就这些**。`pipeline.py` 完全不动，`task_progress.py` 完全不动，前端完全不动，DB schema 完全不动。

### 4.3 改完之后行为变化（用之前 10 用户场景对比）

假设单任务 1 分钟，并发度 = 5：

| 用户 | 串行（现状）| 并发 5（方案 A）|
| --- | --- | --- |
| 用户 1 | 0:00 ~ 1:00 ✅ | 0:00 ~ 1:00 ✅ |
| 用户 2 | 1:00 ~ 2:00 | 0:00 ~ 1:00 ✅ |
| 用户 3 | 2:00 ~ 3:00 | 0:00 ~ 1:00 ✅ |
| 用户 4 | 3:00 ~ 4:00 | 0:00 ~ 1:00 ✅ |
| 用户 5 | 4:00 ~ 5:00 | 0:00 ~ 1:00 ✅ |
| 用户 6 | 5:00 ~ 6:00 | 1:00 ~ 2:00 ✅ |
| 用户 7 | 6:00 ~ 7:00 | 1:00 ~ 2:00 ✅ |
| 用户 8 | 7:00 ~ 8:00 | 1:00 ~ 2:00 ✅ |
| 用户 9 | 8:00 ~ 9:00 | 1:00 ~ 2:00 ✅ |
| 用户 10 | 9:00 ~ 10:00 | 1:00 ~ 2:00 ✅ |
| **最差等待** | **10 分钟** | **2 分钟** |

**5 倍提速，0 个新中间件，30 行代码。**

### 4.4 顺手能加的小功能（可选，不影响主路径）

- **队列位置查询**：在 SSE 推送里多带一个 `queue_position` 字段（一句 SQL COUNT），让排队的用户看到"前面还有 N 个人"。
- **并发监控**：加一个 `/api/admin/workers` 接口返回每个 worker 当前在跑哪个 task_id，方便看运行状态。
- **限流防护**：Groq / DeepSeek 客户端外面再套一层 `asyncio.Semaphore`，比如 Groq 最多同时 3 个请求（即使 worker 有 5 个）—— 防止撞 API 限流。

这些都是**可选项**，主方案不依赖它们。

## 五、需要你拍板的几件事

1. **并发度初始值定多少？** 我建议 `WORKER_CONCURRENCY=5`，你可以改。
2. **claim 锁用 A1（asyncio.Lock）还是 A2（FOR UPDATE SKIP LOCKED）？** 建议 A1，原因：你目前是单进程部署，A1 够用且零 SQL 兼容性问题。将来真要多进程时 5 分钟切换到 A2。
3. **要不要顺手加"队列位置 / 当前并发数"两个小特性？** 我建议加队列位置（SSE 体验明显提升），管理接口先不加。
4. **第三方 API 速率限制要不要现在就加防护？** 建议先不加，跑起来观察 1~2 天看有没有 429，再针对性加。**过早优化**。
