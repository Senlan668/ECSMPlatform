## 1. 多剪辑师共享视频文件（OSS 方案）

### 现状的痛点
现在视频是用户在浏览器里选好之后，**本地用 FFmpeg.wasm 抽音频**，再把音频上传给服务端。视频本身根本没上传，结果就是：
- 剪辑师 A 上传的视频，剪辑师 B 在另一台电脑上看不到。
- 切片完成后想做二次剪辑、复核，也找不到原视频。
- 团队协作完全做不了。

### 改进方案（一句话总结）
**视频上传走 OSS 直传，OSS 地址和 task_id 绑定，后续所有人通过 task_id 就能拿到同一个视频。**

### 详细流程

```
[剪辑师A 浏览器]
   ↓ ① 调后端拿临时凭证（STS Token / 预签名 URL）
[后端: 签发上传凭证]
   ↓ ② 浏览器拿凭证直接 PUT 到 OSS（不经过我们服务器）
[阿里云 OSS / AWS S3 / 腾讯 COS]
   ↓ ③ 上传完成后，回调后端注册 video_url
[后端: 写库 tasks.video_url = oss://bucket/xxx.mp4]
   ↓ ④ 同时抽音频上传，或服务端拉 OSS 视频做 ASR
[剪辑师B]
   ↓ ⑤ 通过 task_id 查到 video_url，前端用签名 URL 直接预览/再切片
```

### 关键技术点（面试可讲）

1. **为什么用直传而不是走后端中转？**
   - 视频动辄几个 GB，走后端会撑爆带宽和内存。
   - 直传走 OSS 内网，速度快、不占我们服务器资源。

2. **凭证怎么签？**
   - 用 STS（临时安全令牌）或者 **预签名 PUT URL**，有效期 15 分钟。
   - 限制目录前缀（`uploads/{user_id}/`）防止越权。

3. **数据库改造**
   ```python
   class Task:
       video_url: str          # oss://bucket/uploads/uuid.mp4
       video_etag: str         # OSS 返回的 MD5，用于秒传判重
       video_size: int
       uploaded_by: str        # 谁上传的
   ```

4. **秒传 / 去重**：上传前算文件 MD5，先查库有没有同 ETag 的，有就直接复用 task。

5. **权限模型**：
   - `task.owner_id` + 团队成员表 `task_members`，控制谁能访问。
   - 拉视频时后端签发**带过期时间的下载 URL**，不直接暴露 OSS 路径。

6. **生命周期管理**：OSS 配置规则，30 天没访问的原视频自动转低频存储，节省成本。

### 面试官可能追问
- **Q：上传中断怎么办？** → OSS 支持分片上传（multipart upload），断点续传。
- **Q：怎么防刷？** → 凭证带 IP 限制 + 单用户配额（每天 100GB）。
- **Q：跨区域怎么办？** → OSS 开 CDN 加速 / 跨地域复制。

## 2. 内容合规 - 导入违禁词建议

### 设计思路（一句话）
**搞个本地违禁词库，对每个 clip 的标题/文案/转录文本扫一遍，命中了就在 clip 数据里标注出来，前端给剪辑师标红提示就行**。最终视频还要人工剪辑确认，不需要做强拦截，更不需要云审核、LLM 复核那套重方案。

### 为什么够用？
- 我们的产品定位是「剪辑助手」，不是「自动发布平台」。
- 最终发布前必然有剪辑师人工 review。
- AI 只负责**给出建议**，决策权在人。
- 重活（图片/视频帧审核）让发布平台（抖音/B站）自己去做，他们的算法比我们强。

### 实现方案

#### 1. 词库（一张表搞定）

```sql
CREATE TABLE sensitive_words (
    id BIGSERIAL PRIMARY KEY,
    word VARCHAR(100) UNIQUE,
    category VARCHAR(30),    -- ad(广告法极限词) / political / vulgar / platform_ban
    suggestion VARCHAR(100), -- 建议替换词，比如 "最好的" → "优秀的"
    enabled BOOLEAN DEFAULT TRUE
);
```

来源：
- 网上有现成的违禁词库（GitHub 上"广告法违禁词"项目几千个词，直接 import）。
- 平台官方公布的：抖音社区规范、B站违禁词。
- 自己积累：剪辑师反馈的新词。

#### 2. 检测逻辑（DFA 算法）

```python
class SensitiveFilter:
    """全局单例，启动时从 DB 加载词库构建字典树"""
    def __init__(self):
        self.trie = build_trie(load_words_from_db())

    def scan(self, text: str) -> list[dict]:
        # O(n)，n=文本长度，跟词库大小无关
        # 返回 [{"word": "最", "pos": 5, "category": "ad", "suggestion": "优秀"}]
        return dfa_match(self.trie, text)
```

DFA 算法（确定有限状态机）就是字典树（Trie）一遍扫描，比正则快几个数量级，10 万词库扫 1 万字耗时 < 1ms。

#### 3. 在 pipeline 里加一步（关键改动）

LLM 分析出 clip_plans 之后，写库之前，每个 clip 跑一遍扫描：

```python
# 在 pipeline.py 里
for plan in clip_plans:
    text = plan["title"] + plan["summary"] + plan["suggested_caption"]
    hits = sensitive_filter.scan(text)

    clip = Clip(
        ...
        compliance_hits=hits,    # 直接存到 clip 表
        compliance_status="warn" if hits else "pass",
    )
```

#### 4. Clip 表加两个字段

```python
class Clip:
    # ... 原有字段
    compliance_hits: list | None = mapped_column(JSON, nullable=True)
    # 示例值：[{"word":"最便宜","pos":12,"category":"ad","suggestion":"超值"}]
    compliance_status: str = mapped_column(String(10), default="pass")
    # pass / warn
```

#### 5. 前端展示

剪辑师在 clip 列表/详情页：
- 命中违禁词的 clip 标题旁边显示一个 ⚠️ 图标。
- 点开看高亮：「标题中的"**最便宜**"是广告法极限词，建议换成"超值"」。
- 剪辑师改了之后再扫一次，没问题就发布。

### 关键技术点

1. **词库热更新**：管理后台增删词后，调一个 `/admin/reload_words` 接口让全局 SensitiveFilter 实例重新加载。多机部署用 Redis pub/sub 广播。
2. **白名单**：有些词是"上下文敏感"的（比如"杀"在"杀价"里没问题），可以在词库加上下文规则，或者直接靠剪辑师人工判断（够用）。
3. **不做强拦截**：只标注、不阻塞流程。剪辑师有权决定要不要改。
4. **性能可忽略**：单 clip 扫描 < 1ms，每个任务最多 10-20 个 clip，总开销可以忽略。

### 面试官可能追问
- **Q：为什么不上云审核 API？** → 我们是工具，不是发布平台。剪辑师人工 review 是必经环节，没必要重复花钱。
- **Q：词库太大怎么办？** → DFA 算法本身和词库大小无关，扫描是 O(n) 文本长度。10w 词都能瞬时扫完。
- **Q：转录文本里有违禁词怎么办？** → 同样扫一遍 transcript_json，标注在对应时间点，前端时间轴显示警告。

## 3. 任务调度 + 高并发（纯 Redis 方案）

### 现状的问题
现在 pipeline 是 FastAPI 直接 `asyncio.create_task` 起后台协程跑：
- 任务都堆在第一台机器的 Python 进程里，加机器没用。
- 100 个用户同时上传，100 个协程一起跑，CPU 爆。
- 没办法多机器协作。

### 用厨房的故事来理解

把系统想象成一个**餐厅厨房**：

| 现实角色 | 系统组件 |
|---|---|
| 顾客 | 用户 |
| 服务员 | FastAPI API 进程 |
| 订单墙 | Redis 的一个 List（队列） |
| 厨师 | Worker 进程（可以有很多个） |
| 取餐窗口 | Postgres |

**流程**：
1. 顾客来下单 → 服务员把订单**贴到订单墙上**，立刻告诉顾客"取餐号 task_id=xxx"。
2. 厨房里有 N 个厨师，**谁手空了谁就去订单墙揭一张单子下来做**。
3. 厨师做完菜，把菜放到取餐窗口（写库）。
4. 顾客凭 task_id 来取餐。

这里就有一个**核心问题**：

> 厨师 A 和厨师 B 同时伸手去揭同一张单子怎么办？两个人都做一遍就浪费灶台了。

这就是**抢单 + 锁**要解决的问题。

### 抢单的核心：Redis 的两条命令

#### 第一步：从订单墙拿单（`BRPOP`）

```python
task_id = redis.brpop("task_queue", timeout=5)
```

- `BRPOP` = Blocking Right POP，**原子操作**：从队列末尾拿一个元素出来。
- "原子"的意思是：**这张单子要么在你手里，要么在订单墙上，不会同时在两个地方**。
- 所以两个厨师同时调 BRPOP，Redis 保证只有一个能拿到这张单子，另一个会拿到下一张（或者拿不到就阻塞等待）。

**到这一步，其实抢单已经解决了**——单子被拿走后就从队列里消失，不可能被第二个人再拿到。

#### 第二步：揭下单子后，再上一把"做菜锁"（`SET NX EX`）

那为什么还要锁？因为可能有这些情况：

- 同一个 task_id 被**重复入队**了（API 处错或者用户连点两次"提交"）。
- 后面要做"任务重试"，重试时会重新塞进队列。

所以厨师揭下单子后，**还要再确认一下：这道菜现在没有别人在做**。

```python
# 厨师揭下单子后，先去打卡
locked = redis.set(
    f"lock:{task_id}",   # key：这张单子的锁
    worker_id,           # value：写上自己的名字，方便后面验证
    nx=True,             # NX = 只有锁不存在时才能设置成功
    ex=600,              # EX = 锁 10 分钟后自动消失
)

if not locked:
    # 已经有别的厨师在做这道菜了，跳过
    continue

# 抢到锁了，开始做菜
run_pipeline(task_id)

# 做完，扔掉锁
redis.delete(f"lock:{task_id}")
```

**关键的 `SET NX EX` 三个参数**：

| 参数 | 含义 | 厨房类比 |
|---|---|---|
| `NX` | Not eXists，只有 key 不存在时才能写 | 灶台上没人占就插自己的牌 |
| `EX 600` | 600 秒后这个 key 自动消失 | 牌子上贴个便条 "10 分钟后自动作废" |
| `value=worker_id` | 锁里记录是谁占的 | 牌子上写自己的名字 |

`NX` 是原子操作，多个厨师同时调，**只有一个能 SET 成功，其他都返回 nil**。

### Worker 完整代码（极简版）

```python
import asyncio
import uuid

async def worker_loop():
    worker_id = f"worker-{uuid.uuid4()}"

    while True:
        # ── 1. 从订单墙拿一张单子 ──
        result = await redis.brpop("task_queue", timeout=5)
        if not result:
            continue                # 5 秒没单子，重新循环
        _, task_id = result

        # ── 2. 抢锁，确认这道菜没人在做 ──
        locked = await redis.set(
            f"lock:{task_id}",
            worker_id,
            nx=True,
            ex=600,                 # 10 分钟够跑完一个 pipeline
        )
        if not locked:
            continue                # 别人在做，跳过

        # ── 3. 做菜 ──
        try:
            await run_pipeline(task_id)
        finally:
            # ── 4. 撤掉锁牌 ──
            await redis.delete(f"lock:{task_id}")


# 启动多个 worker 并发跑
async def main():
    workers = [worker_loop() for _ in range(4)]   # 单进程 4 个并发
    await asyncio.gather(*workers)
```

部署：
- 一台机器跑一个 Python 进程，开 4 个 worker 协程 → 这台机能同时做 4 道菜。
- 加 5 台机器 → 全集群 20 道菜并发。
- **加机器只要新机器能连到 Redis 就行，不用改任何代码**。

### API 端怎么把单子贴上墙

API 那边更简单：

```python
@router.post("/api/tasks")
async def create_task(audio_path: str):
    task = Task(audio_path=audio_path, status="pending")
    db.add(task)
    await db.commit()

    # 把 task_id 贴到订单墙上
    await redis.lpush("task_queue", str(task.id))

    return {"task_id": task.id}
```

**LPUSH 从左边推进队列，BRPOP 从右边弹出 → 先进先出（FIFO）**。

### 锁为什么必须有过期时间（EX 600）？

如果**只 NX 不 EX**：

- 厨师拿到锁后突然电脑断电，再也回不来。
- 锁永远在那里，这道菜**永远没人敢做**。

所以一定要给锁加个过期时间。即使厨师跑路了，10 分钟后锁自动消失，下一个厨师可以接手。

> 失败重试、心跳续约这些更复杂的话题，留到「第 4 节 任务失败处理」里讲。这里只关心**抢单和锁**本身。

### 一图概览

```
[API 进程]
   │ LPUSH task_queue task_id_xxx
   ▼
┌────────────────────────────────────┐
│       Redis                        │
│                                    │
│   task_queue (订单墙)               │
│   ┌──────────────────┐             │
│   │ task_C           │  ← BRPOP    │ ← Worker 从右边拿
│   │ task_B           │             │
│   │ task_A           │  ← LPUSH    │ ← API 从左边推
│   └──────────────────┘             │
│                                    │
│   lock:task_C → worker-1 (TTL=10m) │ ← 谁在做这道菜
│   lock:task_B → worker-2 (TTL=10m) │
│                                    │
└────────────────────────────────────┘
       ▲           ▲           ▲
       │           │           │
   [Worker-1]  [Worker-2]  [Worker-3]    （想加多少加多少）
```

### 面试官可能追问
- **Q：BRPOP 已经保证只能被一个 Worker 拿到，为什么还要锁？**
  → 防止 task_id 被重复入队（重试 / 用户重复提交）造成同一个任务做两遍。锁是双保险。
- **Q：为什么用 List 不用 Stream？**
  → List 够用，简单。Stream 的高级特性（消费组、ack、重投）是给"任务失败要可靠重试"用的，留到第 4 节再讨论。
- **Q：Redis 挂了怎么办？**
  → 用 Sentinel 做主从自动切换，做 AOF 持久化保证队列数据不丢。
- **Q：怎么加机器扩容？**
  → 新机器跑同一份 Worker 代码，连同一个 Redis。立刻就能消费队列，零配置。

## 4. 任务失败的处理 + 数据持久化 + 恢复

### 设计原则（核心）
**失败就完全重头来。只持久化"成功完成"的最终结果，不存中间状态。**

### 为什么选这个方案？

之前想过"断点续传"（ASR 跑完就存，失败从 LLM 继续），但有几个问题：

1. **复杂度爆炸**：每一步要设计幂等键、中间表、状态机，代码量翻倍，bug 翻三倍。
2. **中间数据可能"过期"**：比如 transcript 存了，但用户改了 video_start_offset，旧 transcript 就废了，断点续传反而错。
3. **本项目链路短**：就 ASR → LLM → 保存 clips 三步，整体也就几分钟。失败重头跑一遍可以接受。
4. **失败本身就是小概率事件**：每次重试代价不大，重复消耗 ASR 配额能接受（成本可控）。

> 一句话总结：**简单粗暴 = 可靠**。中间态越多，bug 越多。

### 具体设计

#### 1. Task 表的状态机（极简）

```python
class TaskStatus:
    pending      # 刚入队，还没开始
    running      # 跑着呢
    done         # 成功完成（transcript + clips 都已落库）
    failed       # 失败，等用户点"重试"或被自动调度重新跑
```

只有 `done` 状态的数据是"完整的、可信的"。其他状态都视为"未完成"。

#### 2. 写库时机：**只在最终成功时一次性写入**

看 `pipeline.py` 的现有代码改造：

```python
async def run_video_pipeline(task_id):
    task = load_task(task_id)
    update_status(task, "running", progress=0)

    try:
        # ── 1. 在内存里跑 ASR ──
        transcript = await transcriber.transcribe(task.audio_path)
        # ★ 不写库！只在内存里

        # ── 2. 在内存里跑 LLM ──
        clip_plans = await analyzer.analyze(transcript)
        # ★ 不写库！只在内存里

        # ── 3. 全部成功后，一个事务里写库 ──
        async with db.begin():        # 数据库事务
            task.transcript_json = transcript
            task.status = "done"
            task.progress = 100
            for plan in clip_plans:
                db.add(Clip(task_id=task.id, **plan))
            # commit 在 with 块结束时自动执行

    except Exception as e:
        # ── 失败：清理，标记 failed，把锅甩给重试机制 ──
        await db.rollback()
        task.status = "failed"
        task.error_message = str(e)[:500]
        task.retry_count += 1
        await db.commit()
        raise
```

**关键**：用一个 DB 事务包住所有"成功写入"操作。要么全部 commit，要么全部 rollback。**不会留下"半成品"数据**。

#### 3. 失败重试机制

```python
class Task:
    status: str             # pending/running/done/failed
    retry_count: int = 0    # 已重试次数
    max_retries: int = 3
    next_retry_at: datetime # 下次重试时间
    error_message: str      # 最后一次失败原因
```

**重试流程**：

```python
# 后台调度器，每分钟扫一次
async def retry_scheduler():
    failed_tasks = db.query(Task).where(
        Task.status == "failed",
        Task.retry_count < Task.max_retries,
        Task.next_retry_at <= now()
    )
    for task in failed_tasks:
        task.status = "pending"
        redis.lpush("task_queue", task.id)   # 重新入队
```

**指数退避**：第 1 次失败 → 1 分钟后重试；第 2 次 → 5 分钟；第 3 次 → 30 分钟；超过 3 次彻底放弃，状态 `failed_permanent`，发通知给运营。

#### 4. 进程崩溃怎么办？

进程崩溃和"业务失败"用同一套机制处理：

- Worker 拿任务时加 Redis 分布式锁（见第 3 节），TTL=60s，心跳每 30s 续约。
- 如果 Worker 崩了，60s 后锁自动释放。
- 另起一个**僵尸任务清理器**（每分钟扫一次）：

```python
async def cleanup_zombie_tasks():
    """status=running 但 Redis 里没锁 = 僵尸任务"""
    running = db.query(Task).where(Task.status == "running")
    for task in running:
        if not redis.exists(f"lock:{task.id}"):
            # 锁没了 = Worker 死了
            task.status = "failed"
            task.error_message = "Worker crashed"
            task.retry_count += 1
            redis.lpush("task_queue", task.id)   # 重新入队
```

### 关键技术点

1. **不存中间态，只存终态**：DB 里看到的数据要么是"完整成功的"，要么是"等待重试的"。没有半成品。
2. **单事务 commit**：所有结果在一个 DB 事务里写完。即使写到一半 DB 挂了，整体回滚。
3. **重试 = 重新入队**：复用现有的 Worker 流程，不需要单独写恢复代码。
4. **指数退避**：避免持续失败的任务把队列堵死。
5. **失败上限**：超过 3 次彻底放弃，避免无限重试。
6. **成本考量**：失败重试意味着 ASR 重新跑一遍。但前提是失败概率低（<1%），重复成本可控。

### 优缺点对比

| 维度 | 完全重试（当前方案） | 断点续传 |
|---|---|---|
| 代码复杂度 | ⭐ 极简 | ⭐⭐⭐⭐ 复杂 |
| 失败恢复速度 | 慢（重跑全部） | 快（继续上次） |
| 出 bug 概率 | 低 | 高（状态机难调） |
| 中间数据腐烂风险 | 无 | 有（用户改参数后中间态失效） |
| API 成本 | 高 | 低 |
| **适合场景** | **失败率低、链路短** | 失败率高、链路长、调用贵 |

我们项目失败率低、链路短，所以选完全重试。

### 面试官可能追问
- **Q：用户能看到一个"重试"按钮吗？** → 能。前端检测到 `status=failed` 显示"重试"按钮，点击后调 `/api/tasks/{id}/retry`，后端把 status 改回 pending 重新入队。
- **Q：Redis 挂了，分布式锁怎么办？** → Redis Sentinel 自动主从切换；如果整个 Redis 都挂了，进程级判断（写 DB 时检查 status）兜底。
- **Q：为什么不存 ASR 中间结果省点钱？** → 我们已经走 Groq（便宜的方案），2H 视频 ASR 也就几毛钱，省的钱不值得增加代码复杂度。如果换成 OpenAI Whisper（贵 10 倍）就值得考虑断点续传了。

## 5. Task 和 Clips 大量数据分库分表

### 什么时候需要分？
经验法则：**单表超过 5000 万行**，或单库磁盘超过 500GB，就该考虑了。

- Task：每天 1 万个任务 × 365 天 × 3 年 ≈ 1100 万 → 还能扛。
- Clip：每个 Task 平均 10 个 Clip → 1 亿行 → **必须分表**。

### 分表方案

#### 方案一：按时间分表（推荐）

```
clips_2025_01
clips_2025_02
clips_2025_03
...
```

**优点**：
- 实现简单，迁移友好。
- 老数据可以归档到冷存储（S3 Glacier）。
- 业务上 90% 查询都是查最近的，命中单表。

**缺点**：
- 跨月查询要 UNION。

#### 方案二：按 task_id 哈希分（Sharding）

```
clips_00 ~ clips_15   (16 张表)
路由：table = "clips_" + (task_id_hash % 16)
```

**优点**：
- 数据分布均匀，没有热点。
- 通过 task_id 查 clips 性能稳定。

**缺点**：
- 扩容麻烦（重新分片很痛苦），所以一上来就要预留够。
- 全局排序（按时间）很难，需要先在每个分片排再合并。

#### 最终推荐：**复合分片**
- 一级：按月分库 `db_2025_01`、`db_2025_02`。
- 二级：每个库内按 task_id 哈希分 16 张表。
- 这样**既能归档老数据，又能水平扩展**。

### 中间件选择

| 方案 | 说明 |
|---|---|
| **ShardingSphere** | Java 系主流，规则灵活，支持读写分离 |
| **Vitess** | YouTube 出品，K8s 友好 |
| **应用层路由** | 自己写一层 router（简单场景） |
| **Citus** | Postgres 原生分布式扩展（我们用 PG，最匹配） |
| **CockroachDB** | 直接换分布式 DB，业务无感（重构成本高） |

**本项目推荐 Citus**：因为我们已经用 Postgres，Citus 是 PG 扩展，**SQL 完全兼容**，无需改业务代码。

### 关键技术点

1. **分片键怎么选？**
   - Clip 表选 `task_id`：保证一个 Task 的所有 Clip 在同一分片，JOIN 性能好。
   - Task 表选 `created_at`：天然时间序，归档方便。

2. **全局 ID 怎么生成？**
   - UUID v7（带时间戳，B+ 树插入友好）。
   - 或者雪花算法（Snowflake）。

3. **跨分片查询怎么办？**
   - 走 ES 做二级索引（按用户/按状态查）。
   - 或者用 ClickHouse 做分析查询。

4. **历史数据归档**：
   - 6 个月前的数据导出到 OSS Parquet 文件。
   - 主库只保留热数据，查询飞快。

### ⚠️ 按时间分表的"0 点边界问题"（面试高频追问）

**问题描述**：
按月分表的话，11 月 30 日 23:59:58 创建的 task，所有 clips 都写在 `clips_2025_11`。但任务跑完已经是 12 月 1 日 00:00:30，那这个任务的 clips 应该写哪张表？

**如果按 `clip.created_at` 路由**：
- 这个任务的 clips 写到了 `clips_2025_12`。
- 但任务 task 在 `tasks_2025_11`。
- 查"11 月的某个 task 有哪些 clips"时，JOIN 不到 → **数据丢失**。

#### 解决方案：分片键必须用"父记录的时间"，不是"自己的时间"

```python
# ❌ 错误做法：clip 自己的 created_at
def route_clip_table(clip):
    return f"clips_{clip.created_at:%Y_%m}"

# ✅ 正确做法：跟着 task 的 created_at 走
def route_clip_table(clip):
    task = get_task(clip.task_id)
    return f"clips_{task.created_at:%Y_%m}"
```

**也就是说**：一个 task 跨月了，它所有的 clip 都还写在 task 出生的那张分片表里。

#### 更通用的解法：用"业务 ID 编码时间"

更好的做法是不依赖 `created_at` 列，而是**把时间编码进主键**：

- Task ID 用 **UUID v7** 或 **Snowflake**，前 41 位本身就是毫秒时间戳。
- Clip ID 同理。
- 路由时直接从 ID 里解码时间，不需要查 DB。

```python
def route_table(record_id):
    # 从 Snowflake ID 解出时间戳
    ts_ms = (record_id >> 22) + EPOCH
    dt = datetime.fromtimestamp(ts_ms / 1000)
    return f"clips_{dt:%Y_%m}"
```

**好处**：
- 不用查 DB 就能算出在哪张表。
- Clip 跟着 Task 的时间走（业务上 clip 的 ID 由 task 的时间戳派生）。

#### 进一步：让分片边界"软"一点

实操中还可以**允许"边界数据"双写或重叠**：

- 月末 23:55 ~ 月初 00:05 这 10 分钟内，**新表先开好**，但 task 还能写老表。
- 查询时把"边界月"两张表都 UNION 一下（前端最多多读一张表，性能可接受）。
- 7 天后做一次合并迁移，把边界数据规整。

#### 工程上的兜底

不管怎么设计，做好两点：
1. **路由函数全局唯一**：所有 read/write 走同一个 `route_table()` 函数，绝不允许任何地方"自己拼表名"。
2. **分片信息也存 DB**：每条记录额外存一列 `shard_key`（写入时算好），查询时按这个 key 路由，**比业务字段更可靠**。
   ```sql
   ALTER TABLE clips ADD COLUMN shard_month VARCHAR(7);  -- 比如 "2025_11"
   ```
   这样路由函数变化也能向前兼容。

### 演进路径
```
单表 → 读写分离 → 分表 → 分库分表 → 分布式数据库
 ↑            ↑          ↑
1000万行   5000万行    1亿行
```

### 日志规范（统一格式）

每条日志必须包含的"骨架"字段：

| 字段 | 说明 |
|---|---|
| `timestamp` | ISO 8601，精度到毫秒 |
| `level` | info/warn/error |
| `event` | 事件名，snake_case，比如 `asr_call_succeeded` |
| `trace_id` | 全链路追踪 ID，从 API 入口贯穿到 Worker |
| `task_id` | 业务主线索（**最重要**，按它聚合就能拼出一个任务的完整流程） |
| `service` | 服务名 `api` / `worker` / `frontend` |
| `host` | Pod / 机器名 |
| `...` | 业务字段 |

**代码层面**：

```python
import structlog

logger = structlog.get_logger()

# 在 pipeline 入口绑定 task_id，后续每条日志都自动带上
log = logger.bind(
    task_id=str(task.id),
    trace_id=trace_id,
    user_id=task.user_id,
)

log.info("asr_call_started", provider="groq")
log.info("asr_call_succeeded", latency_ms=8420, segments=156)
```

### 关键技术点

1. **结构化（JSON）**：必须 JSON，云平台才能按字段索引。
2. **TraceID 全链路**：API 入口生成（用 `uuid4` 或 OpenTelemetry），透传到 Worker（放进 Redis 任务消息里）。一个 task_id 在所有服务的日志里都能用 trace_id 串起来。
3. **业务事件 vs 技术日志分开**：业务事件（task_completed, asr_call_succeeded）走"业务日志库"用于分析；技术日志（stack trace, debug）走"运维日志库"用于排查。
4. **采样**：高频/低价值日志（health check）采样 1%，关键业务事件 100% 采集。
5. **脱敏**：手机号、邮箱、API Key 自动脱敏（Fluent Bit Lua 脚本，正则匹配后 mask 中间几位）。
6. **冷热分层**：7 天热（SLS 标准），30 天温（SLS 低频），90+ 天冷（OSS Parquet 归档）。
7. **告警规则**（示例）：
   - 5 分钟内 `task_failed` 数 > 10 → 钉钉告警。
   - `asr_call_failed` 中 provider=groq 占比 > 30% → 自动触发兜底切流。
   - 单用户 1 小时内创建 task > 100 → 反爬告警。

### 成本控制
- 日志按量计费，关键节点全采、调试日志采样、长字段截断到 1KB。
- 上线前关 DEBUG。
- 大日志（stack trace、prompt 全文）单独存到 OSS，日志里只放 URL 引用。

## 7. ASR 多厂商兜底机制

### 当前问题
现在 `transcriber.py` 写死了 Groq Whisper，**Groq 一挂全挂**。需要多厂商兜底。

### 设计原则
1. **统一接口**（已有 `BaseTranscriber` 基类，已经预留好了）。
2. **优先级 + 自动降级**：主厂商失败自动切备用。
3. **能熔断**：连续失败 N 次直接跳过该厂商一段时间。

### 候选 ASR 厂商

| 厂商 | 优势 | 劣势 |
|---|---|---|
| Groq Whisper | 极快、便宜 | 偶尔不稳 |
| OpenAI Whisper | 准确率高 | 贵、慢、国内不稳 |
| 阿里云 ASR | 国内稳、中文好 | 贵 |
| 腾讯云 ASR | 国内稳 | 同上 |
| 火山引擎 | 字节出品，便宜 | 文档一般 |
| 本地 faster-whisper | 自己跑，不要钱 | 慢，要 GPU |

### 实现方案

```python
class FallbackTranscriber:
    def __init__(self):
        self.providers = [
            GroqASRTranscriber(),      # 优先
            AliyunASRTranscriber(),    # 备 1
            LocalWhisperTranscriber(), # 兜底（不依赖外网）
        ]
        self.circuit_breakers = {
            p.name: CircuitBreaker() for p in self.providers
        }

    async def transcribe(self, audio_path):
        last_error = None
        for provider in self.providers:
            cb = self.circuit_breakers[provider.name]
            if cb.is_open():
                continue                # 熔断中，跳过

            try:
                result = await provider.transcribe(audio_path)
                cb.record_success()
                return result
            except Exception as e:
                cb.record_failure()
                last_error = e
                logger.warning(f"{provider.name} failed, trying next")

        raise RuntimeError(f"All ASR providers failed: {last_error}")
```

### 熔断器（Circuit Breaker）

```
状态机：
  Closed (正常) → 失败超阈值 → Open (熔断)
       ↑                          ↓
       └──── 探活成功 ←── Half-Open (探活)
                              ↑
                          冷却时间到
```

```python
class CircuitBreaker:
    def __init__(self, failure_threshold=5, cooldown=60):
        self.fail_count = 0
        self.state = "closed"
        self.opened_at = None

    def is_open(self):
        if self.state == "open":
            if time.time() - self.opened_at > self.cooldown:
                self.state = "half_open"
                return False
            return True
        return False

    def record_failure(self):
        self.fail_count += 1
        if self.fail_count >= self.failure_threshold:
            self.state = "open"
            self.opened_at = time.time()

    def record_success(self):
        self.fail_count = 0
        self.state = "closed"
```

### 关键技术点

1. **不同厂商接口差异**：基类抽象出统一返回 `list[{start, end, text}]`，每个适配器内部转换。
2. **配额管理**：每个厂商记录今日已用 quota，快超了自动切下一个。
3. **效果对比**：可以开"AB 测试"模式，两家都跑，对比准确率（off-line 评估用）。
4. **成本路由**：长音频走便宜的，短音频走快的。
5. **配置热更新**：调整厂商顺序 / 开关，不用重启服务（Nacos / Apollo）。

## 8. LLM 多厂商兜底机制

### 当前问题
`analyzer.py` 写死了 DeepSeek，**DeepSeek 一挂没法切片**。

### 思路和 ASR 一样，但有几个 LLM 特有的难点

1. **不同厂商 Prompt 差异大**：同样的 Prompt，DeepSeek 输出 JSON 很标准，但 GPT-3.5 可能加 ```json``` 包裹。
2. **不同模型上下文长度不同**：DeepSeek 64K vs GPT-3.5 16K vs Claude 200K。
3. **不同厂商定价天差地别**：成本差 10 倍。

### 候选 LLM

| 厂商 | 模型 | 价格 ($/1M tokens) | 特点 |
|---|---|---|---|
| DeepSeek | deepseek-chat | $0.14 / $0.28 | 便宜、中文好 |
| OpenAI | gpt-4o-mini | $0.15 / $0.60 | 稳、通用 |
| Anthropic | claude-3.5-sonnet | $3 / $15 | 强推理 |
| 阿里云 | qwen-plus | ¥0.8 / ¥2 | 国内 |
| 智谱 | glm-4 | ¥0.5 / ¥1.5 | 国内 |
| 月之暗面 | kimi | ¥12 / ¥12 | 长上下文 |

### 实现方案（OpenAI 兼容 API 是个救星）

很多厂商都兼容 OpenAI 的 API 协议（DeepSeek、阿里通义、月之暗面），只需要换 `base_url` 和 `api_key`：

```python
class LLMRouter:
    PROVIDERS = [
        # name, base_url, model, max_context
        ("deepseek", "https://api.deepseek.com", "deepseek-chat", 64000),
        ("qwen", "https://dashscope.aliyuncs.com/...", "qwen-plus", 128000),
        ("openai", "https://api.openai.com/v1", "gpt-4o-mini", 128000),
        ("kimi", "https://api.moonshot.cn/v1", "moonshot-v1-128k", 128000),
    ]

    async def chat(self, prompt: str, **kwargs):
        for name, base_url, model, max_ctx in self.PROVIDERS:
            if self.cb[name].is_open():
                continue
            if estimate_tokens(prompt) > max_ctx:
                continue                 # 上下文不够，跳过

            try:
                client = AsyncOpenAI(base_url=base_url, api_key=KEYS[name])
                resp = await client.chat.completions.create(
                    model=model,
                    messages=[{"role": "user", "content": prompt}],
                    response_format={"type": "json_object"},  # 强制 JSON
                    **kwargs
                )
                self._record_usage(name, resp.usage)
                return resp.choices[0].message.content
            except Exception as e:
                self.cb[name].record_failure()
                continue

        raise RuntimeError("All LLM providers exhausted")
```

### 关键技术点

1. **JSON 强制输出**：用 `response_format={"type": "json_object"}` 避免厂商乱输出。不支持的厂商，用正则提取 ```json``` 块。
2. **重试解析**：LLM 输出非合法 JSON，把错误信息塞回去再让它修。
   ```python
   # 第二次调用 Prompt 追加
   "上次你输出的 JSON 解析失败，错误：{error}，请只输出合法 JSON"
   ```
3. **路由策略**：
   - **成本优先**：先 DeepSeek，再 qwen，再 OpenAI。
   - **质量优先**：先 GPT-4o，再 Claude。
   - **速度优先**：先 DeepSeek（响应最快）。
4. **Token 预算**：超过 max_ctx 的提前裁剪 transcript（保头尾 + 关键词密度高的段）。
5. **流式响应**：用 SSE 边生成边返回，体验更好（项目里已经有 SSE 框架）。
6. **缓存**：相同 Prompt + 相同 transcript 哈希，结果缓存到 Redis，**24h 内不重复调用**。
7. **温度参数**：分析切片用 `temperature=0.3`（更稳定），生成爆款标题用 `0.8`（更有创意）。

### 面试官可能追问
- **Q：怎么判断结果质量？** → JSON Schema 校验 + 业务规则（时间不能重叠 / clip 数 > 0）+ 偶发抽样人工 review。
- **Q：上下文超长怎么办？** → 分段处理 + map-reduce（每 1000 段单独分析，最后汇总）。

## 9. 2H 视频 ASR 串行 → 并行升级

### 现状（看 `transcriber.py` 第 183 行）
```python
for index, (chunk_path, offset) in enumerate(chunks, start=1):
    segments = await self._transcribe_single(chunk_path)
    # ↑ 一个 chunk 跑完才跑下一个
```

2 小时音频按 10 分钟切片 → 12 个 chunk → **串行跑 12 次**，每次 Groq 平均 10-15 秒 → **总耗时 2-3 分钟**。

### 改造目标：12 个 chunk **并发跑** → 总耗时 = 最慢的那个 chunk 的时间（约 15 秒）。

### 改造方案

#### 第一步：把 for 循环改成 `asyncio.gather`

```python
async def transcribe(self, audio_path, audio_duration=None):
    chunks = _split_audio(audio_path, chunk_seconds, known_duration=audio_duration)

    # 并发跑所有 chunk，但用信号量限制并发
    semaphore = asyncio.Semaphore(settings.asr_max_concurrency)  # 比如 6

    async def transcribe_with_limit(chunk_path, offset, idx):
        async with semaphore:
            logger.info(f"Transcribing chunk {idx}")
            segs = await self._transcribe_single(chunk_path)
            for seg in segs:
                seg["start"] += offset
                seg["end"] += offset
            return segs

    tasks = [
        transcribe_with_limit(path, offset, i)
        for i, (path, offset) in enumerate(chunks)
    ]
    results = await asyncio.gather(*tasks, return_exceptions=True)

    # 合并 + 处理失败的 chunk
    all_segments = []
    failed = []
    for i, r in enumerate(results):
        if isinstance(r, Exception):
            failed.append((i, r))
        else:
            all_segments.extend(r)

    if failed:
        # 失败的 chunk 重试一次（串行）
        for idx, err in failed:
            logger.warning(f"Chunk {idx} failed: {err}, retrying")
            r = await self._transcribe_single(chunks[idx][0])
            all_segments.extend(r)

    all_segments.sort(key=lambda x: x["start"])
    return all_segments
```

### 关键技术点

1. **为什么要信号量限流？**
   - Groq API 有 QPS 限制（免费版 RPM=30），并发太高会触发 429。
   - Semaphore 控制最多同时 6 个请求，安全又快。

2. **失败处理**：
   - `return_exceptions=True` 不让一个 chunk 失败拖垮整体。
   - 失败的 chunk 单独重试（甚至切换备用厂商）。

3. **进度上报**：
   - 串行版每 chunk 一个进度点。
   - 并发版改成"完成 N/12"，用 `asyncio.as_completed`：
   ```python
   completed = 0
   for coro in asyncio.as_completed(tasks):
       result = await coro
       completed += 1
       await update_progress(15 + 45 * completed / total)
   ```

4. **顺序保证**：
   - 并发跑结果会乱序，最后用 `sort(key=lambda x: x["start"])` 排回去。
   - 每个 chunk 自带 offset，时间戳正确。

5. **本地切片**：
   - `_split_audio` 用的是 `ffmpeg -c copy`（流拷贝，不重编码），切 12 个 chunk 只需要几秒。
   - 也可以异步切（`asyncio.create_subprocess_exec`）。

6. **更激进的优化**：
   - **流式切片 + 流式 ASR**：边切边发，不用等所有 chunk 切完。
   - **VAD 切片**：用 silero-vad 在静音处切，避免一句话被切断。

### 性能对比

| 方案 | 2H 视频耗时 | 并发 |
|---|---|---|
| 串行 | 180s | 1 |
| 并发 6 | 30s | 6 |
| 并发 12 + 多厂商 | 15s | 12 |

### 面试官可能追问
- **Q：为什么不直接 chunk_size=2h 一次发？**
  → Groq 限制单文件 40MB，2 小时 MP3 远超限制。
- **Q：chunk 边界一句话被切断怎么办？**
  → 切片时加 1-2 秒 overlap，后处理去重（按时间戳合并相同文本）。
- **Q：怎么验证并发后准确率没下降？**
  → 跑同一份音频，串行和并发结果做 diff，人工抽查。

def track_llm_usage(purpose: str):

            start = time.time()
            task_id = kwargs.get("task_id")

### 价格表配置（厂商价格会变）

```python
PRICING = {
    "deepseek-chat": {"input": 0.14, "output": 0.28, "unit": "USD/M"},
    "gpt-4o-mini":   {"input": 0.15, "output": 0.60, "unit": "USD/M"},
    "qwen-plus":     {"input": 0.8,  "output": 2.0,  "unit": "CNY/M"},
}

def calc_cost(model, prompt_tokens, completion_tokens) -> int:
    """返回 micro_yuan（百万分之一元）"""
    p = PRICING[model]
    cny_per_m = to_cny(p["input"], p["unit"])
    cost = (prompt_tokens * p["input"] + completion_tokens * p["output"]) / 1_000_000
    return int(cost * 1_000_000)  # micro_yuan
```

### 用 ClickHouse 做分析

业务库（Postgres）写入 → 同步到 ClickHouse → 跑分析查询。
ClickHouse 聚合查询比 Postgres 快 100 倍。

```sql
-- 每天每个厂商的成本
SELECT
    toDate(called_at) AS day,
    provider,
    SUM(total_tokens) AS tokens,
    SUM(cost_micro_yuan) / 1e6 AS cny
FROM llm_usage
WHERE called_at > now() - INTERVAL 30 DAY
GROUP BY day, provider
ORDER BY day DESC;

-- Top 10 高消耗用户
SELECT user_id, SUM(cost_micro_yuan) / 1e6 AS cny
FROM llm_usage
WHERE called_at > now() - INTERVAL 7 DAY
GROUP BY user_id
ORDER BY cny DESC LIMIT 10;
```

### 关键技术点

1. **价格用整数存（micro_yuan）**：浮点数加和会有精度误差，财务会算崩。
2. **异步写入**：埋点不能阻塞业务，用 `BackgroundTasks` 或本地队列异步写。
3. **配额预扣**：调用前先估算 token，预扣额度，调完结算实际值（避免超额）。
   ```python
   estimated = estimate_tokens(prompt) * 2  # 假设输出和输入差不多长
   if user.balance < estimated_cost:
       raise InsufficientFunds()
   # 调用...
   actual = resp.usage.total_tokens
   refund_or_charge(user, estimated, actual)
   ```
4. **实时大盘**：用 Grafana + Prometheus，关键指标做面板：
   - 今日消耗
   - 各厂商占比
   - 单任务平均成本
   - 异常用户预警
5. **告警**：单用户/单任务超过阈值（比如单任务 > 5 元）自动告警。
6. **导出报表**：月底导出 CSV，对账。

- **Q：高并发下统计会丢吗？**
  → 业务库写入用 INSERT（同步），保证不丢。CK 做聚合分析（最终一致）。
- **Q：怎么防止用户刷接口烧钱？**

如果面试官问"你做过哪些架构优化"，可以按 **业务规模演进** 来讲：

```
单机 MVP
   ↓ 用户多了，要协作
1. OSS 直传 + 视频共享
   ↓ 任务多了，单机扛不住
2. Redis + Celery 异步队列

3. 任务可断点续传 + 失败重试
   ↓ 单厂商不稳定
4. ASR / LLM 多厂商兜底 + 熔断
   ↓ 长视频处理太慢
5. ASR 并发改造
   ↓ 数据涨到亿级
6. 分库分表（Citus）
   ↓ 要算钱
7. Token 统计 + 成本分析
   ↓ 要合规
8. 违禁词三层过滤
   ↓ 要运维
9. 日志统一采集（Fluent Bit + SLS）
```

每一步都是"**遇到什么问题 → 用什么方案 → 关键权衡**"，逻辑顺，故事完整。
