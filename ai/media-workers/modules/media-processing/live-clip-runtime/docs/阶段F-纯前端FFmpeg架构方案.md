```

## 二、为什么这个架构对你的场景特别合适

| 维度 | 旧架构（服务端 FFmpeg）| 新架构（纯前端 FFmpeg）|
| --- | --- | --- |
| **上传流量** | 整段视频 1~10 GB | 只传音频 10~50 MB |
| **服务器存储** | 原视频 + 切片产物 | 仅音频文件 + DB 元数据 |
| **服务器 CPU 占用** | FFmpeg 抽音频/切片瞬时打满 | 几乎为零（只等第三方 API）|
| **服务器并发上限** | 受 CPU 抢核约束（推荐 5）| 受 Groq/DeepSeek RPM 约束（可调 10+）|
| **服务器装 FFmpeg** | 必装、要折腾 PATH | **完全不需要** |
| **隐私** | 视频要传到服务器 | **视频从不离开用户机器** |
| **跨设备查看切片** | 服务器存切片 → 可以 | 切片在用户 PC → 不能（接受这个取舍）|

**核心洞察**：你已经写好了 70% 的前端 FFmpeg 代码，旧架构的"服务端切片"分支其实是**死代码**——前端默认就是只上传音频走"音频直传"路径。这次改造是**清理 + 调优**，不是从头新建。

## 三、本次具体改了什么

### 3.1 服务端 `pipeline.py`：大幅瘦身

旧文件 374 行，新文件 ~190 行（**减半**）：

- 删除整个 "传统视频模式" 分支（`is_audio == False`）

所有 `Clip` 记录的 `file_key` 永远为 `None`——切片产物不在服务器。

### 3.2 服务端 `config.py`

```

```env

```

### 3.4 没改的部分（向后兼容）

- **`TaskStatus` 枚举**：`downloading` / `clipping` 这两个状态名虽然不再贴切，但保留以避免 DB migration。`task_runner.py` 抢到任务时仍写 `downloading`（语义可理解成"开始处理"），pipeline 里没有任何代码会写 `clipping` 了。

- **前端**：一行没改。当前 UI 已经是「上传音频 → 等任务 done → 手动点一键切片下载」，完全符合方案。

### 旧时序（用户视角）

```
0:00   选视频 → 浏览器抽音频（30s~2min）
0:30   上传音频（10MB，~3s）

4:00   服务器: 拷贝到 storage（~5s）
4:05   任务 done，用户在详情页看到切片
4:05+  用户：通过 /api/clips/{id}/download 下载切片
```

### 新时序（用户视角）

```
0:00   选视频 → 浏览器抽音频（30s~2min）
0:30   上传音频（10MB，~3s）
0:33   服务器: ASR 转录（30s~3min）         ← 协程池能并发跑 10 个
3:00   服务器: LLM 分析（10~30s）

3:31   用户点「一键切片」→ 重新选原视频

```

**关键变化**：服务器侧少了 30~60 秒的 FFmpeg 切片时间，腾出来的 CPU 直接拿来并发服务更多用户。

## 五、10 个用户并发场景（新架构下）

假设单任务的服务端处理时间 ~1 分钟（主要在 ASR）：

| 用户 | 串行（最初）| 5 worker（阶段 E）| 10 worker（阶段 F）|
| --- | --- | --- | --- |
| 用户 1~10 | 0~10 min | 0~2 min | **0~1 min** ⚡ |

**10 个用户几乎可以 1 分钟全部完成**——只要不撞 Groq/DeepSeek 的 RPM 限流。

为什么能拉到 10？因为新架构下 worker 协程在 99% 时间都是「挂在网络响应上」，本机 CPU 完全闲着。**瓶颈从机器变成第三方 API 速率**，而不是你的 CPU。

## 六、并行调度机制：把后台想象成一家小食堂

> 这一节假设你完全没有后台基础，所有专业名词都翻译成"食堂"里的东西。看完这一节你应该能跟别人讲清楚"10 个用户同时下单时后台是怎么不出乱子的"。

### 6.1 一句话比喻

**把整个后台想象成一家小食堂**：

| 后台专业名词 | 在食堂里对应什么 |
| --- | --- |
| `tasks` 表（数据库里的一张表）| 贴在墙上的**订单板**，每个用户来下单就贴一张新单 |
| Worker 协程 | 后厨的**厨师**，咱们这家店开了 10 个 |
| `asyncio.Lock` | 订单板旁边唯一一支**取单笔** |
| `run_video_pipeline`（流水线代码）| 厨师拿到单后**做菜的全过程**，要做大概 1 分钟 |

记住这张对应表，下面所有内容你都能套进来理解。

```
┌──────────────────────────────────────────────────────────┐
│            后台 = 一家有 10 个厨师的小食堂                  │
│                                                          │
│         ┌──────────────────────┐                         │
│         │  🧾 订单板 (tasks 表) │                         │
│         │  ─────────────────  │                         │
│         │  ☐ 张三的单 (待做)   │  ← 用户提交时贴上来       │
│         │  ☐ 李四的单 (待做)   │                         │
│         │  ☐ 王五的单 (待做)   │                         │
│         │  ▣ 赵六的单 (做完)   │                         │
│         └──────────────────────┘                         │
│                  ▲                                       │
│                  │ 想划单子必须先去拿这支笔                  │
│                  ▼                                       │
│              ┌──────────┐                                │
│              │ 📝 取单笔 │  ← 全店就这一支                  │
│              │  (Lock)  │     谁拿到谁才能动单子            │
│              └──────────┘                                │
│                  ▲                                       │
│                  │ 抢                                     │
│   ┌──────┬──────┬┴─────┬──────┬──────┐                  │
│   │👨‍🍳 #0 │👨‍🍳 #1 │👨‍🍳 #2 │ ...  │👨‍🍳 #9 │  10 个厨师       │
│   └──────┴──────┴──────┴──────┴──────┘                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 6.2 厨师的"上班一天"在干啥

每个厨师一上班就**重复做下面这件事，永远不停**：

1. 走到订单板前，**伸手去拿取单笔**
   - 笔在 → 拿起来，进入第 2 步
   - 笔不在（被别的厨师拿走了）→ 在旁边等，等到笔被放回来立刻抢

2. 拿到笔了，**看订单板最上面那张"待做"的单**
   - 有 → 在上面写"#X 号厨师认领了"，划掉"待做"标记，**放下笔**
   - 没有（板上全空或全在做了）→ 直接放下笔，去后厨睡 1 秒，回到第 1 步

3. 回到自己的灶台，**开始做菜**（这个过程要 ~1 分钟）
   - 这一步**非常关键**：笔已经在第 2 步还回去了
   - 这一分钟我在闷头做菜，**其他 9 个厨师该接单的接单，互不干扰**

4. 做完菜了，**在订单上盖"出餐"章**（数据库里把这单 status 改成 done）
   - 立刻回到第 1 步，去拿笔接下一单

**全部精髓就在第 2 步和第 3 步的分工**：
- 第 2 步要拿笔，但只要 ~50 毫秒（几句话的功夫）
- 第 3 步不用拿笔，但要花 60 秒

笔被占用的时间太短了，所以 10 个厨师就算抢笔也不会真的卡住。

### 6.3 没有这支笔会怎样？

你可能会问：搞这么麻烦干嘛？让 10 个厨师自己看订单板不就行了？

来看一下**没有取单笔**会发生什么：

```
时间 0 秒：10 个厨师同时盯着订单板，
          全都看见"张三的单"在最上面
时间 1 秒：10 个厨师同时伸手抢这张单 → 10 个人都觉得"是我的"
时间 2 秒：10 个厨师同时在做张三的菜
          → Groq 接口被白白调用 10 次（钱包出血）
          → 张三的菜被做了 10 份
          → 数据库里张三的切片记录被写 10 遍
          → 张三看到自己的进度条疯狂跳动很疑惑
```

显然这就乱套了。**这就是为什么需要一支"取单笔"**——它的唯一作用就是保证「同一时刻只有一个厨师在动订单板」。

**有了取单笔之后**：

```
时间 0.00 秒：厨师 #0 第一个抢到笔
            另外 9 个厨师没抢到，全都在边上等
时间 0.05 秒：厨师 #0 划掉"张三的单"，放下笔，跑去灶台做菜
时间 0.05 秒：等在边上的 9 个人立刻又抢笔，厨师 #1 抢到了
时间 0.10 秒：厨师 #1 看板上最上面已经是"李四的单"（张三的已被划掉）
            划掉它，放下笔，跑去灶台
时间 0.15 秒：厨师 #2 抢到笔 → 划掉"王五的单" → 放笔 → 跑灶台
...
时间 0.50 秒：所有 10 个厨师各拿了一张单，全部跑去灶台做菜
            订单板上不再有"待做"的单了
```

干净利落。**这就是 `asyncio.Lock` 这个东西的全部作用**——一句话总结：

> **它就是那支"全店唯一一支取单笔"，保证同一时刻只有一个厨师在动订单板。仅此而已。**

### 6.4 整个机制能跑起来的关键

请记住下面这个数字对比，**整个并行机制能成立全靠这个**：

| 动作 | 耗时 | 需要拿笔吗？ |
| --- | --- | --- |
| 拿笔 + 划单 + 放笔（第 2 步）| **~50 毫秒** | 是 |
| 做菜（第 3 步，ASR + LLM）| **~60 秒** | **否** |

两个时间差 **1000 多倍**。

这意味着：
- 就算 10 个厨师为了拿笔排队，**总共也只排 0.5 秒**（10 × 50ms）
- 接下来的 60 秒，10 个厨师**真正并行**在各自的灶台做菜
- 拿笔这事根本不会成为瓶颈

**反过来设想一下**：如果我们设计成"做菜全程都得拿着笔"，那店里 10 个厨师的意义就没了——同一时刻只有一个人在做菜，跟单厨师小破店一模一样。**好在我们的设计是"拿笔只为了认单，做菜不需要笔"**。

### 6.5 一个完整故事：晚高峰 10 个人同时下单

```
T = 0 秒
  └─ 10 个用户同时点击"提交任务"
     前台（FastAPI）噼里啪啦在订单板上贴 10 张"待做"单
     此时 10 个厨师都在后厨睡觉（上一轮没活儿了在打盹）

T = 1 秒
  └─ 10 个厨师身上的"1 秒醒一次"闹钟同时响了，全员起床冲向订单板抢笔
  
  T+0.00s  厨师 #0 抢到笔，划掉"张三的单"，放笔，回灶台
  T+0.05s  厨师 #1 抢到笔，划掉"李四的单"，放笔，回灶台
  T+0.10s  厨师 #2 抢到笔，划掉"王五的单"，放笔，回灶台
  ...
  T+0.45s  厨师 #9 抢到笔，划掉"第 10 个用户的单"，放笔，回灶台

T = 1.5 秒
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  此刻 10 个厨师在 10 个灶台上 **同时** 做菜
  全程不用拿笔，谁也不挡谁
  整个店看起来很闲（电脑 CPU 占用接近 0%），因为
  所有厨师都在"等 Groq/DeepSeek 把菜的半成品发回来"
  就像 10 个厨师都把菜放进了 10 个烤箱，
  站那盯着烤箱看，自己啥事也没干
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

T ≈ 61 秒
  └─ 10 个厨师陆续做完各自的菜，盖完"出餐"章
     盖完立刻去抢笔，看有没有新单
  
T ≈ 61.5 秒
  └─ 板上一张待做的单都没有了
     10 个厨师又集体回去睡 1 秒，等下波客人
```

**结论**：10 个用户的菜在 **60 多秒**就全部出餐了。如果是只有 1 个厨师的旧店（串行），同样的活要做 **10 分钟**。

> 顺带说一句：**如果 11 个用户同时来**，第 11 张单只会先贴在板上等着，等任何一个厨师做完手头的菜回来抢笔，下一轮就把它捞走。所以"排队"是真实存在的，只不过把"排队的位置"从 10 名以后开始而已。

### 6.6 订单上的"状态"长什么样

每张订单上有一个**状态标记**，会随着流程变化。看这张图你就明白整个流转了：

```
   ┌─ 用户下单 ─┐
   ▼            │
  [待做]         │  ← 新单子刚贴上来时的样子
    │            │
    │ 厨师拿笔划掉
    ▼            │
  [做菜中]       │  ← 这单已经被某个厨师认领了
   ...           │     "做菜中"细分成几个小阶段：
   ...           │       - 厨师抓起单（downloading）
   ...           │       - 调 Groq 转录中（transcribing）
   ...           │       - 调 DeepSeek 分析中（analyzing）
   ...           │       - 把结果写进数据库中（uploading）
    │            │
    │ pipeline 跑完
    ▼            │
  [出餐 done]    │  ← 终态，用户能在前端看到结果
                 │
   ━━━ 或者 ━━━
                 │
  [翻车 failed]  │  ← 也是终态，前端会显示错误信息
```

简单翻译一下数据库里那些英文状态名：

| 状态名（英文）| 通俗解释 |
| --- | --- |
| `pending` | 待做（贴上墙等被拿）|
| `downloading` / `transcribing` / `analyzing` / `uploading` | 做菜中的不同小阶段，看到这些都表示"正在被某厨师处理"|
| `done` | 出餐了 |
| `failed` | 翻车了 |

> 历史包袱：`downloading` / `clipping` 这种名字是旧"服务端切片"时代留下的。新架构没有切片步骤了，但**我们刻意保留这些名字是为了不动数据库表结构**（动了就要做迁移）。当成"做菜中的某个阶段"理解就够了。

### 6.7 停电了怎么办？菜会丢吗？

这个机制最让人安心的一点：**订单板是用钉子钉在墙上的（数据库存的），不是写在白板上的（内存）**。

所以哪怕整个食堂突然停电、服务器进程被强杀重启，**订单一张都不会丢**。

重启时做一件简单的事：

```
扫一遍订单板，
把所有"做菜中"的单子全部改回"待做"
（因为之前在做这单的厨师肯定没做完，半成品菜废了）

然后重新拉起 10 个厨师，开干。
```

**代价**：被打断的那几单**要重头做一遍**（再调一次 Groq ASR）。但用户的菜不会丢，数据也不会错乱——因为咱们设计成"做完最后一步才一次性把切片写进数据库"，做一半被打断时数据库里**根本没有脏数据**，重做时也是一张干净的画布开始。

### 6.8 为啥不用数据库自带的锁？

你后面可能会听别人说"用 **PostgreSQL FOR UPDATE SKIP LOCKED** 更专业"——意思是让**数据库本身**当那支取单笔，厨师每次要划单子都去数据库前台借笔。

| 方案 | 通俗对比 | 适用场景 |
| --- | --- | --- |
| 当前的 `asyncio.Lock` | **店里自备一支笔**，所有厨师共用 | 一家分店（单进程部署）|
| `FOR UPDATE SKIP LOCKED` | **跑去数据库前台借笔**，更通用 | 多家分店共用一块订单板（多进程部署）|

两个方案**效果一样**。但只要哪天店开始**两个分店共用一块订单板**（也就是 `uvicorn --workers 4` 多进程部署），自家这支笔就不够用了——一号分店的笔二号分店看不见。那时再切到去数据库借笔。

**现在还只有一个分店，所以自家这支笔最简单、最够用**，5 行代码搞定。详见"决策 2"。

### 6.9 你可能想问的几个问题

| 你可能想问 | 简单回答 |
| --- | --- |
| 10 个厨师真的是 10 个员工吗？| 不是真的开 10 个员工。是 Python 里把同一段做菜代码"复制 10 份分身"。CPU 还是那 1 颗，只是分身们大部分时间都在"等 Groq/DeepSeek 返回结果"（相当于在烤箱前发呆），所以能凑出 10 个同时干活的效果 |
| 抢笔会不会成瓶颈？| 不会。抢笔 + 划单总共 0.05 秒，做菜要 60 秒。就算 10 个人排队抢笔总共也只 0.5 秒，相比 60 秒可以忽略 |
| 做菜时其他厨师要等吗？| 不用。笔早就放回去了，10 个厨师同时各做各的，互不影响 |
| 排队的客人能知道自己排第几吗？| 能。看一下板上"我前面还有几张待做单"就知道了。前端会显示成"前面还有 N 个人排队"（通过 SSE 推送的 `queue_position` 字段）|
| 客人下单时要不要也用这支笔？| **不要**。下单只是往板上**贴一张新单**（不动旧单），所以下单跟厨师抢笔完全不冲突。10 个客人同时下单也毫无问题 |
| 队列存在哪？要不要 Redis？| **队列就是订单板（`tasks` 表）本身**。没有 Redis、没有 Celery、没有任何额外的中间件 |
| 停电了菜会丢吗？| 不会。订单板钉在墙上（数据库），重启后把"做菜中"的单全部改回"待做"，重新分给厨师做 |

## 七、关键设计决策的理由记录

### 决策 1：切片产物为什么不上传回服务器

**用户决定**："先下载到本地"，不要服务端转存。

**理由**：
- 公司内部用，切片主要是给操作的人自己用
- 服务端存储是真金白银的成本（一段直播 → 10 段切片 → 几百 MB）
- 用户随时可以重新打开任务，再切一次（clip plans 在 DB 永久保存）
- **取舍**：失去了"换台机器在服务器看切片"的能力。可以接受。

### 决策 2：为什么不用 PostgreSQL `FOR UPDATE SKIP LOCKED`

**当前部署**：单 uvicorn 进程，所以阶段 E 的 `asyncio.Lock` 已经够了。
**升级路径**：将来如果要 `uvicorn --workers 4` 多进程部署，再换 `FOR UPDATE SKIP LOCKED`（5 行代码）。

> 详细对比见 §6.4。简言之：单进程下两种方案功能等价，`asyncio.Lock` 代码量更少、零 SQL 兼容性问题，所以选它；将来真要多进程再迁移。

### 决策 3：为什么不删 `clipper.py`

保留以让 `task_duration.py` 给**旧任务**回填 `video_duration`。新机器上不装 FFmpeg 不影响——`hydrate_missing_task_durations` 内部 `try/except` 会安静吞掉。

代码留着是免费的；删掉万一回滚要恢复就麻烦。**保守起见保留**。

## 九、如果 Groq / DeepSeek 限流了怎么办？

**触发条件**：`WORKER_CONCURRENCY=10` + 10 个用户齐刷刷上来 + 单个用户音频很长（多次调 ASR）。

**应对**：
1. **先观察**：跑几天，看错误日志里有没有 `429` / `rate_limit_exceeded`
2. **真出问题再优化**：
   - 加客户端层级 `asyncio.Semaphore` 限制对 Groq 的并发（独立于 worker 数）
   - 升级 Groq / DeepSeek 套餐
   - 加指数退避重试（很多 SDK 自带）
3. **过早优化是恶**：现在不加，等用户量真上来再说。
