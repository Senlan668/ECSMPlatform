# 本地一键切片交互重设计文档

> **目标**：移除任务详情页“一键切片”里手动输入本地绝对路径的交互，改为用户重新选择原视频文件后，在浏览器本地完成切片并下载 ZIP，同时不影响现有“AI 自动直播切片”主流程。

---

## 1. 背景与问题

当前任务详情页的“一键切片”入口要求用户手动输入原视频的本地绝对路径，然后由后端调用本机 FFmpeg 裁切视频。

这个方案的问题很明确：

- 用户需要去 Finder 或资源管理器复制路径，交互笨重
- 纯 Web 环境下浏览器拿不到可复用的本地绝对路径，体验天然别扭
- 用户最常见的真实诉求不是“输入路径”，而是“把刚才那个原视频拿来切”
- 该交互和上传页“拖拽 / 选文件”的使用心智不一致

本次设计只解决任务详情页的“一键切片”体验，不重构分析链路，也不改变上传页的工作方式。

---

## 2. 设计目标

- 去掉手动输入本地路径的交互
- 不重新上传整段原视频
- 不改动现有“浏览器提取音频 -> 上传音频 -> 后端 AI 分析”的主流程
- 改动范围尽量限制在任务详情页和新增的前端本地切片服务
- 优先保证 macOS Chrome、Windows Chrome、Edge 可用
- 用户在当前页面这一次操作内完成免手输，不要求跨刷新、跨天保留本地文件关联

### 非目标

- 不改上传页
- 不改任务创建接口、转录、分析、SSE、任务列表
- 不把“一键切片”做成桌面应用能力
- 不要求 Safari 达到和 Chrome / Edge 同等体验
- 不处理“刷新页面后仍记住本地文件句柄”这一类持久化能力

---

## 3. 范围约束

本次设计只替换 [TaskDetailPage.tsx](/Users/lizexi/Documents/AI/Agent/ai-slice/frontend/src/pages/TaskDetailPage.tsx) 中现有“一键切片”弹窗和它依赖的导出实现。

以下能力必须保持不变：

- 上传页仍使用 [audioExtractor.ts](/Users/lizexi/Documents/AI/Agent/ai-slice/frontend/src/services/audioExtractor.ts) 在浏览器端提取音频
- 后端任务分析仍通过 `/api/tasks`、ARQ、SSE 完成
- 任务详情页中的 SRT 导出、文案导出、剪映草稿导出保持现状
- 后端现有 `/api/tasks/{task_id}/export/clips` 接口可以先保留，但新的任务详情页不再依赖它

## 4. 用户体验方案

### 4.1 新交互流程

用户在任务详情页点击“`一键切片`”后：

1. 打开弹窗
2. 弹窗展示拖拽区域和“选择原视频”按钮
3. 用户拖入或选择本地原视频文件
4. 弹窗显示：
   - 文件名
   - 文件大小
   - 与当前任务文件名的匹配状态
5. 用户点击“开始切片”
6. 前端在浏览器本地逐段裁切视频并打包 ZIP
7. 浏览器自动下载 ZIP 文件

### 4.2 弹窗文案调整

原来的路径输入框和“复制路径”提示全部移除，替换为更直接的引导：

- 标题：`✂️ 一键切片`
- 副文案：`选择本任务对应的原视频文件，浏览器会在本地完成切片，不会重新上传整段视频`
- 主操作：`选择原视频`
- 次要交互：支持直接拖入文件

### 4.3 切片过程反馈

弹窗中必须显示明确阶段，而不是单一 loading 状态。建议阶段如下：

- 正在加载本地切片引擎
- 正在读取视频文件
- 正在切片 `X / N`
- 正在打包 ZIP
- 准备下载

用户需要可以主动取消当前切片任务，避免浏览器长时间占用时无反馈。

## 5. 技术方案

### 5.1 总体策略

“AI 自动直播切片”的数据生产仍由后端完成；“一键切片”的视频裁切改为前端本地完成。

即：

- **分析链路不变**：视频 -> 浏览器提音频 -> 上传音频 -> 后端转录/分析 -> 产出 clips
- **导出链路改造**：原视频文件 -> 浏览器本地 FFmpeg.wasm -> 本地逐段切片 -> 浏览器 ZIP 下载

这样能同时满足：

- 不再手输路径
- 不重新上传整段视频
- 不侵入现有后端分析流程

### 5.2 模块拆分

新增一个独立前端服务，例如：

- `frontend/src/services/videoClipExporter.ts`

该服务负责：

- 加载并复用 FFmpeg.wasm
- 接收 `File` 与 `clips`
- 在浏览器本地逐段导出 MP4
- 汇总结果并打包 ZIP
- 对外暴露进度事件与取消能力

页面层 [TaskDetailPage.tsx](/Users/lizexi/Documents/AI/Agent/ai-slice/frontend/src/pages/TaskDetailPage.tsx) 只负责：

- 文件选择和拖拽交互
- 进度展示
- 错误展示
- 下载触发

### 5.3 ZIP 打包

前端当前没有现成 ZIP 依赖，需要新增轻量打包库，推荐 `fflate`。

要求：

- 仅用于前端内存中打包多个 MP4 文件
- 生成单个 ZIP Blob 后交给现有下载逻辑
- 尽量避免引入过重依赖

### 5.4 与现有代码的隔离

不建议把上传页的 [audioExtractor.ts](/Users/lizexi/Documents/AI/Agent/ai-slice/frontend/src/services/audioExtractor.ts) 直接重构成一个“万能 FFmpeg 基建层”后再回改上传页。

原因：

- 本次任务的最高优先级是降低回归风险
- 用户明确要求“不影响之前的 AI 自动直播切片功能”
- 上传页目前工作正常，没必要把这次交互优化扩散成全局重构

可接受的复用只有：

- 复用 `/public/ffmpeg` 静态资源
- 复用与 FFmpeg.wasm 初始化相关的基本模式

但本地切片服务应保持实现隔离。

## 6. 数据与时间轴处理

### 6.1 切片数据来源

切片时间段继续来自任务详情接口返回的 `task.clips`。

每个 clip 至少使用这些字段：

- `clip_index`
- `title`
- `start_time`
- `end_time`

### 6.2 时间偏移处理

当前系统在音频直传模式下，会把检测到的视频 `video_start_offset` 应用到转录结果，再由 LLM 产出 clip 时间。

这意味着：

- 任务详情中用于展示给用户看的时间，可能会为了“手动在剪映裁切”而减去 `video_start_offset`
- 但真正用于本地视频裁切的时间，应继续使用任务接口里的原始 `clip.start_time` / `clip.end_time`

本次设计明确规定：

- **UI 展示可继续沿用当前 display range 逻辑**
- **本地 FFmpeg.wasm 切片必须使用任务原始 clip 时间，不使用前端减偏移后的展示时间**

这样才能与当前后端导出逻辑保持一致，避免切出来的时间整体偏移。

### 6.3 文件名生成

输出 ZIP 内的文件命名规则沿用现有风格：

- `01_标题.mp4`
- `02_标题.mp4`

标题中的非法字符继续替换为安全字符，文件名长度保留合理截断。

最终下载文件名建议为：

- `<视频名>_AI切片.zip`

## 7. 浏览器兼容与降级策略

### 7.1 主支持范围

本设计的主支持范围是：

- Windows Chrome
- macOS Chrome
- Edge

### 7.2 降级策略

如果当前浏览器无法稳定运行本地 FFmpeg.wasm 切片，则弹窗直接给出明确提示，不再退回“手输路径”模式。

提示策略：

- FFmpeg.wasm 初始化失败：提示用户改用 Chrome / Edge
- 浏览器资源不足导致任务失败：保留失败信息并建议关闭其他占用较高页面后重试

本次设计不要求为 Safari 提供等价替代交互。

## 8. 错误处理策略

### 8.1 文件选择校验

在开始切片前做轻量校验：

- 文件扩展名或 MIME 类型看起来像视频
- 文件名与当前任务 `video_filename` 完全一致或高度相似
- 文件大小不是明显异常值

如果文件名不一致：

- 不强制阻止继续
- 但显示显著警告：`你选择的可能不是本任务的原视频，切片时间点可能不匹配`

### 8.2 切片失败策略

本次设计采用“部分成功优先”而不是“全有或全无”。

具体行为：

- 单段切片失败时，记录该段失败原因并继续处理后续片段
- 所有片段完成后，如果存在失败项，则给出汇总结果
- 结果文案示例：`已成功导出 9 段，失败 2 段`

这样比整批直接失败更符合真实使用场景。

### 8.3 用户取消

切片过程中保留“取消”按钮。

取消后应：

- 终止当前 FFmpeg.wasm 任务
- 清理中间状态
- 保持弹窗可重新选择文件或重新开始

## 12. 设计结论

本方案通过把“一键切片”从“输入路径 -> 后端切片”改成“选择文件 -> 前端本地切片”，在不影响现有 AI 分析主流程的前提下，解决了当前最显著的交互问题。

它的核心价值是：

- 去掉最笨重的手输路径
- 保持纯 Web 形态
- 不增加原视频二次上传成本
- 把改动范围控制在任务详情页与新增前端服务内

该设计已经足够进入实现计划阶段，不依赖额外产品决策。
