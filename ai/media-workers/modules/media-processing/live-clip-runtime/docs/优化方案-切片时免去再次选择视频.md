# 优化方案：切片时免去再次选择视频文件

## 一、问题背景

### 1.1 用户视角的疑惑

进入系统时上传了一次视频，AI 分析处理完成，进入任务详情页点「✂️ 一键切片」时，又被要求**再选一次视频文件**。看起来像是"上传两次同样的视频"，体验割裂。

### 1.2 实际数据流转

| 阶段 | 用户动作 | 实际发生的事 | 是否真的上传 |
|------|----------|--------------|-------------|
| 进入系统 | 拖入视频 | 浏览器用 FFmpeg.wasm 在**本地**提取音频，仅把 ~30MB 的 MP3 发到后端 | ✅ 仅上传音频 |
| 一键切片 | 再选视频 | 浏览器用 FFmpeg.wasm 在**本地**按 AI 给的时间点切片打包成 ZIP 下载 | ❌ 一字节都不传 |

**核心事实**：原视频从未离开用户电脑，"两次上传"实际只有一次上传（30MB 音频），第二次只是浏览器需要重新拿到 `File` 对象的引用。

### 1.3 为什么不能省掉第二次选择

- **后端没存原视频**：为了"上传体积缩减 99%"的核心卖点，后端 `./storage/uploads/` 里只有 MP3，没有原视频。
- **浏览器没法记住磁盘路径**：出于安全沙箱限制，JS 不能凭一个绝对路径去访问磁盘文件，必须由用户主动通过 `<input type="file">` 或 `showOpenFilePicker()` 重新授权。
- **页面跳转丢失 `File` 引用**：从 `UploadPage` 跳到 `TaskDetailPage`，第一次选中的 `File` 对象在内存里就被释放/丢失了。

---

## 二、四种方案对比

### 方案总览

| 方案 | 核心思路 | 改动量 | 跨会话保留 | 体验 | 兼容性 | 推荐度 |
|------|----------|--------|-----------|------|--------|--------|
| **A：同会话缓存 `File`** | 全局 store 暂存 File 对象 | 极小（~100 行） | ❌ 关页面就丢 | 同会话无感 | 全浏览器通用 | ⭐⭐⭐⭐⭐ |
| **B：File System Access API 持久化** | IndexedDB 存 FileHandle | 中等（~300 行 + 兜底） | ✅ 关浏览器仍记住 | 接近原生 | Chrome/Edge 完整；Safari/Firefox 受限 | ⭐⭐⭐ |
| **C：OPFS / IndexedDB 缓存 Blob** | 把整段视频复制到浏览器存储 | 大 | ✅ 但易失效 | 复制阶段卡顿 | 配额受限 | ⭐ |
| **D：后端保留原视频** | 退回直传视频方案 | 巨大 | ✅ | 上传慢 | 通用 | ❌ |

---

### 方案 A：同会话缓存 `File` 对象 ✅ 推荐先做

#### 原理

页面级全局 store（Zustand / Context / 模块级 `Map`）保存 `taskId → File`。同一标签页内跳转、刷新前都能用。

#### 落地步骤

1. 新建 `frontend/src/stores/videoFileStore.ts`：

   ```ts
   const store = new Map<string, File>();
   export const videoFileStore = {
     set: (taskId: string, file: File) => store.set(taskId, file),
     get: (taskId: string) => store.get(taskId),
     delete: (taskId: string) => store.delete(taskId),
   };
   ```

2. `UploadPage.handleUploadSuccess` 创建任务成功后写入：

   ```ts
   videoFileStore.set(task.id, file);
   navigate(`/tasks/${task.id}`);
   ```

3. `TaskDetailPage` 点击「一键切片」前先查 store：
   - 命中 → 直接跳过弹窗、走切片流程
   - 未命中 → 退回当前的 `LocalClipExportModal` 让用户选

4. 任务删除时调用 `videoFileStore.delete(taskId)` 释放内存。

#### 优点

- **零依赖、零兼容性问题**：纯内存 Map，所有浏览器都行
- **改动量小**：核心约 100 行内
- **覆盖最高频路径**：用户上传完等几分钟看到结果就立刻切片，这是最常见的使用方式

#### 缺点

- 刷新、关标签页、换设备就会丢，需要 fallback 流程
- 长视频 `File` 对象会在内存里多占用一份引用（但 `File` 是磁盘指针，不是真的拷贝到内存，所以几乎无开销）

---

### 方案 B：File System Access API 持久化 FileHandle

#### 原理

Chromium 内核浏览器支持 `showOpenFilePicker()` 返回的 `FileSystemFileHandle` 可以被序列化存进 IndexedDB，下次按 `taskId` 取出来，调 `handle.getFile()` 重新拿到 `File`。仅需要一次权限授权（部分场景仍需点一下确认）。

#### 落地步骤

1. 上传时把入口从 `<input type="file">` 换成 `showOpenFilePicker()`，拿到 `handle`
2. 上传成功后：`indexedDB.put('videoHandles', taskId, handle)`
3. 一键切片时：`const handle = await indexedDB.get('videoHandles', taskId); const file = await handle.getFile();`
4. 必要时调 `handle.requestPermission({ mode: 'read' })` 重新授权

#### 优点

- **跨会话有效**：关浏览器、重启电脑都记得住
- **用户感知最接近原生应用**

#### 缺点

- **兼容性割裂**：
  - Chrome / Edge：完整支持
  - Safari：仅部分支持，handle 持久化能力有限
  - Firefox：桌面端基本不支持
- **降级路径不可省**：必须保留 `<input type="file">` 兜底
- **失效场景多**：用户移动/重命名/删除了源文件就报错，要 UI 引导重新选
- **首次仍需用户主动选**：第一次还是会走选择器，并不能完全免掉

---

### 方案 C：OPFS / IndexedDB 缓存视频 Blob

#### 原理

上传时除了提取音频，再把原视频写一份进 OPFS（Origin Private File System）或 IndexedDB。下次直接从浏览器存储读，不需要用户选文件。

#### 优点

- 跨会话有效，且不依赖磁盘上的原文件是否还在原地

#### 缺点

- **复制成本大**：把 GB 级视频写一份进浏览器存储，I/O 极慢，会让上传环节直接慢一倍以上
- **配额风险**：浏览器存储有配额（一般几 GB），多个任务并存会被驱逐
- **违背项目核心卖点**：项目主打"上传体积缩减 99%、内存占用 50MB"，这个方案直接打脸
- **隐私争议**：把用户视频写进浏览器存储，需要明确同意

❌ 不推荐。

---

### 方案 D：后端保留原视频

#### 原理

退回最早期的"用户直传视频到服务器"方案，所有处理都在服务端，切片时直接用服务端文件。

#### 优点

- 体验最简单，跟传统 SaaS 一样
- 跨设备可用（用户从手机看任务也能切片）

#### 缺点

- **彻底放弃当前架构**：README 里反复强调的"99% 体积缩减""零云服务依赖""单机轻量"全部作废
- 6 小时直播录像可能十几个 GB，本地存储 + 上传带宽双重压力
- 与 README 第 23 行 "上传体积缩减 99%" 的卖点直接冲突

❌ 不推荐。

---

## 三、推荐落地路径

### 第一步（必做）：方案 A

- 投入：1~2 小时
- 收益：覆盖最常见使用场景（上传完立刻看结果立刻切片），第二次选择基本可省
- 风险：零

### 第二步（可选增强）：在方案 A 之上叠加方案 B

- 投入：额外 0.5~1 天（含降级、权限处理、Safari/Firefox 兜底）
- 收益：跨会话也能记住，体验质感更高
- 风险：兼容性测试成本，需要在多浏览器跑

### 不要做：方案 C 和 D

与项目"浏览器端轻量处理、服务端零视频依赖"的整体定位冲突。

## 四、其他可衍生的体验优化

跟"再选一次"这个问题相关，但不在主线上的小优化点：

| 优化点 | 现状 | 改进 |
|--------|------|------|
| 一键切片的文件名提示 | 已有"建议选择与任务文件名相同的原视频" | 命中 store 时显示"已自动复用上传时的视频"标签 |
| 任务列表跳转切片 | 每个任务都要重新选 | 列表卡片悬浮可显示"视频引用：✅ 可直接切片 / ❌ 需重新选择" |
| 文件不匹配提示 | `evaluateVideoSelection` 已有 warning | 配合 store 时，按文件名+大小+时长综合判断更可靠 |

---

## 五、决策建议

**直接做方案 A**。它命中了 80% 用户场景、改动小、零风险，且不会和未来上方案 B 冲突。等用户实际反馈"换设备/重启后还想要"的时候再加方案 B。
