# 端到端数据流转（用户视角）

> 主线：用户**上传视频** → 服务器**分析出切片时间点** → 用户**在浏览器把视频切成 ZIP 下载到本地**
>
> 关键判断：**原视频从头到尾留在用户电脑上**。服务端只见过 MP3（音频）和文字（转录 + 分析结果），从未见过完整的视频画面。

---

## 0. 三个角色 / 四个边界

```
┌────────────────────────┐    边界①   ┌──────────────────────┐    边界③   ┌────────────────────┐
│        浏览器           │ ─────────► │     后端服务器        │ ─────────► │     云端 AI         │
│   (React + WASM)        │            │  (FastAPI + Worker)  │            │  Groq + DeepSeek    │
│                         │ ◄───────── │                      │ ◄───────── │                    │
│                         │    边界②   │                      │    边界④   │                    │
└────────────────────────┘            └──────────────────────┘            └────────────────────┘
   ↑ 持有 原视频/导出的ZIP                 持有 音频MP3/转录文本/切片元数据         无状态(每次调用独立)
```

| 边界 | 流向 | 经过的数据 |
|------|------|----------|
| ① 浏览器 → 服务器 | HTTP POST | MP3 音频文件 / 任务元数据 JSON / 操作指令（重试/重命名/AI生成） |
| ② 服务器 → 浏览器 | HTTP + SSE | 任务状态 / 进度百分比 / 切片列表 / 切片 mp4 文件流（本流程未使用） |
| ③ 服务器 → AI | HTTPS | MP3 文件（Groq）/ 转录文本拼成的 prompt（DeepSeek）|
| ④ AI → 服务器 | HTTPS | 转录 segments / 切片方案 JSON / 标题数组 / 剪辑思路 JSON |

**关键约束**：边界①只走音频和文字，**永不走完整视频**。

## 1. 时间轴总览

```
┌────────────────────────────────────────────────────────────────────────────────────────────┐
│ 用户                                                                                        │
│  │ 选场景 → 选视频                                                                          │
│  ▼                                                                                          │
│ ┌────────────────────────────────────────────────────────────────────────────────────────┐ │
│ │ A. 前端: FFmpeg.wasm 抽音频 (0~70%) ─ 输入 File ──► 输出 MP3 Blob + startOffset      │ │
│ ├────────────────────────────────────────────────────────────────────────────────────────┤ │
│ │ B. 前端 → 后端: POST /api/upload/audio (70~95%)                                       │ │
│ │      ▲ multipart MP3 Blob                                                              │ │
│ │      ▼ { audio_path: 服务器本地路径 }                                                  │ │
│ ├────────────────────────────────────────────────────────────────────────────────────────┤ │
│ │ C. 前端 → 后端: POST /api/tasks                                                       │ │
│ │      ▲ { video_path:audio_path, scene_mode, video_start_offset, video_duration }       │ │
│ │      ▼ { id, status:'pending', ... }                                                   │ │
│ ├────────────────────────────────────────────────────────────────────────────────────────┤ │
│ │ D. 前端: navigate(/tasks/:id) → GET /api/tasks/:id (首屏) + EventSource(/progress)    │ │
│ ├────────────────────────────────────────────────────────────────────────────────────────┤ │
│ │ E. 后端 Worker (task_runner) 拾起 pending 任务 → 跑 pipeline                          │ │
│ │   E1. pending → downloading (10%)  排队完成                                            │ │
│ │   E2. downloading → transcribing (15→60%)                                              │ │
│ │        ── 后端 → Groq: POST /audio/transcriptions (MP3 文件)                          │ │
│ │        ── Groq → 后端: { segments:[{start,end,text}, ...] }                            │ │
│ │   E3. transcribing → analyzing (60→80%)                                                │ │
│ │        ── 后端 → DeepSeek: POST /chat/completions (转录文本 + 场景 prompt, 可能多批次) │ │
│ │        ── DeepSeek → 后端: [{ clip_id, title, start_time, end_time, ... }, ...]        │ │
│ │   E4. analyzing → uploading (80→95%)  对齐时间/去重/过滤短片段                          │ │
│ │   E5. uploading → done (95→100%) 把切片元数据 INSERT 到 clips 表 (file_key = NULL)    │ │
│ │   SSE 每秒把 progress / progress_message / status 推给浏览器                            │ │
│ ├────────────────────────────────────────────────────────────────────────────────────────┤ │
│ │ F. 前端: status=done → 重新 GET /api/tasks/:id → 渲染切片卡片列表                     │ │
│ ├────────────────────────────────────────────────────────────────────────────────────────┤ │
│ │ G. 用户点 "一键切片" → 重新选/拖入原视频 → 前端 FFmpeg.wasm 切片                       │ │
│ │     输入: 原视频 File + clips[{start_time,end_time}]                                   │ │
│ │     输出: ZIP (各片段 mp4) → a[download] 触发本地下载                                  │ │
│ └────────────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                              │
│  另外两个可选异步动作（用户点单个切片卡片才触发）：                                          │
│   - POST /api/clips/:id/viral-titles  → DeepSeek → ["标题1",...,"标题5"]                    │
│   - POST /api/clips/:id/editing-guide → DeepSeek → { special_effects, music, ... }          │
└────────────────────────────────────────────────────────────────────────────────────────────┘
```

## 2. 阶段详解（每步：输入 → 输出）

### 阶段 A：浏览器端抽音频（核心降本步骤）

| 项 | 内容 |
|----|------|
| 代码 | `frontend/src/services/audioExtractor.ts::extractAudio()` |
| 触发 | `FileUploader.processFile` 在用户拖入/选择视频后立即开始 |
| **输入** | `videoFile: File` （MP4/MKV/MOV/FLV/...，可达 GB 级）|
| 内部技术 | FFmpeg.wasm + WORKERFS 挂载，**不把视频整体读进内存** |
| FFmpeg 参数 | `-vn -acodec libmp3lame -ar 16000 -ac 1 -b:a 64k` |
| **输出** | `{ blob: MP3 Blob (~30MB), filename, startOffset, videoDuration }` |
| 副产物 | 通过解析 FFmpeg log 探测出 `start: XX.XX`（PTS 偏移，OBS 分段录制必备）和总时长 |
| 用户可见 | 全局进度 0–70% |

**为什么要在浏览器抽**：原始 2.5GB 视频 → 30MB MP3，上传体积缩减 99%；同时彻底避免大文件上传超时和服务器存储压力。后端的 `extract_audio()` 函数还在，但当前流程不会被走到。

---

### 阶段 B：上传 MP3 到服务器

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/upload/audio`（multipart/form-data） |
| 代码 | 前端 `services/api.ts::uploadAudio()`，后端 `app/api/upload.py::upload_audio()` |
| **前端发出** | 表单字段 `file: Blob`，文件名 `原视频名.mp3` |
| 后端行为 | 1MB 分块流式落盘 → `storage/uploads/{timestamp}_{safe_name}.mp3` |
| **后端返回** | `{ audio_path, original_filename, size_bytes }` |
| 用户可见 | 全局进度 70–95% |

> **数据落地点 #1**：服务器磁盘 `storage/uploads/*.mp3`。**音频是服务器唯一拥有的"媒体内容"**。

---

### 阶段 C：创建任务

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/tasks` |
| 代码 | 前端 `api.ts::createTask()`，后端 `app/api/tasks.py::create_task()` |
| **前端发出** | `{ video_path: <audio_path>, video_filename, video_start_offset, video_duration, scene_mode }` |
| 数据库动作 | `INSERT INTO tasks (status='pending', progress=0, source_path=audio_path, scene_mode, video_start_offset, video_duration, ...)` |
| **后端返回** | 完整 `TaskResponse`（含 `id: uuid`，`clips: []`） |
| 后续 | 前端 `navigate(/tasks/${id})` |

> **数据落地点 #2**：PostgreSQL `tasks` 表新增一行，`status='pending'` 等待 worker 拾起。
>
> 字段坑：列名叫 `video_path`/`source_path`，但实际存的是**音频路径**。这是历史遗留命名，与文件类型无关。

---

### 阶段 D：进入详情页，订阅进度

| 项 | 内容 |
|----|------|
| 首屏拉取 | `GET /api/tasks/:id` → 返回任务详情（此刻 `clips: []`） |
| 长连接 | `new EventSource('/api/tasks/:id/progress')` |
| SSE 数据 | 每秒一条 `data: {"progress":42,"message":"...","status":"transcribing"}` |
| 终态处理 | `status` 变为 `done` → 关闭 SSE → 再 `GET /api/tasks/:id` 拿完整 clips |
| 兜底 | SSE 断开 → 5 秒一次 `GET /api/tasks/:id` 轮询 |
| 后端实现 | `tasks.py::task_progress_sse` 每秒查 DB 序列化推送 |

> 关键认知：**SSE 推送的字段只有 `progress / message / status`，不带切片数据**。任何字段变化都必须等终态切到 done 后再发一次详情请求才能拿到 clips。

---

### 阶段 E：后端 Pipeline（异步、不占 API 线程）

这是最复杂的一段。它在 FastAPI 启动时 `start_task_runner()` 启动的后台协程里跑（`task_runner.py` + `workers/pipeline.py`）。

**Worker 主循环**（`task_runner.py::_task_runner_loop`）：
1. 每秒扫一次 `tasks` 表
2. 找到 `status='pending'` 的最早任务，CAS 改为 `downloading` 占据
3. 调 `run_video_pipeline(task_id)` 走完整条流水线
4. pipeline 抛异常时 status 变 `failed`

#### E1 - E2：ASR 转录（status: `transcribing`，progress 15→60%）

| 项 | 内容 |
|----|------|
| 代码 | `services/transcriber.py::GroqASRTranscriber.transcribe()` |
| 触发 | 因为 `source_path` 以 `.mp3` 结尾，pipeline 走"音频直传模式"，**跳过本地 extract_audio()** |
| **AI 输入** | 服务器本地 MP3 文件 (`storage/uploads/*.mp3`) |
| 切块策略 | 如果音频 > 25 分钟（`groq_asr_chunk_minutes`），用 ffmpeg `-c copy` 切块逐段调用 |
| 远程调用 | `POST https://api.groq.com/openai/v1/audio/transcriptions` <br/>multipart：file + model + `response_format=verbose_json` + `timestamp_granularities[]=segment` |
| **AI 返回** | `{ segments: [{ start, end, text }, ...] }` |
| 偏移修正 | 如果 `task.video_start_offset > 1.0`（OBS 分段录制），把每个 segment 的 start/end 加回去 |
| **落地** | `task.transcript_json = transcript`（JSON 字段，整段转录入库） |

> **数据落地点 #3**：PostgreSQL `tasks.transcript_json` 存的是带时间戳的纯文字转录。
>
> 转录 segment 结构：
> ```json
> [
>   { "start": 12.4, "end": 18.7, "text": "今天我们来聊聊..." },
>   { "start": 18.7, "end": 25.1, "text": "第一个问题是..." }
> ]
> ```

#### E3：DeepSeek 切片分析（status: `analyzing`，progress 60→80%）

| 项 | 内容 |
|----|------|
| 代码 | `services/analyzer.py::ClipAnalyzer.analyze()` |
| 场景 | 根据 `task.scene_mode` 选择 prompt 模板（livestream / interview / lecture） |
| 分批 | 按 Token 预算分批（中文约 1.5 字符 = 1 token，单批 ≤ 6000 tokens） |
| **AI 输入** | DeepSeek `POST /chat/completions`，messages 一条 user：场景 prompt 模板 + `[12.4s→18.7s] 今天我们来聊聊...` 这样的逐行转录 |
| 远程调用 | OpenAI SDK 兼容协议，`api.deepseek.com`，模型 `deepseek-chat`，temperature=0.3，timeout=120s |
| 重试 | 单批失败最多 3 次（指数退避 2s/4s/6s） |
| **AI 返回** | 一段 JSON 数组（可能被 markdown 围栏包裹，代码层有兜底 regex 提取）：<br/>`[{ clip_id, title, start_time, end_time, duration, type, summary, virality_score, suggested_caption }, ...]` |
| 后处理 | 1. 多批结果重新连续编号 `clip_id`<br/>2. `_align_to_transcript`：把 LLM 给的时间裁到最近的 transcript segment 边界（防止 LLM 编时间）<br/>3. `_deduplicate`：双向重叠率 > 阈值时丢掉低分项<br/>4. 过短 / 过长片段按场景配置过滤或截断<br/>5. 按 `virality_score` 降序排列 |

> 切片方案中间结果（内存中的 `clip_plans` 列表）：
> ```json
> [
>   {
>     "clip_id": 1,
>     "title": "炸场金句：千万别这样做",
>     "start_time": 312.5,
>     "end_time": 398.2,
>     "duration": 85.7,
>     "type": "高能时刻",
>     "summary": "主播激情吐槽行业乱象...",
>     "virality_score": 9,
>     "suggested_caption": "看完直接泪目..."
>   }
> ]
> ```

#### E4 - E5：写库收尾（status: `uploading` → `done`，progress 80→100%）

| 项 | 内容 |
|----|------|
| 因为输入是音频 | pipeline 走 "音频直传模式" 的分支，**跳过 FFmpeg 实际切视频**，直接把切片方案逐条 `INSERT INTO clips`，每条 `file_key = NULL`（**没有视频文件可下载**） |
| 最后一步 | `status='done'`，`progress=100`，`progress_message="分析完成！找到 N 个精彩片段时间点，请在剪映中按时间点裁切"` |
| 清理 | `temp_dir/{task_id}` 临时目录递归删除（`storage/uploads/*.mp3` 保留） |

> **数据落地点 #4**：PostgreSQL `clips` 表，每个 clip 一行：
> ```
> id | task_id | clip_index | title | summary | clip_type | start_time | end_time
>    | duration | virality_score | suggested_caption | file_key=NULL
>    | viral_titles=NULL | editing_guide=NULL
> ```

---

### 阶段 F：用户看到结果

| 项 | 内容 |
|----|------|
| 触发 | SSE 推 `status='done'` → 前端关闭 SSE 后再发一次 `GET /api/tasks/:id` |
| **后端返回** | 任务 + 所有 clips，`download_url` 字段会被后端拼成 `/api/clips/{id}/download`（**但本流程因为 `file_key=NULL` 不会拼**，所以是 `null`） |
| 前端渲染 | `TaskDetailPage` 把 clips 喂给 `ClipCard` 数组渲染 |
| 卡片底部按钮 | 因为 `download_url=null`，按钮文字是"一键复制时间，去剪映中裁切"，不是"下载切片" |

---

### 阶段 G：浏览器端本地切片（用户真正"拿到"视频片段的地方）

| 项 | 内容 |
|----|------|
| 触发 | 用户在详情页点 "✂️ 一键切片" → 弹 `LocalClipExportModal` 让用户**重新拖入原视频文件** |
| 代码 | `frontend/src/services/videoClipExporter.ts::exportVideoClipsLocally()` |
| **输入** | `{ videoFile: File, clips: [{start_time, end_time, ...}], videoStartOffset }` |
| 内部技术 | FFmpeg.wasm + WORKERFS 挂载原视频（同样零拷贝惰性读取，不进 WASM 堆） |
| 切片命令 | `ffmpeg -ss <start> -i <input> -t <duration> -c copy -avoid_negative_ts make_zero -y <output>`（`-c copy` = 不重编码，秒切） |
| 偏移补偿 | 数据库存的 `start_time` 是相对音频时间，`buildLocalClipSpecs` 会减回 `videoStartOffset` 得到相对原视频的真实时间 |
| 打包 | `fflate.zipSync(archiveEntries)` 在浏览器把所有切片 mp4 打成 ZIP |
| **输出** | `Blob (application/zip)` → `<a href={URL.createObjectURL(blob)} download>` 触发原生下载 |
| 失败处理 | 单 clip 失败不阻塞整体，最后汇总 `failed: LocalClipExportFailure[]` 给 UI 显示 |

> **数据落地点 #5**（终态）：用户磁盘上的 ZIP 文件。**整段流程到此结束**。

---

### 阶段 H：（可选）AI 二次生成

用户点单个 ClipCard 上的 "🔥 生成爆款标题" 或 "🎬 生成剪辑思路" 才会触发，与主流程异步独立。

| 接口 | 前端发送 | 后端处理 | AI 调用 | 后端返回 | 落库 |
|------|---------|---------|---------|---------|------|
| `POST /api/clips/:id/viral-titles` | 无请求体，timeout=30s | 从 DB 查 clip 拿 title/summary/clip_type/suggested_caption | DeepSeek（`viral_title_generator.py`），temperature=0.8 | `{ clip_id, viral_titles: ["...","...","...","...","..."] }` | `clips.viral_titles = [...]`（覆盖式） |
| `POST /api/clips/:id/editing-guide` | 无请求体，timeout=45s | 从 DB 查 clip 拿 title/summary/clip_type/duration/virality_score | DeepSeek（`editing_guide_generator.py`），temperature=0.5 | `{ clip_id, editing_guide: { special_effects, music, subtitles, rhythm, cover } }` | `clips.editing_guide = {...}`（覆盖式） |

两者都有 fallback：LLM 返回空或解析失败时返回写死的兜底文案。

## 3. 核心数据结构对照

### 3.1 流转中数据形态变化

```
[阶段 A 输入]                [阶段 A 输出 / B 输入]
videoFile: File             MP3 Blob (30MB)
2.5GB MP4                   16kHz mono 64kbps
└──FFmpeg.wasm──►

[阶段 B 输出]
audio_path: "D:\...\storage\uploads\1715680000000_xxx.mp3"

[阶段 E2 输出 / E3 输入]
transcript: [
  { start: 12.4, end: 18.7, text: "..." },
  ...
]

[阶段 E3 输出（LLM 直出）]
clip_plans: [
  { clip_id: 1, title, start_time, end_time, duration,
    type, summary, virality_score, suggested_caption },
  ...
]

[阶段 E5 输出（DB 持久化）]
clips 表: clip_index/title/summary/clip_type/start_time/end_time
        /duration/virality_score/suggested_caption/file_key=NULL

[阶段 F 给前端]
TaskClip: { id, clip_index, title, ..., download_url: null,
            viral_titles: null, editing_guide: null }

[阶段 G 输入]
videoFile (用户重新选的) + clips[]
[阶段 G 输出]
ZIP: { clip_001_xxx.mp4, clip_002_xxx.mp4, ... }
```

### 3.2 时间字段语义

| 时间字段 | 在哪 | 相对的零点 | 备注 |
|---------|------|----------|------|
| `Task.video_start_offset` | DB | 视频文件 PTS 起点 | OBS 分段录制的偏移；前端探测 |
| `Task.video_duration` | DB | — | 原视频总时长（秒），用于 UI 显示 |
| transcript segment.start/end | DB JSON | 音频起点（已加 `video_start_offset`） | Groq 给的相对时间被改写 |
| `Clip.start_time / end_time` | DB | 与转录相同（音频起点 + offset） | LLM 给的时间被对齐到 transcript 边界 |
| `ClipCard` 显示时间 | UI | 视为相对原视频起点 | 经 `getClipDisplayRange` 计算 |
| 阶段 G 浏览器切片 | FFmpeg `-ss` 参数 | 视为相对原视频起点 | `buildLocalClipSpecs` 减回 offset |

## 4. 全流程数据落地点一览

| # | 数据 | 存储位置 | 持久性 | 何时写 | 何时删 |
|---|------|---------|--------|--------|--------|
| 1 | 原视频 File | **用户磁盘**（浏览器只持引用） | 取决用户 | 用户选的瞬间 | 关闭页面即丢，需要切片时要重新选 |
| 2 | MP3 音频 | 服务器 `storage/uploads/*.mp3` | 持久 | 阶段 B | 删除任务时清理 |
| 3 | Task 行 | PostgreSQL `tasks` 表 | 持久 | 阶段 C | 用户主动删除 |
| 4 | 转录 JSON | `tasks.transcript_json` | 持久 | 阶段 E2 | 随 Task 删除 |
| 5 | Clip 行 | PostgreSQL `clips` 表 | 持久 | 阶段 E5 | 随 Task 级联删除 / 重试时清空 |
| 6 | 切片视频 mp4 | （本流程**不存在**） | — | — | — |
| 7 | viral_titles | `clips.viral_titles` (JSON) | 持久 | 阶段 H 用户点生成时 | 随 Clip 删除 |
| 8 | editing_guide | `clips.editing_guide` (JSON) | 持久 | 阶段 H 用户点生成时 | 随 Clip 删除 |
| 9 | 切片 ZIP | **用户磁盘**（下载文件夹） | 取决用户 | 阶段 G | 用户自行清理 |

> 关键洞察：**整条流水线里服务器从未拥有"视频画面"**。它只见过 MP3 + 文字 + 时间戳。视频从用户磁盘到 ZIP 输出回用户磁盘，构成一个完整的本地闭环。

## 5. 一句话概括端到端

> 用户**给浏览器一个视频文件**。
> 浏览器**给服务器一份 30MB 的 MP3 和场景选择**。
> 服务器**找 Groq 把 MP3 转成带时间戳的文字**，再**找 DeepSeek 从文字里挑出精彩片段的时间区间**。
> 用户在浏览器看到**一份切片清单**（标题 + 时间 + 推荐文案 + 爆款指数）。
> 想拿到实际视频时用户**再把原视频拖回浏览器**，浏览器用 FFmpeg.wasm **按清单本地切片**打 ZIP 给用户。
