# 提示词按分类应用设计

## 目标

完成提示词库卡片的“应用”功能：根据提示词分类跳转到对应创作入口，并把提示词内容一次性回填到目标输入框。应用只负责回填，不自动开始生成或启动工作流。

## 已确认规则

- `poster` 分类进入海报生成的“自定义生成”面板。
- `workflow` 分类进入内容工作流的第一步。
- `other` 分类没有对应创作入口，隐藏“应用”按钮，只保留复制内容能力。
- 公共提示词可以直接应用，不要求先复制到个人收藏。
- 应用成功后记录一次使用次数；计数失败不阻塞应用。

## 当前问题

- `PromptCard` 已触发 `use` 事件，但 `PromptLibraryPage` 没有监听，点击“应用”没有任何效果。
- 前端和后端已有 `usePrompt()` 与 `POST /prompts/{id}/use`，提示词库页面没有调用。
- 海报自定义生成已有 `prefill` 回填能力，内容工作流已有主题输入状态，但提示词库没有把数据传递给它们。
- 当前没有覆盖提示词库“应用”行为的前端测试。

## 交互设计

### 卡片状态

- 只有 `poster` 和 `workflow` 分类卡片显示“应用”按钮；`other` 和其他未知分类均不渲染该按钮，底部保留使用次数。
- 点击应用后，当前卡片按钮显示“应用中”，并暂时禁用列表内全部应用按钮，避免第二次点击覆盖共享回填数据。
- 路由跳转失败时恢复按钮状态，并在提示词库页面显示“应用失败，请重试”。

### 海报提示词

点击 `poster` 分类的“应用”后：

1. 保存一次性回填数据。
2. 跳转到 `/poster?tab=custom`。
3. `PosterPage` 消费回填数据，切换到 `custom` 面板。
4. `PanelCustom` 的提示词输入框使用该提示词完整内容覆盖原输入。
5. 不自动调用生图接口。

### 工作流提示词

点击 `workflow` 分类的“应用”后：

1. 保存一次性回填数据。
2. 跳转到 `/workflow`。
3. `WorkflowPage` 消费回填数据，将内容写入 `topicDirection`。
4. 页面保持在第一步，不调用 `handleStart()`，用户确认内容后自行开始。

## 一次性回填协议

新增独立工具模块 `frontend/src/utils/promptApply.js`，避免提示词库直接依赖目标页面内部状态。存储键固定为 `prompt_apply_payload`。

统一 payload：

```json
{
  "prompt_id": "uuid",
  "category": "poster",
  "content": "提示词正文",
  "created_at": 1784000000000
}
```

工具模块提供：

- 保存 payload 到 `sessionStorage`。
- 按目标分类消费 payload，读取后立即删除。
- 清除未完成跳转留下的 payload。
- payload 有效期为 5 分钟；超过有效期视为无效数据。
- JSON 解析失败、分类不匹配、正文为空或超过有效期时删除无效数据并返回 `null`。

使用 `sessionStorage` 的原因是数据只服务于当前标签页的一次跳转，不应跨浏览器会话或长期保留。

## 页面职责

### PromptCard

- 仅在 `prompt.category` 属于 `poster`、`workflow` 白名单时显示应用按钮。
- 接收 `applying` 属性控制当前按钮的加载和禁用状态。
- 接收 `applyDisabled` 属性，在其他卡片正在跳转时禁用按钮。
- 只发送 `use` 事件，不负责路由、存储或 API 调用。

### PromptLibraryPage

- 监听 `@use="handleApply"`。
- 使用 `applyingPromptId` 标识唯一进行中的应用流程；只要该值存在，全部卡片应用按钮都不可再次触发。
- 校验分类，仅允许 `poster` 和 `workflow`。
- 保存一次性回填数据，并按分类执行 `router.push()`。
- 路由成功后以不阻塞方式调用 `usePrompt(prompt.id)`；失败只记录警告，不撤销跳转。
- 路由失败时清除回填数据、恢复按钮并显示错误提示。

### PosterPage

- 页面初始化时消费 `poster` 分类 payload。
- 设置 `activeTab = 'custom'`，清空旧生成结果。
- 构造现有 `PanelCustom` 可识别的 `prefill`，只填充 `prompt`，不改变用户默认尺寸、风格或色调。
- 不调用生成事件。

### WorkflowPage

- 页面初始化时消费 `workflow` 分类 payload。
- 重置到工作流第一步并把正文写入 `topicDirection`。
- 不复用会自动调用 `handleStart()` 的 `startWithParams()`。
- 不自动创建线程或发起流式请求。

## 使用计数

- 只有路由成功后才触发 `usePrompt()`。
- 计数请求异步执行，不延迟目标页面展示。
- 计数失败只输出警告，不向用户显示“应用失败”，因为回填已经成功。
- 本期不修改后端计数接口和数据模型。

## 错误处理

- 分类为 `other` 或未知分类时不执行跳转；正常 UI 不应暴露该入口。
- 回填正文为空时阻止应用，并显示“提示词内容为空”。
- 写入 `sessionStorage` 失败时留在提示词库并显示错误。
- 路由失败时清除已写入的 payload，防止后续误回填。
- 目标页面读取到损坏或不匹配的数据时静默丢弃，不影响页面正常打开。

## 不在本期范围

- 不修改复制按钮及 HTTP 环境的剪贴板兼容问题。
- 不实现公共提示词 Fork 到个人收藏。
- 不自动开始海报生成或内容工作流。
- 不让用户在应用时再次选择目标页面。
- 不对提示词内容做转换、拼接或二次生成。

## 验证与验收

前端测试至少覆盖：

- `poster` 应用 payload 保存、目标路由和一次性消费。
- `workflow` 应用 payload 保存、目标路由和一次性消费。
- 海报页回填后处于 `custom` 面板且不会自动生成。
- 工作流页回填后停留第一步且不会自动启动。
- `other` 分类不显示应用按钮。
- 当前卡片显示应用中，并在跳转完成前禁用全部卡片的应用按钮。
- 路由失败会清理 payload 并显示错误。
- 使用计数失败不影响路由和回填。
- 空正文、损坏 payload、未知分类和超过 5 分钟的 payload 被安全拒绝。
