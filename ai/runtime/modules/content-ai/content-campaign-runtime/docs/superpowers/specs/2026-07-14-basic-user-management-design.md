# 基础版人员管理设计

## 目标

把现有仅能授予或取消管理员权限的页面，补齐为可长期使用的基础账号管理闭环：只有管理员能够创建账号，管理员可以搜索、筛选、分页查看用户，并可重置密码、启停账号和调整管理员角色。

本期不建设部门、角色组、细粒度权限、使用量统计、登录日志或操作审计。

## 已确认规则

- 账号只允许管理员创建，关闭用户自行注册。
- 新账号使用“用户名 + 初始密码”，首次登录不强制修改密码。
- 用户仍可在个人中心主动修改自己的密码。
- 账号移除采用停用，不硬删除；其作品、模板和其他历史数据继续保留。
- 角色仅保留管理员和普通用户两级。
- 管理员负责人员、公共模板和公共模型等全站管理能力。

## 当前问题

- 页面一次加载全部用户，没有搜索、筛选和分页。
- 每行直接铺设高强调的角色按钮，列表密度低，也容易误操作。
- 不能在系统内创建账号、停用账号或重置密码。
- 登录页和 `/auth/register` 仍允许公开注册，与管理员创建账号的目标冲突。
- `users.is_admin` 与环境变量、首用户兜底的有效管理员判断可能不一致，导致有权进入页面的账号在列表中仍显示为普通用户。

## 页面设计

### 页面头部

- 标题使用“人员管理”，去除不必要的英文副标题和说明性大字。
- 右侧提供“新增用户”主按钮。
- 刷新作为图标按钮放在筛选工具栏末端，并提供悬停提示。

### 筛选工具栏

- 用户名关键字搜索，提交后从第一页重新查询。
- 角色筛选：全部、管理员、普通用户。
- 状态筛选：全部、正常、已停用。
- 搜索与筛选均由服务端执行。

### 用户列表

表格字段为：

- 用户：用户名；若有昵称则作为辅助信息显示。
- 角色：管理员或普通用户。
- 状态：正常或已停用。
- 创建时间。
- 操作。

当前登录账号附加“当前用户”标识。操作列只保留“更多”菜单，菜单按当前状态显示：

- 重置密码。
- 设为管理员或取消管理员。
- 停用账号或启用账号。

列表使用服务端分页，默认每页 20 条，可切换 20、50、100 条。切换搜索或筛选条件后回到第一页。

### 新增用户弹窗

字段包括：

- 用户名，3 至 50 个字符。
- 初始密码，6 至 100 个字符。
- 确认密码，仅前端校验，不发送到后端。
- 设为管理员，默认关闭。

用户名重复时保留已填写内容，并在用户名字段下展示错误。创建成功后关闭弹窗、刷新当前列表并显示成功提示。

弹窗只允许通过右上角关闭按钮或明确的取消按钮关闭；点击遮罩和按 Esc 不关闭，避免丢失已填写内容。

### 角色、状态和密码操作

- 角色调整、账号停用和密码重置均使用确认弹窗。
- 重置密码由管理员输入新密码和确认密码；系统不展示旧密码，也不自动生成后长期展示明文密码。
- 停用说明明确告知：该用户将无法登录和继续调用受保护接口，但已有数据不会删除。
- 操作期间只禁用当前用户对应的操作，不阻塞整张表。
- 操作失败时保留当前列表、筛选条件和弹窗输入，并展示后端返回的明确原因。

## 权限与安全规则

- 所有 `/api/v1/admin/users` 管理接口必须在后端验证当前用户是管理员，不能只依赖前端隐藏菜单。
- 管理员不能取消自己的管理员权限，也不能停用自己的账号。
- 系统必须至少保留一名处于启用状态的管理员；取消角色或停用账号前，事务需先用 `SELECT ... FOR UPDATE` 锁定全部启用管理员记录，再基于锁定后的最新状态校验和更新，避免并发操作同时移除最后两名管理员。
- 停用用户不能登录；已持有令牌的停用用户在后续调用受保护接口时也必须被拒绝。
- `get_current_user` 和 `get_current_user_optional` 都要识别停用状态，避免可选认证接口继续把停用用户当作有效用户。
- 用户名保持唯一；密码复用当前哈希算法和长度规则。
- 删除 `/auth/register` 路由，使匿名注册请求返回 `404`；登录页同时移除注册模式与注册 API 调用。

## 管理员来源收口

`users.is_admin` 继续作为日常权限管理的唯一持久化来源。现有 `ADMIN_USERNAMES` 和首用户规则只用于本次升级时兼容旧数据，不应长期形成页面不可撤销的隐形管理员。

具体规则：

- 升级时如果数据库中已经存在启用状态的 `is_admin=true` 用户，不执行兼容迁移，权限判断只使用数据库字段。
- 升级时如果已有用户但没有启用管理员，优先把 `ADMIN_USERNAMES` 中匹配到的最早启用用户持久化为管理员；没有匹配项时，把最早创建且启用的用户持久化为管理员，以保持旧系统的首用户行为。
- 完成升级兼容后，运行时删除环境变量和首用户动态授权分支，所有权限判断只读取 `users.is_admin` 和 `users.is_active`。
- 用户列表展示持久化后的真实角色，保证页面角色与实际权限一致。

### 空数据库和应急引导

关闭公开注册后，空数据库没有账号可进入人员管理，因此提供 `scripts/create_admin.py` 作为部署和应急工具：

- 命令行接收用户名，并交互式输入和确认密码，密码不通过命令参数传递，避免进入 shell 历史。
- 用户不存在时创建一个启用管理员；用户已存在时将其启用并设为管理员，若需要修改密码则交互式确认。
- 工具直接复用项目密码哈希和数据库配置，不保存或打印密码明文。
- 全新部署必须先运行该工具创建首个管理员，再由该管理员在页面中创建其他账号。
- 正常页面流程不暴露此能力；它仅用于空库初始化或所有管理员意外不可用时的恢复。

## 数据模型

在 `users` 表增加：

- `is_active BOOLEAN NOT NULL DEFAULT TRUE`：账号是否可登录和访问受保护接口。

沿用现有字段：

- `is_admin`：是否为管理员。
- `username`、`password_hash`、`nickname`、`created_at`、`updated_at`。

不增加软删除时间、最后登录时间、登录失败次数和审计日志字段。

## API 设计

### 查询用户

`GET /api/v1/admin/users`

查询参数：

- `keyword`：用户名模糊搜索，可空。
- `role`：`all`、`admin`、`user`，默认 `all`。
- `status`：`all`、`active`、`inactive`，默认 `all`。
- `page`：从 1 开始，默认 1。
- `page_size`：20、50 或 100，默认 20。

响应改为分页对象：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0,
  "total_pages": 0
}
```

每个用户项包含 `id`、`username`、`nickname`、`is_admin`、`is_active` 和 `created_at`。创建、角色调整和状态调整接口均返回同一用户对象结构。

### 创建用户

`POST /api/v1/admin/users`

请求体包含 `username`、`password` 和 `is_admin`。默认创建启用账号。成功返回新用户对象；用户名冲突返回 `409`，输入不合法返回 `422`。

### 调整管理员角色

沿用 `PUT /api/v1/admin/users/{user_id}/admin`，请求体为 `is_admin`。

### 启停账号

`PUT /api/v1/admin/users/{user_id}/status`

请求体为 `is_active`。

### 重置密码

`PUT /api/v1/admin/users/{user_id}/password`

请求体为 `password`。后端直接保存新密码哈希，成功返回 `204 No Content`，不返回密码明文。

## 数据流

1. 管理员进入页面，前端带筛选和分页参数请求用户列表。
2. 管理员执行创建、角色调整、启停或重置密码操作。
3. 后端先验证操作者权限，再执行业务安全校验和数据库更新。
4. 成功后前端关闭对应弹窗并重新加载当前页；若当前页因筛选条件不再包含该用户，则自然从列表消失。
5. 停用用户再次登录或调用受保护接口时，认证层统一拒绝。

## 错误处理

- 非管理员访问管理接口返回 `403`。
- 用户不存在返回 `404`。
- 用户名重复返回 `409`。
- 取消自己的管理员权限或停用自己返回 `400`。
- 操作会导致没有启用管理员时返回 `409`，提示必须至少保留一名可用管理员。
- 已停用账号登录返回 `403` 和“账号已停用，请联系管理员”。
- 携带停用账号令牌访问可选认证接口时也返回 `403`，不能静默降级为匿名访问。
- 列表加载失败显示页面内错误状态并允许重试；已有列表数据不因单次操作失败而清空。

## 迁移与兼容

- 数据库初始化与启动迁移都补充 `is_active BOOLEAN NOT NULL DEFAULT TRUE`，保证旧用户默认可用。
- 用户列表接口从数组改为分页对象，前后端必须同版本发布。
- 删除登录页注册入口和前端 `register` 调用；删除公开注册路由，不能只改 UI。
- 现有管理员角色数据保留；首次上线时按“管理员来源收口”规则完成一次兼容迁移。
- 空数据库部署或管理员应急恢复使用 `scripts/create_admin.py`，不重新开放公开注册。

## 验证与验收

后端测试至少覆盖：

- 普通用户不能查询或修改人员信息。
- 管理员可分页、搜索并按角色和状态筛选。
- 管理员可创建普通用户和管理员，用户名重复被拒绝。
- 管理员可启停其他用户、重置密码和调整角色。
- 不能停用自己、不能取消自己的管理员权限。
- 不能停用或降级最后一名启用管理员。
- 停用用户不能登录，已有令牌也不能继续访问受保护接口。
- 升级兼容管理员会被持久化，页面角色与实际权限一致。
- 空数据库可通过命令行工具创建首个管理员，密码不会进入命令参数或日志。
- `/auth/register` 路由不存在，匿名注册请求返回 `404`。

前端测试至少覆盖：

- 登录页不再出现注册入口。
- 人员管理页只有管理员可见。
- 搜索、筛选和分页参数正确传递。
- 新增用户、重置密码、角色调整和启停操作均有确认流程。
- 点击弹窗遮罩或按 Esc 不会关闭新增用户弹窗。
- API 失败后保留筛选条件和弹窗输入。

最后运行完整后端测试、前端测试、生产构建和 `git diff --check`。
