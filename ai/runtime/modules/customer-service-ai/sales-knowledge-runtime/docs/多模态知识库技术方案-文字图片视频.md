# 多模态知识库技术方案 —— 让 AI 客服回复文字 / 图片 / 视频

> 目标：用户问一个问题（如「我大专可以学吗？」），AI 客服能**语义命中**并依次回复：
> 1. **文字**：「大专当然可以学，我们有很多案例……」
> 2. **图片**：大专学员的 offer 喜报、聊天记录截图
> 3. **视频**：「大专能不能学」这个问题的视频解答
>
> 本方案解决三件事：文本数据结构与 chunk 拆分、图片语义命中、视频语义命中，全部落在现有的 **PostgreSQL + pgvector** 技术栈上。

---

## 〇、核心设计思想：万物归一到「文本向量空间」

用户的提问永远是**一段文本**。所以最简单、最省钱、召回最稳的做法是：

> **把图片、视频也表达成「一段可检索的文本描述」，和文字知识一起向量化进同一个 1024 维空间，用一条 SQL 完成跨模态召回。**

```
                          用户提问（文本）
                               │
                     text-embedding-v3 → query 向量(1024)
                               │
        ┌──────────────────────┼──────────────────────┐
        ▼                      ▼                      ▼
  文字知识条目            图片素材                视频素材
 (recommended_response)  (描述+标签+OCR)        (标题+ASR转写+摘要)
        │                      │                      │
        └──────── 全部向量化进同一张检索表 (rag_assets) ────────┘
                               │
                   pgvector  ORDER BY embedding <=> query
                               │
                   召回 top-K，按 modality 分组
                               │
              文字回复  +  图片URL卡片  +  视频URL卡片
```

**为什么不直接对图片/视频做"图像向量"？**

| 方案 | 原理 | 优点 | 缺点 | 本方案取舍 |
|------|------|------|------|------------|
| **文本描述向量化**（主方案） | 图片/视频转成文字描述再 embedding | 复用现有 text-embedding-v3 与 pgvector，零新组件；query 是文本，天然对齐 | 依赖描述质量 | ✅ **默认采用** |
| 多模态向量（CLIP 类） | 图像与文本映射到同一向量空间 | 不写描述也能图文互搜 | 需引入多模态模型（如阿里 `multimodal-embedding-v1`），维度不同需独立索引 | 🔶 作为增强项（见 §3.5） |

> 一句话：**图片/视频的"语义"，本质上是它"能回答什么问题"。把这句话写出来去向量化，就命中了。**

## 一、文本数据：结构 / chunk / 存储 / 查询

### 1.1 数据中心导出的文本数据结构

文本知识有两类来源，建议数据中心统一导出为 **JSONL**（一行一条，便于断点续传与增量）：

#### 类型 A：结构化问答/话术（推荐主力，对应 `knowledge_articles`）

```json
{
  "id": "kb_000123",
  "type": "qa",
  "scene": "学历顾虑-大专能否学习",
  "scene_category": "course",
  "customer_says": "我大专可以学吗？基础不太好",
  "recommended_response": "大专完全可以学，我们大专背景的学员占比超过40%，很多都拿到了15k+的offer……",
  "key_points": ["先打消顾虑", "给数据/案例", "引导留资"],
  "confidence": 0.92,
  "source": "wechat_chat",
  "updated_at": "2026-05-20T10:00:00"
}
```

#### 类型 B：长文档（课程大纲、政策说明，对应 `knowledge_chunks`）

```json
{
  "id": "doc_0007",
  "type": "document",
  "title": "AI 课程招生政策说明",
  "section": "学历与基础要求",
  "content": "本课程面向零基础学员……大专、本科均可报名……",
  "source": "course_doc",
  "doc_id": "policy_2026",
  "updated_at": "2026-05-18T09:00:00"
}
```

**字段约定（关键）**：

| 字段 | 作用 | 是否参与向量化 |
|------|------|----------------|
| `customer_says` / `title+section` | **检索锚点**：用户怎么问，就放什么 | ✅ 权重最高 |
| `recommended_response` / `content` | 提供语义上下文，丰富向量 | ✅ |
| `scene` / `scene_category` | 过滤维度（`WHERE` 用） | ❌ 仅作元数据 |
| `confidence` | 质量排序 | ❌ 仅作排序 |

### 1.2 Chunk 拆分策略

**结论：结构化条目天然就是一个 chunk，不需要传统切分；只有长文档才需要拆。**

| 数据类型 | Chunk 策略 | 理由 |
|----------|-----------|------|
| 结构化问答（类型 A） | **一条 = 一个 chunk** | 本身就是独立场景，平均 100~200 字，最佳粒度 |
| 长文档（类型 B） | **按语义段落切，目标 300~500 字，重叠 50 字** | 太大稀释语义、太小丢上下文 |
| 超长话术（`recommended_response` > 500 字） | 按「要点」拆为多条 | 避免单向量承载过多信息 |

长文档切分示例：

```python
def split_document(text: str, target=400, overlap=50) -> list[str]:
    """按段落聚合到目标长度，段间保留 overlap 字重叠，避免切断语义。"""
    paras = [p.strip() for p in text.split("\n") if p.strip()]
    chunks, buf = [], ""
    for p in paras:
        if len(buf) + len(p) <= target:
            buf += p + "\n"
        else:
            if buf:
                chunks.append(buf.strip())
            # 保留上一块尾部 overlap 字作为衔接
            buf = (buf[-overlap:] if buf else "") + p + "\n"
    if buf.strip():
        chunks.append(buf.strip())
    return chunks
```

### 1.3 向量化文本拼接（决定召回质量的关键）

向量化的不是单一字段，而是**拼接后的"检索文本"**：

```python
# 类型 A：结构化问答
embed_text = f"场景:{a.scene}\n客户说:{a.customer_says}\n回复:{a.recommended_response}"

# 类型 B：长文档 chunk
embed_text = f"标题:{title}\n小节:{section}\n内容:{chunk_content}"
```

> 把 `customer_says` 放进去，是为了让"你们太贵了""大专能学吗"这类**用户原话**成为强匹配锚点。

### 1.4 PostgreSQL + pgvector 存储

项目已有 `knowledge_articles`、`knowledge_chunks` 两张表（均为 `embedding vector(1024)`）。**本方案在此之上新增一张统一检索表 `rag_assets`**，把三类模态的"检索单元"汇总到一起，源表仍作为内容真相（source of truth）：

```sql
CREATE TABLE rag_assets (
    id              bigserial PRIMARY KEY,
    modality        varchar(10)  NOT NULL,        -- 'text' / 'image' / 'video'
    retrieval_text  text         NOT NULL,        -- 实际被向量化的文本
    embedding       vector(1024),                 -- text-embedding-v3 输出

    -- 展示内容（命中后回给前端/客服系统的东西）
    answer_text     text,                          -- 文字回复正文
    media_url       text,                          -- 图片/视频 TOS URL（text 模态为空）
    cover_url       text,                          -- 视频封面图 URL
    duration_sec    integer,                       -- 视频时长

    -- 过滤与排序维度
    category        varchar(50),                   -- course/sales/objection...
    tags            jsonb DEFAULT '[]',            -- ["大专","本科","15k","北京"]
    confidence      float DEFAULT 0.0,

    -- 来源追溯
    source_table    varchar(40),                   -- knowledge_articles/materials/videos
    source_id       bigint,
    status          varchar(20) DEFAULT 'active',  -- active/disabled
    created_at      timestamp DEFAULT now(),
    updated_at      timestamp DEFAULT now()
);
```

**索引**（数据量到万级以上建 HNSW，与 `pgvector入门指南.md` 一致）：

```sql
-- 向量索引（HNSW，开箱即用、支持增量插入）
CREATE INDEX idx_rag_assets_embedding ON rag_assets
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 100);

-- 过滤辅助索引
CREATE INDEX idx_rag_assets_modality   ON rag_assets(modality, status);
CREATE INDEX idx_rag_assets_category   ON rag_assets(category);
CREATE INDEX idx_rag_assets_tags_gin   ON rag_assets USING gin (tags);
```

> 不想新增表也可以：直接给 `materials` 加 `embedding vector(1024)` 列，三张表各自检索后在应用层合并。**但统一表 `rag_assets` 的好处是"一条 SQL 跨模态召回"，强烈推荐。**

### 1.5 查询（语义检索）

```sql
SET hnsw.ef_search = 100;  -- 越大越精确，默认 40

SELECT id, modality, answer_text, media_url, cover_url, tags, confidence,
       1 - (embedding <=> CAST(:qvec AS vector)) AS similarity
FROM rag_assets
WHERE status = 'active'
  AND (:category IS NULL OR category = :category)
ORDER BY embedding <=> CAST(:qvec AS vector)
LIMIT :top_k;
```

Python 封装：

```python
def semantic_search(db, query: str, top_k=20, category=None,
                    modality=None, min_sim=0.35):
    qvec = get_embeddings([query])[0]            # 复用 app/services/embedding.py
    sql = """
        SELECT id, modality, answer_text, media_url, cover_url,
               tags, confidence,
               1 - (embedding <=> CAST(:qvec AS vector)) AS similarity
        FROM rag_assets
        WHERE status='active'
          AND (:category IS NULL OR category = :category)
          AND (:modality IS NULL OR modality = :modality)
        ORDER BY embedding <=> CAST(:qvec AS vector)
        LIMIT :top_k
    """
    rows = db.execute(text(sql), {
        "qvec": str(qvec), "category": category,
        "modality": modality, "top_k": top_k,
    }).fetchall()
    # 相似度阈值过滤：低于阈值视为"没命中"，避免答非所问
    return [r for r in rows if r.similarity >= min_sim]
```

## 二、图片回复：如何命中语义

### 2.1 核心：图片不向量化，"图片的文本描述"向量化

一张大专学员 offer 喜报截图，它能回答的问题是「大专能学吗 / 大专有成功案例吗 / 大专找到工作了吗」。我们把这句"它能回答什么"写成 `retrieval_text` 去向量化。

**`retrieval_text` 由四部分拼成**（来源越全，命中越准）：

```python
def build_image_retrieval_text(m: Material, ocr_text: str, caption: str) -> str:
    parts = [
        m.title or "",                      # 人工标题
        m.remark or "",                     # 人工备注（现有字段，RAG导出描述）
        " ".join(m.tags or []),             # 标签：大专/本科/北京/15k
        caption,                            # VLM 自动生成的图片描述
        ocr_text[:200],                     # OCR 提取的图中文字（offer金额、公司名等）
    ]
    return "。".join([p for p in parts if p.strip()])
```

| 描述来源 | 怎么来 | 价值 |
|----------|--------|------|
| `title` / `remark` | 人工填（现有素材管理已有） | 最准，优先 |
| `tags` | 人工打标（大专/城市/薪资） | 结构化命中锚点 |
| **VLM caption** | 多模态大模型看图说话（如 `qwen-vl`） | 自动覆盖没人工描述的图 |
| **OCR** | RapidOCR（项目打码链路已集成） | 抓 offer 金额、岗位、公司等图内文字 |

### 2.2 离线建库流程（图片 → rag_assets）

```
materials 表 (category=report/brand 的图片)
        │
        ├─ 1. 下载图片 (tos_service.download_object)
        ├─ 2. OCR 提取图内文字 (RapidOCR，复用打码链路)
        ├─ 3. VLM 生成 caption (qwen-vl："这张图展示了什么")
        ├─ 4. 拼 retrieval_text = 标题+备注+标签+caption+OCR
        ├─ 5. text-embedding-v3 → 1024维向量
        └─ 6. UPSERT 进 rag_assets
              modality   = 'image'
              media_url  = 打码图的 TOS 公开URL (优先 masked 版本)
              answer_text= "这是{标签}学员的成功案例 👇"
              tags       = m.tags
```

```python
def index_image_material(db, m: Material):
    img_bytes = tos_service.download_object(m.oss_key)
    ocr_text  = run_ocr(img_bytes)                    # 复用打码链路的 RapidOCR
    caption   = qwen_vl_caption(img_bytes)            # 多模态描述
    rtext     = build_image_retrieval_text(m, ocr_text, caption)
    emb       = get_embeddings([rtext])[0]

    # 隐私优先：能用打码图就别用原图
    public_url = resolve_masked_url(db, m)

    upsert_rag_asset(db,
        modality="image",
        retrieval_text=rtext,
        embedding=emb,
        answer_text=f"这是{ '、'.join(m.tags or []) }相关的学员案例 👇",
        media_url=public_url,
        category=m.category,
        tags=m.tags,
        source_table="materials",
        source_id=m.id,
    )
```

> **隐私红线**：图片素材入库时，`media_url` 一律取**打码版本**（`source_material_id` 关联的 `category=masked` 图，取 `id` 最大者）。与现有素材架构 §4.2 完全一致。

### 2.3 检索与回复

图片命中走同一个 `semantic_search`，只是 `modality='image'`：

```python
images = semantic_search(db, "我大专可以学吗", modality="image",
                         top_k=3, min_sim=0.40)  # 图片阈值略高，宁缺毋滥
```

返回给客服系统的结构：

```json
{
  "type": "image",
  "text": "这是大专相关的学员案例 👇",
  "images": [
    "https://bucket.tos.../masked/uuid1.png",
    "https://bucket.tos.../masked/uuid2.png"
  ]
}
```

### 2.4 阈值与兜底

- **相似度阈值**（建议 image ≥ 0.40）：低于阈值不回图，避免「问大专却甩出 Java 喜报」。
- **数量上限**：一次最多回 2~3 张，多了像广告。
- **兜底**：语义没命中时，回退到现有的**标签关键词匹配**（`export_materials_rag.py` 的 `TAG_EXPAND_MAP`），双保险。

### 2.5 （增强项）真·多模态向量

若希望"不写描述也能图文互搜"，可引入阿里 `multimodal-embedding-v1`：图片和文本映射到**同一向量空间**。做法：

```sql
-- 多模态向量维度不同，单独建列/表，独立索引
ALTER TABLE rag_assets ADD COLUMN mm_embedding vector(1024);  -- 维度以模型为准
CREATE INDEX idx_rag_mm ON rag_assets USING hnsw (mm_embedding vector_cosine_ops);
```

检索时对 query 文本用多模态模型编码，再和图片的多模态向量比。**作为文本描述方案的补充**（描述命中不到时兜底），不建议一上来就只用它——成本高、对中文长尾问法不如文本描述稳。

## 三、视频回复：如何命中语义

### 3.1 核心：视频 → "可检索文本" → 向量

视频和图片同理，但信息更丰富。一个「大专能不能学」的讲解视频，它的语义 = **标题 + 口播内容（ASR 转写）+ 内容摘要**。

```python
def build_video_retrieval_text(v) -> str:
    parts = [
        v.title,                       # "大专学历能不能学AI？三分钟讲清"
        v.summary,                     # LLM 对转写的摘要
        " ".join(v.tags or []),        # 大专/学历/答疑
        v.transcript[:500],            # ASR 转写前 500 字（口播即语义）
    ]
    return "。".join([p for p in parts if p and p.strip()])
```

### 3.2 视频数据表

视频体积大，建议在 `materials`（复用 TOS 存储）基础上单独建 `video_assets` 存视频专有字段：

```sql
CREATE TABLE video_assets (
    id            bigserial PRIMARY KEY,
    title         varchar(300) NOT NULL,
    oss_key       varchar(500) NOT NULL,    -- 视频文件 TOS key
    cover_key     varchar(500),             -- 封面图 key（首帧/自定义）
    duration_sec  integer,
    transcript    text,                     -- ASR 全文转写
    summary       text,                     -- LLM 摘要
    category      varchar(50),
    tags          jsonb DEFAULT '[]',
    created_at    timestamp DEFAULT now()
);
```

> 视频本身不存进 `rag_assets`，只把它的"检索文本"作为一条 `modality='video'` 的记录写进 `rag_assets`，`media_url` 指向视频 URL、`cover_url` 指向封面。

### 3.3 离线建库流程（视频 → rag_assets）

```
上传视频 (TOS)
   │
   ├─ 1. ASR 语音转文字（whisper / 阿里听悟 / 火山ASR）→ transcript
   ├─ 2. LLM 摘要 transcript → summary（"本视频讲解大专能否学AI及就业情况"）
   ├─ 3. 抽首帧/关键帧做封面 (ffmpeg) → cover_key
   ├─ 4. 写 video_assets 表
   ├─ 5. retrieval_text = 标题+摘要+标签+转写片段
   ├─ 6. text-embedding-v3 → 向量
   └─ 7. UPSERT 进 rag_assets (modality='video')
```

```python
def index_video(db, video_file, title, tags, category):
    oss_key   = tos_service.upload_object(video_file)         # 存视频
    cover_key = extract_cover(video_file)                     # ffmpeg 抽帧
    transcript= asr_transcribe(video_file)                    # 语音转写
    summary   = llm_summarize(transcript)                     # 摘要

    v = save_video_asset(db, title=title, oss_key=oss_key,
                         cover_key=cover_key, transcript=transcript,
                         summary=summary, tags=tags, category=category)

    rtext = build_video_retrieval_text(v)
    emb   = get_embeddings([rtext])[0]
    upsert_rag_asset(db,
        modality="video",
        retrieval_text=rtext,
        embedding=emb,
        answer_text=f"这个视频专门讲了「{title}」，可以看下 👇",
        media_url=public_url(oss_key),
        cover_url=public_url(cover_key),
        duration_sec=v.duration_sec,
        category=category, tags=tags,
        source_table="video_assets", source_id=v.id,
    )
```

### 3.4 检索与回复

```python
videos = semantic_search(db, "大专能不能学", modality="video",
                        top_k=1, min_sim=0.42)  # 视频更重，通常只回1个最相关
```

返回结构：

```json
{

  "video": {

### 3.5 （增强项）长视频片段定位

长视频（>10 分钟）建议按 ASR 时间戳分段，每段单独成一条 `rag_assets`，`media_url` 带上 `#t=开始秒数`，命中后直接跳到对应片段：

```
transcript 带时间戳 → 按话题/每60秒切段 → 每段一条向量
media_url = ".../uuid.mp4#t=125"   ← 命中"大专"片段直接定位到 2分05秒
```

## 四、统一检索与图文视频混排回答

### 4.1 一次提问，三类内容一起召回

```python
def answer_with_media(db, user_query: str, category: str = None):
    # 1) 三模态各自召回（也可一条SQL召回全部再分组）
    texts  = semantic_search(db, user_query, modality="text",  top_k=3, min_sim=0.35)
    images = semantic_search(db, user_query, modality="image", top_k=3, min_sim=0.40)
    videos = semantic_search(db, user_query, modality="video", top_k=1, min_sim=0.42)

    replies = []

    # 2) 文字回复：用最相关的话术 + LLM 润色（或直接用 recommended_response）
    if texts:
        reply_text = llm_polish(user_query, [t.answer_text for t in texts])
        replies.append({"type": "text", "text": reply_text})

    # 3) 图片回复：命中才发，最多2~3张
    if images:
        replies.append({
            "type": "image",
            "text": images[0].answer_text,
            "images": [i.media_url for i in images[:3]],
        })

    # 4) 视频回复：命中最相关的1个
    if videos:
        v = videos[0]
        replies.append({
            "type": "video",
            "text": v.answer_text,
            "video": {"url": v.media_url, "cover": v.cover_url,
                      "duration": v.duration_sec},
        })

    return replies   # 前端/客服系统按顺序依次发送三条消息
```

### 4.2 编排原则

| 原则 | 说明 |
|------|------|
| **文字必发，图视频按需** | 文字是主回复；图片/视频只有"够相关"才追加，避免刷屏 |
| **阈值递增** | text 0.35 < image 0.40 < video 0.42，越重的媒体越保守 |
| **数量克制** | 文字1条、图片≤3张、视频≤1个，像真人客服而非广告机 |
| **顺序固定** | 文字 → 图片 → 视频，符合"先解答、再举证、后深入"的销售节奏 |
| **rerank（可选）** | 召回 top-20 后，用 rerank 模型（如 `gte-rerank`）精排，进一步提准 |

### 4.3 与现有「火山引擎客服」的衔接

现状是导出 CSV 给火山知识库。本方案有两种落地姿势：

| 落地方式 | 做法 | 适用 |
|----------|------|------|
| **A. 自建检索服务** | 后端直接跑 `rag_assets` + pgvector 检索，客服系统调你的 `/api/answer` 接口 | 想完全掌控召回逻辑、图文视频混排 |
| **B. 继续喂火山** | 把 `rag_assets`（含 image/video 的 URL）导出成增强版 CSV/JSONL 灌入火山 | 想沿用现有火山客服 UI |

> 推荐 **A**：图片/视频的多条消息混排、阈值控制、片段定位等精细逻辑，自建检索服务才能完全实现。火山方案的 answer 字段塞 URL 只能做到"图文一段话"，难以做到"分条发 + 视频卡片"。

## 六、实施步骤

1. **建表与索引**：新建 `rag_assets`、`video_assets`，建 HNSW + GIN 索引（一条 SQL 脚本）。
2. **文本入库**：写 `scripts/index_text_assets.py`，把 `knowledge_articles` / 数据中心 JSONL 灌进 `rag_assets`（复用现有 `embedding.py`）。
3. **图片入库**：写 `scripts/index_image_assets.py`，遍历 `materials`（report/brand），OCR + VLM caption + embedding → `rag_assets`（OCR 复用打码链路的 RapidOCR）。
4. **视频入库**：接入 ASR（whisper/火山ASR）+ ffmpeg 抽帧，写 `scripts/index_video_assets.py`。
5. **检索服务**：新增 `app/services/multimodal_rag.py`，实现 `semantic_search()` + `answer_with_media()`。
6. **API**：新增 `POST /api/answer`，入参 `{query, category}`，出参分条消息数组。
7. **增量更新**：素材/视频新增或打码后，触发重新生成对应 `rag_assets` 记录（按 `source_table + source_id` UPSERT）。

## 八、关键设计要点速查

| 问题 | 答案 |
|------|------|
| 图片怎么向量化？ | **不向量化图片本身**，向量化它的文本描述（标题+备注+标签+VLM caption+OCR） |
| 视频怎么向量化？ | 向量化「标题 + ASR 转写 + 摘要」 |
| 三类内容怎么统一查？ | 全进 `rag_assets`，按 `modality` 字段区分，一条 SQL 跨模态召回 |
| 怎么避免答非所问？ | 相似度阈值（text 0.35 / image 0.40 / video 0.42）+ 数量上限 + 标签兜底 |
| 隐私怎么保证？ | 图片 `media_url` 一律用打码版本（沿用现有 `source_material_id` 关联） |
| 维度多少？ | 1024（DashScope `text-embedding-v3`），与现有 `knowledge_articles` 一致 |
| 用什么索引？ | HNSW（`vector_cosine_ops`），万级以上数据开箱即用、支持增量 |
| 要不要上 CLIP 多模态？ | 默认不用；文本描述方案更省更稳，CLIP 作为兜底增强项 |
