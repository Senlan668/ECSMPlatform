# 数据清洗链路说明

## 一、全局数据流总览

```
微信 SQLite 数据库 (MSG0-5.db + MicroMsg.db)
  │
  ▼ [阶段0] ETL 导入（仅结构预处理：时间过滤 + 群消息解析 + 去重）
  │                    ⚠️ 不做内容过滤，所有文本消息原样入库
  │
raw_chats 表（原始消息，不可修改，已在 PostgreSQL/SQLite 中）
  │
  ▼ [阶段1+2] AdminService "一键清洗"（内部依次执行：）
  │            ① DataFilter.should_include() → 内容过滤（丢弃垃圾/系统/媒体/过短）
  │            ② DataFilter.desensitize() → 脱敏（手机号/身份证/银行卡等）
  │            ③ 5分钟时间窗切分 → 合并同角色连续消息 → 去重复行
  │            ④ 自动预标注（分类/质量分/Q&A提取）
  │
staging_conversations 表（暂存区，status=pending）
  │
  ▼ [阶段3] 人工审核（approve / reject / 编辑）
  │
staging_conversations 表（status=approved）
  │
  ├──▶ [下游A] 导出训练数据（ShareGPT / Alpaca / OpenAI / JSONL / RAG CSV）
  ├──▶ [下游B] 向量知识库（embedding → knowledge_chunks → 语义搜索 / RAG 问答）
  ├──▶ [下游C] 知识提炼（LLM → knowledge_articles → 结构化问答对）
  └──▶ [下游D] 发布到 labeled_conversations（可选归档）
```

---

## 二、各阶段详细处理逻辑

### 阶段 0：ETL 导入

| 项目 | 说明 |
|------|------|
| **输入** | 微信本地 SQLite 文件：`MicroMsg.db`（通讯录）+ `MSG0.db ~ MSG5.db`（分库消息） |
| **输出** | `contacts` 表、`sessions` 表、`raw_chats` 表 |
| **代码** | `backend/app/services/etl.py` → `WeChatETL` |
| **触发** | `POST /api/admin/etl` 或 CLI `python scripts/run_etl.py` |

**处理规则：**

1. **加载通讯录** → 构建 wxid → 显示名映射（优先级：Remark > NickName > Alias > wxid）
2. **遍历 MSG0-5.db** → 读取 `MSG` 表全部消息
3. **时间过滤** → 丢弃 `timestamp < 1759248000`（2025-10-01 之前）的消息
4. **群消息解析** → 对群聊消息拆分 `wxid:\n内容` 格式，提取真实发送者
5. **去重** → 按 `(session_id, timestamp, content)` 跳过已存在记录
6. **语音补链** → 匹配 `msg_id` 对应的 `.mp3` 文件路径
7. **写入** → 保存到 `raw_chats` 表，状态为 `pending`

> ETL 阶段只做结构性预处理，不涉及内容语义判断。

---

### 阶段 1：内容过滤与脱敏（DataFilter）

> ⚠️ **注意**：DataFilter 不是独立运行的阶段，它**内嵌在 AdminService 的一键清洗流程中**被调用。
> 数据已经存入 raw_chats 表之后，执行"一键清洗"时才触发过滤。

| 项目 | 说明 |
|------|------|
| **输入** | 单条 `raw_chats` 记录的 `content` + `msg_type` |
| **输出** | 分类结果 `ContentCategory` + 脱敏后文本 |
| **代码** | `backend/app/services/filter.py` → `DataFilter` |
| **调用时机** | `AdminService.preprocess_session()` 遍历消息时逐条调用 |

**内容分类逻辑（`classify_content`）：**

按优先级从高到低依次判断：

| 优先级 | 条件 | 分类 | 后续处理 |
|--------|------|------|----------|
| 1 | `msg_type == 10000` | SYSTEM | 丢弃 |
| 2 | `msg_type != 1`（非文本） | MEDIA | 丢弃 |
| 3 | 内容长度 < 5 字符 | SHORT | 丢弃 |
| 4 | 以 `<?xml` 或 `<msg>` 开头 | SYSTEM | 丢弃 |
| 5 | 匹配敏感信息正则（手机号/身份证/银行卡等） | SENSITIVE | 丢弃 |
| 6 | 匹配闲聊模式（纯语气词/纯表情/纯标点） | CHITCHAT | 丢弃 |
| 7 | 包含销售高价值关键词 | VALUABLE | **保留** |
| 8 | 包含垃圾关键词（兼职/日赚/博彩等） | SPAM | 丢弃 |
| 9 | 以上均不匹配 | VALUABLE | **保留**（销售场景倾向保留） |

**脱敏逻辑（`desensitize`）：**

| 类型 | 示例输入 | 脱敏输出 |
|------|----------|----------|
| 手机号 | `13812345678` | `138****5678` |
| 身份证 | `320123199001011234` | `320***********1234` |
| 银行卡 | `6222021234561234` | `6222****1234` |
| 邮箱 | `test@example.com` | `t***@example.com` |
| API Key | `provider-key-example` | `provider-***` |
| 密码 | `密码：mypass123` | `密码：******` |

### 阶段 2：对话块构建与预标注（AdminService）

| 项目 | 说明 |
|------|------|
| **输入** | 某个 `session_id` 下所有通过阶段 1 过滤的 `raw_chats` 记录 |
| **输出** | `staging_conversations` 表记录（status=pending） |
| **代码** | `backend/app/services/admin.py` → `AdminService.preprocess_session()` |
| **触发** | `POST /api/admin/preprocess`（一键清洗） |

**处理流程：**

```
1. 排除群聊（session_id 以 @chatroom 结尾）
2. 查询 raw_chats（时间 >= 2025-10-01，按时间升序）
3. 逐条消息处理：
   ├── DataFilter.should_include() → 不通过则标记 rejected 并跳过
   ├── DataFilter.desensitize() → 内容脱敏
   └── 判断是否开新块（与上一条间隔 > 300秒 = 5分钟）
4. 块内组装：
   ├── original_text: 保留原始内容
   ├── cleaned_text: 脱敏后内容
   └── conversation_json: 结构化 [{role, content, sender_name, msg_id, timestamp}]
5. 块级后处理（_finalize_block）：
   ├── 合并同一角色连续消息（role 相同 → content 用 \n 拼接）
   ├── normalize_conversation_json() → 去除 turn 内部连续重复行
   └── rebuild_cleaned_text() → 从 JSON 重建文本
6. 自动预标注：
   ├── auto_category: 调用 DataFilter.classify_content 分类
   ├── auto_quality_score: 规则评分或 LLM 评分（0-10）
   ├── auto_question / auto_answer: 简单规则提取 Q&A
   └── auto_flags: 标记 has_sensitive / is_junk / too_short
7. 去重：按 (session_id, start_time, end_time) 跳过已存在块
8. 写入 staging_conversations（status=pending）
```

**规则评分逻辑（无 LLM 时）：**
- 基础分 5.0
- 内容长度 > 100 字符 → +1.0
- 包含问号（`?` 或 `？`）→ +1.0
- 上限 10.0

**Q&A 提取逻辑（规则版）：**
- question：第一条包含 `?` 或 `？` 的行
- answer：前 3 条长度 > 10 字符的非问句行

---

### 阶段 3：人工审核

| 项目 | 说明 |
|------|------|
| **输入** | `staging_conversations`（status=pending） |
| **输出** | `staging_conversations`（status=approved / rejected） |
| **操作** | 通过 Admin 后台界面或 API 进行 |

人工可执行的操作：
- **审核通过**（approve）→ 进入下游处理
- **拒绝**（reject）→ 不参与后续流程
- **编辑** → 修改 `cleaned_text`、`conversation_json`、`human_category`、`human_question`、`human_answer`
- **批量操作** → 一次性批量 approve / reject / delete

## 三、清洗后数据的去向与用途

### 下游 A：导出训练数据（微调用）

| 项目 | 说明 |
|------|------|
| **数据源** | `staging_conversations`（status=approved）+ `custom_conversations`（手工高质量数据） |
| **输出格式** | ShareGPT / Alpaca / OpenAI JSONL / RAG CSV |
| **代码** | `backend/app/routers/export.py` |
| **用途** | LLM 微调训练（LLaMA-Factory / OpenAI Fine-tuning / Stanford Alpaca） |

**导出前的 7 阶段训练数据清洗（`_clean_conversation_for_training`）：**

| 阶段 | 处理内容 | 目的 |
|------|----------|------|
| 1 | 若第一条是 gpt，前面补一条 `human: "你好"` | 确保 human 先开口 |
| 2 | 移除末尾连续的 human 消息 | 确保以 gpt 回复结束 |
| 3 | 价格脱敏（`_desensitize_price`） | 避免训练出具体价格 |
| 3.5 | 清理残留 `[资料链接]` 占位符 | 去除无效占位 |
| 4 | 风格验证修复（标点/行长度/价格违规） | 保持输出风格统一 |
| 5 | 过滤过短对话（< min_turns，默认 2 条） | 去除无训练价值数据 |
| 6 | 截断过长对话（> max_turns，默认 40 条） | 控制单条训练长度 |
| 7 | 最终清理：残留占位符、`您→你`、多余空行 | 统一语言风格 |

**导出格式对照：**

| 格式 | 结构 | 适用工具 |
|------|------|----------|
| ShareGPT | `{"conversations": [{"from": "system/human/gpt", "value": "..."}]}` | LLaMA-Factory, FastChat |
| Alpaca | `{"instruction": "...", "input": "...", "output": "..."}` | Alpaca-LoRA |
| OpenAI | `{"messages": [{"role": "system/user/assistant", "content": "..."}]}` | OpenAI Fine-tuning API |
| RAG CSV | `question,answer,category,tags` | 火山引擎知识库 |

---

### 下游 B：向量知识库（RAG 问答用）

| 项目 | 说明 |
|------|------|
| **数据源** | `staging_conversations`（status=approved） |
| **输出** | `knowledge_chunks` 表（含 embedding 向量） |
| **代码** | `backend/app/services/knowledge.py` → `build_from_labeled_data()` |
| **用途** | 语义搜索 + RAG 问答 |

**处理流程：**

```
1. 查询所有 approved 的 staging_conversations
2. 对每条记录：
   ├── 取 cleaned_text（脱敏后文本）
   ├── 长度 < 20 字符则跳过
   ├── 调用 embedding 模型生成向量（DashScope text-embedding-v3 / MiniLM）
   ├── 提取关键词
   └── 生成摘要（优先用 auto_question）
3. 写入 knowledge_chunks（chunk_type='labeled'）
```

**使用方式：**
- 用户输入问题 → embedding → 在 `knowledge_chunks` 中做向量相似度搜索
- 返回 Top-K 相关对话块作为上下文 → 喂给 LLM 生成回答

---

### 下游 C：知识提炼（结构化知识条目）

| 项目 | 说明 |
|------|------|
| **数据源** | `knowledge_chunks` 或 `staging_conversations`（approved） |
| **输出** | `knowledge_articles` 表 |
| **代码** | `backend/app/services/extractor.py` |
| **用途** | RAG 优先检索源、结构化话术库 |

**输出字段：**
- `scene`：适用场景
- `customer_says`：客户话术
- `recommended_response`：推荐回复
- `embedding`：向量（用于语义匹配）

---

### 下游 D：发布到生产标注表（可选）

| 项目 | 说明 |
|------|------|
| **数据源** | `staging_conversations`（status=approved） |
| **输出** | `labeled_conversations` 表 |
| **代码** | `backend/app/services/admin.py` → `publish_to_production()` |
| **用途** | 数据归档、兼容旧版标注流 |

## 五、关键设计决策

| 决策 | 原因 |
|------|------|
| 时间门槛 2025-10-01 | 只关注近期销售数据，避免处理过多历史噪声 |
| 默认排除群聊 | 群聊消息噪声大、角色不明确，不适合训练 |
| 5 分钟时间窗切分 | 平衡对话完整性与单条训练数据长度 |
| 脱敏是多层冗余的 | 阶段 1 脱敏 + 阶段 2 写入 + 导出时再次脱敏，确保敏感信息不泄露 |
| 价格脱敏是硬红线 | 避免微调后的模型泄露真实定价策略 |
| 销售场景默认保留 | `classify_content` 的 fallback 是 VALUABLE，宁可多保留、人工再审 |
| staging 是审核中枢 | 所有下游（训练/知识库/导出）均以 staging approved 为数据源 |
