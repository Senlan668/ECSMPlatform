# 多模态知识库 · 下篇 —— 离线处理与增强

> 本篇是「离线路线」：在[上篇](./多模态知识库-上篇-人工维护与即时命中.md)已建好的**同一张 `rag_assets` 表**上，用机器（OCR / VLM / ASR）补充图片视频的内容信息，**重新生成向量**，让召回更全、更准。
>
> 关键：下篇**不新增检索表、不改 media_url、不动人工标签**。它只是把更多"机器读出来的文本"追加进 `retrieval_text`，再 embedding 一次。`enrich_status` 从 `manual` 变成 `enriched`。

---

## 〇、为什么需要离线处理（上篇的三个缺口）

| 缺口 | 上篇（人工）的问题 | 下篇（离线）怎么补 |
|------|--------------------|--------------------|
| **图片里的文字读不到** | offer 金额、公司名、岗位都在图里，人工不一定抄进描述 | **OCR** 自动提取图内文字 |
| **图片语义靠人工想象** | 人工只写一句描述，覆盖不了所有问法 | **VLM 看图说话**，自动生成丰富描述 |
| **视频口播内容缺失** | 人工只写一句简介，视频里讲的内容全丢了 | **ASR 转写** + LLM 摘要，把口播变成可检索文本 |

> 一句话：**上篇是"人工告诉机器这是什么"，下篇是"机器自己看懂这是什么"。两者叠加，召回最强。**

---

## 一、离线增强的统一机制（图片视频通用）

所有离线增强都遵循同一个**幂等三步**，作用在上篇插入的那一行上：

```
              rag_assets 中的一行 (enrich_status=manual)
                            │
   ┌────────────────────────┼─────────────────────────┐
   │ Step 1  机器抽取        │ 图片: OCR + VLM          │
   │                        │ 视频: ASR + 摘要 + 抽帧   │
   ├────────────────────────┼─────────────────────────┤
   │ Step 2  重写检索文本     │ retrieval_text =         │
   │                        │   人工字段 + 机器抽取文本  │
   ├────────────────────────┼─────────────────────────┤
   │ Step 3  重新向量化       │ embedding = embed(rtext) │
   │         回写            │ enrich_status = enriched │
   └────────────────────────┴─────────────────────────┘
                            │
                  ✅ 同一行，召回能力增强，URL/标签不变
```

**检索文本的"分层拼接"约定**（人工在前、机器在后，人工权重天然更高）：

```python
def merge_retrieval_text(manual_part: str, machine_parts: list[str]) -> str:
    """人工描述放最前（最可信），机器抽取的内容补在后面。"""
    parts = [manual_part] + [p for p in machine_parts if p and p.strip()]
    return "。".join(parts)
```

幂等回写（按 `source_table+source_id` UPSERT，可反复跑）：

```python
def re_embed_asset(db, source_table, source_id, retrieval_text, extra=None):
    emb = get_embeddings([retrieval_text])[0]
    db.execute(text("""
        UPDATE rag_assets
        SET retrieval_text=:rt, embedding=CAST(:emb AS vector),
            enrich_status='enriched', updated_at=now()
        WHERE source_table=:st AND source_id=:sid
    """), {"rt": retrieval_text, "emb": str(emb),
           "st": source_table, "sid": source_id})
    db.commit()
```

## 二、图片离线处理：OCR + VLM

### 2.1 机器抽取什么

| 抽取手段 | 工具 | 抽到什么 | 价值 |
|----------|------|----------|------|
| **OCR** | RapidOCR（项目打码链路已集成） | 图中所有文字：offer 金额、公司、岗位、日期 | 让"字节15k""15000元"这类图内信息可被搜到 |
| **VLM 看图说话** | `qwen-vl` 等多模态大模型 | 一段自然语言图片描述 | 覆盖人工没写到的语义、场景 |

### 2.2 离线处理流程

```python
def enrich_image(db, m: Material):
    img = tos_service.download_object(m.oss_key)        # 用原图做OCR/VLM（信息全）
    ocr_text = run_ocr(img)                             # RapidOCR
    caption  = qwen_vl_caption(img)                     # "图中展示一份字节跳动的offer，薪资15k..."

    # 人工部分（上篇已有） + 机器部分
    manual = "。".join(filter(None, [m.title, m.remark, " ".join(m.tags or [])]))
    rtext  = merge_retrieval_text(manual, [caption, ocr_text[:200]])

    re_embed_asset(db, "materials", m.id, rtext)
    # 注意：media_url 不变，仍是上篇存的打码版URL（隐私不回退）
```

> **隐私红线不变**：OCR/VLM 用**原图**（信息最全），但 `rag_assets.media_url` 始终保持上篇的**打码版** URL。机器只读原图、不暴露原图。

### 2.3 数据怎么存（增强前后对比）

**增强前（上篇 manual）：**

```
id            | 2001
retrieval_text| 深圳大专学员offer-字节15k。大专学历转行AI拿到offer的真实案例。大专 深圳 15k 字节 offer
enrich_status | manual
embedding     | [0.022, -0.018, ...]
media_url     | https://.../masked/uuid-bbb.png   ← 打码版
```

**增强后（下篇 enriched，同一行被 UPDATE）：**

```
id            | 2001  (同一行)
retrieval_text| 深圳大专学员offer-字节15k。大专学历转行AI拿到offer的真实案例。大专 深圳 15k 字节 offer。
                〔VLM〕图中展示一份字节跳动的录用通知，岗位为AI算法工程师，薪资15000元/月，
                入职时间2026年6月。〔OCR〕字节跳动 录用通知 AI算法工程师 月薪15000 报到时间...
enrich_status | enriched   ← 变了
embedding     | [0.025, -0.015, ...]   ← 重新算了
media_url     | https://.../masked/uuid-bbb.png   ← 不变
tags          | ["大专","深圳","15k","字节","offer"]   ← 人工标签不变
```

### 2.4 怎么查（查询方式完全不变）

检索 SQL 与上篇**一字不改**——增强只是让同一行的向量更"懂"内容。

```python
images = semantic_search(db, "字节的offer有多少钱", modality="image",
                         top_k=3, min_sim=0.40)
```

效果差异（同一个问法）：

```
问："字节的offer薪资多少"
  增强前(manual):   similarity≈0.38  ← 人工描述没提"薪资多少"，勉强命中或漏掉
  增强后(enriched): similarity≈0.71  ← OCR读到"月薪15000"，VLM描述了薪资，强命中 ✅
```

## 三、视频离线处理：ASR 转写 + 摘要 + 抽帧

### 3.1 机器抽取什么

| 抽取手段 | 工具 | 抽到什么 |
|----------|------|----------|
| **ASR 转写** | whisper / 火山ASR / 阿里听悟 | 视频口播全文（带时间戳） |
| **LLM 摘要** | DeepSeek 等 | 把转写浓缩成摘要 |
| **抽帧封面** | ffmpeg | 首帧/关键帧做封面（人工没传封面时） |

### 3.2 离线处理流程

```python
def enrich_video(db, v: VideoAsset):
    vfile = tos_service.download_object(v.oss_key)
    transcript = asr_transcribe(vfile)              # 带时间戳的全文转写
    summary    = llm_summarize(transcript)          # LLM 摘要
    if not v.cover_key:
        v.cover_key = extract_cover(vfile)          # ffmpeg 抽帧补封面

    # 回写 video_assets 内容真相
    v.transcript = transcript
    v.summary    = v.summary or summary             # 人工简介优先，没有才用LLM摘要
    db.commit()

    # 重写检索文本：人工(标题+简介+标签) + 机器(摘要 + 转写片段)
    manual = "。".join(filter(None, [v.title, v.summary, " ".join(v.tags or [])]))
    rtext  = merge_retrieval_text(manual, [summary, transcript[:500]])
    re_embed_asset(db, "video_assets", v.id, rtext)
```

### 3.3 数据怎么存（增强前后对比）

**`video_assets` 内容真相表：**

```
增强前:  transcript=(null)   summary="（人工）针对大专能否学AI的解答"   cover_key=(可能为空)
增强后:  transcript="大家好，今天讲讲大专学历到底能不能学AI...（全文1200字带时间戳）"
        summary 保持人工版（人工优先）   cover_key="materials/covers/auto-uuid.jpg"
```

**`rag_assets` 检索投影行：**

```
增强前(manual):
  retrieval_text| 大专学历能不能学AI？3分钟讲清。针对大专能否学AI的解答。大专 学历 答疑 就业
  enrich_status | manual

增强后(enriched，同一行):
  retrieval_text| 大专学历能不能学AI？3分钟讲清。针对大专能否学AI的解答。大专 学历 答疑 就业。
                  〔摘要〕本视频解答了大专、自考学历能否学AI，列举了3位大专学员的就业数据...
                  〔转写〕大家好，今天讲讲大专学历到底能不能学AI，先说结论：完全可以...
  enrich_status | enriched
  embedding     | (重新计算)
  media_url     | (不变)  cover_url | (若新抽帧则更新)
```

### 3.4 长视频片段定位（离线进阶设计）

长视频（>10 分钟）只存一条向量，命中后用户还要自己找。离线处理时**按 ASR 时间戳分段**，每段单独成一条 `rag_assets`，`media_url` 带 `#t=` 直接跳转：

```
ASR转写(带时间戳)
   │  按话题/每60秒切段
   ▼
video_assets id=301 → 拆成多条 rag_assets:
   ┌─────────────────────────────────────────────────────────────┐
   │ source_table='video_segments', source_id=拼接(301_0)          │
   │ retrieval_text="〔00:00-01:05〕先说结论：大专完全可以学..."     │
   │ media_url=".../uuid-vvv.mp4#t=0"                             │
   ├─────────────────────────────────────────────────────────────┤
   │ source_id=拼接(301_1)                                         │
   │ retrieval_text="〔02:05-03:10〕大专学员就业数据：3人拿到offer" │
   │ media_url=".../uuid-vvv.mp4#t=125"   ← 命中直接跳2分05秒       │
   └─────────────────────────────────────────────────────────────┘
```

> 建议给分段单独建 `video_segments` 表（`video_id, start_sec, end_sec, text`），`rag_assets.source_table='video_segments'`。短视频不分段，按整段一条即可。

### 3.5 怎么查（与上篇一致）

```python
videos = semantic_search(db, "大专学员就业数据怎么样", modality="video",
                        top_k=1, min_sim=0.42)
# 增强后命中到具体片段，返回 media_url 带 #t=125，前端播放器直接跳转
```

## 四、离线任务的调度与幂等

### 4.1 触发时机

| 触发方式 | 场景 |
|----------|------|
| **新增触发** | 上篇人工上传后，丢一条任务进队列，离线 worker 异步增强 |
| **批量回扫** | 定时扫 `rag_assets WHERE enrich_status='manual'`，批量补 |
| **更新触发** | 素材重新打码、视频替换 → 标记 `manual` 重新增强 |

```python
def run_enrich_batch(db, modality, batch=50):
    rows = db.execute(text("""
        SELECT source_table, source_id FROM rag_assets
        WHERE enrich_status='manual' AND modality=:m AND status='active'
        LIMIT :n
    """), {"m": modality, "n": batch}).fetchall()
    for r in rows:
        if modality == "image":
            enrich_image(db, get_material(db, r.source_id))
        elif modality == "video":
            enrich_video(db, get_video(db, r.source_id))
        print(f"enriched {r.source_table}#{r.source_id}")
```

### 4.2 幂等保证

- 一切按 `(source_table, source_id)` UPSERT/UPDATE，**重复跑结果一致**，不产生重复行。
- 失败可重试：增强失败不改 `enrich_status`，下次批量回扫会再捞到。
- 人工内容永远优先：`retrieval_text` 人工部分恒在最前，机器内容只追加不覆盖；`media_url` 锁定打码版。

## 五、两篇协作总览

```
┌──────────────────────── 同一张 rag_assets 表 ───────────────────────────┐
│                                                                         │
│  【上篇】人工上传                          【下篇】离线增强                  │
│  ─────────────                          ─────────────                  │
│  填标题/描述/标签        ──── 即时可命中 ──┐                                │
│  写 materials/video_assets               │                              │
│  拼人工retrieval_text                     │  异步队列/定时回扫              │
│  embedding                               ▼                              │
│  INSERT enrich_status=manual    ┌─────────────────────┐                 │
│                                 │ 图片: OCR + VLM       │                 │
│                                 │ 视频: ASR+摘要+抽帧+分段│                │
│                                 │ 重写retrieval_text    │                │
│                                 │ 重新embedding         │                │
│                                 │ UPDATE→enriched       │                │
│                                 └─────────────────────┘                 │
│                                                                         │
│  查询：两阶段对前端完全透明，semantic_search SQL 一字不改                    │
│       manual 行也能命中（只是召回弱些），enriched 行召回更强                  │
└─────────────────────────────────────────────────────────────────────────┘
```

| 维度 | 上篇（人工） | 下篇（离线） |
|------|-------------|-------------|
| 何时生效 | 上传即生效 | 异步处理后增强 |
| retrieval_text 来源 | 标题+描述+标签 | + OCR/VLM/ASR/摘要 |
| 机器成本 | 0 | VLM/ASR 一次性 |
| media_url | 打码版 URL | **不变** |
| enrich_status | manual | enriched |
| 检索 SQL | 同一套 | **完全一样** |

## 六、实施步骤（下篇）

1. **建增强 worker**：`scripts/enrich_assets.py`，实现 `enrich_image()` / `enrich_video()` + 批量回扫。
2. **接 OCR**：复用打码链路的 RapidOCR，封装 `run_ocr(bytes)->str`。
3. **接 VLM**：调 `qwen-vl`（或同类），封装 `qwen_vl_caption(bytes)->str`。
4. **接 ASR**：选型（whisper 自部署 / 火山ASR / 阿里听悟），封装 `asr_transcribe(file)->str`（带时间戳）。
5. **接 ffmpeg**：`extract_cover(file)->oss_key` 抽帧上传。
6. **长视频分段**：建 `video_segments` 表，按时间戳切段写多条 `rag_assets`。
7. **调度**：新增上传后投递异步任务 + 每日定时回扫 `enrich_status='manual'`。
8. **验证**：对比同一问法在 manual / enriched 下的 `similarity`，确认召回提升。

## 七、关键设计要点速查（两篇通用）

| 问题 | 答案 |
|------|------|
| 图片/视频向量化的是什么？ | 它的「检索文本」`retrieval_text`，不是图像/视频本身 |
| 上篇下篇是两张表吗？ | **同一张 `rag_assets`**，下篇是对上篇行的原地 UPDATE |
| 怎么区分两条路线的数据？ | `enrich_status` 字段：`manual` / `enriched` |
| 离线会改返回的 URL 吗？ | 不会，`media_url` 始终是上篇的打码版 |
| 人工和机器描述谁优先？ | 人工恒在 `retrieval_text` 最前，机器只追加 |
| 检索 SQL 要为两条路线写两套吗？ | 不用，**完全同一套** `semantic_search` |
| 离线任务能重复跑吗？ | 能，按 `(source_table, source_id)` 幂等 UPSERT |
| 长视频怎么精确定位？ | 按 ASR 时间戳分段成多条，`media_url` 带 `#t=秒数` |
| 向量维度 / 模型 / 索引？ | 1024 / `text-embedding-v3` / HNSW `vector_cosine_ops` |
