## 项目一：AI 实时语音对话智能客服

### 项目自述（开场白）

> 这是我们整个 AI 体系最前台的项目。业务背景是电商私域销售：客户在微信、WhatsApp 上咨询商品然后下单。我们把销售的历史对话数据导出，经过**数据中台**清洗后建立 RAG 知识库，构建了一个能**实时语音对话**的智能客服，承接从商品咨询到引导下单的全流程。
>
> 技术上我主要做了四块：一是基于 Prompt Pilot 做提示词工程和自动化评测；二是 RAG 检索链路的优化，用 HNSW 混合检索加量化把存储降了 75%、检索提速 4 倍；三是 LLM 推理性能优化，靠固定检索片段顺序、合并 system prompt 触发前缀缓存，把首字延迟从 15 秒压到 2 秒内；四是 RTC 语音链路，做了情绪化 TTS 和弱网恢复。

---

### Q1. 你说首字延迟从 15s 压到 2s，具体是怎么做到的？前缀缓存的原理是什么？

**回答：**
核心是想办法让请求的**前缀部分在多次请求间保持完全一致**，从而命中推理引擎（如 vLLM）的 **Prefix Caching**。原理是：大模型推理分 prefill 和 decode 两个阶段，prefill 要对整个 prompt 算一遍 attention，长 prompt 的 RAG 场景这一步特别慢。如果前缀的 token 序列和上一次完全相同，引擎就能复用之前算好的 KV Cache，跳过这部分 prefill。

我做了三件事来"对齐前缀"：
1. **固定 RAG 检索片段的排序**：检索出来的片段如果每次顺序不同，前缀就变了，缓存全失效。我强制按稳定 key 排序。
2. **合并 System Prompt 消息体**：把零散的多条 system 消息合并成一个固定块，放在最前面，保证这部分永远不变。
3. **把动态内容（用户当前问题）尽量往后放**，让"不变的部分"集中在前缀。

这样复杂 RAG 场景下 TTFT 从 15s 降到 2s 内。

**可能的追问：那如果知识库更新了，检索片段变了，缓存不就失效了吗？**

**回答：**
对，这是个 trade-off。缓存命中的前提是前缀稳定，知识更新确实会让那一部分失效。但实际场景里：①同一个会话内的多轮对话，system prompt 和已检索片段大概率复用；②热点问题的检索结果是高度集中的，命中率很高。我们不追求 100% 命中，而是让高频路径受益。另外知识更新是低频的，更新后第一次请求慢一点、后续又能复用，整体收益依然显著。

### Q2. HNSW_HYBRID 混合检索 + Int8 量化是什么？为什么能降 75% 存储还提速？

**回答：**
**HNSW** 是一种基于图的近似最近邻索引，查询是对数复杂度，比暴力扫描快很多。**Hybrid（混合检索）**指同时用 dense（向量语义）和 sparse（关键词/BM25）两路召回再融合，靠 `dense_weight` 调两者权重——纯语义检索对"专有名词、型号、价格"这类强关键词命中差，混合后能补上。

**Int8 量化**是把原本 float32 的向量压成 int8，1 个维度从 4 字节变 1 字节，存储直接降到约 1/4，所以**存储降 75%**;同时向量更小、内存带宽和距离计算开销都下降，检索也更快。配合 base-rerank 轻量重排和精简 Limit，整体**检索速度提升 4 倍**。

**可能的追问：量化会损失精度，你怎么保证检索质量没下降？**

**回答：**
量化确实有精度损失，所以我用的是**两阶段策略**：先用 int8 量化向量做**粗召回**（top-20 之类，要的是召回率，对精度不敏感），再用 **base-rerank 轻量重排模型**对候选做精排取 top-k。粗召回阶段量化的微小误差不影响"把相关的捞进来"，精排阶段再用更精确的模型保证排序质量。这样既享受了量化的存储和速度收益，又用 rerank 把精度补回来。针对语音场景专有名词命中差的问题，也是靠这个混合 + rerank 的组合解决的。

### Q3. 语音场景（RTC）和普通文本客服比，难点在哪？你的弱网恢复怎么做的？

**回答：**
语音难点有三个：①**延迟敏感**，文字客服等 3 秒可以接受，语音对话超过 1~2 秒用户就觉得卡；②**链路长**，多了 ASR（语音转文字）和 TTS（文字转语音）两段，任何一段慢都会放大延迟；③**网络不稳定**，移动场景丢包严重。

弱网恢复我用了两个 RTC 经典手段配合：
- **FEC（前向纠错）**：发送端额外发冗余包，接收端丢了几个包也能用冗余包恢复，不用等重传。
- **自适应 Jitter Buffer**：动态调整缓冲区大小，网络抖动大时多缓冲一点平滑播放，网络好时缩小缓冲降延迟。

两者协同，实测在 **50% 高丢包**环境下还能让语音对话无感知恢复。另外情绪化回复是基于 ASR 实时捕获用户情感波动，配合 TTS 声音复刻做出带情绪的回应。

**可能的追问：ASR + LLM + TTS 串起来延迟很高，你怎么优化端到端延迟？**

**回答：**
几个手段叠加：①LLM 这层靠前缀缓存把 TTFT 压到 2s 内（前面讲过）；②接入**微调后的豆包模型**，微调后能用很短的 system prompt 达到同样效果，prompt 短了 prefill 就快；③**流式串联**——LLM 一边出 token，TTS 就一边合成、一边播，不等整句生成完；④Max Token 分级控制，按 95% 覆盖率边界限制输出长度，避免模型啰嗦拖长 TTS 时间。

### Q4. Prompt Pilot 的自动化评测怎么做的？怎么衡量一次提示词改动是变好还是变坏？

**回答：**
核心是**构建评测集 + 自动化评分 + 版本对比**。我先沉淀一批有标准答案或评分标准的测试用例作为评测集，每次改动 prompt 后，自动跑全套评测集，由打分模型按维度（准确性、一致性、是否踩红线等）打分，和上一个版本横向对比，用数据决定是否上线，而不是凭感觉。

对于得分低的**负样本**，再用智能优化算法做提示词重构，并把这些高质量样本回流**数据中台**沉淀，作为后续模型精调的语料。

**可能的追问：评测用模型打分，那打分模型本身不靠谱怎么办？**

**回答：**
两个办法兜底：①**评分标准尽量客观化、结构化**，不让模型自由发挥，而是给明确的 rubric（每个维度什么情况几分），降低主观性；②**关键版本上线前做人工抽检校准**，对比模型分和人工分的一致性。模型打分的价值在于**快速、可规模化地排除明显变差的改动**，相当于"自动化回归测试"，最终决策仍有人工把关，类似 CI 里跑单测但发版还要人 review。

## 项目二：销售数据中台

### 项目自述（开场白）

> 这是整个 AI 体系的**底座**。我们发现所有 AI 业务——客服、微调、考核、素材、直播——本质都在消费数据，所以我把数据生产统一收口做了数据中台。它把微信原始聊天记录，经过 ETL、内容过滤脱敏、对话块切分、人工审核，加工成**可训练、可检索、可考核**的高质量数据资产，再分发给各个下游。
>
> 整条链路是：微信 SQLite → ETL 落原始库 → 一键清洗（过滤+脱敏+切分+预标注）→ 人工审核 → 多下游（导出微调数据 / 建向量知识库 / 提炼结构化知识）。我重点做了 9 级内容分类过滤、多层冗余脱敏、5 分钟时间窗对话切分，以及一份数据多格式导出。

---

### Q1. 数据清洗这条链路你为什么要分这么多阶段？直接一步清洗不行吗？

**回答：**
分阶段是为了**职责单一、可回溯、可人工干预**。我分成：

- **阶段 0 ETL**：只做结构性预处理（时间过滤、群消息解析、去重），**不做任何内容过滤**，原样落 `raw_chats` 原始库且不可变。这样原始数据永远在，后面清洗规则改了可以重跑。
- **阶段 1+2 一键清洗**：内容过滤 + 脱敏 + 5 分钟窗切分 + 预标注，结果进 `staging_conversations` 暂存区。
- **阶段 3 人工审核**：approve / reject / 编辑。
- **下游**：从 approved 数据导出。

关键设计是 **raw 层不可变、staging 层是审核中枢**。如果一步到位清洗，规则一旦有 bug 或要调整，原始数据已经被污染了，没法回滚。分层后每层输入输出清晰，出问题能定位到具体阶段。

**可能的追问：原始数据不可变，那存储不会爆炸吗？**

**回答：**
会增长，但可控。①ETL 阶段就做了**时间门槛**（只保留 2025-10-01 之后的数据），把历史噪声挡在门外；②做了 `(session_id, timestamp, content)` **去重**，重复消息不入库；③原始文本数据本身体积小（纯文本），相比它带来的"可重跑、可追溯"价值，这点存储成本完全划算。真正大的是图片/视频，那些走对象存储（TOS）单独管理，不进这张表。

---

### Q2. 脱敏怎么做的？为什么说脱敏是"多层冗余"的？价格为什么是硬红线？

**回答：**
脱敏分两类。**敏感个人信息**（手机号、身份证、银行卡、邮箱、API Key、密码）用正则匹配后做掩码，比如 `13812345678 → 138****5678`。

"多层冗余"指我在**三个环节**都做脱敏：①清洗阶段（DataFilter）；②写入 staging 阶段；③导出训练数据时再脱一次。看似重复，但这是**纵深防御**——任何一层有遗漏，后面还能兜住，确保敏感信息绝不流入训练数据或知识库。

**价格是硬红线**，因为如果真实定价、优惠策略被微调进模型权重，模型可能在对话里**脱口报出内部价格或底价**，这是商业风险。所以导出时专门有 `_desensitize_price` 把价格脱敏，宁可让模型学会"引导而不直接报价"。

**可能的追问：脱敏会不会把有用信息也误伤了，影响训练效果？**

**回答：**
会有一点，但我做了取舍。脱敏针对的是**对销售话术风格学习无价值、但有泄露风险**的信息——手机号、身份证这些本来就不该让模型学。价格脱敏后，模型学到的是**"怎么应对价格异议"的话术结构**（先认同、再讲价值、引导），而不是具体数字，这反而是我们想要的——具体价格应该走 RAG 知识库实时注入，而不是固化在模型里。这正好呼应"微调管风格、RAG 管知识"的分工。

---

### Q3. 一份清洗后的数据，你怎么同时支撑微调和 RAG 两个完全不同的下游？

**回答：**
靠**同源分流 + 多格式导出**。同一批 approved 数据，按下游需求加工成不同形态：

- **微调下游**：要的是多轮对话，导出成 ShareGPT / Alpaca / OpenAI JSONL 格式，并经过 **7 阶段训练专用清洗**（补 human 先开口、确保 gpt 收尾、价格脱敏、风格修复、过短过滤、过长截断、占位符清理），保证训练数据格式规范、风格统一。
- **RAG 下游**：要的是可检索的知识片段，对文本做 embedding 写入 `knowledge_chunks`，再用 LLM 提炼成结构化的 `knowledge_articles`（场景 / 客户话术 / 推荐回复 / 置信度）。

这就是"**微调管风格、RAG 管知识**"的职责分离：对话风格固化进模型，事实知识放知识库随时更新。

**可能的追问：导出微调数据时那 7 个清洗阶段，哪个最关键？为什么？**

**回答：**
我认为**"确保 human 先开口、gpt 收尾"和"风格修复"**最关键。

前者是因为微调对**对话格式**很敏感：如果训练数据里 gpt 先说话，或者以 human 提问结尾，模型会学到错误的对话轮次模式，推理时容易答非所问或不收尾。所以阶段 1 补一条 human 开场、阶段 2 移除末尾连续的 human 消息。

风格修复关键是因为微调的核心目的就是**学风格**，标点、句长、是否用"亲"这类客服腔如果不统一，模型学出来的风格就是混乱的。统一风格后，盲评里风格一致性才能从 2.1 提到 4.3。

## 项目三：AI 直播切片 Agent

### 项目自述（开场白）

> 公司有直播带货场景，每场直播回放好几个小时，剪辑师要从里面找精彩片段做短视频，特别耗时。我做了一个端到端的 AI 切片 Agent：用户传一段长视频，AI 自动听完、挑出 10 个左右最有传播价值的片段，配好标题文案，前端再用浏览器里的 FFmpeg.wasm 一键裁切打包下载。
>
> 架构是前端 React 19 + FFmpeg.wasm、后端 FastAPI、外部用 Groq Whisper 做 ASR 和 DeepSeek 做内容分析。几个我比较得意的设计：浏览器内提音频把上传体积压到 1%、用 PostgreSQL 当任务队列实现零云依赖、进度统一写库 + SSE 推送。另外 ASR 转写结果会回流到数据中台留存。

---

### Q1. 为什么要在浏览器里用 FFmpeg.wasm 提音频，而不是直接传视频到后端？

**回答：**
因为直播回放动辄 2~10 GB，直传后端有几个问题：上传慢、占带宽、后端要扛大文件存储和转码压力。但 AI 分析其实**只需要音频**。所以我在浏览器里用 FFmpeg.wasm 先把视频转成 16kHz 单声道 64kbps 的 MP3，2.5GB 的视频能压到 ~30MB，**上传体积降低 99%**，用户"等上传"的时间从分钟级降到秒级。

本质是**用用户浏览器的闲置算力换服务器算力**——重活（提音频、最后切片）都放浏览器干，后端只做浏览器干不了的事（调 AI、任务编排）。

**可能的追问：浏览器 wasm 性能有限，提音频会很慢，扛得住吗？有边界吗？**

**回答：**
桌面 Chromium 浏览器 + 1~3 小时视频是舒适区，实测 1 小时视频提音频 6~12 分钟。能扛住靠几个优化：①**WORKERFS 零拷贝挂载**，不把 2GB 视频塞进 wasm 内存（否则 32 位 wasm 4GB 寻址直接 OOM）；②音频参数压到极致（16kHz/单声道/64k），编码计算量小；③配 COOP/COEP 头开 SharedArrayBuffer 让 wasm 多线程。

边界我也清楚：手机浏览器、老版本 Safari、6 小时以上超长视频扛不住。所以留了**兜底路径**——这些场景走 `/api/upload/video` 视频直传，后端用本地 ffmpeg 干活。架构原则是"能在浏览器干的就在浏览器干，干不了的留后门"。

---

### Q2. 你为什么用 PostgreSQL 当任务队列，而不是 Celery / RabbitMQ / Redis？

**回答：**
因为项目定位是**零云依赖、单机可跑**，引入 Redis/RabbitMQ/Celery 会增加部署和运维复杂度。视频切片任务的特点是：任务量不大、单任务耗时长（几分钟到几十分钟）、需要持久化和可观测。这种场景下 PostgreSQL 完全够用：

- **入队**：API 插一行 `status=pending` 就秒回，不阻塞 HTTP。
- **出队**：Task Runner 后台协程每秒 `SELECT` 一条 pending，抢到就改成 downloading 开始跑。
- **崩溃恢复**：进程被杀重启时，把残留在"进行中"状态的任务重置回 pending，自动续上——这是内存队列做不到的。

数据库天然持久化，省了一整套中间件。

**可能的追问：用数据库轮询当队列，并发上来了会不会有性能问题或者重复执行？**

**回答：**
对于这个项目的量级不会，但确实是要注意的点。①**重复执行**：领取任务时用 `UPDATE ... WHERE status='pending'` 配合行锁（或 `SELECT FOR UPDATE SKIP LOCKED`）保证一条任务只被一个 worker 抢到，不会重复跑；②**性能**：当前是单机串行执行（有意为之，避免多个 ffmpeg + LLM 并发把内存打爆），每秒一次轻量 SELECT 对 PG 毫无压力。

如果未来量级真的大到需要高并发，那才是该上专业消息队列的时候——但在那之前，提前引入是过度设计。我会根据数据量选方案，这点和我们 AI 考核里"百条数据用 SQL、十万条才上 RAG"的判断逻辑是一致的。

---

### Q3. 长视频（比如 6 小时）你怎么处理？ASR 和 LLM 都有上下文/请求上限。

**回答：**
两个地方都做了分段。

**ASR 侧**：Groq Whisper 单次请求有文件大小和时长上限，所以超过 25 分钟的音频，用本地 `ffmpeg -c copy` **无损流复制**按 25 分钟切成 N 段，串行发给 Groq，每段返回的相对时间戳加上 offset 还原成绝对时间，最后合并排序。`-c copy` 不重编码，切几秒就完，耗时主要在网络往返。串行而非并发是有意的——避免触发 Groq 速率限制。

**LLM 侧**：转录文本按 ~6000 token 分批，每批独立调一次 DeepSeek，最后合并 + **双向重叠去重**（两个片段重叠率 >30% 取爆款分高的那个）。这样 6 小时视频也不会爆上下文，还能把重复的相似片段合并成最佳一条。

**可能的追问：分批分析会不会丢失跨批次的上下文，导致切片质量下降？**

**回答：**
有一定影响，但对这个场景可接受。因为我们要找的是"**精彩片段**"，而精彩片段通常是**局部完整**的——一段话术、一个爆梗、一个促单瞬间，基本在一个时间窗内说完，不太依赖几十分钟前的上下文。批次切分时我也会在边界做适当重叠，避免把一个完整片段切断。再加上场景化 Prompt（直播/面试/讲座对"精彩"的定义不同）和最后的重叠去重，整体质量是够的。如果是做"全局摘要"那种强依赖上下文的任务，才需要更复杂的处理。

---

### Q4. SSE 和 WebSocket 怎么选的？进度为什么要写库再读，不能直接推？

**回答：**
选 **SSE** 是因为进度推送是**单向**的（只有服务器告诉浏览器进度，前端不需要往后端发消息），SSE 基于普通 HTTP、`new EventSource(url)` 一行就能用，比 WebSocket 简单得多。WebSocket 是双向的，适合聊天、协作那种场景，这里用不上。

进度**写库再读**是为了让**进度源唯一**。干活的 Task Runner 只管"做完一步就 UPDATE 一下 progress 字段"，SSE 接口只管"每秒 SELECT 一次 progress 推给前端"。两者**不直接通信**，全靠数据库这个中间人。好处是：①断线重连能接着读，进度不丢；②多个客户端同时订阅，看到的进度天然一致；③进程崩了重启，进度还在库里。

**可能的追问：每秒轮询数据库推送，延迟和数据库压力怎么权衡？**

**回答：**
1 秒的粒度对"看进度条"这个体验完全够了，用户感知不到 1 秒延迟。数据库压力上，这是一个**按主键查单行**的极轻量查询，加了索引基本是微秒级，即使几十个并发 SSE 连接也没压力。如果要追求更实时（比如毫秒级），可以改成"Runner 更新时主动发事件通知"的推模式，但那样架构复杂度上去了，而收益对这个场景没意义。所以我选了简单稳定的轮询——**简单方案能满足需求时就不上复杂方案**。

---

## 项目四：自媒体智能内容运营 Agent

### 项目自述（开场白）

> 电商要持续产出商品图、商品介绍、宣传文案，人工做很费时。我用 LangGraph 做了一个多阶段的内容生成 Agent 来降本增效——把选题、写作、配图拆成独立的子图编排，自动跑完整条内容生产线。
>
> 技术重点有几块：用 LangGraph 的 SubGraph 拆分流程、PostgreSQL Checkpointer 做状态持久化支持断点重试；SSE 流式输出把感知首字延迟降 60%；配图批量生成做异步并行降耗时 70%；多模型动态路由降成本 40%。另外生成的商品图会直接进**素材中心**，反哺多模态知识库。

---

### Q1. 你用 LangGraph 的 SubGraph 拆分工作流，相比一个大流程图有什么好处？

**回答：**
SubGraph（子图）就是把"选题、写作、配图"各自封装成一个独立的子图，主图只负责编排它们。好处主要三个：

1. **故障恢复粒度细化**：配合 PostgreSQL Checkpointer，状态持久化后可以从任意历史节点回滚重试。如果配图这一步失败了，不用从选题重新跑整张图，只重跑配图子图就行——**恢复粒度从整图级别细化到单节点级别**。
2. **可复用、可测试**：每个子图职责单一，能单独测试、单独复用。
3. **可读性和可维护性**：主图清晰表达业务流程，细节藏在子图里。

**可能的追问：Checkpointer 持久化具体存了什么？流式输出会不会破坏它的一致性？**

**回答：**
Checkpointer 存的是**每个节点执行后的 State 快照**——包括中间产物、当前进度、上下文，存在 PostgreSQL 里。这样任意节点崩了都能从最近的 checkpoint 恢复。

流式确实是个坑点。我专门保证了**流式和非流式模式下 Checkpointer 的数据结构一致**——流式输出是通过监听 `astream_events` 的 `on_chat_model_stream` 事件拿 token 级数据，但写入 State 和 checkpoint 的还是完整结果，不会因为是流式就存成碎片。这样无论哪种模式，恢复和回滚的行为都一致。

---

### Q2. 配图批量生成的异步并行优化具体怎么做的？为什么能降 70%？

**回答：**
原来多张配图是**串行**生成的——生成完第一张再生成第二张，N 张图就是 N 倍时间。我做了两步改造：

1. **同步 API 转协程**：图片生成 API 是同步阻塞的，我用 `asyncio.gather` 配合 `run_in_executor` 把它包装成协程，多张图**并发**生成。原来串行的 N 倍耗时变成约等于单张耗时，这是降 70% 的主要来源。
2. **指数退避重试**：LLM/图片服务会抖动，我加了 1s→2s→4s 的指数退避重试，避免偶发失败导致整批重来。

**可能的追问：并发请求会不会触发服务商限流？你怎么控制并发度？**

**回答：**
会，所以不能无脑全并发。我会控制**并发上限**（比如用信号量 Semaphore 限制同时在飞的请求数），在"并发提速"和"不触发限流"之间找平衡。指数退避重试也是配合这个用的——万一还是被限流了，退避重试能让请求错峰重发，而不是一起撞上去。这跟直播切片里 ASR 故意串行避免 Groq 限流是同一类权衡，只是这里数据量小、可以适度并发。

---

### Q3. 多模型动态路由是怎么决策的？怎么保证便宜模型不会拉低质量？

**回答：**
思路是**按任务复杂度分级路由**：简单任务（比如改写一句标题、格式整理）走轻量的 Flash 模型，复杂任务（创意文案、长文生成）走旗舰模型。这样在保证质量的前提下**降低 40% 推理成本**——因为大量任务其实是简单的，没必要都用贵模型。

保证质量靠两点：①路由规则保守，**拿不准的归到复杂类走旗舰模型**，宁可多花钱也不牺牲质量；②用 `with_structured_output` 做 **Pydantic Schema 强约束输出**，无论哪个模型，输出格式都被卡死，结构上不会崩。

**可能的追问：你提到 JSON Mode 和 Token 统计不兼容，这是什么问题，怎么解决的？**

**回答：**
这是个实战踩的坑。`with_structured_output` 底层在某些实现下会走 function calling 或特殊通道，导致**拿不到准确的 Token 消耗统计**（usage 字段缺失或不准），而我又需要靠 Token 统计做成本核算和模型选型。

解决办法是：对需要精确统计 Token 的链路，改用 **JSON Mode（`response_format`）+ 手动解析**——让模型按 JSON 输出，我自己解析，这样 usage 字段是完整的，既拿到了结构化输出又保住了 Token 统计。本质是在"框架便利"和"可观测性"之间做了取舍，可观测性对我们做成本优化更重要。

---

### Q4. 这个项目和数据中台、素材中心是怎么联动的？

**回答：**
是"**生成即沉淀**"的闭环。Agent 生成的商品图不是用完就扔，而是**直接写入素材中心**（项目六的 `materials` 表）。进入素材中心后，这些图会经过离线增强（OCR 提取图中文字、VLM 看图生成描述），变成可被语义检索的多模态知识，最终成为 **AI 客服的多模态回复素材**——客户问某个商品时，客服能直接甩出对应的商品图。

所以数据流是：自媒体 Agent 生图 → 素材中心 → 离线增强 → 多模态知识库 → AI 客服调用。这就是我们整个体系"我中有你、你中有我"的一个典型例子，背后都是**数据中台在串联**。

## 项目五：基于 MCP 协议的多 Agent 共享服务集群

### 项目自述（开场白）

> 随着公司 AI 项目越来越多——客服、自媒体、考核……每个项目都各自实现一套 LLM 调用、RAG 检索、记忆管理、Prompt 管理，重复造轮子还难维护。我做了一次架构层面的统一收口：基于 MCP 协议，把这些公共能力抽成**独立的共享服务**，所有业务复用同一套底座，新业务接入只需写自己的 Agent 编排层。
>
> 具体是把 LLM 网关、RAG 检索、会话记忆、Prompt 管理做成 4 个独立的 MCP Server，通过 Streamable HTTP 暴露标准化的 Tool/Prompt 原语；前面再加一个 REST→MCP 的桥接网关做认证、配额、路由。Agent 本身无状态，所有状态下沉到共享服务。

---

### Q1. 什么是 MCP？你为什么选 MCP 而不是自己定义一套 RPC 接口？

**回答：**
MCP（Model Context Protocol）是一套**标准化的协议**，用来让大模型应用以统一的方式调用外部能力——它定义了 Tool（工具调用）、Prompt（提示词模板）、Resource 等标准原语。

我选 MCP 而不是自己撸一套 RPC，原因是：①**标准化**，Tool/Prompt 这些原语是协议级定义的，新业务接入认知成本低，不用学一套私有约定；②**生态兼容**，MCP 正在成为 Agent 领域的事实标准，未来能直接对接支持 MCP 的客户端和工具；③**职责清晰**，它天然鼓励"把能力做成可被发现、可被调用的 Tool"这种解耦设计，正好契合我"把公共能力抽成共享服务"的目标。自定义 RPC 短期能跑，但长期是技术债。

**可能的追问：Streamable HTTP 是什么？为什么不用 stdio 或 SSE 传输？**

**回答：**
MCP 支持多种传输方式。**stdio** 适合本地进程间（比如桌面端插件），我们是**分布式多服务**部署，进程不在一台机器上，stdio 不适用。早期 MCP 用 HTTP+SSE 双通道，但有连接管理复杂、长连接维护成本高的问题。**Streamable HTTP** 是更新的传输方式，用单个 HTTP 端点支持请求响应和流式，连接管理更简单。我用它配合内置的 MCP Client Manager 管理**短连接**，按需建连，省去维护一堆长连接的开销，更适合我们这种服务端集群。

---

### Q2. 你设计了 REST→MCP 的桥接网关，为什么需要这一层？它做了什么？

**回答：**
需要这一层是因为：上层业务项目大多是普通的 REST 服务，让它们直接讲 MCP 协议改造成本高；而且**认证、配额、路由这些横切关注点**需要统一收口，不该散落在每个业务里。

网关做了四层中间件链：**认证 → 日志 → 配额 → 路由**：
- **认证**：按项目维度做 API Key 校验，多租户隔离。
- **日志**：用 Trace ID 串起全链路追踪。
- **配额**：按项目计量 Token 消耗，做成本管控。
- **路由**：通过内置的 MCP Client Manager 把请求转发到对应的 MCP Server。

这样业务方只要调 REST，复杂的 MCP 通信和治理全在网关里。

**可能的追问：多租户的成本管控具体怎么落地？Token 配额是怎么算和扣的？**

**回答：**
关键是**让所有 LLM/Embedding 调用都收口到统一 LLM 网关**。任何业务要调模型，都经过网关，网关在调用完成后拿到本次的 Token 消耗，**统一回传到配额模块按 project_id 扣减**。因为入口唯一，所以没有任何一个业务能绕过计量。配额可以按项目设上限，超了就拒绝或降级。这套设计的前提就是"网关是唯一入口"——这也是为什么我把 LLM 调用抽成共享网关而不是让各业务自己调，**不光是复用，更是为了可计量、可管控**。

---

### Q3. 双层记忆架构——对话记忆按 session 隔离，用户画像跨项目共享，为什么这么设计？

**回答：**
因为这两种记忆的**生命周期和共享范围完全不同**：

- **对话记忆**是短期的、会话内的，按 `project_id + session_id` 隔离——A 项目这次对话的上下文，不该泄露到 B 项目或下次会话。支持最近 N 轮召回和会话清空。
- **用户画像**是长期的、跨项目的认知——比如客服 Agent 通过对话发现"这个用户预算有限、对某成分过敏"，这种事实**应该跨项目共享**。所以按 `user_id` 存，A 项目的客服提取的画像，B 项目的写作 Agent 也能读到，实现**跨项目用户认知同步**。

举例：客服 Agent 在聊天中提取"用户过敏"写入画像，自媒体/写作 Agent 给这个用户生成内容时就能即时读到，避免推荐过敏商品。

**可能的追问：用户画像跨项目共享、并发写入，怎么保证不冲突、不脏写？**

**回答：**
用 PostgreSQL 的 **`UPSERT`（`ON CONFLICT DO UPDATE`）实现幂等写入**。多个 Agent 可能同时发现并写同一个用户的画像，用 UPSERT 能保证"存在就更新、不存在就插入"，不会插出重复行。对于同一字段的并发更新，靠数据库的行锁保证原子性。画像更新本身是"事实累加"型的（提取到新事实就补充），不是强一致性的转账场景，所以这套幂等 + 行锁的方案足够，不需要引入分布式锁这种重武器。

---

### Q4. Agent 七步编排流水线和"Agent 无状态、状态下沉"是什么意思？有什么好处？

**回答：**
七步是指一次完整的 Agent 处理：**召回历史 → 读取画像 → 检索知识 → 渲染 Prompt → LLM 推理 → 保存记忆 → 提取事实**，全部通过统一的 GatewayClient 调共享服务完成。

"**Agent 无状态、状态下沉**"是说：Agent 自己不持有任何状态——对话历史在记忆服务、知识在 RAG 服务、画像在记忆服务、模板在 Prompt 中心。Agent 只是个**编排器**，把这些共享服务串起来。

好处：①**水平扩展容易**，Agent 无状态就能随便加实例、负载均衡，挂了重启也不丢数据；②**业务和基础设施彻底分离**，换模型、改知识库、调模板都不用动 Agent 代码；③**新业务接入快**，复用全套底座，只写自己的编排逻辑。

**可能的追问：每一步都调远程共享服务，链路这么长，延迟和可用性怎么保证？**

**回答：**
确实链路变长了，这是解耦的代价，我用几个手段控制：①**全链路 Trace ID 追踪**，能快速定位是哪一步慢、哪一步挂；②部署在阿里云 Serverless（FC/SAE），**按依赖拓扑分层发布**（先发 MCP 服务、再网关、最后业务），并支持自动弹性伸缩应对流量波动；③**SIGTERM 优雅终止**保证发版/缩容时在途请求不丢；④独立镜像支持**灰度发布和秒级回滚**，单个服务出问题能快速止血。延迟上，部分步骤（比如读画像、召回历史）其实可以并行发起，不必严格串行。整体是用"可观测 + 可弹性 + 可快速回滚"来换取解耦带来的灵活性。

---

## 项目六：多模态素材中心 & AI 销售考核系统

### 项目自述（开场白）

> 这个项目其实是两件事，但共享一条数据线。**素材中心**统一管理商品图、详情、视频，核心价值是为 AI 客服提供多模态回复能力，同时这些素材又补充进数据中台。**AI 销售考核**则是消费数据中台导出的高质量知识，自动出题、自动评分，把过去的人工销售培训和考核给自动化了。
>
> 素材中心我设计了一个统一向量底座 `rag_assets`，让文字、图片、视频三种模态共用一张表、一套检索。关键理念是：**不向量化媒体本身，而是向量化它的"检索文本"**，也就是"这张图/这个视频能回答什么问题"。考核系统我做了 LLM 出题 + LLM 评分的全流程，还规划了数据量涨到十万级时的 RAG 演进方案。

---

### Q1. 图片和视频怎么做语义检索？你说"不向量化媒体本身"是什么意思？

**回答：**
关键洞察是：**用户提问是文本，所以我应该拿文本去匹配文本**。直接对图片/视频做向量化（CLIP 那种跨模态向量）也能做，但跨模态对齐效果不稳定，而且和我们已有的文本 embedding 体系不统一。

所以我的做法是给每个图片/视频构造一段 **`retrieval_text`（检索文本）**——描述"这张图/这个视频能回答什么问题"，只对这段文本做 embedding。用户提问也是文本，两者**天然在同一个向量空间对齐**，检索质量高、还能复用文本那套 `text-embedding-v3` + pgvector + HNSW 的基础设施。命中后再返回媒体的在线 URL 和展示文案。

**可能的追问：那这个 retrieval_text 从哪来？写得不好不就검索不到了吗？**

**回答：**
对，这正是核心难点，所以我设计了**两条路线叠加**：

- **人工路线（manual）**：上传时人工填标题、描述、标签。重点是**描述要写成"用户会怎么问"**——比如别写"一张截图"，要写"大专学历也能学 AI 并成功就业的证明"，这样用户问"大专能学吗"才命中。这条路线**零等待、零机器成本，上传即可命中**。
- **离线增强路线（enriched）**：人工肯定写不全，所以离线再用机器补——图片用 **OCR** 提取图里的文字（offer 金额、公司名）、用 **VLM 看图说话**生成丰富描述；视频用 **ASR** 转写口播 + LLM 摘要。把这些机器抽取的文本**追加**到 retrieval_text 后面重新 embedding。

实测同一个问法"字节的 offer 薪资多少"，增强前相似度约 0.38（勉强命中），增强后 OCR 读到"月薪 15000"，相似度提到 0.71，强命中。

---

### Q2. 人工路线和离线增强路线在同一张表上，你怎么管理两者的关系？会不会互相覆盖？

**回答：**
设计上**它们作用在同一行**，不是两张表。我用 `enrich_status` 字段区分：人工上传插入的行是 `manual`，离线处理后**原地 UPDATE** 成 `enriched`。三个保证不冲突：

1. **人工内容永远优先**：retrieval_text 拼接时人工部分恒在最前（权重最高），机器抽取的内容只**追加在后面，绝不覆盖**人工写的。
2. **幂等回写**：所有写入按 `(source_table, source_id)` 做 UPSERT/UPDATE，**重复跑结果一致**，不产生重复行，离线任务失败了重试也安全。
3. **隐私红线不回退**：OCR/VLM 用原图分析（信息最全），但 `media_url` 始终锁定**打码版** URL——机器只读原图、绝不暴露原图。

而且**检索 SQL 完全不用改**，两条路线对前端透明，manual 行也能命中（只是召回弱些），enriched 行召回更强。

**可能的追问：长视频命中了，用户还得自己在视频里找那段内容，体验不好，你怎么解决？**

**回答：**
我做了**按 ASR 时间戳分段**。长视频（>10 分钟）离线处理时，按话题或每 60 秒把转写切段，**每段单独成一条 `rag_assets`**，`media_url` 带上 `#t=秒数`。这样用户问"大专学员就业数据怎么样"，命中的是具体那个片段（比如 2 分 05 秒处），返回的 URL 是 `...mp4#t=125`，**前端播放器直接跳转到那一秒**，不用用户自己拖进度条。短视频则不分段，整段一条就够。

### Q3. AI 考核的出题和评分是怎么做的？为什么出题用 temperature 0.7、评分用 0.3？

**回答：**
**出题**：从数据中台产出的 `knowledge_articles`（结构化话术，按场景分类 + 置信度排序取素材）拼进 Prompt，让 LLM 以"资深销售培训考官"角色生成开放式场景题，按难度分级（简单/中等/困难），每题带参考答案。**评分**：把"题目 + 参考答案 + 学员回答"拼进 Prompt，让 LLM 以"资深评审专家"角色从实战性、需求把握、销售策略、话术自然度、红线检查五个维度逐题打分，最后按"平均分 ×10"映射成百分制。

temperature 的区别是**有意的**：
- **出题用 0.7（高）**：希望题目**多样、有创造性**，每次生成的场景不一样，避免千篇一律。
- **评分用 0.3（低）**：评分要**稳定、可复现**，同一份答案多次评分结果应该接近，不能忽高忽低，所以压低随机性。

**可能的追问：LLM 评分主观性强、可能不稳，你怎么保证考核公平？**

**回答：**
几个办法：①**低 temperature** 保证同一答案多次评分一致；②**结构化评分维度 + 明确分档**（8-10 优秀 / 5-7 合格 / 0-4 不合格），不让模型自由发挥；③**保留人工复核环节**——状态机里 `ai_graded` 之后还有 `human_reviewed`，AI 评分作为初筛和参考，关键考核由销售主管二次校准。AI 的价值是把人从"批量初评"里解放出来，而不是完全取代人判。这和我们 AI 客服里"AI 处理大部分、复杂的转人工"是一个思路。

### Q4. 现在出题为什么不用 RAG？数据量大了你打算怎么演进？

**回答：**
现在**故意不用 RAG**，因为当前知识库是**百条级别**的小数据量。我直接用 SQL 按分类 + 置信度取 top-30 条全量塞进 Prompt，这种"全量灌入"在小数据下效果最好——没有检索误差，模型能看到全部素材。引入 RAG 反而要多维护 embedding 和向量库，是过度设计。

但我设计了**清晰的演进路径**，按数据规模分档：
- < 50 条：全量灌入（现状）
- 50~500 条：分层随机采样（避免老取同一批 top-30 导致题目重复）
- 500~5000 条：SQL + 关键词匹配缩小范围
- **> 5000 条 / 十万级：上 RAG（向量检索）+ 重排序 + 去重**

**可能的追问：十万条数据时，用户说"帮我随便出套题"，没有明确主题，RAG 拿什么去检索？**

**回答：**
这是 RAG 最棘手的场景——模糊请求没有检索方向。我的解法是**分阶段检索，先把模糊变具体**：

- **Phase 1（不用 RAG）确定方向**：让 LLM 先规划出题主题，比如生成 ["价格异议", "竞品对比", "犹豫拖延", "售后顾虑"...] 这几个子主题；或者更智能的——**查用户历史低分题的分类，针对薄弱项出题**。
- **Phase 2（RAG 核心）多轮检索**：拿每个子主题分别做向量检索，各召回 5~8 条。
- **Phase 3 去重保多样**：用 **MMR（最大边际相关）** 去掉过于相似的素材，保证每个方向都有覆盖，最终选 30 条。
- **Phase 4**：和现在一样拼 Prompt 出题。

核心思路是：**用户请求模糊时，RAG 不是第一步，而是第二步——第一步永远是用 LLM 规划或业务规则把"模糊"变"具体"**。

## 项目七：基于聊天数据的大模型微调

### 项目自述（开场白）

> 这个项目的目标是让 AI 客服回复**更像真人销售**。光靠 RAG + 通用大模型，回复总有股"AI 味"，话术不够地道。所以我基于数据中台导出的高质量对话数据，对豆包大模型做了微调，再把微调后的模型回灌到 AI 客服系统当 LLM 层。
>
> 我的核心方法论是"**微调管风格、RAG 管知识**"——微调让模型学会销售的说话风格和话术策略，具体的产品、价格知识则交给 RAG 实时注入。训练上走"本地小模型快速验证 + 云端大模型生产训练"两阶段，并规划了 SFT → DPO 的训练路径，还建了三层评估体系来量化效果。

---

### Q1. 为什么要"微调管风格、RAG 管知识"分开？全用微调把知识也学进去不行吗？

**回答：**
不行，这是我特意做的**职责分离**。原因是微调和 RAG 各有擅长：

- **微调擅长学风格**：说话的语气、话术结构、销售策略（先认同再讲价值、用提问挖需求）——这些是相对稳定、需要"内化"到模型里的，适合微调。
- **RAG 擅长管知识**：课程详情、价格、优惠政策这些事实知识**时效性强、变化快、要求精确**。如果把这些微调进模型权重，会有三个问题：①**时效性差**——价格一改就得重新训练；②**覆盖度低**——训练数据覆盖不到的知识就答不出；③**精确度不足**——模型可能"记串"或编造。

分开后，**知识更新只需改知识库、不用重训模型**，既保证风格地道又保证知识准确。这也呼应了价格脱敏那条——价格本来就不该进模型权重。

**可能的追问：那微调数据和 RAG 数据是怎么从同一批清洗数据里分出来的？**

**回答：**
在数据中台导出环节就分流了。同一批 approved 的对话数据：
- **走 SFT**：保留**完整多轮对话上下文**，因为要学的是"在这个对话情境下销售怎么接话"，上下文不能丢。
- **走 RAG 知识库**：从对话里**提炼**出课程详情、价格体系、优惠政策这些结构化知识片段（`knowledge_articles`），只要事实、不要对话形式。

一份原始数据，按"学风格"还是"存知识"两个目的，加工成两种形态。

---

### Q2. 为什么用"本地小模型 + 云端大模型"两阶段，而不是直接上云端训？

**回答：**
为了**降低试错成本**。直接上云端大模型训练，一轮又慢又贵，如果数据有问题或方向错了，浪费大量时间和钱。所以：

- **本地阶段（快速验证）**：用 **LLaMA-Factory + Qwen2.5-3B-Instruct** 做 QLoRA 微调（rank=64, alpha=128, lr=2e-4），单卡 RTX 4090 约 40 分钟一轮。目的是**快速验证数据质量和训练方向**——数据清洗得对不对、风格学得动不动，小模型上跑一遍就知道。
- **云端阶段（生产训练）**：本地验证通过后，把数据转成 OpenAI messages 格式，上**火山引擎方舟**对豆包 1.5 做生产级 LoRA（rank=32, lr=1e-5）。

注意超参不是照搬的，是**按模型规模等比缩放**的——大模型用更小的学习率和 rank。最终风格一致性盲评从基座的 2.1 提到 4.3（5 分制）。

**可能的追问：QLoRA 和 LoRA 有什么区别？为什么本地用 QLoRA、云端用 LoRA？**

**回答：**
**LoRA** 是在原模型旁边加一组低秩矩阵（A、B 两个小矩阵），只训练这些小矩阵、冻结原权重，大幅减少要训练的参数量。**QLoRA** 在 LoRA 基础上再加一步——把**基座模型量化到 4bit** 再训练，进一步把显存占用压下来。

本地用 QLoRA 是因为 **RTX 4090 显存有限**（24G），3B 模型全精度加载 + 训练会爆显存，量化到 4bit 才跑得动，而且本地只是验证、精度损失可接受。云端用 LoRA 是因为方舟平台资源充足，不需要靠量化省显存，用标准 LoRA 精度更好，适合出生产模型。本质是**根据资源约束选方案**。

---

### Q3. 你规划了 SFT → DPO 两阶段，为什么要 DPO？为什么不用 RLHF？

**回答：**
两阶段解决两个递进的问题：

- **SFT 阶段**：解决"**从不会到会说**"——让模型模仿销售的话术风格，先学会像个销售那样说话。
- **DPO 阶段**：解决"**会说但说不好**"的质量精排——用 SFT 模型对同一个问题采样多条回复，由销售主管标注哪个更好（chosen）、哪个差（rejected），训练模型去区分回复的优劣，往更好的方向靠。

**选 DPO 而不是 RLHF**，关键原因是 DPO **跳过了奖励模型（Reward Model）的训练环节**。RLHF 要先训一个奖励模型，再用强化学习（PPO）调，流程长、不稳定、对数据量要求高。DPO 直接用偏好对（chosen/rejected）做监督式训练，在我们**偏好数据量有限（几百条）**的场景下，更稳定、成本更低、更容易落地。

**可能的追问：偏好数据靠销售主管标，几百条够吗？质量怎么保证？**

**回答：**
对 DPO 来说几百条是能起步的——DPO 本身就是为"偏好数据有限"设计的，这也正是我选它而非 RLHF 的理由。质量上：①标注人是**真正的销售主管**，懂什么是好话术，不是外行众包；②标注的是**相对偏好**（A 比 B 好）而非绝对打分，人对"两个里挑一个更好的"判断比"打几分"更可靠、更一致；③可以持续累积，线上跑出来的 case 不断补充偏好数据迭代。如果数据实在不够，还能用更强的模型做初筛、人工只做最终确认来扩量。

### Q4. 你建了三层评估体系，为什么要三层？哪层最有说服力？

**回答：**
三层是因为**单一指标都有盲区**，得分层覆盖、互相补充：

1. **自动化指标层**（BLEU / ROUGE / 语义相似度）：训练过程中**快速、低成本**地排除明显失败的版本。缺点是这些指标和"销售效果好不好"相关性弱——话术和参考答案不一样但更好，BLEU 反而低。所以它只用来**初筛**。
2. **人工 A/B 盲评层**（50 用例 × 3 评估员，用 Cohen's Kappa 校准评估员一致性）：这是**核心指标**，转化引导能力从 1.8 提到 3.9。人能判断话术是否地道、有没有踩雷。
3. **业务指标层**（线上 A/B 分流）：最终看真实业务数据——客户回复率 +23%、有效会话时长 +35%、咨询→试听转化率和人工持平（AI 18.2% vs 人工 19.7%）。

**最有说服力的是业务指标层**——因为它直接证明了模型带来了**真实的商业价值**，而不只是"看起来更好"。自动化指标和人工盲评都是过程指标，业务指标才是结果。但三层缺一不可：没有前两层快速迭代，根本到不了上线验证业务指标这一步。

**可能的追问：转化率和人工持平（18.2% vs 19.7%），其实还略低于人工，这算成功吗？**

**回答：**
我认为算，而且是很有价值的结果。要看**综合性价比**：AI 客服的转化率**接近人工**（差 1.5 个百分点，统计上基本持平），但它能 **7×24 小时在线、瞬时响应、可无限并发、边际成本几乎为零**。人工销售有上限、有成本、有情绪波动。所以在"用接近人工的转化率换取巨大的成本和规模优势"这个意义上，是成功的。而且这是当前版本，配合 DPO 持续迭代和数据回流，转化率还有提升空间。客服的定位也不是完全取代人——高意向、复杂的客户依然可以转人工，AI 承接大量标准化咨询，这才是最优解。

## 附：贯穿全场的"数据闭环"话术（面试杀手锏）

面试官如果问"**这些项目之间是什么关系**"或"**讲讲你的整体架构**"，用这套讲，能瞬间拔高：

> 我们所有项目其实是**一个由数据中台驱动的闭环**，AI 的一切都是数据：
>
> 1. 微信/WhatsApp 销售数据 → **数据中台**清洗脱敏；
> 2. 中台数据 → ① 给 **AI 客服**建 RAG 知识库；② 导出 SFT/DPO 数据做**模型微调**，微调模型再回灌客服；③ 提炼结构化知识给 **AI 销售考核**出题；
> 3. **自媒体 Agent** 生成的商品图 → 进**素材中心** → 离线增强成多模态知识 → 反哺中台，给 AI 客服多模态回复能力；
> 4. **直播切片** 的 ASR 转写结果 → 回流中台留存复用；
> 5. **MCP & Agent 集群** → 给上面所有业务做统一的基础设施底座（LLM 网关、RAG、记忆、Prompt 复用）。
>
> 所以这七个项目不是孤立的，是**我中有你、你中有我**，数据在中间流动、互相反哺，形成一个自我增强的飞轮。
