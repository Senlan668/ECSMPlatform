# 多模态知识库 · 上篇 —— 人工维护与即时命中

> 本篇是「人工路线」：图片/视频在**初次上传时由人工维护标签和描述信息**，写入向量库后即可被 RAG 语义命中，返回图片/视频的在线链接。
>
> 配套阅读：[下篇 —— 离线处理与增强](./多模态知识库-下篇-离线处理与增强.md)。下篇在本篇建好的同一张表上，用机器（OCR/VLM/ASR）补充描述、提升召回。
>
> 技术栈沿用现状：**PostgreSQL + pgvector（pg17）** + DashScope `text-embedding-v3`（1024 维）+ TOS 对象存储。

---

## 〇、两条路线总览（先建立全局认知）

```
                     ┌──────────────────────────────────────┐
                     │            rag_assets 统一向量库        │
                     │   modality: text / image / video      │
                     │   embedding: vector(1024)             │
                     │   enrich_status: manual / enriched    │  ← 区分两条路线
                     └──────────────────────────────────────┘
                          ▲                         ▲
            上篇（本文）：人工维护          下篇：离线机器增强
            ┌──────────────────┐         ┌──────────────────────┐
            │ 上传时人工填：    │         │ OCR / VLM 看图说话    │
            │  标题/描述/标签   │ ──同一行──> ASR 转写 / 摘要      │
            │ → 立即可命中       │  原地增强 │ → 重写retrieval_text  │
            │                  │         │ → 重新 embedding      │
            └──────────────────┘         └──────────────────────┘
```

**核心约定（两篇共用，务必记住）：**

> 图片、视频**本身不做向量化**。我们向量化的是它的「检索文本」`retrieval_text`——也就是"这张图/这个视频能回答什么问题"。用户提问是文本，`retrieval_text` 也是文本，天然对齐。
>
> - **上篇**：`retrieval_text` 完全来自**人工填写**的标题、描述、标签。
> - **下篇**：在人工内容基础上，**追加机器抽取**的 OCR 文字、VLM 描述、ASR 转写，重新生成向量。

## 一、统一存储底座：`rag_assets` 表

三种模态共用一张检索表，一条 SQL 即可跨模态召回。源数据（`materials` 图片表、`video_assets` 视频表）仍是内容真相，`rag_assets` 是它们的"检索投影"。

```sql
CREATE TABLE rag_assets (
    id              bigserial PRIMARY KEY,
    modality        varchar(10)  NOT NULL,          -- 'text' / 'image' / 'video'

    -- 检索核心
    retrieval_text  text         NOT NULL,          -- 真正被向量化的文本
    embedding       vector(1024),                   -- text-embedding-v3 输出

    -- 命中后返回给客服/前端的展示内容
    answer_text     text,                            -- 文字回复正文
    media_url       text,                            -- 图片/视频在线URL（text模态为空）
    cover_url       text,                            -- 视频封面URL
    duration_sec    integer,                         -- 视频时长（秒）

    -- 过滤 / 排序维度
    category        varchar(50),                     -- course/sales/objection...
    tags            jsonb DEFAULT '[]',              -- ["大专","北京","15k"]
    confidence      float DEFAULT 1.0,               -- 人工内容默认高置信

    -- 来源追溯 + 路线标记
    source_table    varchar(40),                     -- materials / video_assets / knowledge_articles
    source_id       bigint,
    enrich_status   varchar(10) DEFAULT 'manual',    -- manual（人工）/ enriched（已离线增强）
    status          varchar(20) DEFAULT 'active',    -- active / disabled
    created_at      timestamp DEFAULT now(),
    updated_at      timestamp DEFAULT now(),

    UNIQUE (source_table, source_id)                 -- 一个源对象对应一行，便于 UPSERT
);
```

索引（与 `pgvector入门指南.md` 一致，万级以上用 HNSW）：

```sql
CREATE INDEX idx_rag_embedding ON rag_assets
USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 100);

CREATE INDEX idx_rag_modality ON rag_assets(modality, status);
CREATE INDEX idx_rag_category ON rag_assets(category);
CREATE INDEX idx_rag_tags_gin ON rag_assets USING gin (tags);
```

> `enrich_status` 字段是两篇的桥梁：上篇插入的行是 `manual`，下篇处理后改成 `enriched`。

## 二、纯文本内容：怎么存、怎么切、怎么查

### 2.1 数据从哪来

文本知识有两类，数据中心统一导出 **JSONL**（一行一条，便于增量与断点续传）。

#### 数据示意 ①：结构化问答（主力）

```json

  "id": "kb_000123",
  "type": "qa",
  "scene": "学历顾虑-大专能否学习",
  "scene_category": "course",

  "recommended_response": "大专完全可以学！我们大专背景的学员占比超过40%，很多零基础的都拿到了15k+的offer，关键是跟着体系化的课程走。",
  "key_points": ["先打消顾虑", "给数据/案例", "引导留资"],
  "confidence": 0.95
}
```

#### 数据示意 ②：长文档（课程大纲、政策说明）

```json
{
  "id": "doc_0007",
  "type": "document",
  "title": "AI课程招生政策说明",
  "section": "学历与基础要求",
  "content": "本课程面向零基础学员开放……无论大专、本科还是自考学历均可报名。我们提供从Python基础到大模型应用的完整路径……（共约1200字）"
}
```

### 2.2 怎么切片（Chunk）

**结论：结构化问答天然就是一个 chunk，不用切；只有长文档才切。**

| 类型 | 切片策略 | 理由 |
|------|----------|------|
| 结构化问答 ① | **一条 = 一个 chunk** | 本身就是独立场景，100~200 字，粒度刚好 |
| 长文档 ② | 按段落聚合到 **300~500 字**，段间 **重叠 50 字** | 太大稀释语义，太小丢上下文 |

长文档切片代码：

```python
def split_document(text: str, target=400, overlap=50) -> list[str]:
    paras = [p.strip() for p in text.split("\n") if p.strip()]
    chunks, buf = [], ""
    for p in paras:
        if len(buf) + len(p) <= target:
            buf += p + "\n"
        else:
            if buf:
                chunks.append(buf.strip())
            buf = (buf[-overlap:] if buf else "") + p + "\n"  # 尾部重叠衔接
    if buf.strip():
        chunks.append(buf.strip())
    return chunks
```

切片结果示意（长文档 ② 被切成多条）：

```
doc_0007#0  "AI课程招生政策说明。学历与基础要求。本课程面向零基础学员开放……（约400字）"
doc_0007#1  "……（与上一片重叠50字衔接）我们提供从Python基础到大模型应用的完整路径……"
```

### 2.3 向量化拼接（决定召回质量）

向量化的不是单一字段，而是**拼接出的检索文本**——把"用户怎么问"放进去当锚点：

```python
# 结构化问答
retrieval_text = f"场景:{a['scene']}\n客户说:{a['customer_says']}\n回复:{a['recommended_response']}"

# 长文档 chunk
retrieval_text = f"标题:{title}\n小节:{section}\n内容:{chunk}"
```

### 2.5 怎么查（语义检索）

```sql
SET hnsw.ef_search = 100;

SELECT id, modality, answer_text, tags, confidence,
       1 - (embedding <=> CAST(:qvec AS vector)) AS similarity
FROM rag_assets
WHERE status='active' AND modality='text'
  AND (:category IS NULL OR category = :category)
ORDER BY embedding <=> CAST(:qvec AS vector)
LIMIT :top_k;
```

## 三、图片：人工维护什么 / 存什么 / 怎么查

### 3.1 人工初次上传时维护哪些信息

图片上传走现有素材库（`materials` 表，TOS 存储、打码链路已具备）。**人工在上传时需要维护这几项**，它们直接决定能否被语义命中：

| 人工维护字段 | 存在哪 | 作用 | 示例 |
|--------------|--------|------|------|
| **标题** `title` | materials.title | 一句话说明这张图是什么 | 「深圳大专学员offer-字节15k」 |
| **描述** `remark` | materials.remark | 这张图能回答什么问题（**最重要**，直接作检索锚点） | 「大专学历转行AI拿到offer的真实案例，证明学历不是门槛」 |
| **标签** `tags` | materials.tags | 结构化维度，过滤+命中 | `["大专","深圳","15k","字节","offer"]` |
| **分类** `category` | materials.category | report(喜报)/brand(聊天截图) | report |
| **打码** | materials.source_material_id | 隐私保护，命中后返回打码版 URL | （走现有笔刷打码链路） |

> **关键**：`remark`（描述）要写成"用户会怎么问"。比如别写"一张截图"，而写"大专学历也能学AI并成功就业的证明"——这样用户问"大专能学吗"才命中。

## 四、视频：人工维护什么 / 存什么 / 怎么查

### 4.1 人工初次上传时维护哪些信息

视频新建 `video_assets` 表存（视频体积大、字段多，不混进 materials）。**人工上传时维护**：

| 人工维护字段 | 作用 | 示例 |
|--------------|------|------|
| **标题** `title` | 视频主题 | 「大专学历能不能学AI？3分钟讲清」 |
| **简介** `summary`（人工版） | 视频讲了什么、能解答什么（检索锚点） | 「针对大专、自考等非全日制学历能否学AI、好不好就业的详细解答」 |
| **标签** `tags` | 维度命中 | `["大专","学历","答疑","就业"]` |
| **分类** `category` | course/objection... | objection |
| **封面** `cover_key` | 上传时可手动指定封面，否则下篇用 ffmpeg 抽帧 | covers/uuid.jpg |

> `summary` 人工版只需一两句话，把"这视频回答什么问题"写清楚即可。完整的 ASR 转写在下篇离线补。

## 五、上传即命中：完整流程串起来

```
人工上传图片/视频
   │  填：标题 + 描述/简介 + 标签 + 分类（+ 打码）
   ▼
写 materials / video_assets（内容真相）
   │
   ▼
拼 retrieval_text（仅人工字段）→ text-embedding-v3 → 向量
   │
   ▼
UPSERT rag_assets（modality=image/video, enrich_status=manual）
   │
   ▼
✅ 立即可被语义命中（无需等离线处理）
```

用户提问后的统一应答（文字+图片+视频混排）：

```python
def answer_with_media(db, query, category=None):
    texts  = semantic_search(db, query, modality="text",  top_k=3, min_sim=0.35)
    images = semantic_search(db, query, modality="image", top_k=3, min_sim=0.40)
    videos = semantic_search(db, query, modality="video", top_k=1, min_sim=0.42)

    replies = []
    if texts:  replies.append({"type": "text",  "text": llm_polish(query, texts)})
    if images: replies.append({"type": "image", "text": images[0].answer_text,
                               "images": [i.media_url for i in images[:3]]})
    if videos:
        v = videos[0]
        replies.append({"type": "video", "text": v.answer_text,
                        "video": {"url": v.media_url, "cover": v.cover_url,
                                  "duration": v.duration_sec}})
    return replies   # 按 文字→图片→视频 顺序分条发送
```

混排编排原则：文字必发、图视频按需；阈值递增（text 0.35 < image 0.40 < video 0.42）；图片≤3张、视频≤1个。

## 六、本篇小结 & 与下篇的衔接

**上篇做到的**：图片/视频上传时人工写好标题/描述/标签 → 立即可被语义命中返回在线链接。**零等待、零机器成本**。

**上篇的局限（下篇要解决的）**：
1. 人工描述写得好不好，直接决定召回——写漏了就命中不到。
2. 图片里的文字（offer 金额、公司名）、视频里的口播内容，人工不可能全抄进去。
3. 长尾问法覆盖不全。

**下篇怎么接**：在**同一行 `rag_assets`** 上，离线用 OCR/VLM 补图片描述、用 ASR 补视频转写，把这些机器抽取的文本**追加进 `retrieval_text` 并重新 embedding**，`enrich_status` 从 `manual` 改成 `enriched`。源表、media_url、人工标签全部保留，只是召回能力增强。
