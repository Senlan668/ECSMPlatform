# 学生管理与喜报素材打通设计

日期：2026-05-06

## 目标

把学生管理模块与喜报素材模块打通，使“一个学生只挂一张主喜报图”成为系统内统一能力，并支持两个入口：

1. 在学生管理里上传或选择喜报并绑定给学生
2. 在喜报模块里先上传未关联喜报，之后再补关联学生

本次只处理新数据和后续编辑数据，不批量回填历史学生与历史喜报。

## 已确认规则

1. 一个学生只允许有一张主喜报图
2. 一张喜报图只允许绑定给一个学生
3. 学生换新喜报时，旧喜报素材保留在素材库中，但解除关联
4. 喜报模块允许先上传为“未关联喜报”，之后再补关联
5. 历史学生与历史喜报不强制迁移，保持原状

## 方案备选

### 方案 A：学生表单向外键到素材表

在 `students` 表增加 `main_report_material_id -> materials.id`。

优点：

- 改动最小
- 单一真相，避免双写
- 适合“一个学生一张主图”的约束
- 现有素材库上传/预览逻辑可最大程度复用

缺点：

- 从素材视角看“当前绑定了谁”时，需要反查学生表或额外查询

### 方案 B：双向字段

学生表存 `main_report_material_id`，素材表再存 `bound_student_id`。

优点：

- 两侧查询都直接

缺点：

- 双写，容易失真
- 替换和解绑场景更容易产生脏数据

### 方案 C：单独关联表

新增 `student_report_bindings(student_id, material_id, is_primary, created_at)`。

优点：

- 方便以后扩展多图、历史版本

缺点：

- 对当前需求偏重
- 实现和维护成本明显更高

## 推荐方案

采用方案 A：`students.main_report_material_id -> materials.id`。

这是当前约束下最稳的设计。学生是“主喜报”的拥有者，素材库继续作为统一文件池存在。两个入口都只是在创建或选择 `materials` 记录后，更新学生外键。

## 数据模型设计

### Student

新增字段：

- `main_report_material_id: Integer | NULL`

语义：

- 为空表示学生当前没有主喜报
- 不为空时，指向一条 `materials.id`

### Material

本次不新增持久化关联字段。

保留当前结构，只要求被绑定的素材满足：

- `category = 'report'`
- 不是文件夹记录

### 读取视图

学生详情和列表响应中增加：

- `main_report_material_id`
- `main_report_material`

`main_report_material` 为一个轻量摘要对象，至少包含：

- `id`
- `filename`
- `title`
- `file_type`
- `category`
- `oss_key`
- `created_at`

素材列表响应中增加：

- `bound_student_id`
- `bound_student_name`

这两个字段由查询时反查学生表得到，不在 `materials` 表中落库。

## 约束规则

### 绑定约束

绑定学生主喜报时必须满足：

1. 素材存在
2. 素材 `category='report'`
3. 素材不是文件夹
4. 素材当前未绑定其他学生

如果素材已绑定其他学生，接口返回 `409 Conflict`。

### 替换规则

给学生绑定新喜报时：

1. 直接更新该学生的 `main_report_material_id`
2. 旧喜报不删除
3. 旧喜报自动变成未关联

### 解绑规则

学生解绑主喜报时：

1. 仅将 `main_report_material_id` 置空
2. 不删除素材

### 删除规则

删除学生时：

1. 正常删除学生记录
2. 不删除喜报素材

删除已绑定喜报时：

推荐策略：禁止删除，并提示先解绑。

原因：

- 规则清晰
- 避免“删除素材导致学生详情突然缺图”的隐式副作用
- 前端更容易理解和处理

## 后端接口设计

### 学生接口扩展

#### 1. 学生列表与详情

扩展现有：

- `GET /api/students/list`
- `GET /api/students/{student_id}`

新增返回字段：

- `main_report_material_id`
- `main_report_material`

#### 2. 绑定现有喜报

新增：

- `PUT /api/students/{student_id}/main-report`

请求体：

```json
{
  "material_id": 123
}
```

行为：

- 校验学生和素材
- 校验素材是否已绑定给其他学生
- 若合法，则更新该学生的 `main_report_material_id`

#### 3. 解绑主喜报

新增：

- `DELETE /api/students/{student_id}/main-report`

行为：

- 将该学生 `main_report_material_id` 置空

### 素材接口扩展

#### 1. 学生管理内上传并绑定

复用现有素材上传接口，在上传时支持可选参数：

- `student_id`

请求场景：

- 仅当上传的是喜报素材且从学生管理发起时携带

行为：

1. 创建 `category='report'` 的素材
2. 若带了 `student_id`，则进一步执行学生绑定逻辑

#### 2. 喜报列表附带绑定信息

扩展现有喜报列表接口，增加派生字段：

- `bound_student_id`
- `bound_student_name`

#### 3. 未关联喜报筛选

扩展现有素材列表接口筛选参数：

- `unbound_only=true`

语义：

- 仅返回没有被任何学生作为主喜报引用的 `report` 素材

用于学生管理里“从现有喜报中选择”。

## 前端交互设计

### 学生管理模块

#### 列表

新增“主喜报”列：

- 有主喜报时显示缩略图或“已关联喜报”
- 无主喜报时显示“未设置”

#### 编辑弹窗

新增“主喜报”区域，支持：

1. 预览当前主喜报
2. 上传新喜报并自动绑定
3. 从现有未关联喜报中选择绑定
4. 解绑主喜报
5. 更换主喜报

推荐交互：

- 上传按钮：直接选文件并上传
- 选择按钮：打开喜报选择器，只展示未关联喜报
- 当前已绑定时，展示缩略图、文件名、上传时间和“更换/解绑”

### 喜报模块

#### 上传

保持现状：

- 允许先上传为未关联喜报

#### 列表 / 详情

增加绑定状态：

- 未关联
- 已关联：`学生名`

增加“关联学生”动作：

- 仅对未关联喜报展示
- 打开学生选择器，选中后完成绑定

已绑定喜报若尝试重新关联其他学生：

- 前端直接禁用入口，或调用接口后展示 `409` 提示

### 选择器复用

建议实现两个轻量选择器组件：

1. 喜报选择器
- 给学生管理使用
- 数据源：`report + unbound_only=true`

2. 学生选择器
- 给喜报模块使用
- 数据源：学生列表

## 数据流

### 场景 1：学生管理上传喜报

1. 学生管理选择图片
2. 上传到素材接口，带 `category='report'` 和 `student_id`
3. 后端创建素材记录
4. 后端将该素材绑定为学生主喜报
5. 前端刷新学生和素材状态

### 场景 2：学生管理选择现有喜报

1. 打开未关联喜报列表
2. 选中一张喜报
3. 调用学生绑定接口
4. 前端刷新学生和素材状态

### 场景 3：喜报模块上传后补关联

1. 在喜报模块上传素材
2. 素材初始状态为未关联
3. 用户点击“关联学生”
4. 调用学生绑定接口
5. 前端刷新喜报状态和学生详情

### 场景 4：学生更换喜报

1. 选择新喜报或上传新喜报
2. 更新学生 `main_report_material_id`
3. 旧喜报保留，但自动变成未关联

## 错误处理

后端错误码建议：

- `404`：学生不存在 / 喜报不存在
- `400`：素材不是喜报 / 素材是文件夹
- `409`：喜报已绑定其他学生

前端提示文案建议：

- “该喜报已关联到其他学生，请先解绑后再使用”
- “只能绑定喜报分类图片”
- “请先解绑当前主喜报后删除素材”

## 兼容性与迁移

### 历史学生

不回填 `main_report_material_id`，默认 `NULL`。

### 历史喜报

不做批量关联处理，继续作为历史素材存在。

### 数据迁移

本次只需要一次轻量 schema 变更：

- 给 `students` 增加 `main_report_material_id`

如果继续沿用当前项目的兼容补列机制，应同步：

1. 模型字段
2. schema sync
3. 针对旧表的回归测试

## 测试设计

### 后端

至少覆盖：

1. 给学生绑定未关联喜报成功
2. 给学生绑定非 `report` 素材失败
3. 已绑定喜报不能绑定给第二个学生
4. 学生更换主喜报后旧图仍存在且未关联
5. 删除学生不删除素材
6. 已绑定喜报删除被拒绝

### 前端

至少覆盖：

1. 学生数据模型能正确映射 `main_report_material`
2. 学生列表/编辑弹窗能展示有无主喜报状态
3. 绑定后 UI 正确刷新
4. 替换后旧喜报在学生侧消失、在喜报侧变为未关联

## 范围外

本次不做：

1. 历史学生与历史喜报的自动匹配
2. 一个学生多张喜报
3. 喜报版本历史
4. OCR 或自动按姓名匹配喜报
5. 喜报与学生之间的批量导入映射

## 实施建议

按以下顺序实现：

1. 后端模型和接口
2. 学生管理内的查看/绑定/解绑/上传
3. 喜报模块内的绑定学生入口和状态展示
4. 回归测试和交互细化

这样可以先打通主链路，再补第二入口，风险最低。
