# 喜报素材拖拽归档设计

日期：2026-05-09

## 目标

为“成交喜报”素材页增加拖拽归档能力，使运营生成的喜报图片可以直接拖到文件夹卡片上完成归档，并支持拖回根目录。

本次只处理“喜报素材移动到文件夹/根目录”这一条能力，不混入素材重命名、批量操作、课程文档或聊天素材拖拽。

## 已确认规则

1. 只在“成交喜报”视图启用拖拽
2. 只能拖动喜报素材卡片，不能拖文件夹
3. 把喜报卡片拖到文件夹卡片上，松手后立即移动
4. 支持把喜报拖回“成交喜报”根目录
5. 不弹确认框，直接执行移动
6. 搜索态下不启用拖拽，避免结果列表与真实目录语义冲突

## 方案备选

### 方案 A：原生拖拽 + 独立移动接口

前端使用原生 HTML Drag and Drop，把喜报卡片设为 `draggable`，把文件夹卡片和根目录面包屑设为 drop target。后端增加独立的素材移动接口，仅负责更新 `folder_id`。

优点：

- 语义清晰，移动行为边界独立
- 后续扩展“批量移动”“右键移动”时可以复用同一后端能力
- 不污染现有素材元数据更新接口

缺点：

- 需要补一个新接口

### 方案 B：复用现有素材更新接口

前端拖拽后仍调用 `PUT /api/materials/{id}`，只是额外提交 `folder_id`。

优点：

- 少一个接口

缺点：

- “改标题/备注”和“移动文件夹”耦合到同一接口，边界变脏
- 后续很容易演变成一个什么都能改的大而杂接口

### 方案 C：仅做前端假移动

拖拽后只更新前端列表顺序或目录显示，不立即持久化。

优点：

- 实现最快

缺点：

- 刷新即丢
- 容易造成前端状态和数据库状态不一致
- 不能上线

## 推荐方案

采用方案 A：原生拖拽 + 独立移动接口。

这个方案最符合当前项目的演进方式。项目已经有素材目录结构和 `folder_id` 字段，但没有单独的“移动素材”语义。把移动行为独立出来，既能快速满足当前需求，也能避免把素材更新接口继续做脏。

## 范围定义

### 本次包含

1. 在“成交喜报”页把喜报素材卡片拖入文件夹
2. 在“成交喜报”页把喜报素材拖回根目录
3. 移动成功后前端即时刷新当前列表和文件夹文件数
4. 拖拽中的目标高亮和反馈提示

### 本次不包含

1. 批量拖拽移动
2. 课程文档页拖拽归档
3. 聊天素材页拖拽归档
4. 跨分类文件夹移动
5. 拖拽排序
6. 移动历史或撤销操作

## 数据模型与后端设计

现有 `materials` 表已经具备：

- `id`
- `category`
- `file_type`
- `folder_id`

因此本次不新增数据库字段。

### 新增接口

新增：

- `PUT /api/materials/{material_id}/move`

请求体：

```json
{
  "folder_id": 123
}
```

语义：

- `folder_id = number`：移动到指定文件夹
- `folder_id = null`：移动回根目录

### 校验规则

后端需要严格校验：

1. 素材存在
2. 素材不是文件夹
3. 素材分类必须为 `report`
4. 目标文件夹存在；若 `folder_id` 为 `null` 则跳过此项
5. 目标文件夹必须是 `file_type='folder'`
6. 目标文件夹分类必须与素材分类一致
7. 不能移动到当前所在文件夹

### 绑定学生的喜报

已绑定学生主喜报的素材，允许移动文件夹。

理由：

- 绑定关系的真相在学生表，不在文件夹
- 文件夹只影响素材归档位置，不影响它是不是学生主喜报
- 只要分类仍然是 `report`，移动文件夹不会破坏关联约束

### 返回结果

接口成功后返回更新后的素材对象，至少包含：

- `id`
- `folder_id`
- `category`
- `filename`
- `title`

方便前端直接更新当前列表。

## 前端交互设计

### 拖拽入口

仅在“成交喜报”页的喜报卡片开启拖拽：

- 根目录下的喜报卡片可以拖拽
- 文件夹内部的喜报卡片也可以拖拽
- 文件夹卡片本身不可拖拽

### 可放置目标

两个目标：

1. 文件夹卡片
2. 面包屑中的“成交喜报”根节点

根节点代表“移动回根目录”。

### 交互反馈

拖动时：

- 被拖卡片降低透明度
- 当前命中的文件夹卡片高亮边框和背景
- 命中根目录面包屑时，高亮“成交喜报”标题区域

放手后：

- 立即发请求
- 不弹确认框

### 成功反馈

成功后：

1. 当前列表中的该喜报立即消失（因为它已经不属于当前目录）
2. 文件夹文件数刷新
3. toast 提示：
   - 移到文件夹：`已移动到「本科」`
   - 移回根目录：`已移回根目录`

### 失败反馈

失败后：

1. 卡片保留原位
2. 高亮状态清空
3. toast 显示后端错误信息

## 前端状态设计

`MaterialView` 增加最小状态：

- `draggingMaterialId: number | null`
- `dropFolderId: number | null`
- `dropOnRoot: boolean`
- `movingMaterialId: number | null`

语义：

- `draggingMaterialId`：当前被拖拽的素材
- `dropFolderId`：当前高亮的目标文件夹
- `dropOnRoot`：当前是否命中根目录
- `movingMaterialId`：正在提交移动请求的素材，防止重复操作

## 前端行为规则

### 允许拖拽的前提

以下情况下启用拖拽：

1. `activeTab === 'report'`
2. 当前不是搜索态
3. 当前素材不是文件夹
4. 当前没有正在进行中的移动请求

### 禁止移动的情况

前端先行拦截这些情况，避免无意义请求：

1. 拖到当前所在文件夹
2. 非 `report` 标签页
3. 搜索结果视图
4. 文件夹目标不存在

即使前端拦截，后端仍需保留完整校验。

## 数据刷新策略

移动成功后需要刷新两类数据：

1. 当前目录素材列表
2. 当前层级的文件夹列表（为了刷新 `file_count`）

若当前在某个子文件夹中：

- 把素材移到别的文件夹或根目录后，该素材应从当前列表移除

若当前在根目录中：

- 把素材拖进某个文件夹后，该素材应从根目录素材列表移除

## API 约定

### 请求示例

移动到文件夹：

```json
{
  "folder_id": 456
}
```

移回根目录：

```json
{
  "folder_id": null
}
```

### 错误码建议

- `404`：素材不存在 / 文件夹不存在
- `400`：素材是文件夹 / 素材不是喜报 / 请求参数非法
- `409`：移动目标与当前目录相同 / 目标分类不一致

## 测试要求

### 后端

至少覆盖：

1. 喜报素材移动到同分类文件夹成功
2. 喜报素材移回根目录成功
3. 非喜报素材移动被拒绝
4. 文件夹记录不能作为被移动对象
5. 移动到当前目录被拒绝
6. 移动到不同分类文件夹被拒绝

### 前端

至少覆盖：

1. 根目录和文件夹目标路径生成正确
2. 当前目录目标会被识别为无效投放
3. 搜索态下不启用拖拽
4. 拖放成功后能触发刷新并清理拖拽状态

## 验收标准

1. 在“成交喜报”页可以把喜报卡片拖进任意同分类文件夹
2. 可以把喜报拖回根目录
3. 移动后页面即时反映最新位置
4. 文件夹文件数同步更新
5. 搜索态不出现拖拽误操作
6. 不会影响已绑定学生喜报的关联关系

## 非目标

本次明确不做：

1. 批量拖拽
2. 课程素材拖拽归档
3. 聊天素材拖拽归档
4. 拖拽排序
5. 移动撤销
