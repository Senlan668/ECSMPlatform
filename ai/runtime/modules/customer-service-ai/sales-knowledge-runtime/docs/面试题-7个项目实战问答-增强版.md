# AI 销售体系 · 7 个项目面试实战问答增强版

> 目标：在原版基础上补足面试追问密度。每个项目都按「项目定位 → 高频问题 → 压迫式追问 → 回答抓手」组织。回答时不要死背整段，优先记住三件事：**业务约束、技术取舍、结果指标**。
>
> 主线：这 7 个项目不是孤立功能，而是一个由**销售数据中台**驱动的 AI 业务闭环：原始销售数据进入中台，清洗审核后分别供给 RAG、微调、考核、素材、多 Agent 服务；线上产生的新数据再回流中台继续增强。

---

## 一、总览类问题

### Q1. 这 7 个项目之间到底是什么关系？

**回答：**
我会把它们讲成一个「数据驱动的 AI 销售飞轮」。底座是销售数据中台，它把微信/WhatsApp 的原始销售对话清洗成可训练、可检索、可考核的数据资产。上层有几个消费方：AI 客服用它建 RAG 和话术能力；微调用它学销售风格；AI 考核用它出题评分；多模态素材中心把图片视频也变成可检索资产；直播切片和自媒体 Agent 产生的新内容再回流中台；MCP 多 Agent 集群则提供统一的 LLM、RAG、记忆和 Prompt 基础设施。

**追问：为什么一定要先做数据中台，不能每个业务自己处理数据？**

**回答：**
如果每个业务自己清洗，会出现三类问题：第一，规则不一致，同一条数据在客服、微调、考核里口径不同；第二，敏感信息容易漏，尤其是价格、手机号、身份信息；第三，数据无法回流复用。中台统一 raw、staging、approved 三层后，每个下游只消费 approved 数据，质量、脱敏、追溯都能统一。

### Q2. 你在这些项目里最核心的技术能力是什么？

**回答：**
不是单点调 API，而是端到端把 AI 应用落地：数据链路设计、RAG 检索、模型微调、LLM 评测、异步任务、前后端工程、上线指标评估都参与过。尤其是我一直围绕「数据从哪来、怎么清洗、怎么被模型消费、线上结果怎么回流」做闭环。

### Q3. 如果面试官质疑这些项目太杂，你怎么解释？

**回答：**
表面上有客服、切片、素材、考核、Agent，但底层技术栈其实高度一致：数据中台负责数据资产，RAG 负责知识注入，LLM 网关负责模型调用，评测体系负责质量，异步任务负责长耗时流程。不同项目只是业务场景不同，复用的是同一套 AI 工程能力。

## 项目一：AI 实时语音对话智能客服

### 项目定位

这是销售链路最前台的 AI 应用：用历史销售对话训练话术风格，用 RAG 注入商品/课程知识，用实时语音链路承接客户咨询、异议处理和引导下单。

### Q1. 这个项目的核心难点是什么？

**回答：**
难点不是「能回答」，而是「像真人一样实时回答」。它同时要求知识准确、话术自然、低延迟、语音链路稳定。文本客服 3-5 秒还能忍，语音场景首字超过 2 秒用户就会感觉卡顿，所以我重点做了 TTFT 优化、流式生成、TTS 串联和弱网恢复。

### Q2. 首字延迟从 15s 降到 2s 具体怎么做？

**回答：**
核心是触发 Prefix Caching。大模型推理分 prefill 和 decode，长 RAG prompt 的 prefill 很重。如果 system prompt、知识片段这部分前缀在多轮请求中保持完全一致，推理引擎可以复用 KV Cache。我做了三件事：固定检索片段排序、合并多条 system prompt、把动态用户问题放到后面。这样高频咨询命中前缀缓存，TTFT 明显下降。

**追问：知识库更新后前缀不稳定怎么办？**

**回答：**
知识更新确实会导致部分缓存失效，但销售场景知识更新频率远低于咨询频率。同一会话、多数热点问题的检索结果高度重复，所以缓存优化针对的是高频路径，不追求 100% 命中。更新后的第一次请求慢一点，后续又能重新缓存。

### Q3. RAG 检索为什么要用 Hybrid，而不是纯向量？

**回答：**
销售场景里有大量强关键词：课程名、型号、价格、优惠、城市、班型。纯向量擅长语义相似，但对专有名词、数字、短关键词不稳定；BM25 对关键词强，但不懂同义表达。Hybrid 把 dense 向量召回和 sparse 关键词召回融合，再用 rerank 精排，兼顾语义和精确匹配。

### Q4. Int8 量化为什么能降存储，还不会明显影响效果？

**回答：**
float32 向量每个维度 4 字节，int8 每个维度 1 字节，理论上存储直接降到 1/4。精度损失通过两阶段补：量化向量只负责 top-N 粗召回，保证相关内容先捞进来；后面用 rerank 或更高精度模型精排。粗召回阶段不要求排序完全准，只要求别漏掉候选。

### Q5. 语音链路 ASR + LLM + TTS 怎么控制端到端延迟？

**回答：**
我按链路拆：ASR 用流式或短句增量识别；LLM 用前缀缓存、短 system prompt、max token 控制；TTS 不等完整回复，LLM 一出稳定片段就开始合成播放。整体是流水线并行，而不是 ASR 完成后等 LLM 完整生成，再等 TTS 完整合成。

### Q6. 弱网恢复怎么做？

**回答：**
语音实时通信里不能完全依赖重传，因为重传回来也可能错过播放时间。我用了 FEC 和自适应 jitter buffer。FEC 用冗余包恢复部分丢包；jitter buffer 根据网络抖动动态调缓冲，网络差时牺牲一点延迟换连续播放，网络好时缩小缓冲降低延迟。

### Q7. 情绪化 TTS 是噱头还是有业务价值？

**回答：**
有业务价值，但不能过度。销售咨询里客户可能焦虑、犹豫、不耐烦，AI 如果永远机械语气，会降低信任感。情绪化 TTS 的目标不是表演，而是让语速、语调、停顿更贴近当前对话，比如安抚异议时更稳，促单时更明确。它服务的是转化体验。

### Q8. 如何防止 AI 客服胡说价格或承诺？

**回答：**
靠三层约束：第一，价格这类事实不进微调权重，走 RAG 实时注入；第二，Prompt 明确禁止编造价格、优惠、保就业等承诺；第三，输出前做规则检查和敏感话术拦截，命中红线就改写或转人工。

### Q9. Prompt Pilot 自动化评测怎么判断 prompt 变好了？

**回答：**
我会维护一批标准场景用例，比如价格异议、学历顾虑、竞品对比、售后担忧。每次改 prompt 后自动跑评测集，用结构化 rubric 评估准确性、话术自然度、红线合规、是否促成下一步。分数超过基线才上线，低分 case 进入负样本池继续优化。

### Q10. 如果线上用户问的问题知识库没有，你怎么处理？

**回答：**
不能硬答。先判断检索相似度和置信度，低于阈值时走兜底策略：澄清问题、回复可确认的信息、提示转人工，或者引导留下联系方式。未知问题会记录下来，进入数据中台作为后续知识补齐和评测样本。

## 项目二：销售数据中台

### 项目定位

数据中台是整个 AI 销售体系的底座。它把微信原始聊天记录加工成可追溯、可审核、可导出、可检索、可训练的数据资产。

### Q1. 数据链路为什么要分 raw、staging、approved？

**回答：**
raw 层保存原始消息，保证可追溯和可重跑；staging 层承接清洗、脱敏、切分、预标注，是人工审核工作区；approved 数据才给下游使用。这样规则改了可以从 raw 重跑，人工审核也不会污染原始数据。

### Q2. 微信数据 ETL 的难点是什么？

**回答：**
微信聊天不是一个简单表。消息分散在 MSG0-5.db，通讯录在 MicroMsg.db；群聊里真实发送人可能藏在 BytesExtra；联系人要做 wxid 到备注/昵称映射；还要过滤非文本消息、系统消息、撤回提示、重复消息。ETL 阶段我只做结构解析和去重，不做内容改写，保证原始层可信。

### Q3. 为什么 raw_chats 不可修改？

**回答：**
因为 raw 是事实来源。清洗规则、脱敏规则、分类规则都会迭代，如果 raw 被改坏，后面没有办法验证和重跑。不可变 raw 的价值类似日志源表，牺牲一点存储，换来可审计、可回溯、可复现。

### Q4. 5 分钟时间窗切分对话有什么依据？

**回答：**
销售聊天天然是碎片化的。太短会把一个完整异议处理拆散，太长会混入多个主题。5 分钟窗口是经验折中：在同一 session 内，把时间接近、话题连续的消息合并成一个 conversation block。后续也可以按消息数量、主题变化、角色轮次做二次切分。

### Q5. 脱敏为什么要多层冗余？

**回答：**
因为敏感信息泄露是高风险，不应该依赖单点规则。我在清洗、写入 staging、导出训练数据三个阶段都做脱敏。任何一层漏掉，后面还有机会兜住。尤其是手机号、身份证、银行卡、API Key、价格底线，都不能流进训练数据或外部知识库。

### Q6. 价格为什么不能直接进入微调数据？

**回答：**
价格、优惠、底价是强时效且高风险的信息。微调进权重后，模型可能在不该报价的时候报出历史价格，甚至编造优惠。更合理的是微调学「如何处理价格异议」的话术结构，具体价格由 RAG 或业务系统实时注入。

### Q7. 怎么做自动分类和质量评分？

**回答：**
先用规则做初筛：长度、关键词、角色轮次、是否包含销售动作、是否有客户问题和销售回应。再用 LLM 对高价值候选做分类和质量评分，比如 sales、course、objection、closing、followup、qa、casual、junk。机器预标注只做提效，最终 approved 仍由人工审核决定。

### Q8. 如何支持一份数据同时给 RAG 和微调用？

**回答：**
同源分流。微调需要完整多轮对话，重点保留角色、上下文、话术风格，导出 ShareGPT、Alpaca、OpenAI JSONL。RAG 需要事实和可检索片段，重点提炼 scene、customer_says、recommended_response、key_points，再做 embedding。目的不同，形态不同。

### Q9. 如果清洗规则误杀了高质量数据怎么办？

**回答：**
首先 raw 层保留原始数据，所以可以回溯。其次被过滤消息会保留 status、flags 和过滤原因，方便抽样检查。规则迭代后可以按 session 或时间范围重跑清洗，把误杀数据重新进入 staging。

### Q10. 你怎么证明数据中台有价值？

**回答：**
看下游效果：客服 RAG 命中率提升、微调风格一致性提升、考核出题素材可复用、人工标注效率提升、敏感信息泄露率下降。数据中台不是单独为了建表，而是让所有 AI 项目都有稳定的数据供给和质量控制。

## 项目三：AI 直播切片 Agent

### 项目定位

把长直播回放自动转成可传播短视频：浏览器提取音频，后端 ASR 转写，LLM 识别精彩片段，前端 FFmpeg.wasm 裁切打包。

### Q1. 为什么在浏览器里提音频？

**回答：**
直播视频动辄几个 GB，但 AI 分析只需要音频。浏览器用 FFmpeg.wasm 把视频转成 16kHz 单声道音频，体积可以压到原来的 1% 左右，上传更快，后端也不用承担大文件转码压力。本质是用客户端算力换服务器带宽和存储。

### Q2. FFmpeg.wasm 的边界是什么？

**回答：**
桌面 Chrome 处理 1-3 小时视频比较合适，但手机、Safari、超长视频会遇到内存和性能问题。所以我会检测浏览器能力，能用 wasm 就前端处理，不能用就走后端 ffmpeg 兜底。架构上不把 wasm 当唯一方案。

### Q3. 为什么 PostgreSQL 可以当任务队列？

**回答：**
因为任务量不高、单任务耗时长、需要持久化进度。API 插入 pending 任务后立即返回，后台 runner 轮询领取任务，进度写库，失败可重试。这个场景上 RabbitMQ/Celery 会增加部署复杂度，PG 足够稳定。

### Q4. 如何避免任务重复执行？

**回答：**
领取任务时用事务和行锁，比如 `SELECT FOR UPDATE SKIP LOCKED`，或者条件更新 `WHERE status='pending'`，保证同一时刻只有一个 worker 抢到任务。任务处理也要设计成幂等，重复跑不会产生多份错误结果。

### Q5. 长视频 ASR 怎么处理上下文和时间戳？

**回答：**
长音频按固定时长切片，比如 20-25 分钟一段。每段 ASR 返回相对时间戳，再加上切片 offset 还原到全局时间。切片边界留少量重叠，避免把一句话切断。最终合并时按时间排序并去重。

### Q6. LLM 怎么从转写里找精彩片段？

**回答：**
Prompt 里明确评分维度：信息密度、情绪强度、冲突点、成交引导、可独立成片、标题吸引力。模型输出结构化 JSON，包括 start、end、reason、title、copy、score。后端再校验时间范围、片段长度、重叠比例，过滤不合法结果。

### Q7. 分批分析会不会漏掉跨片段精彩内容？

**回答：**
会有风险，所以分批时要做 overlap，最终再全局合并。直播切片找的是局部可传播片段，通常 30-180 秒内能自洽，不像长文总结依赖全局上下文。对跨片段内容，重叠窗口和二次合并可以覆盖大多数情况。

### Q8. SSE 和 WebSocket 怎么选？

**回答：**
进度推送是服务端到浏览器的单向通知，SSE 足够简单，基于 HTTP，断线还可以自动重连。WebSocket 更适合双向实时交互，比如聊天或协作编辑。这里用 SSE 是按需求选最轻方案。

### Q9. 进度为什么写库而不是直接推内存事件？

**回答：**
写库让进度源唯一。任务 runner、SSE 接口、前端刷新页面都从数据库读同一份状态。进程重启、浏览器断线、多个页面同时看进度都不会丢。内存事件虽然实时，但可靠性和恢复能力差。

### Q10. 这个 Agent 产出的数据怎么回流？

**回答：**
ASR 转写、精彩片段标题、爆点原因、成片文案都可以回流数据中台。后续可以作为销售话术素材、直播培训素材、短视频选题库，也可以进入多模态素材中心做视频片段检索。

## 项目四：自媒体智能内容运营 Agent

### 项目定位

用 Agent 自动完成选题、文案、配图、发布素材生成，服务私域销售和自媒体增长。重点不是单次生成，而是可编排、可复用、可评测的内容生产流水线。

### Q1. 为什么用 LangGraph/SubGraph，而不是一个普通流程？

**回答：**
内容生产不是线性脚本，包含选题、资料检索、文案生成、审校、配图、改写、发布等多个可复用节点。SubGraph 能把复杂流程拆成独立子图，每个子图有自己的输入输出和失败重试。这样后续新平台、新风格、新审核规则可以复用节点，而不是改一个巨大脚本。

### Q2. Agent 工作流怎么处理失败重试？

**回答：**
每个节点要定义可重试和不可重试错误。比如模型超时、图片生成接口 429 可以指数退避重试；内容违规、输入缺字段属于业务错误，直接进入人工处理。节点输出结构化结果，失败时记录 error_code 和上下文，方便断点续跑。

### Q3. 配图批量生成为什么要异步并行？

**回答：**
图片生成是 IO 密集且耗时长，如果串行生成 10 张图，用户等待会非常久。我把文案和图片 prompt 先生成出来，再并发调用图片模型，控制并发上限防止限流。失败的单张图可独立重试，不影响整篇内容。

### Q4. 多模型动态路由怎么做？

**回答：**
按任务复杂度和成本路由：简单改写、标题扩写走便宜快模型；策略规划、长文生成、审核判断走强模型；图片 prompt 优化可用中等模型。路由不是只看价格，还看历史质量、超时率、token 成本和任务风险。

### Q5. 怎么保证便宜模型不拉低质量？

**回答：**
有兜底和评测。便宜模型输出后先做规则校验和 LLM judge，低于阈值就升级到强模型重跑。对高风险节点，比如最终发布文案和合规审查，不让便宜模型单独决策。

### Q6. 内容质量怎么评估？

**回答：**
分生成前和生成后。生成前看选题是否匹配目标人群、是否有销售场景；生成后看标题吸引力、信息密度、转化动作、平台违规风险、风格一致性。上线后再看阅读、停留、私信、留资等业务指标，把优质内容回流成样本。

### Q7. 如何避免生成内容同质化？

**回答：**
选题层做去重，避免重复主题；Prompt 层注入不同人设、场景和角度；生成后用 embedding 检测与历史内容相似度，过高则要求重写。内容库越大，越需要用相似度和标签控制多样性。

### Q8. 这个项目和素材中心怎么联动？

**回答：**
Agent 生成或选择的图片、视频、文案都可以进入素材中心，打标签、做 OCR/VLM/ASR 增强后，变成可检索的多模态资产。客服遇到类似问题时，可以直接召回这些图文视频作为证据或案例。

### Q9. 怎么处理平台合规和敏感词？

**回答：**
发布前做两层检查：规则层检查敏感词、夸大承诺、价格底线、医疗金融等禁区；模型层判断语义风险，比如变相保就业、绝对化承诺。命中风险后自动改写，改写不了就转人工审核。

### Q10. 如果重做这个项目，你会优化什么？

**回答：**
我会更早抽象出统一的内容资产 schema，把选题、脚本、图片、视频、发布结果都结构化存储。这样后续复盘爆款、训练选题模型、回流素材中心会更顺畅，而不是先做功能后补数据结构。

## 项目五：基于 MCP 协议的多 Agent 共享服务集群

### 项目定位

把多个 Agent 都会用到的能力沉淀成共享服务：LLM 网关、RAG、记忆、用户画像、Prompt 管理、工具调用。MCP 提供统一协议，REST 网关负责对业务系统友好接入。

### Q1. MCP 是什么？为什么不用自定义 RPC？

**回答：**
MCP 可以理解为 Agent 与外部工具/资源交互的标准协议。相比自定义 RPC，它让工具发现、参数 schema、资源读取、调用返回都有统一规范。后续新增工具时，Agent 不需要硬编码每个接口细节，只要按协议发现和调用。

### Q2. 为什么还要 REST 到 MCP 的桥接网关？

**回答：**
因为业务系统和前端更习惯 REST，而 Agent 内部适合 MCP。桥接网关把 REST 请求转换成 MCP 调用，同时统一做鉴权、限流、日志、Trace ID、错误码和模型路由。它是业务世界和 Agent 工具世界之间的适配层。

### Q3. LLM 网关解决什么问题？

**回答：**
统一模型调用入口，屏蔽不同厂商 API 差异。上层只传任务类型、消息和约束，网关负责选择模型、处理重试、限流、熔断、fallback、成本统计和日志。这样业务不用散落一堆 OpenAI、DeepSeek、豆包的调用代码。

### Q4. 双层记忆怎么设计？

**回答：**
短期对话记忆按 session 隔离，保证当前对话上下文连贯；长期用户画像按 user_id 共享，沉淀跨项目的稳定事实，比如学历、预算、关注点、历史异议。session 记忆解决「这轮聊到哪了」，画像解决「这个人长期是什么样」。

### Q5. 用户画像共享会不会串数据？

**回答：**
要做租户、用户、项目边界。画像必须绑定 tenant_id/user_id，敏感字段分级存储，Agent 只能读自己权限范围内的画像。写入时用 UPSERT 幂等更新，记录来源和更新时间，避免无来源事实污染画像。

### Q6. Agent 无状态、状态下沉是什么意思？

**回答：**
Agent 进程本身只做编排，不保存关键状态。对话历史在记忆服务，知识在 RAG 服务，Prompt 在 Prompt 中心，用户画像在画像服务。这样 Agent 可以水平扩容，挂了重启不丢数据，业务逻辑和基础设施也解耦。

### Q7. 多 Agent 调共享服务，如何保证并发安全？

**回答：**
数据库层用唯一键和 UPSERT 保证幂等，比如同一个 `(tenant_id, user_id, fact_key)` 只保留一条画像事实。需要抢占式任务时用行锁。大部分画像是事实累加型，不是强事务转账，所以幂等写入 + 版本号 + 更新时间足够。

### Q8. 链路变长后怎么排查问题？

**回答：**
必须有全链路 Trace ID。一次用户请求从业务 API、网关、MCP 服务、LLM 调用、RAG 检索、记忆读写都带同一个 trace_id。日志里记录模型、token、耗时、召回片段、错误码。否则多 Agent 系统一出问题很难定位。

### Q9. 怎么做模型 fallback？

**回答：**
按错误类型分：超时、限流、服务不可用可以切换备用模型；内容安全拒绝不能简单 fallback，需要改写或转人工；质量低可以升级强模型重试。fallback 要记录原因，否则会掩盖主模型长期不稳定的问题。

### Q10. MCP 集群对其他项目的价值是什么？

**回答：**
它把重复能力平台化。客服、考核、自媒体、素材中心都不用各自实现 LLM 调用、RAG、记忆和 Prompt 管理，只需要调用共享服务。这样新 Agent 接入速度更快，质量和成本也更可控。

## 项目六：多模态素材中心 & AI 销售考核系统

### 项目定位

素材中心把文字、图片、视频统一成可检索资产；AI 考核系统消费数据中台的结构化话术，自动出题、评分和复盘，服务销售培训。

### Q1. 图片和视频怎么做语义检索？

**回答：**
核心是万物归一到文本向量空间。用户问题是文本，所以图片和视频也先转成可检索文本：图片用标题、标签、人工描述、OCR、VLM caption；视频用标题、ASR 转写、摘要、标签。然后统一 embedding 到 `rag_assets` 表，用 modality 区分 text/image/video。

### Q2. 为什么不直接用 CLIP 多模态向量？

**回答：**
CLIP 可以做图文对齐，但中文销售长尾问法和业务术语不一定稳定，还要引入新模型、新维度、新索引。文本描述方案能复用现有 embedding 和 pgvector，成本低、可解释、可人工修正。CLIP 可以作为兜底增强，不一定是第一选择。

### Q3. retrieval_text 写得不好怎么办？

**回答：**
所以设计了人工 + 离线增强两条路。人工描述要写成用户会怎么问，比如「大专学员拿到 15k offer 案例」；离线用 OCR 抽图中文字、VLM 生成描述、ASR 转视频口播、LLM 做摘要。人工内容优先，机器内容追加，不覆盖人工判断。

### Q4. 怎么避免图片/视频答非所问？

**回答：**
图片和视频比文字更重，所以阈值更保守。比如 text 相似度 0.35 可用，image 0.40，video 0.42；数量也限制，图片最多 2-3 张，视频最多 1 个。低置信度不发媒体，宁缺毋滥。

### Q5. 长视频命中后怎么定位到具体片段？

**回答：**
离线 ASR 时保留时间戳，把长视频按话题或 60 秒窗口切成多个 rag_assets，每条的 media_url 带 `#t=秒数`。命中后前端播放器直接跳到对应时间点，用户不用自己拖进度。

### Q6. 素材隐私怎么保证？

**回答：**
展示 URL 永远使用打码后的素材。OCR/VLM 可以读取原图来保证识别质量，但写入 `media_url` 时只写 masked 版本。检索系统只负责召回，不应该把原始敏感图片暴露给前端。

### Q7. AI 考核怎么出题？

**回答：**
从 `knowledge_articles` 里选场景化话术素材，让 LLM 扮演销售培训考官生成题目。题目按难度分级，包含客户背景、问题场景、考察点和参考答案。小数据量时直接 SQL 取 top 素材，大数据量再引入 RAG 检索和 MMR 去重。

### Q8. AI 考核怎么评分？

**回答：**
把题目、参考答案、学员回答一起给模型，按结构化 rubric 打分，比如需求识别、异议处理、价值表达、成交推进、合规红线。评分 temperature 要低，保证稳定。AI 分只是初评，关键考试保留人工复核。

### Q9. 为什么出题 temperature 高、评分 temperature 低？

**回答：**
出题需要多样性，temperature 可以高一点，比如 0.7，避免每次题目都一样。评分需要一致性，同一答案不能今天 60 明天 90，所以用 0.2-0.3，并配明确分档和 JSON 输出约束。

### Q10. 数据量到十万条，考核出题怎么演进？

**回答：**
先让 LLM 或业务规则把模糊需求拆成具体主题，比如价格异议、学历顾虑、竞品对比；再对每个主题做 RAG 召回；之后用 rerank 和 MMR 保证相关性和多样性；最后拼 prompt 出题。模糊请求下，RAG 不是第一步，第一步是规划检索方向。

## 项目七：基于聊天数据的大模型微调

### 项目定位

用数据中台导出的高质量销售对话做 SFT/DPO，让 AI 客服学会真人销售的话术风格、接话方式和转化策略。具体事实知识仍由 RAG 管理。

### Q1. 为什么要微调，Prompt + RAG 不够吗？

**回答：**
Prompt + RAG 能解决知识准确，但很难稳定解决风格。销售话术里的语气、节奏、追问方式、异议处理结构，需要模型内化。单靠长 prompt 会增加延迟，也容易在复杂对话中漂移。微调让模型默认就按销售风格说话，RAG 再补事实。

### Q2. 为什么「微调管风格，RAG 管知识」？

**回答：**
风格相对稳定，适合进权重；知识变化快，适合放知识库。课程价格、优惠政策、班型、活动时间如果微调进模型，更新成本高且容易答错。微调学「怎么说」，RAG 提供「说什么事实」。

### Q3. 微调数据怎么从聊天记录变成训练样本？

**回答：**
从 approved conversations 导出，先做训练专用清洗：角色规整、human 先开口、assistant 收尾、去掉空轮次、脱敏价格和隐私、过滤过短过长、统一风格。再转换成 ShareGPT 或 OpenAI messages 格式。核心是保证每条样本都是高质量多轮对话。

### Q4. 为什么本地小模型先验证？

**回答：**
直接云端训大模型成本高、反馈慢。本地用 Qwen2.5-3B + QLoRA 快速跑一轮，可以验证数据清洗是否正确、风格能否学到、loss 是否异常、输出是否崩。方向对了再上云端豆包做生产训练。

### Q5. QLoRA 和 LoRA 区别是什么？

**回答：**
LoRA 是冻结基座权重，只训练低秩适配矩阵，节省训练参数。QLoRA 进一步把基座模型量化到 4bit，显著降低显存。本地 4090 资源有限适合 QLoRA，云端资源更充足可以用标准 LoRA 追求更好精度。

### Q6. SFT 和 DPO 各解决什么问题？

**回答：**
SFT 解决「会不会说」，让模型模仿高质量销售对话；DPO 解决「说得好不好」，用 chosen/rejected 偏好对让模型偏向更好的回答。SFT 是模仿，DPO 是偏好优化。

### Q7. 为什么不用 RLHF？

**回答：**
RLHF 要训练 reward model，再用 PPO 等强化学习方法优化，流程长、成本高、稳定性要求高。我们偏好数据量有限，DPO 可以直接用 chosen/rejected 训练，省掉 reward model，更适合中小规模业务落地。

### Q8. 怎么评估微调效果？

**回答：**
三层评估：自动指标用于快速排除坏版本；人工盲评看话术自然度、风格一致性、转化引导；线上 A/B 看客户回复率、有效会话时长、咨询到试听转化。最有说服力的是业务指标，但上线前必须先过离线评测。

### Q9. 微调会不会过拟合销售话术？

**回答：**
会有风险，所以要控制数据多样性、去重、验证集和早停。过拟合的表现是回答模板化、只会固定话术、遇到新问题不会转弯。解决办法是保留多场景样本、加入反例和拒答样本、用 RAG 补知识，不让模型死记具体事实。

### Q10. 微调后模型怎么接回客服系统？

**回答：**
把微调模型作为 LLM 层替换或灰度接入，RAG、Prompt、工具调用链路不变。先小流量 A/B，对比基座模型的回复质量、延迟、转化指标和投诉率。如果效果稳定，再逐步扩大流量。灰度期间保留回滚到基座模型的能力。

## 压迫式通用追问

### Q1. 这些指标真实吗？面试官问你怎么测的。

**回答抓手：**
不要只说结果，要说测量口径。例如 TTFT 是从请求进入 LLM 服务到首 token 返回；检索提速是在相同 top_k、相同数据量、相同机器下对比；转化率是线上 A/B 分流，不是离线主观判断。讲清楚口径，指标才可信。

### Q2. 如果只能讲一个最有含金量的项目，你讲哪个？

**回答抓手：**
讲数据中台或实时客服。数据中台体现架构底座和闭环思维；实时客服体现 RAG、推理性能、语音链路、评测和业务指标。根据岗位选择：偏平台讲中台，偏 AI 应用讲客服。

### Q3. 你最大的技术取舍是什么？

**回答抓手：**
强调「按规模选方案」。百条知识不用 RAG，直接 SQL 全量灌入；单机长任务不用 MQ，PG 队列足够；图片检索先不用 CLIP，用文本描述向量化更稳；价格不进微调，交给 RAG。体现工程判断，而不是技术堆砌。

### Q4. 你怎么保证 AI 系统安全可控？

**回答抓手：**
数据侧脱敏和审核，检索侧置信度阈值，生成侧 Prompt 红线和输出校验，业务侧转人工和日志追踪，模型侧离线评测和灰度发布。安全不是一个开关，是多层防线。

### Q5. 如果线上效果变差，你怎么排查？

**回答抓手：**
先分层定位：数据是否更新、检索是否召回错、Prompt 是否变更、模型是否切换、外部 API 是否异常、用户流量是否变化。用 trace_id 看单次请求链路，用离线评测看版本回归，用业务指标看整体趋势。
