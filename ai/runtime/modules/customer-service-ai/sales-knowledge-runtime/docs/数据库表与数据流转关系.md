# 数据库表与数据流转关系

## 一、所有表都在同一个数据库里

本项目**只有一个数据库**（PostgreSQL 或 SQLite），所有表都在这个库里。不存在"从一个数据库搬到另一个数据库"的情况。

唯一的外部数据源是微信本地的 SQLite 文件（`MSG0-5.db`、`MicroMsg.db`），ETL 把数据从这些文件**导入**到我们的应用数据库后，后续所有操作都在同一个库内完成。

## 三、数据流转详解（一条消息的一生）

```
                    微信 SQLite 文件
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
     MicroMsg.db      MSG0.db ... MSG5.db
     (通讯录)          (聊天记录分库)
          │              │
          ▼              ▼
    ┌─────────┐    ┌──────────┐
    │contacts │    │raw_chats │  ←── ETL 写入（原始内容不动）
    │sessions │    │          │
    └─────────┘    └──────────┘
                         │
                         │ AdminService.preprocess_session()
                         │ 内部做：过滤 → 脱敏 → 切分 → 合并 → 预标注
                         ▼
               ┌───────────────────┐
               │staging_conversations│  ←── status=pending
               │                   │
               │  人工审核在这里：   │
               │  • approve        │  → status=approved ✓
               │  • reject         │  → status=rejected ✗
               │  • 编辑标签/内容   │  → 更新字段，status 不变
               └───────────────────┘
                         │
           ┌─────────────┼─────────────┬────────────────┐
           ▼             ▼             ▼                ▼
     导出训练数据   knowledge_chunks  knowledge_articles  labeled_conversations
     (JSON 文件)    (向量搜索)        (结构化知识)        (可选归档)
```

## 四、每张表的字段说明

### 第1层：`raw_chats`（原始消息）

| 字段 | 说明 | 谁写入 |
|------|------|--------|
| content | 原始消息文本 | ETL（不再修改） |
| status | `pending` / `rejected` | 一键清洗时标记（被过滤的标 rejected） |
| clean_content | 人工修改后的内容 | 预留字段，目前不常用 |
| auto_category | 自动分类 | 一键清洗时填入 |
| auto_flags | 自动标记 | 一键清洗时填入 |

> **注意**：`raw_chats.status` 是对**单条消息**的标记，表示"这条消息在清洗时是否被过滤掉了"。这不是人工审核。

---

### 第2层：`staging_conversations`（对话块暂存区）⭐ 核心工作表

| 字段 | 说明 | 谁写入 |
|------|------|--------|
| original_text | 原始对话文本（含原始内容） | 一键清洗 |
| cleaned_text | 脱敏后的对话文本 | 一键清洗 |
| conversation_json | 结构化对话 `[{role, content, sender_name, msg_id}]` | 一键清洗 |
| session_id | 来源会话 | 一键清洗 |
| source_message_ids | 对应的 raw_chats.id 列表 | 一键清洗 |
| start_time / end_time | 对话块时间范围 | 一键清洗 |
| auto_question | AI 提取的问题 | 一键清洗 |
| auto_answer | AI 总结的答案 | 一键清洗 |
| auto_category | 自动分类 | 一键清洗 |
| auto_quality_score | 自动质量分（0-10） | 一键清洗 |
| auto_flags | 自动标记 | 一键清洗 |
| **human_question** | 人工修正的问题 | **人工审核** |
| **human_answer** | 人工修正的答案 | **人工审核** |
| **human_category** | 人工分类 | **人工审核** |
| **human_notes** | 人工备注 | **人工审核** |
| **status** | `pending` → `approved` / `rejected` | **人工审核** |
| reviewed_by | 审核人 | 人工审核 |
| reviewed_at | 审核时间 | 人工审核 |

**这张表就是人工审核的工作区。审核 = 修改这张表的字段，不会创建新记录或新表。**

---

### 第3层：下游产出表

#### `knowledge_chunks`（向量知识库）

| 字段 | 说明 | 数据来源 |
|------|------|----------|
| content_block | 对话块文本 | staging_conversations.cleaned_text |
| embedding | 1024维向量 | embedding 模型生成 |
| topic_summary | 主题摘要 | auto_question 或截取 |
| chunk_type | `chat` / `labeled` | 区分来源 |

#### `knowledge_articles`（结构化知识条目）

| 字段 | 说明 | 数据来源 |
|------|------|----------|
| scene | 销售场景描述 | LLM 从 chunk/staging 提炼 |
| customer_says | 客户典型说法 | LLM 提炼 |
| recommended_response | 推荐话术 | LLM 提炼 |
| embedding | 向量 | scene + customer_says 生成 |

#### `labeled_conversations`（归档标注）

| 字段 | 说明 | 数据来源 |
|------|------|----------|
| conversation_text | 对话文本 | staging.cleaned_text 复制 |
| conversation_json | 结构化对话 | staging.conversation_json 复制 |
| human_category | 人工分类 | staging 复制 |
| human_quality | 质量评级 | 根据 auto_quality_score 推算 |

> 这张表是可选的归档步骤（`publish_to_production`），当前主流程不依赖它。

#### `custom_conversations`（手工高质量数据）

| 字段 | 说明 | 数据来源 |
|------|------|----------|
| conversation_json | 对话内容 | 人工手写 / 导入 |
| category | 分类 | 人工指定 |
| quality | 质量等级 | 默认 high |

> 不经过清洗流程，直接参与导出训练数据。

## 五、常见疑问澄清

### Q1：人工审核的 approve/reject 数据存在哪？

**答**：就在 `staging_conversations` 表，`status` 字段从 `pending` 改为 `approved` 或 `rejected`，其他人工填写的字段（`human_category`、`human_question` 等）也在这同一条记录上更新。

### Q2：raw_chats 上的 status 和 staging_conversations 上的 status 有什么区别？

| | raw_chats.status | staging_conversations.status |
|--|------------------|------------------------------|
| 粒度 | 单条消息 | 一个对话块（多条消息组成） |
| 谁改 | 机器自动标记（一键清洗时） | 人工审核 |
| 含义 | "这条消息能不能用" | "这段对话是否通过审核" |
| 值 | pending / rejected | pending / approved / rejected |

### Q3：数据会从一个数据库搬到另一个数据库吗？

**答**：不会。唯一跨库的操作是 ETL（微信 SQLite → 应用数据库）。之后所有表都在同一个 PostgreSQL（或 SQLite）里，数据只是在不同表之间流转。

### Q4：清洗后的数据最终去哪？

```
staging_conversations (approved)
    │
    ├→ 导出为 JSON/JSONL 文件 → 用于 LLM 微调训练
    ├→ 写入 knowledge_chunks → 用于 RAG 语义搜索
    ├→ 写入 knowledge_articles → 用于结构化话术库
    └→ (可选) 复制到 labeled_conversations → 归档
```

## 六、表与表之间的关联关系

```
contacts ─── wxid ──→ raw_chats.sender_wxid
sessions ─── session_id ──→ raw_chats.session_id
                              │
                              │ source_message_ids (ID 列表)
                              ▼
                     staging_conversations
                              │
                              │ content 复制
                              ▼
                    ┌─── knowledge_chunks
                    │         │
                    │         │ source_chunk_id
                    │         ▼
                    │    knowledge_articles
                    │
                    └─── labeled_conversations (可选)
```

所有关联都是**逻辑外键**（存 ID 列表或 session_id 字符串），没有数据库层面的物理外键约束。
