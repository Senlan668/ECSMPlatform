# 第 13 章：共享服务六 — 工具注册中心（Tool Registry）

> 一句话：从"代码里硬编码 Tool"到"动态注册、动态发现、动态调用"，让 Tool 成为可管理的资产。

---

## 13.1 问题：Tool 的管理困境

```
初期（3 个 Server，10 个 Tool）：
  没问题，代码里写死就行

中期（10 个 Server，50 个 Tool）：
  → 哪些 Tool 可用？去翻各个 Server 的代码
  → 两个 Server 注册了同名 Tool？调用冲突
  → 某个 Tool 下线了？调用方不知道，报错了才发现

后期（20 个 Server，200 个 Tool）：
  → 完全失控
```

---

## 13.2 Tool 注册中心的角色

```
                  ┌──────────────────┐
                  │  Tool 注册中心    │
                  │                  │
   Server 启动时  │  ┌────────────┐  │  Client 查询时
   ─────────────► │  │ Tool 注册表│  │ ◄──────────────
   注册自己的 Tool │  │            │  │  获取可用 Tool
                  │  │ name       │  │  列表和路由
                  │  │ description│  │
                  │  │ schema     │  │
                  │  │ server_url │  │
                  │  │ version    │  │
                  │  │ scope      │  │
                  │  └────────────┘  │
                  └──────────────────┘
```

---

## 13.3 注册、发现、调用流程

```
阶段 1：注册
  Server A 启动 → 向注册中心注册自己的 Tool 列表
  Server B 启动 → 向注册中心注册自己的 Tool 列表

阶段 2：发现
  项目请求到达网关 → 网关查询注册中心
  → "当前项目可用的 Tool 有哪些？"
  → 注册中心返回列表（根据权限过滤）

阶段 3：调用
  LLM 选择 Tool → 网关查注册中心获取路由
  → "query_database 在哪个 Server？"
  → 注册中心返回 Server 地址
  → 网关转发请求
```

---

## 13.4 公共 Tool vs 项目私有 Tool

```
Tool 的可见性：

  公共 Tool（所有项目可用）：
    ├── chat_completion    (LLM 网关)
    ├── rag_search         (RAG 服务)
    ├── memory_recall      (记忆服务)
    └── send_email         (通知服务)

  项目私有 Tool（仅特定项目可用）：
    ├── proj_A: process_refund       (客服退款)
    ├── proj_A: escalate_ticket      (工单升级)
    ├── proj_B: generate_quote       (报价生成)
    └── proj_B: update_crm           (CRM 更新)
```

注册中心在返回 Tool 列表时，根据 `project_id` 过滤：

```
项目 A 看到的 Tool = 公共 Tool + proj_A 私有 Tool
项目 B 看到的 Tool = 公共 Tool + proj_B 私有 Tool
```

## 13.5 版本管理

```
query_database
  ├── v1.0  只支持 SELECT
  ├── v2.0  增加了 INSERT/UPDATE 支持
  └── v3.0  增加了事务支持（不兼容 v2 的参数格式）

策略：
  - v2 和 v3 并行运行
  - 新项目默认用 v3
  - 旧项目保持 v2，按计划迁移
  - v1 标记为 deprecated，6 个月后下线

注册中心维护版本映射：
  proj_A → query_database@v2
  proj_B → query_database@v3
  proj_C → query_database@v3
```

## 13.6 Tool 描述规范

注册中心需要一套**标准化的 Tool 描述规范**，确保所有 Tool 的质量一致：

```yaml
name: query_database
version: "3.0"
scope: public
category: data
owner: platform-team
description: |
  执行 SQL 查询。支持 SELECT/INSERT/UPDATE/DELETE。
  当用户需要查询、修改业务数据时使用。
  注意：DELETE 操作需要用户确认。
input_schema:
  type: object
  properties:
    sql:
      type: string
      description: "SQL 语句"
    confirm_destructive:
      type: boolean
      description: "是否确认执行破坏性操作（DELETE/UPDATE）"
  required: [sql]
tags: [database, query, data]
sla:
  max_latency_ms: 5000
  availability: 99.9%
```

> **关键**：description 和 tags 决定了 LLM 能否正确发现和使用这个 Tool。注册中心应该对描述质量做门禁检查。
