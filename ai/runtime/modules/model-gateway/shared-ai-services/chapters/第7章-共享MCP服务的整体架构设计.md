# 第 7 章：共享 MCP 服务的整体架构设计

> 一句话：四层架构 —— 网关层做认证路由，服务层放共享 MCP Server，基础能力层对接模型和存储，存储层持久化数据。

## 7.2 网关层：所有请求的入口

MCP 网关是整个架构的**咽喉**，所有项目的请求都经过它：

```
项目请求 → 网关
              ├── 1. 认证：这个请求来自哪个项目？有权限吗？
              ├── 2. 路由：应该转发给哪个 MCP Server？
              ├── 3. 限流：这个项目的配额用完了吗？
              ├── 4. 日志：记录请求和响应（全链路 Trace）
              └── 5. 转发：将请求发送给目标 Server
```

网关**不做业务逻辑**，只做横切关注点（Cross-Cutting Concerns）。

### 网关的协议角色

```
项目 A                  MCP 网关                   LLM Server
(MCP Client)                                     (MCP Server)
    │                      │                          │
    │── MCP 请求 ─────────►│                          │
    │  (SSE/HTTP)          │── 转发（可以是 stdio）──►│
    │                      │                          │
    │                      │◄── 返回结果 ─────────────│
    │◄── MCP 响应 ─────────│                          │
    │                      │                          │
```

网关对外暴露 SSE/Streamable HTTP（让项目通过网络访问），对内可以用 stdio 启动 Server（简化部署）。

## 7.3 服务注册与发现

当共享 Server 数量增多，需要一个机制让网关知道"有哪些 Server、它们在哪里、提供什么能力"：

```
┌──────────────┐
│  服务注册表   │
│              │
│ LLM 网关     │ → host:port, tools:[chat,summarize,translate]
│ RAG 服务     │ → host:port, tools:[search,upload], resources:[knowledge://]
│ 记忆服务     │ → host:port, tools:[save,recall]
│ Prompt 中心  │ → host:port, prompts:[code_review,translate]
│              │
└──────────────┘

启动时：Server 注册自己
运行时：网关查表路由
下线时：Server 注销
健康检查：定期探活，自动摘除不健康节点
```

实现方式从简到繁：

| 方案 | 适合规模 | 说明 |
|---|---|---|
| 静态配置文件 | 3-5 个 Server | 网关配置里写死地址，够用就行 |
| Consul / etcd | 5-20 个 Server | 动态注册发现，健康检查 |
| Kubernetes Service | 大规模 | 利用 K8s 原生服务发现 |

> **务实建议**：初期用静态配置就够了。不要过度设计。

## 7.4 多租户设计

多个项目共享同一个 Server，必须解决**数据隔离**：

```
项目 A 的知识库 ≠ 项目 B 的知识库
项目 A 的记忆 ≠ 项目 B 的记忆
项目 A 的配额 ≠ 项目 B 的配额
```

隔离策略：

```
方案 1：命名空间隔离（推荐起步方案）
  每个请求带 project_id，数据按 project_id 分区
  同一个数据库/向量库，通过字段隔离

  knowledge_base 表：
  | id | project_id | content | vector |
  | 1  | proj_A     | ...     | ...    |
  | 2  | proj_B     | ...     | ...    |

方案 2：实例隔离
  每个项目独立的数据库/向量库实例
  隔离性最强，成本最高

方案 3：混合方案
  核心数据（知识库、记忆）→ 实例隔离
  配置数据（Prompt、Tool 注册）→ 命名空间隔离
```

## 7.5 通信协议选择

| 协议 | 优点 | 缺点 | 推荐场景 |
|---|---|---|---|
| **SSE** | MCP 原生支持，实现简单 | 单向流（Server→Client），需配合 HTTP POST | 默认选择 |
| **Streamable HTTP** | MCP 新标准，支持断线重连 | SDK 支持还在完善中 | 生产环境首选 |
| **WebSocket** | 全双工，低延迟 | MCP 未原生支持，需自行适配 | 高频实时场景 |
| **gRPC** | 高性能，强类型 | MCP 未原生支持，需适配层 | 内部服务间通信 |

**推荐**：对外用 SSE/Streamable HTTP（标准 MCP 协议），服务内部通信可以用 gRPC 提升性能。

---

## 7.6 部署模式

```
模式 1：单体部署（适合起步）
  所有 MCP Server 部署在同一台机器/容器
  ✅ 简单  ❌ 无法独立扩展

模式 2：微服务部署（适合中等规模）
  每个 MCP Server 独立容器，用 K8s 编排
  ✅ 独立扩展  ❌ 运维复杂度上升

模式 3：Serverless（适合低频场景）
  按需启动，用完即停（AWS Lambda / 阿里云函数计算）
  ✅ 成本最低  ❌ 冷启动延迟

推荐演进路径：
  单体起步 → 拆分高负载服务 → 逐步微服务化
  不要一开始就搞全套微服务
```

## 本章小结

```
四层架构 = 网关层 → 服务层 → 基础能力层 → 存储层
网关职责 = 认证 + 路由 + 限流 + 日志（不做业务逻辑）
服务发现 = 初期静态配置，后期按需引入注册中心
多租户  = 命名空间隔离起步，核心数据按需升级为实例隔离
协议    = 对外 SSE/Streamable HTTP，内部可选 gRPC
部署    = 单体起步，按需拆分，不要过度设计
```
