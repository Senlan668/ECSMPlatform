# 第 6 章：如何识别和抽象通用 MCP 服务？

> 一句话：用"复用频率 × 稳定性 × 无业务耦合"三个维度判断，从最痛的点开始抽象，粒度不要太细也不要太粗。

---

## 6.1 三维判断标准

一个 MCP Server 是否值得抽象为共享服务，用三个问题判断：

```
问题 1：复用频率 — 多少个项目需要它？
  ≥3 个项目 → 高复用 ✅
  2 个项目  → 中复用 ⚠️
  1 个项目  → 不需要共享 ❌

问题 2：稳定性 — 接口是否经常变化？
  接口稳定，半年不改 → 适合共享 ✅
  每周都在改接口    → 暂不适合 ❌

问题 3：业务耦合 — 是否包含特定业务逻辑？
  纯通用能力（查DB、调LLM）→ 适合共享 ✅
  包含业务规则（工单分配逻辑）→ 不适合 ❌
```

三个维度都满足 = **立即抽象**。满足两个 = **评估后决定**。只满足一个 = **暂不抽象**。

## 6.3 分类矩阵

把清单按"通用 vs 业务"和"高频 vs 低频"放入四象限：

```
          高复用频率
              │
    ┌─────────┼─────────┐
    │ ①立即抽象│②优先抽象 │
    │         │         │
    │ LLM调用 │ RAG检索  │
    │ 数据库   │ 会话记忆 │
    │         │ 文件操作 │
通用┼─────────┼─────────┼业务
    │ ③可选   │ ④不抽象  │
    │         │         │
    │ 语义缓存 │ CRM对接  │
    │ Prompt库│ 工单逻辑 │
    │ 日志服务 │ 业务规则 │
    │         │         │
    └─────────┼─────────┘
              │
          低复用频率
```

## 6.4 确定第一批共享服务

**不要一口气抽象所有东西。** 按优先级分批：

```
第一批（解决最痛的问题）：
  1. LLM 统一网关    ← 解决 Key 散落 + 模型切换 + 成本管控
  2. RAG 即服务      ← 解决最大的重复开发量

第二批（完善基础设施）：
  3. 会话记忆服务    ← 多项目统一记忆管理
  4. Prompt 管理中心 ← 团队协作和最佳实践沉淀

第三批（优化和治理）：
  5. 语义缓存        ← 降低成本
  6. Tool 注册中心   ← 动态发现和管理

同步建设（贯穿全程）：
  7. 日志与可观测性  ← 从第一天就要有
  8. 认证与鉴权      ← 从第一天就要有
```

## 6.5 粒度把控

### 太粗的问题

```
把所有能力塞进一个巨型 MCP Server：
  "AI-Everything-Server"
  ├── LLM 调用
  ├── RAG 检索
  ├── 数据库查询
  ├── 文件操作
  ├── 邮件发送
  └── ……

问题：单点故障、无法独立扩展、代码耦合、部署困难
```

### 太细的问题

```
每个功能都拆成独立 Server：
  llm-chat-server
  llm-summary-server
  llm-translate-server
  rag-search-server
  rag-upload-server
  rag-rerank-server
  ……

问题：Server 爆炸、管理复杂、调用链过长、运维成本高
```

### 正确的粒度

```
按"领域"划分，一个领域一个 Server：

  LLM 网关（所有 LLM 相关）
  ├── chat_completion
  ├── summarize
  ├── translate
  └── embed

  RAG 服务（所有知识库相关）
  ├── search
  ├── upload
  ├── rerank
  └── list_collections

  记忆服务（所有记忆相关）
  ├── save
  ├── recall
  ├── summarize_history
  └── clear
```

**判断原则**：同一个 Server 内的 Tool 是否**共享同一批基础资源**（同一个数据库、同一个模型、同一套配置）？如果是，放在一起。如果不是，拆开。

---

## 本章小结

```
判断标准 = 复用频率 × 稳定性 × 无业务耦合
分类工具 = 四象限矩阵
落地策略 = 分批推进，第一批解决最痛的问题
粒度原则 = 按领域划分，不要太粗（巨石）也不要太细（碎片）
```
