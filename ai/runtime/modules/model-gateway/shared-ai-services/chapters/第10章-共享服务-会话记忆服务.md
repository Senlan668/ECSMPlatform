# 第 10 章：共享服务三 — 会话记忆服务

> 一句话：把会话历史和用户偏好统一管理，支持跨会话、跨项目的记忆召回，让 AI 真正"记住"用户。

---

## 10.1 为什么记忆需要跨项目？

```
用户在客服系统说："我对花生过敏"
  → 客服系统记住了

用户在销售系统说："推荐个零食"
  → 销售系统不知道用户过敏信息
  → 推荐了花生酥 💀
```

如果记忆只存在单个项目中，用户在不同项目间的体验是**割裂的**。共享记忆服务让所有项目共享同一份用户认知。

---

## 10.2 记忆的两层设计

```
┌──────────────────────────────────────┐
│             短期记忆                   │
│  当前会话的对话历史（最近 N 轮）       │
│  存储：Redis（快、自动过期）           │
│  生命周期：会话结束后归档或丢弃        │
└──────────────────┬───────────────────┘
                   │ 摘要 / 提取
                   ▼
┌──────────────────────────────────────┐
│             长期记忆                   │
│  用户偏好、关键事实、历史摘要          │
│  存储：PostgreSQL + 向量数据库        │
│  生命周期：持久化，按需清理            │
└──────────────────────────────────────┘
```

---

## 10.3 MCP 接口设计

```
Tool: memory_save
  输入：user_id, content, memory_type(short/long), metadata
  作用：保存一条记忆

Tool: memory_recall
  输入：user_id, query, memory_type, limit
  作用：召回相关记忆（语义检索 or 时间排序）

Tool: memory_summarize
  输入：user_id, session_id
  作用：将一次会话压缩为长期记忆摘要

Tool: memory_clear
  输入：user_id, memory_type, before_date
  作用：清除指定记忆

Resource: memory://user/{user_id}/profile
  作用：用户的长期画像摘要（只读）
```

---

## 10.4 记忆的压缩策略

对话越长，记忆越多，直接全量塞给 LLM 会**撑爆上下文窗口**。需要压缩：

```
策略 1：滑动窗口
  只保留最近 20 轮对话，更早的丢弃
  ✅ 简单  ❌ 丢失重要历史信息

策略 2：摘要压缩
  每 20 轮对话用 LLM 生成摘要，替代原文
  ✅ 保留关键信息  ❌ 摘要可能丢失细节

策略 3：分层策略（推荐）
  最近 10 轮：保留原文
  更早的：压缩为摘要
  关键事实：提取为长期记忆

  ┌─────────────────────────────────┐
  │ 长期记忆（关键事实）    占 10%   │
  │ "用户对花生过敏"                │
  │ "偏好中文回答"                 │
  ├─────────────────────────────────┤
  │ 历史摘要               占 20%   │
  │ "上周讨论了退货流程…"           │
  ├─────────────────────────────────┤
  │ 最近对话原文           占 70%   │
  │ [最近 10 轮完整对话]           │
  └─────────────────────────────────┘
```

## 10.5 跨项目记忆的权限模型

不是所有项目都应该看到所有记忆：

```
记忆分级：
  ├── 全局记忆（用户基本偏好）  → 所有项目可读
  ├── 领域记忆（购物偏好）      → 相关项目可读
  └── 私有记忆（会话细节）      → 仅产生项目可读

权限矩阵：
  │              全局记忆   购物记忆   客服会话   销售会话
  │ 客服系统      ✅ 读      ✅ 读     ✅ 读写    ❌
  │ 销售系统      ✅ 读      ✅ 读写   ❌         ✅ 读写
  │ 知识问答      ✅ 读      ❌        ❌         ❌
```

---

## 10.6 检索方式

| 方式 | 适合场景 | 实现 |
|---|---|---|
| **时间排序** | "最近聊了什么" | ORDER BY created_at DESC |
| **语义检索** | "用户说过过敏吗" | 向量相似度搜索 |
| **混合** | 综合召回 | 时间加权 + 语义相似度 |

> **推荐**：短期记忆用时间排序，长期记忆用语义检索。
