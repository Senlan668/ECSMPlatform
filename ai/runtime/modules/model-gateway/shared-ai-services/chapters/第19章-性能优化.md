# 第 19 章：性能优化

> 一句话：LLM 调用延迟是大头（占 80%+），优化重点是减少不必要的调用、缩短调用链、用异步和缓存加速。

---

## 19.1 延迟分解

先搞清楚时间花在哪里：

```
一次典型请求的延迟分解：

  网关处理          20ms   ████
  记忆召回          50ms   ██████████
  RAG 检索         100ms   ████████████████████
  LLM 推理         800ms   ████████████████████████████████████████████（大头）
  结果返回          10ms   ██
  ─────────────────────────
  总计             980ms

结论：LLM 推理占 80%+ 的延迟
→ 优化 LLM 调用次数和速度是关键
→ 网关层优化几毫秒意义不大
```

---

## 19.2 关键优化手段

### ① 减少 LLM 调用次数

```
语义缓存（第 12 章）  → 相似问题直接返回缓存
结果预计算            → 高频固定问题预先生成答案
智能路由              → 简单问题不调 LLM，直接规则匹配
```

### ② 加速单次 LLM 调用

```
流式输出              → 首字节延迟从 2s 降到 200ms（体感快很多）
Prompt 压缩           → 减少 input token，直接减少推理时间
模型降级              → 简单任务用小模型（4o-mini 比 4o 快 3-5 倍）
```

### ③ 并行化

```
串行（慢）：                    并行（快）：
  记忆召回 50ms                   ┌── 记忆召回 50ms
  ↓                               │
  RAG 检索 100ms                  ├── RAG 检索 100ms
  ↓                               │
  数据库查询 70ms                  └── 数据库查询 70ms
  ↓                                    │
  总计 220ms                      总计 100ms（取最长的那个）
                                  
  然后再调 LLM                    然后再调 LLM
```

能并行的环节一定要并行。

### ④ 连接池与长连接

```
短连接（每次新建）：
  建立连接 30ms + 请求 50ms = 80ms

长连接（复用）：
  请求 50ms

MCP Server 之间用连接池，避免重复建立 TCP/TLS 连接。
SSE 连接保持长连接，不要每次请求都重新握手。
```

---

## 19.3 异步处理

不是所有操作都需要同步等待：

```
同步（必须等结果）：
  ✅ LLM 推理（用户等着看回答）
  ✅ RAG 检索（LLM 需要上下文）
  ✅ 数据库查询（LLM 需要数据）

异步（不需要等结果）：
  ✅ 记录日志 → 写入消息队列，异步入库
  ✅ 保存记忆 → 异步写入
  ✅ 发送通知 → 异步发送
  ✅ 更新缓存 → 异步更新
  ✅ 评估打分 → 异步执行

架构：
  主流程（同步）：查询 → LLM → 返回用户
  副流程（异步）：日志/记忆/通知/评估 → 消息队列 → 后台 Worker
```

---

## 19.4 批量推理

当多个请求可以合并时：

```
请求 1："翻译：Hello"        ──┐
请求 2："翻译：World"        ──┼── 合并为一次 LLM 调用
请求 3："翻译：Good morning" ──┘

合并前：3 次 LLM 调用，3 × 800ms = 2400ms
合并后：1 次 LLM 调用，1 × 1000ms = 1000ms

适用场景：
  ✅ 批量数据处理（分类、提取、翻译）
  ❌ 实时对话（用户在等，不能攒批）
```

## 19.5 性能基准测试

```
建立基准：

  测试场景           目标延迟    目标 QPS
  简单问答           < 1s       50
  RAG 知识问答       < 2s       30
  多 Tool 调用       < 3s       20
  文件生成           < 5s       10

测试方法：
  工具：k6 / locust / wrk
  持续时间：10 分钟稳定负载
  逐步加压：10 → 20 → 50 → 100 QPS
  观察：延迟 P50/P95/P99、错误率、资源使用率

定期运行（每周 / 每次发版前），对比历史数据，发现性能退化。
```
