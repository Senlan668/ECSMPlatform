# 第 15 章：调用日志与可观测性

> 一句话：从用户请求到 LLM 返回的全链路 Trace，统一的日志格式，关键指标监控和告警。

---

## 15.1 全链路 Trace

一次用户请求可能穿越多个 MCP Server，需要一个 `trace_id` 串联：

```
用户问："我的订单到哪了？"

trace_id: t-20250115-abc123
│
├── [网关] 接收请求         0ms
├── [记忆服务] recall       +50ms    → 召回用户历史
├── [数据库] query_orders   +120ms   → 查询订单状态
├── [LLM 网关] chat         +800ms   → GPT-4o 推理
│   ├── input_tokens: 520
│   ├── output_tokens: 180
│   └── model: gpt-4o
├── [网关] 返回响应         +850ms
│
总耗时：850ms
总 Token：700
```

每个环节都带上同一个 `trace_id`，出问题时可以**一键追踪整条链路**。

---

## 15.2 统一日志格式

所有 MCP Server 使用统一的结构化日志格式：

```json
{
  "trace_id": "t-20250115-abc123",
  "span_id": "span-002",
  "parent_span_id": "span-001",
  "timestamp": "2025-01-15T10:30:00.120Z",
  "service": "database-server",
  "method": "tools/call",
  "tool": "query_orders",
  "project_id": "proj_A",
  "duration_ms": 70,
  "status": "success",
  "metadata": {
    "sql": "SELECT * FROM orders WHERE user_id = ?",
    "rows_returned": 3
  }
}
```

> 用 OpenTelemetry 标准，兼容 Jaeger / Zipkin / Grafana Tempo 等追踪系统。

## 15.4 告警规则

| 告警 | 条件 | 级别 |
|---|---|---|
| LLM 成功率下降 | 5 分钟内 < 95% | 🔴 严重 |
| P99 延迟飙升 | 超过 5 秒 | 🟡 警告 |
| Token 消耗异常 | 1 小时内超过日预算 50% | 🟡 警告 |
| 单项目 QPS 激增 | 超过配额 200% | 🟡 警告 |
| MCP Server 不可用 | 健康检查连续失败 3 次 | 🔴 严重 |
| API Key 余额不足 | 余额 < $100 | 🟡 警告 |

## 15.5 日志存储方案

```
热数据（最近 7 天）：
  Elasticsearch / ClickHouse
  → 快速查询、全文搜索

温数据（7-90 天）：
  对象存储（S3 / MinIO）+ Parquet 格式
  → 成本低，按需加载分析

冷数据（90 天以上）：
  归档存储
  → 合规需求保留，极少访问

推荐技术栈：
  日志采集：OpenTelemetry Collector
  存储查询：ClickHouse（性价比最高）
  可视化：Grafana
  追踪：Grafana Tempo 或 Jaeger
```
