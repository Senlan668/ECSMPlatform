# 第 5 章：单项目 MCP 架构的痛点

> 一句话：当多个 AI 项目各搞一套 MCP，重复开发、Key 散落、日志缺失、成本失控等问题会迅速恶化。

---

## 5.1 痛点全景

假设你的公司有 3 个 AI 项目：智能客服、销售助手、内部知识问答。每个项目都有自己的 MCP 体系：

```
项目 A（智能客服）               项目 B（销售助手）             项目 C（知识问答）
├── LLM Server (GPT-4)         ├── LLM Server (GPT-4)        ├── LLM Server (Claude)
├── RAG Server                  ├── RAG Server                 ├── RAG Server
├── DB Server                   ├── DB Server                  ├── DB Server
├── Email Server                ├── CRM Server                 └── File Server
└── Memory Server               └── Calendar Server
```

看到问题了吗？**前三个 Server 每个项目都有，而且功能几乎相同。**

---

## 5.2 六大痛点

### ① 重复造轮子

```
RAG Server：项目 A 写了一个，项目 B 又写了一个，项目 C 再写一个
三个版本实现不同、质量不同、bug 不同

同样的事还发生在：LLM 调用、数据库查询、文件操作……
```

**成本**：3 个项目 × 5 个通用 Server = 15 套重复代码。团队把 60% 的时间花在造轮子上。

### ② API Key 散落

```
项目 A 的 .env：OPENAI_API_KEY=sk-xxx-aaa
项目 B 的 .env：OPENAI_API_KEY=sk-xxx-bbb
项目 C 的 .env：ANTHROPIC_API_KEY=sk-ant-ccc

→ Key 散落在各项目、各服务器、各开发者电脑上
→ 离职交接丢失、Key 泄露无法快速轮换
→ 无法统一管控哪个项目用了多少额度
```

### ③ 没有统一日志

```
项目 A 用 print 打日志
项目 B 用 Python logging
项目 C 压根没记日志

出了问题：
"用户说回答不对" → 哪个项目？哪个 Tool？什么参数？LLM 返回了什么？
→ 无法追踪，只能猜
```

### ④ 模型切换要改 N 个项目

```
需求："GPT-4 太贵了，普通查询切到 GPT-4o-mini"

→ 改项目 A 的 LLM Server
→ 改项目 B 的 LLM Server
→ 改项目 C 的 LLM Server
→ 分别测试、分别部署
→ 下次想试试 Claude？再来三遍……
```

### ⑤ Token 成本失控

```
月底账单：OpenAI $2,000 + Anthropic $800

→ 哪个项目花了多少？不知道
→ 哪些查询是浪费的？不知道
→ 有没有重复查询可以缓存？不知道
→ 预算快超了能不能限流？做不到
```

### ⑥ 知识和能力无法沉淀

```
项目 A 团队优化了一个很好的 RAG 检索策略
项目 B 团队完全不知道
项目 C 团队在踩同样的坑

→ 最佳实践无法跨项目传播
→ 每个团队都在独立试错
```

---

## 5.3 核心问题：哪些是通用的？哪些是业务的？

把所有 MCP Server 放进一个二维矩阵：

```
                    通用（所有项目都需要）
                         │
              ┌──────────┼──────────┐
              │          │          │
              │  LLM调用  │  RAG检索 │
              │  会话记忆  │  文件操作│
              │  数据库查询│  日志记录│
   稳定 ──────┼──────────┼──────────┼────── 易变
   (少变化)   │          │          │      (常变化)
              │  Prompt  │  CRM对接 │
              │  模板库   │  工单系统 │
              │          │  业务规则 │
              │          │          │
              └──────────┼──────────┘
                         │
                    业务特有
```

**左上角（通用 + 稳定）= 最应该抽象为共享服务的能力**

```
第一优先级（通用 + 稳定）：
  ✅ LLM 统一网关
  ✅ RAG 检索服务
  ✅ 会话记忆服务
  ✅ Prompt 管理中心

第二优先级（通用但有一定变化）：
  ✅ 数据库查询（需要多租户隔离）
  ✅ 文件操作（需要权限控制）
  ✅ 语义缓存

保留在项目内（业务特有）：
  ❌ CRM 对接逻辑
  ❌ 特定业务规则
  ❌ 项目专属的工作流编排
```

---

## 5.4 从痛点到架构需求

每个痛点对应一个架构需求：

| 痛点 | 架构需求 |
|---|---|
| 重复造轮子 | → 通用能力抽象为共享 MCP 服务 |
| API Key 散落 | → 统一的密钥管理和 LLM 网关 |
| 没有统一日志 | → 集中式日志和可观测性平台 |
| 模型切换改 N 个项目 | → LLM 网关统一路由，一处切换全局生效 |
| Token 成本失控 | → 按项目/用户计量 + 配额 + 语义缓存 |
| 能力无法沉淀 | → Prompt 管理中心 + Tool 注册中心 |

```
现状                              目标
────                              ────
N 个项目 × M 个 Server            共享服务层 + 项目业务层
= N×M 套重复代码                  = M 个共享服务 + N 个轻量业务层

散落的 API Key                    → 统一密钥管理
各自的日志                        → 集中可观测性
各自选模型                        → 统一 LLM 网关
无成本管控                        → 计量 + 配额 + 缓存
```
