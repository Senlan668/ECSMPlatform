# 第 18 章：高可用与容灾设计

> 一句话：共享服务是所有项目的命脉，必须设计水平扩展、多区域部署、优雅降级和灰度发布能力。

---

## 18.1 单点故障风险

共享服务架构的最大风险：**一个 Server 挂了，所有项目都受影响。**

```
之前（各搞各的）：
  项目 A 的 RAG 挂了 → 只影响项目 A

之后（共享服务）：
  共享 RAG 服务挂了 → 项目 A/B/C/D 全挂 💀
```

共享带来效率，但也带来了**爆炸半径扩大**的风险。必须用高可用设计来对冲。

---

## 18.2 水平扩展

每个共享 MCP Server 至少部署 **2 个实例**，通过负载均衡分发请求：

```
              负载均衡器
             ╱    │    ╲
            ╱     │     ╲
  LLM 网关-1  LLM 网关-2  LLM 网关-3
  RAG 服务-1  RAG 服务-2
  记忆服务-1  记忆服务-2

扩展策略：
  CPU > 70%  → 自动扩容（加实例）
  CPU < 30%  → 自动缩容（减实例）
  QPS > 阈值 → 自动扩容
```

关键要求：**Server 必须是无状态的**（状态放在 Redis/数据库中），才能随意增减实例。

## 18.3 多区域部署

```
场景：公司在北京和上海都有团队

  北京区域                     上海区域
  ┌──────────────┐           ┌──────────────┐
  │ MCP 网关     │           │ MCP 网关     │
  │ LLM 网关 ×2 │           │ LLM 网关 ×2 │
  │ RAG 服务 ×2 │           │ RAG 服务 ×2 │
  │ Redis        │ ◄──同步──► │ Redis        │
  │ 向量DB       │ ◄──同步──► │ 向量DB       │
  └──────────────┘           └──────────────┘
         │                          │
    北京团队就近访问            上海团队就近访问

DNS 就近路由：
  北京用户 → 北京区域
  上海用户 → 上海区域
  某区域故障 → 自动切换到另一区域
```

> **务实建议**：小团队不需要多区域。单区域多实例已经能覆盖大多数场景。

---

## 18.4 优雅降级

当核心依赖不可用时，不是直接报错，而是**降低服务质量但继续服务**：

```
降级策略清单：

  LLM 不可用：
    GPT-4o 挂了 → 自动切到 Claude
    所有云模型挂了 → 切到本地小模型（质量降低但能用）
    所有模型都挂了 → 返回预设的兜底回复

  RAG 不可用：
    向量库挂了 → 退化为关键词搜索
    搜索全挂了 → LLM 直接回答（不带知识库上下文）

  记忆服务不可用：
    Redis 挂了 → 无记忆模式（当前会话内短期记忆仍可用）

  网关不可用：
    主网关挂了 → DNS 切到备用网关
    所有网关挂了 → 项目可以配置本地 Fallback（直连 LLM API）
```

降级不是故障，是**有计划的退让**。每个降级策略都要提前设计和测试。

---

## 18.5 灰度发布与蓝绿部署

```
蓝绿部署（适合大版本更新）：

  蓝（当前版本）──── 100% 流量
  绿（新版本）────── 0% 流量，待命
                     │
  验证通过后：        │
                     ▼
  蓝（旧版本）──── 0% 流量（保留回滚用）
  绿（新版本）──── 100% 流量

灰度发布（适合渐进式更新）：

  v2.0 ──── 90% 流量
  v2.1 ──── 10% 流量（先让部分项目试用）
              │
  验证没问题：  │
              ▼
  v2.0 ──── 0%
  v2.1 ──── 100%

灰度维度：
  ├── 按项目：项目 A 先用新版本
  ├── 按流量比例：10% → 30% → 50% → 100%
  └── 按用户：内部用户先用
```
