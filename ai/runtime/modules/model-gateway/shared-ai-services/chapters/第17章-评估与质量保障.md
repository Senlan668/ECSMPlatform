# 第 17 章：评估与质量保障（Eval）

> 一句话：建立统一的评估体系，自动化测量 AI 输出质量，收集用户反馈形成闭环，每次变更做回归测试。

---

## 17.1 为什么需要统一评估？

```
没有统一评估时：
  "换了个 Prompt，效果好像变好了？" → 凭感觉
  "升级了模型，用户满意度变了吗？"   → 不知道
  "RAG 检索结果准确吗？"            → 偶尔看几条

有了统一评估：
  每次变更前后有量化对比
  每个项目的质量指标持续追踪
  自动发现质量退化并告警
```

---

## 17.2 评估指标体系

```
┌─────────────────────────────────────────────────────┐
│                   评估指标分层                        │
├──────────────┬──────────────────────────────────────┤
│              │                                      │
│  LLM 输出    │  准确性（事实正确吗？）               │
│  质量        │  相关性（回答了问题吗？）              │
│              │  完整性（信息齐全吗？）               │
│              │  安全性（有害内容？幻觉？）            │
│              │                                      │
│  RAG         │  检索准确率（Top-5 命中率）           │
│  质量        │  检索召回率（相关文档都找到了吗？）     │
│              │  答案归因（回答是否基于检索结果？）     │
│              │                                      │
│  Tool 调用   │  调用准确率（调对了吗？）              │
│  质量        │  参数正确率（参数对吗？）              │
│              │  执行成功率（执行成功了吗？）          │
│              │                                      │
│  用户体验    │  满意度评分（👍👎）                    │
│              │  任务完成率                           │
│              │  平均对话轮数（越少越好）              │
│              │                                      │
└──────────────┴──────────────────────────────────────┘
```

---

## 17.3 自动化评估方法

```
方法 1：LLM-as-Judge（用 LLM 评估 LLM）
  准备测试集：[问题, 参考答案] × N
  让 LLM A 生成回答
  让 LLM B（评估者）打分
  ✅ 扩展性好  ⚠️ 评估者自身有偏差

方法 2：规则检查
  输出是否包含必要字段？
  输出长度是否在合理范围？
  是否包含有害/敏感内容？
  JSON 格式是否正确？
  ✅ 确定性强  ❌ 只能检查表面质量

方法 3：人工抽检
  每天随机抽取 50 条对话
  人工标注质量评分
  ✅ 最准确  ❌ 成本高、速度慢

推荐：规则检查（全量）+ LLM-as-Judge（全量）+ 人工抽检（抽样）
```

---

## 17.4 用户反馈闭环

```
用户对话结束
  │
  ▼
展示评价按钮（👍👎 + 可选文字反馈）
  │
  ▼
反馈入库 → 关联 trace_id
  │
  ▼
定期分析：
  ├── 差评集中在哪个 Tool？→ 针对性优化
  ├── 差评集中在哪类问题？→ 补充知识库
  ├── 差评关联哪个 Prompt 版本？→ 回滚或优化
  └── 差评率趋势变化？→ 是否有系统性退化

    反馈 → 分析 → 优化 → 验证 → 反馈
              闭环
```

---

## 17.5 变更回归测试

每次修改 Prompt、切换模型、更新 RAG 策略之前：

```
变更前：
  跑测试集 → 得到基线分数（如 85 分）

变更后：
  跑同一个测试集 → 得到新分数

对比：
  新分数 ≥ 基线  → ✅ 允许发布
  新分数 < 基线  → ❌ 阻止发布，需要分析原因

可以集成到 CI/CD：
  git push → 触发评估 → 评估通过 → 自动部署
                        评估失败 → 阻止部署
```
