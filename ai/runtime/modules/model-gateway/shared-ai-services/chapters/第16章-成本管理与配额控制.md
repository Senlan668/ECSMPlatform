# 第 16 章：成本管理与配额控制

> 一句话：按项目/团队/用户三个维度计量 Token 消耗，设预算告警，配额限流，用缓存和模型降级优化成本。

---

## 16.1 多维度 Token 计量

```
每次 LLM 调用记录：
  {
    project_id: "proj_A",
    team: "customer-service",
    user_id: "user_123",
    model: "gpt-4o",
    input_tokens: 520,
    output_tokens: 180,
    cost: $0.0088,
    timestamp: "2025-01-15T10:30:00Z"
  }

聚合维度：
  按项目：项目 A 本月消耗 $500
  按团队：客服团队本月消耗 $800
  按用户：user_123 本月消耗 $12
  按模型：GPT-4o 占 60%，GPT-4o-mini 占 35%，Claude 占 5%
  按 Tool：chat_completion 占 80%，summarize 占 15%
```

---

## 16.2 预算与告警

```
预算体系：

  公司月总预算：$5,000
    │
    ├── 项目 A（客服）：$2,000/月
    │   ├── 日预算：~$67
    │   └── 告警：达到 80% 时告警，100% 时限流
    │
    ├── 项目 B（销售）：$1,500/月
    │
    ├── 项目 C（知识库）：$1,000/月
    │
    └── 预留（新项目/突发）：$500/月

告警规则：
  │ 日消耗 > 日预算 80%   → 通知项目负责人
  │ 月消耗 > 月预算 90%   → 通知部门负责人
  │ 月消耗 > 月预算 100%  → 自动降级（切到便宜模型）
  │ 月消耗 > 月预算 120%  → 自动限流（拒绝非关键请求）
```

---

## 16.3 配额与限流

```
限流维度：

  项目级：项目 A 最多 100 QPS
  用户级：单用户最多 10 QPS
  模型级：GPT-4o 全局最多 50 QPS（受 API 限制）
  Tool级：send_email 每天最多 1000 次

限流算法：
  令牌桶（Token Bucket）
  → 允许短时突发，但长期平均不超过配额

超额处理：
  ├── 排队等待（延迟增加但不拒绝）
  ├── 降级到便宜模型（GPT-4o → GPT-4o-mini）
  └── 拒绝并返回友好提示
```

---

## 16.4 成本优化手段

```
              成本优化漏斗

  原始请求      10,000 次/天       $100/天
      │
  ① 语义缓存命中 35%               -$35
      │
  ② 模型降级                       -$20
     简单查询用 mini 模型
      │
  ③ Prompt 压缩                    -$10
     精简 system prompt
     压缩历史消息
      │
  ④ 批量推理                       -$5
     合并相似请求
      │
  实际 LLM 成本                    $30/天
                                   节省 70%
```

各手段效果对比：

| 手段 | 节省比例 | 实现难度 | 对质量的影响 |
|---|---|---|---|
| 语义缓存 | 20-40% | 中 | 几乎无 |
| 模型降级 | 15-30% | 低 | 轻微降低 |
| Prompt 压缩 | 10-20% | 低 | 几乎无 |
| 批量推理 | 5-15% | 高 | 增加延迟 |
| 结果预计算 | 10-30% | 中 | 无（但只适合固定场景） |
