# 第 9 章：共享服务二 — RAG 即服务

> 一句话：把"文档上传 → 解析 → 向量化 → 存储 → 检索"整条管线封装为共享 MCP Server，所有项目按 collection 隔离。

---

## 9.1 为什么 RAG 最值得共享？

RAG（Retrieval-Augmented Generation）是 AI 项目中**重复建设最严重**的模块：

```
每个 RAG 系统都要解决：
  ✦ 文档解析（PDF/Word/网页 → 纯文本）
  ✦ 文本分块（chunk 策略）
  ✦ 向量化（选 Embedding 模型）
  ✦ 向量存储（选向量数据库）
  ✦ 检索策略（向量检索 / 混合检索 / Rerank）

→ 每个项目都在重复解决这 5 个问题
→ 且每个项目的解决质量参差不齐
```

---

## 9.2 架构设计

```
                    RAG MCP Server
                    │
  ┌─────────────────┼─────────────────┐
  │                 │                 │
  ▼                 ▼                 ▼
上传管线          检索引擎          管理接口
  │                 │                 │
  ├─ 文档解析       ├─ 向量检索       ├─ 创建集合
  ├─ 智能分块       ├─ 关键词检索     ├─ 列出集合
  ├─ 向量化         ├─ 混合检索       ├─ 删除文档
  └─ 入库           └─ Rerank 重排    └─ 查看统计
  │                 │
  ▼                 ▼
┌─────────────────────────────────────┐
│  Embedding 服务    向量数据库        │
│ (统一/可切换)     (Milvus/Qdrant)    │
└─────────────────────────────────────┘
```

MCP Tool 接口：

```
Tool: rag_upload       → 上传文档到指定集合
Tool: rag_search       → 语义检索
Tool: rag_hybrid_search → 混合检索（向量 + 关键词）
Tool: rag_list_collections → 列出当前项目的所有集合

Resource: rag://collections/{name}/stats → 集合统计信息
```

## 9.3 文档处理管线

```
上传 PDF
  │
  ▼
① 解析（PDF→文本）
  ├─ 简单文档：PyPDF2 / pdfplumber
  ├─ 复杂排版：Unstructured / LlamaParse
  └─ 扫描件：OCR（Tesseract / 云服务）
  │
  ▼
② 分块（文本→chunks）
  ├─ 固定长度分块（512 tokens，重叠 50）
  ├─ 语义分块（按段落/标题自然切分）
  └─ 递归分块（先大块再细分）
  │
  ▼
③ 向量化（chunks→vectors）
  └─ 调用统一 Embedding 服务
     （OpenAI text-embedding-3-small / BGE / Jina）
  │
  ▼
④ 入库（vectors→向量数据库）
  └─ 带上 metadata：project_id, collection, source, chunk_index
```

### 分块策略选择

| 策略 | 适合 | 不适合 |
|---|---|---|
| 固定长度 | 通用场景，简单可靠 | 可能切断完整段落 |
| 语义分块 | 结构化文档（有标题/段落） | 纯文本流水内容 |
| 递归分块 | 长文档 | 短文档 |

> **务实建议**：起步用固定长度（512 tokens, overlap 50），80% 的场景够用。

## 9.4 检索策略

```
用户查询 → "我们的退货政策是什么？"
               │
               ▼
         ┌─────┴──────┐
         │             │
    向量检索        关键词检索
   (语义相似)      (BM25 精确匹配)
         │             │
         └─────┬───────┘
               │
          混合融合（RRF）
               │
               ▼
          Rerank 重排
     (用交叉编码器精排)
               │
               ▼
        Top-K 结果返回
```

| 策略 | 优点 | 缺点 |
|---|---|---|
| 纯向量检索 | 语义理解强 | 可能丢失精确关键词匹配 |
| 纯关键词检索 | 精确匹配好 | 不理解语义 |
| **混合检索** | 两者优点兼得 | 实现稍复杂 |
| **混合 + Rerank** | 效果最好 | 多一次模型调用，延迟增加 |

> **推荐**：混合检索作为默认策略，对精度要求高的场景加 Rerank。

## 9.5 多租户隔离：Collection 模型

```
向量数据库
  ├── proj_A_产品手册      ← 项目 A 私有
  ├── proj_A_FAQ           ← 项目 A 私有
  ├── proj_B_销售话术      ← 项目 B 私有
  ├── shared_公司制度      ← 全局共享
  └── shared_技术文档      ← 全局共享
```

权限模型：

```
项目 A 能访问：proj_A_* + shared_*
项目 B 能访问：proj_B_* + shared_*
管理员能访问：所有
```

通过 `project_id` + Collection 前缀实现隔离，查询时自动过滤。

---

## 9.6 向量数据库选型

| 方案 | 特点 | 推荐场景 |
|---|---|---|
| **Milvus** | 功能最全，支持混合检索 | 大规模生产环境 |
| **Qdrant** | Rust 编写，性能好，API 友好 | 中等规模，快速上手 |
| **Weaviate** | 内置多模态支持 | 需要多模态检索 |
| **pgvector** | PostgreSQL 插件，运维简单 | 数据量 <100 万，想减少组件 |
| **Chroma** | 超轻量，嵌入式 | 本地开发和原型 |

> **务实建议**：数据量小用 pgvector（不引入新组件），数据量大用 Qdrant 或 Milvus。
