# 记忆服务架构与流程解析

## 这个服务解决什么问题

LLM 本身没有记忆，每次调用都是一张白纸。不把历史喂回去，它连上一句说了什么都不知道：

```
没有记忆服务：
  用户：我要专业版          用户：多少钱？
  AI：  好的，哪款产品？    AI：  请问您在问什么的价格？ ← 忘了"专业版"

有了记忆服务：
  用户：我要专业版          用户：多少钱？
  AI：  好的，哪款产品？    AI：  专业版每月 299 元。   ← 记住了上文
```

但光记住"当前这轮对话"还不够。用户在客服项目说了"我芒果过敏"，到了零食推荐项目就不知道了——因为两个项目的对话历史是隔离的。

所以记忆服务提供**两层能力**：

| 层次 | 解决什么 | 隔离粒度 | 生命周期 |
|------|---------|---------|---------|
| **对话记忆** | 让 AI 记住当前对话上下文 | project_id + session_id | 跟随会话，可清空 |
| **用户画像** | 让所有项目共享用户偏好/事实 | 仅 user_id | 跟随用户，长期存在 |

## 它在多项目架构中的角色

```
                        四个共享 MCP 服务
                ┌────────────────────────────────────────┐
                │                                        │
项目 A ────→    │  ① LLM 网关    → 统一调用大模型          │
(智能客服)      │  ② RAG 服务    → 检索知识库              │
                │  ③ 记忆服务    → 对话记忆 + 用户画像  ◀── 本文档
项目 B ────→    │  ④ Prompt 中心 → 管理提示词模板          │
(写作助手)      │                                        │
                └────────────────────────────────────────┘
```

### 客服项目处理一次提问——记忆服务在哪里被调用

```
用户："专业版多少钱？"
        │
┌─ 项目 A (agent.py) ─────────────────────────────────────────────┐
│                                                                  │
│  ① recall_memory → 记忆服务                                     │
│     拿回对话历史：["用户: 我要专业版", "AI: 好的，哪款..."]       │
│                                                                  │
│  ② recall_user_facts → 记忆服务                                 │
│     拿回用户画像：[allergy=芒果, budget=月预算500]                │
│                                                                  │
│  ③ search_knowledge → RAG 服务                                  │
│  ④ get_prompt → Prompt 中心                                     │
│  ⑤ chat_completion → LLM 网关                                   │
│     → "专业版每月 299 元，年付享 8 折。"                          │
│                                                                  │
│  ⑥ save_memory → 记忆服务（存本轮对话）                         │
│  ⑦ save_user_fact → 记忆服务（如 AI 识别到新的用户偏好）        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**①⑥ 形成对话记忆闭环**，**②⑦ 形成用户画像闭环**。

### 跨项目协作场景

```
项目 A（客服）：
  用户说 "我芒果过敏"
  → save_user_fact(user_id="张三", fact_key="allergy", fact_value="芒果")

项目 B（零食推荐）：
  用户说 "给我推荐零食"
  → recall_user_facts(user_id="张三")
  → 得到 [allergy=芒果]
  → AI 推荐时自动排除芒果类
```

## 两个源文件，各管什么

```
memory_service/
├── server.py    # 入口：注册 6 个 MCP Tool，接收请求后委托 store
└── store.py     # 存储：SQLite 读写，两张表的全部 CRUD 逻辑
```

### server.py（入口与路由）

- 初始化 `FastMCP`（端口 9003）和 `MemoryStore`（指向 `data/sqlite/memory.db`）。
- 注册 6 个 Tool，每个 Tool 只做**打日志 + 调 store 方法**，自己不含业务逻辑。

### store.py（存储引擎）

- 管理两张表：`memory_messages`（对话记忆）和 `user_facts`（用户画像）。
- 线程安全：所有操作在 `threading.Lock` 保护下执行。
- 自动初始化：构造时自动创建目录、建表、建索引。

## 数据模型

### 表 1：`memory_messages`（对话记忆）

```sql
CREATE TABLE IF NOT EXISTS memory_messages (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id TEXT NOT NULL,
    session_id TEXT NOT NULL,
    role       TEXT NOT NULL,        -- "user" 或 "assistant"
    content    TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_memory_ps_id
ON memory_messages (project_id, session_id, id);
```

隔离方式：`project_id + session_id` → 不同项目、不同会话之间数据互不可见。

### 表 2：`user_facts`（用户画像）

```sql
CREATE TABLE IF NOT EXISTS user_facts (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id        TEXT NOT NULL,
    fact_key       TEXT NOT NULL,
    fact_value     TEXT NOT NULL,
    source_project TEXT NOT NULL DEFAULT '',
    updated_at     TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(user_id, fact_key)       -- 同用户同 key 只保留最新值
);
CREATE INDEX IF NOT EXISTS idx_user_facts_uid
ON user_facts (user_id);
```

隔离方式：仅按 `user_id` → **跨项目共享**，任何项目都能读写同一用户的画像。

### 两张表的隔离对比

```
memory_messages 表                    user_facts 表
┌───────────────────────────┐       ┌───────────────────────────┐
│ project="customer-service"│       │ user_id="张三"            │
│ ├── session="s-001" [...]│       │ ├── allergy = 芒果        │ ← 客服写入
│ └── session="s-002" [...]│       │ ├── budget  = 月500以内   │ ← 客服写入
│                           │       │ └── style   = 简洁幽默   │ ← 写作助手写入
│ project="writing-assistant│       │                           │
│ └── session="s-001" [...]│       │ user_id="李四"            │
│   ↑ 和上面的 s-001 互不干扰│       │ └── ...                   │
└───────────────────────────┘       └───────────────────────────┘
    按 project+session 隔离               按 user_id 共享
```

## 6 个 Tool 的详细流程

### 对话记忆：save_memory

```
传入 project_id, session_id, role, content
  │
  ① 校验 role → 必须是 "user"/"assistant"
  ② 校验 content → 不能为空
  ③ 加锁 → INSERT INTO memory_messages → commit → 返回自增 id
  │
返回 {"status":"ok", "id":1, "project_id":..., "session_id":...}
```

### 对话记忆：recall_memory

```
传入 project_id, session_id, last_n(默认10)
  │
  ① 规整 last_n：强制 1 <= n <= 500
  ② SELECT ... WHERE project_id=? AND session_id=?
     ORDER BY id DESC LIMIT ?     ← 先倒序取最新 N 条
  ③ rows.reverse()                ← 再翻转成时间正序
  │
返回 {"messages":[{"role":"user","content":"..."},  ...], "count":N}
```

**先 DESC 再 reverse** 是获取「最近 N 条并正序返回」的经典技巧。返回的 `messages` 可直接拼进 LLM 的 messages 参数。

### 对话记忆：clear_memory

```
传入 project_id, session_id
  │
  DELETE FROM memory_messages WHERE project_id=? AND session_id=?
  │
返回 {"status":"ok", "deleted":4}
```

### 用户画像：save_user_fact

```
传入 user_id, fact_key, fact_value, source_project(可选)
  │
  ① 校验三个必填字段
  ② INSERT ... ON CONFLICT(user_id, fact_key) DO UPDATE
     → 已存在则覆盖 value 和 updated_at，不存在则新增
  │
返回 {"status":"ok", "user_id":..., "fact_key":..., "fact_value":...}
```

关键机制：`UNIQUE(user_id, fact_key)` + `ON CONFLICT DO UPDATE` 实现**幂等写入**——重复调用不会产生重复数据，只更新最新值。

### 用户画像：recall_user_facts

```
传入 user_id
  │
  SELECT fact_key, fact_value, source_project, updated_at
  FROM user_facts WHERE user_id=?
  ORDER BY updated_at ASC
  │
返回 {"user_id":..., "facts":[{"key":"allergy","value":"芒果","source":"customer-service"}, ...], "count":3}
```

任何项目都可调用，不需要传 project_id。

### 用户画像：delete_user_fact

```
传入 user_id, fact_key
  │
  DELETE FROM user_facts WHERE user_id=? AND fact_key=?
  │
返回 {"status":"ok", "deleted":1}
```

## 为什么选 SQLite

| 考量 | SQLite | MySQL / PostgreSQL |
|------|--------|-------------------|
| 安装 | **零安装**，Python 标准库自带 | 需要装数据库服务 |
| 部署 | 数据就是一个文件 `memory.db` | 独立进程 + 配置 |
| 适用场景 | 单机、低并发、教学演示 | 生产环境、高并发 |

首次运行自动创建目录和数据库，真正开箱即用：

```python
self._db_path.parent.mkdir(parents=True, exist_ok=True)
self._init_schema()
```

## 线程安全机制

所有数据库操作遵循同一模式：

```python
with self._lock:              # 全局互斥锁
    conn = self._connect()    # 每次新建连接
    try:
        ...                   # SQL 操作
        conn.commit()
    finally:
        conn.close()          # 用完立即关闭
```

简单可靠，适合低并发场景。如需高并发可换成连接池 + WAL 模式。

## 与其他共享服务的对比

| | 记忆服务 | LLM 网关 | RAG 服务 | Prompt 中心 |
|--|---------|---------|---------|------------|
| **一句话** | 记住说过什么 + 用户是谁 | 统一调用大脑 | 查找参考资料 | 组织提示词 |
| **存储** | SQLite（两张表） | 无状态 | ChromaDB | YAML 文件 |
| **隔离** | 对话: project+session / 画像: user_id | project_id | project_id | 按模板名 |
| **端口** | 9003 | 9001 | 9002 | 9004 |

四个服务合在一起：**记住上下文 → 了解用户 → 查找资料 → 组织语言 → 调用大脑**，就是一个完整的 AI 对话能力底座。
