# 记忆服务调用过程与输入输出

用一个具体的业务场景，把 6 个 Tool 从头到尾走一遍。每一步列出：调了什么、传进去什么、返回什么。

---

## 场景设定

模拟两个项目共用记忆服务的完整过程：

```
角色：用户"张三"
项目 A（智能客服）：张三咨询产品，对话中透露"芒果过敏"
项目 B（零食推荐）：张三要推荐零食，服务读取画像后自动规避芒果
```

MCP 服务地址：`http://127.0.0.1:9003/mcp`

---

## 第一部分：对话记忆（项目 A 内部）

### 第 ① 步：保存用户消息 — save_memory

用户在客服项目中发了一条消息。

**输入**（Tool arguments）：

```json
{
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001",
  "role": "user",
  "content": "你好，我想了解一下你们的零食礼盒，但我芒果过敏，能帮我推荐吗？"
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| project_id | str | 是 | 项目标识，决定数据隔离范围 |
| session_id | str | 是 | 会话标识，同一用户同一对话线程 |
| role | str | 是 | 只接受 `"user"` 或 `"assistant"` |
| content | str | 是 | 消息正文，不能为空 |

**输出**：

```json
{
  "status": "ok",
  "id": 1,
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001"
}
```

| 字段 | 说明 |
|------|------|
| status | `"ok"` 表示成功 |
| id | SQLite 自增主键，天然代表时间顺序 |

**错误情况**：

```json
// role 传了非法值
{"status": "error", "message": "role 必须是 \"user\" 或 \"assistant\""}

// content 为空
{"status": "error", "message": "content 不能为空"}
```

---

### 第 ② 步：保存 AI 回复 — save_memory

客服 Agent 生成了回复，也要存下来。

**输入**：

```json
{
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001",
  "role": "assistant",
  "content": "您好！我们有多款零食礼盒。既然您芒果过敏，我帮您排除含芒果的产品，推荐这三款：坚果礼盒（¥128）、饼干组合（¥98）、果干混合装（¥158，不含芒果干）。"
}
```

**输出**：

```json
{
  "status": "ok",
  "id": 2,
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001"
}
```

注意 `id` 从 1 变成了 2，自增说明时间在后。

---

### 第 ③ 步：用户追问 — 再存一轮

**输入**：

```json
{
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001",
  "role": "user",
  "content": "坚果礼盒里有什么？"
}
```

**输出**：

```json
{"status": "ok", "id": 3, "project_id": "customer-service", "session_id": "sess-zhangsan-001"}
```

---

### 第 ④ 步：召回对话历史 — recall_memory

处理新一轮提问前，Agent 先召回之前的对话作为 LLM 上下文。

**输入**：

```json
{
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001",
  "last_n": 10
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| project_id | str | 是 | — | 项目标识 |
| session_id | str | 是 | — | 会话标识 |
| last_n | int | 否 | 10 | 最多返回条数，上限 500 |

**输出**：

```json
{
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001",
  "count": 3,
  "messages": [
    {
      "role": "user",
      "content": "你好，我想了解一下你们的零食礼盒，但我芒果过敏，能帮我推荐吗？"
    },
    {
      "role": "assistant",
      "content": "您好！我们有多款零食礼盒。既然您芒果过敏，我帮您排除含芒果的产品，推荐这三款：坚果礼盒（¥128）、饼干组合（¥98）、果干混合装（¥158，不含芒果干）。"
    },
    {
      "role": "user",
      "content": "坚果礼盒里有什么？"
    }
  ]
}
```

| 字段 | 说明 |
|------|------|
| messages | 按时间**从早到晚**排列，可直接拼进 LLM 的 `messages` 参数 |
| count | 本次实际返回的条数 |

**内部机制**：先 `ORDER BY id DESC LIMIT 10` 取最新 10 条，再 `reverse()` 翻转成正序。

**空会话输出**（session_id 不存在）：

```json
{
  "project_id": "customer-service",
  "session_id": "non-existent",
  "count": 0,
  "messages": []
}
```

---

### 第 ⑤ 步：清空对话 — clear_memory

会话结束或用户要求"忘掉之前的对话"时调用。

**输入**：

```json
{
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001"
}
```

**输出**：

```json
{
  "status": "ok",
  "project_id": "customer-service",
  "session_id": "sess-zhangsan-001",
  "deleted": 3
}
```

| 字段 | 说明 |
|------|------|
| deleted | 实际删除的消息条数 |

清空后再 `recall_memory` 同一会话，将返回 `count: 0`。只影响该 project_id + session_id 下的数据。

## 第二部分：用户画像（跨项目共享）

### 第 ⑥ 步：客服项目写入用户事实 — save_user_fact

客服 Agent 在对话中识别出"张三芒果过敏"，将这条事实写入用户画像。

**输入**：

```json
{
  "user_id": "zhangsan",
  "fact_key": "allergy",
  "fact_value": "芒果过敏",
  "source_project": "customer-service"
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| user_id | str | 是 | — | 用户标识，跨项目唯一 |
| fact_key | str | 是 | — | 事实类别，如 allergy、budget、style |
| fact_value | str | 是 | — | 事实内容 |
| source_project | str | 否 | `""` | 写入来源，便于溯源 |

**输出**：

```json
{
  "status": "ok",
  "user_id": "zhangsan",
  "fact_key": "allergy",
  "fact_value": "芒果过敏"
}
```

**关键机制——幂等覆盖**：

数据库有 `UNIQUE(user_id, fact_key)` 约束，写入时使用 `ON CONFLICT DO UPDATE`。如果张三已经有一条 `allergy` 记录，再次写入会**覆盖旧值**而不是新增一条：

```json
// 第一次写入
{"user_id": "zhangsan", "fact_key": "allergy", "fact_value": "芒果过敏"}
// → 插入新行

// 第二次写入（更新）
{"user_id": "zhangsan", "fact_key": "allergy", "fact_value": "芒果和花生过敏"}
// → 覆盖旧值，不会出现两条 allergy
```

### 第 ⑦ 步：再写一条 — 预算偏好

同样是客服项目识别出的用户信息。

**输入**：

```json
  "user_id": "zhangsan",
  "fact_key": "budget",
  "fact_value": "月预算 500 以内",
  "source_project": "customer-service"
}
```

**输出**：

```json
{
  "status": "ok",
  "user_id": "zhangsan",
  "fact_key": "budget",
  "fact_value": "月预算 500 以内"
}
```

### 第 ⑧ 步：写作项目也写一条 — 风格偏好

张三在写作助手项目中表达了偏好，项目 B 也往同一个用户画像里写入。

**输入**：

```json
{
  "user_id": "zhangsan",
  "fact_key": "style_preference",
  "fact_value": "喜欢简洁幽默的文风",
  "source_project": "writing-assistant"
}
```

**输出**：

```json
{
  "status": "ok",
  "user_id": "zhangsan",
  "fact_key": "style_preference",
  "fact_value": "喜欢简洁幽默的文风"
}
```

现在 `user_facts` 表中张三有 3 条事实，来自 2 个不同项目。

---

### 第 ⑨ 步：零食推荐项目读取用户画像 — recall_user_facts

项目 B 要给张三推荐零食，先读取他的全部画像。**注意：不需要传 project_id，任何项目都能读。**

**输入**：

```json
{
  "user_id": "zhangsan"
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| user_id | str | 是 | 用户标识 |

**输出**：

```json
{
  "user_id": "zhangsan",
  "count": 3,
  "facts": [
    {
      "key": "allergy",
      "value": "芒果过敏",
      "source": "customer-service",
      "updated_at": "2026-03-31 10:15:00"
    },
    {
      "key": "budget",
      "value": "月预算 500 以内",
      "source": "customer-service",
      "updated_at": "2026-03-31 10:15:01"
    },
    {
      "key": "style_preference",
      "value": "喜欢简洁幽默的文风",
      "source": "writing-assistant",
      "updated_at": "2026-03-31 11:30:00"
    }
  ]
}
```

| 字段 | 说明 |
|------|------|
| facts | 按 `updated_at` 从早到晚排列 |
| key | 事实类别 |
| value | 事实内容 |
| source | 哪个项目写入的，便于溯源 |
| count | 事实总条数 |

**这就是跨项目共享的核心**：客服写的 allergy 和 budget，零食推荐项目直接拿到。零食推荐 Agent 可以把这些 facts 拼进 Prompt，让 LLM 自动避开芒果类产品。

**用户不存在时的输出**：

```json
{
  "user_id": "non-existent-user",
  "count": 0,
  "facts": []
}
```

---

### 第 ⑩ 步：删除过时事实 — delete_user_fact

张三打电话说"我芒果过敏已经好了"，需要删掉这条事实。

**输入**：

```json
{
  "user_id": "zhangsan",
  "fact_key": "allergy"
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| user_id | str | 是 | 用户标识 |
| fact_key | str | 是 | 要删除的事实类别 |

**输出**：

```json
{
  "status": "ok",
  "user_id": "zhangsan",
  "fact_key": "allergy",
  "deleted": 1
}
```

| 字段 | 说明 |
|------|------|
| deleted | 实际删除条数。如果 fact_key 不存在，返回 `0` |

删除后再 `recall_user_facts("zhangsan")`，只剩 2 条（budget 和 style_preference）。

## 完整场景时间线

```
时间轴 ─────────────────────────────────────────────────────→

10:15  项目A（客服）                      11:30  项目B（写作助手）

  用户："我芒果过敏，推荐零食"               用户："帮我写篇文章"
    │                                         │
    ├─ save_memory (对话)                      ├─ save_user_fact
    ├─ save_user_fact (allergy=芒果)           │   (style=简洁幽默)
    ├─ save_user_fact (budget=月500)           │
    ├─ recall_memory → 拿到对话上下文           │
    ├─ chat_completion → 生成回答              │
    └─ save_memory (AI 回复)                   │
                                               │
                              12:00  项目C（零食推荐）
                                               │
                                ├─ recall_user_facts("zhangsan")
                                │   → 拿到 3 条事实（来自项目A和B）
                                │   → allergy=芒果, budget=月500, style=简洁幽默
                                │
                                ├─ 推荐零食时自动排除芒果
                                └─ "推荐：坚果礼盒 ¥128..."
```

## 总结：6 个 Tool 的输入输出速查表

### 对话记忆（按 project + session 隔离）

| Tool | 输入 | 输出 | 说明 |
|------|------|------|------|
| `save_memory` | `project_id` `session_id` `role` `content` | `{"status":"ok", "id":1, ...}` | role 只接受 user/assistant |
| `recall_memory` | `project_id` `session_id` `last_n`(可选,默认10) | `{"messages":[...], "count":N}` | 按时间正序，可直接喂给 LLM |
| `clear_memory` | `project_id` `session_id` | `{"status":"ok", "deleted":N}` | 只清该会话，不影响其他 |

### 用户画像（按 user_id 跨项目共享）

| Tool | 输入 | 输出 | 说明 |
|------|------|------|------|
| `save_user_fact` | `user_id` `fact_key` `fact_value` `source_project`(可选) | `{"status":"ok", ...}` | 同 key 幂等覆盖 |
| `recall_user_facts` | `user_id` | `{"facts":[...], "count":N}` | 任何项目都能读 |
| `delete_user_fact` | `user_id` `fact_key` | `{"status":"ok", "deleted":N}` | 不存在时 deleted=0 |
