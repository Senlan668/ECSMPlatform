# 一次请求的完整生命周期

> 以一次 `chat_completion` 调用为例，追踪请求从客户端进入到豆包 API 返回的每一步，
> 展示每个模块边界上的**实际输入输出数据**。

---

## 场景

项目 A（智能客服）的用户问了一句："你们产品支持哪些功能？"

---

## 全链路总览

```
测试客户端                server.py              router.py          doubao.py           豆包云端 API
    │                        │                      │                  │                    │
    │─── ① call_tool ──────→│                      │                  │                    │
    │                        │─── ② resolve ──────→│                  │                    │
    │                        │←── ③ 返回 ──────────│                  │                    │
    │                        │─── ④ get_config ───→│                  │                    │
    │                        │←── ⑤ 返回 ──────────│                  │                    │
    │                        │─── ⑥ chat ─────────────────────────→│                    │
    │                        │                                       │─── ⑦ POST ───────→│
    │                        │                                       │←── ⑧ 200 OK ─────│
    │                        │←── ⑨ 返回 ──────────────────────────│                    │
    │←── ⑩ 结果 ────────────│                                                           │
```

---

## ① 客户端 → server.py

**动作**：MCP Client 发起 `call_tool`

```python
session.call_tool("chat_completion", {
    "messages": [{"role": "user", "content": "你们产品支持哪些功能？"}],
    "project_id": "customer-service",
    "model": "auto",
    "temperature": 0.7,
    "max_tokens": 2000
})
```

**底层发生了什么**：

MCP SDK 把这个调用序列化为 JSON-RPC 请求，通过 HTTP POST 发送：

```
POST http://127.0.0.1:9001/mcp
Content-Type: application/json

{
    "jsonrpc": "2.0",
    "method": "tools/call",
    "params": {
        "name": "chat_completion",
        "arguments": {
            "messages": [{"role": "user", "content": "你们产品支持哪些功能？"}],
            "project_id": "customer-service",
            "model": "auto",
            "temperature": 0.7,
            "max_tokens": 2000
        }
    },
    "id": 1
}
```

FastMCP 收到后，根据 Tool 名找到 `chat_completion` 函数，将 `arguments` 解包为函数参数。

---

## ② server.py → router.py

**动作**：server.py 拿到 `model="auto"`，问路由器该用哪个模型

```
调用：router.resolve("auto")
输入：model = "auto"
```

**router.py 内部逻辑**：

```
1. model == "auto"  →  命中默认分支
2. 替换为 self.default_model  →  启动时从 config.yaml 读到的值
3. config.yaml 中 routing.default = "doubao-pro"
4. "doubao-pro" 在 self.models 字典中存在  →  校验通过
```

## ③ router.py → server.py

```
输出："doubao-pro"
```

意义：`"auto"` 这个语法糖被解析成了具体的逻辑模型名。

---

## ④ server.py → router.py

**动作**：server.py 拿到模型名，继续获取该模型的参数配置

```
调用：router.get_model_config("doubao-pro")
输入：model = "doubao-pro"
```

**router.py 内部逻辑**：

```
1. resolve("doubao-pro")  →  非 auto，直接校验，返回 "doubao-pro"
2. 从 self.models["doubao-pro"] 取出 config.yaml 中的定义
3. 合并 name 字段返回
```

## ⑤ router.py → server.py

```
输出：{
    "name": "doubao-pro",
    "description": "豆包主力模型，适合复杂推理和长文本生成",
    "max_tokens": 4096,
    "temperature": 0.7
}
```

**server.py 拿到后计算 effective_max_tokens**：

```
min(调用方传入的 2000, 模型配置上限 4096) = 2000

为什么取 min？
- 调用方说"我最多要 2000 个 token" → 尊重调用方意愿
- 模型配置说"这个模型最多支持 4096" → 保护模型不超限
- 两者取较小值，双方都满意
```

---

## ⑤½ server.py 内部：逻辑模型名 → 物理端点的翻译

这是整个链路中最关键的设计点。到这一步，server.py 手里有两样东西：

```
来自 router.py 的：   "doubao-pro"              ← 逻辑模型名（人类语义）
来自 .env 的：        "doubao-seed-1-8-251228"   ← 物理端点 ID（API 实际需要的）
```

**这两个值是在启动时就准备好的，不是实时查询的**：

```python
# server.py 启动阶段（模块级代码，只执行一次）

CHAT_ENDPOINT = os.environ["ARK_CHAT_MODEL"]
#                           ↓
#                 .env 文件内容：
#                 ARK_CHAT_MODEL=doubao-seed-1-8-251228
#                           ↓
# CHAT_ENDPOINT = "doubao-seed-1-8-251228"    ← 启动时就固定了
```

**翻译发生的位置**——就在 server.py 的 `chat_completion` 函数中，第 73 行：

```python
result = await doubao.chat_completion(
    model=CHAT_ENDPOINT,        # ← 这里！不传 "doubao-pro"，而传物理端点
    ...
)
```

**完整的名字变换链路**：

```
调用方传入         router 解析         server.py 翻译为物理端点         doubao.py 发给 API
─────────         ──────────         ─────────────────────          ────────────────────
"auto"      →     "doubao-pro"       ×丢弃，不传给 doubao.py
                                     CHAT_ENDPOINT             →    "doubao-seed-1-8-251228"
```

```
变量流向图：

.env                            server.py                     doubao.py
┌─────────────────────┐         ┌───────────────────────┐     ┌──────────────────┐
│ ARK_CHAT_MODEL=     │         │                       │     │                  │
│ doubao-seed-1-8-    │──启动→──│ CHAT_ENDPOINT =       │──→──│ model 参数 =     │
│ 251228              │  读取   │ "doubao-seed-1-8-     │ 传入│ "doubao-seed-1-8-│
└─────────────────────┘         │  251228"              │     │  251228"         │
                                │                       │     └──────────────────┘
config.yaml                     │                       │
┌─────────────────────┐         │                       │     返回给调用方
│ routing:            │         │                       │     ┌──────────────────┐
│   default:          │──启动→──│ router.resolve()      │──→──│ "model":         │
│   "doubao-pro"      │  读取   │ → "doubao-pro"        │ 附加│ "doubao-pro"     │
└─────────────────────┘         │                       │     └──────────────────┘
                                └───────────────────────┘

                                server.py 是唯一同时接触两个名字的地方
                                它从 router 拿逻辑名，从 .env 拿物理端点
                                向下传物理端点给 doubao.py
                                向上返回逻辑名给调用方
```

**为什么 router 不直接返回物理端点？**

因为物理端点是敏感的环境信息（Endpoint ID 可以用来调用 API），放在 `.env` 不提交 Git。而 `config.yaml` 要提交 Git 给团队共享。如果 config.yaml 里写死端点 ID，就等于把环境信息泄露到代码仓库了。

**当前演示项目的简化**：所有逻辑模型（doubao-pro / doubao-lite）都指向同一个物理端点。生产环境中，可以在 `.env` 配置多个端点变量（如 `ARK_CHAT_MODEL_PRO`、`ARK_CHAT_MODEL_LITE`），由 server.py 根据 router 的解析结果选择对应的端点。

## ⑥ server.py → doubao.py

**动作**：server.py 组装好参数，交给 API 客户端去调用

```
调用：doubao.chat_completion(
    model = "doubao-seed-1-8-251228",       ← 物理端点（来自 .env）
    messages = [{"role": "user", "content": "你们产品支持哪些功能？"}],
    temperature = 0.7,
    max_tokens = 2000                       ← 经过 min() 夹逼后的值
)
```

doubao.py 收到的参数里**没有任何逻辑模型名的痕迹**。它完全不知道 "doubao-pro" 的存在，只看到一个 Endpoint ID，只负责把请求发出去、把结果拿回来。

---

## ⑦ doubao.py → 豆包 API

**动作**：doubao.py 内部通过 OpenAI SDK 发出 HTTP 请求

```
POST https://ark.cn-beijing.volces.com/api/v3/chat/completions
Authorization: Bearer 8fe7ae22-35c9-46d9-...
Content-Type: application/json

{
    "model": "doubao-seed-1-8-251228",
    "messages": [
        {"role": "user", "content": "你们产品支持哪些功能？"}
    ],
    "temperature": 0.7,
    "max_tokens": 2000
}
```

OpenAI SDK 自动处理了：
- 拼接 `base_url` + `/chat/completions` 得到完整 URL
- 把 `api_key` 放入 `Authorization: Bearer ...` 请求头
- 序列化请求体、解析响应体

---

## ⑧ 豆包 API → doubao.py

**动作**：豆包 API 返回 HTTP 200 OK

```json
{
    "id": "chatcmpl-abc123",
    "model": "doubao-seed-1-8-251228",
    "choices": [
        {
            "index": 0,
            "message": {
                "role": "assistant",
                "content": "我们的产品支持以下核心功能：\n1. 智能对话：支持多轮自然语言交互\n2. 知识检索：基于语义的精准搜索\n3. ..."
            },
            "finish_reason": "stop"
        }
    ],
    "usage": {
        "prompt_tokens": 23,
        "completion_tokens": 156,
        "total_tokens": 179
    }
}
```

OpenAI SDK 把这个 JSON 解析成 Python 对象（`ChatCompletion`）。

---

## ⑨ doubao.py → server.py

**动作**：doubao.py 从 SDK 响应中提取关键字段，转为简洁 dict

```
输出：{
    "content": "我们的产品支持以下核心功能：\n1. 智能对话...",
    "usage": {
        "prompt_tokens": 23,
        "completion_tokens": 156,
        "total_tokens": 179
    }
}
```

**doubao.py 做了什么裁剪**：

```
丢弃了：id, model, choices[0].index, choices[0].finish_reason
保留了：choices[0].message.content → 扁平化为 "content"
保留了：usage 三个字段
```

为什么裁剪？上游不需要 SDK 的完整响应结构，只关心"回复了什么"和"花了多少 token"。

**注意**：返回值里**没有**模型名和项目 ID。doubao.py 不关心这些业务信息，它只负责"把请求发出去，把结果拿回来"。

---

## ⑩ server.py → 客户端

**动作**：server.py 在结果上附加元数据，返回给 MCP 客户端

```
最终输出：{
    "content": "我们的产品支持以下核心功能：\n1. 智能对话...",
    "usage": {
        "prompt_tokens": 23,
        "completion_tokens": 156,
        "total_tokens": 179
    },
    "model": "doubao-pro",              ← server.py 附加
    "project_id": "customer-service"    ← server.py 附加
}
```

**两个附加字段的用意**：

| 字段 | 值 | 当前用途 | 后续用途（Step 6 网关） |
|------|-----|---------|---------------------|
| `model` | `"doubao-pro"` | 客户端知道用了哪个逻辑模型 | 日志记录 |
| `project_id` | `"customer-service"` | 原样回传 | 配额计量、日志关联 |

注意返回的是逻辑名 `"doubao-pro"` 而**不是**物理端点 `"doubao-seed-1-8-251228"`。客户端永远看不到实际的 Endpoint ID。

**FastMCP 的包装过程**：

```
dict                     json.dumps()              MCP TextContent            JSON-RPC 响应
{"content":...}    →    '{"content":...}'    →    TextContent(text=...)  →   HTTP 200 返回客户端
```

## 各模块的信息边界

每个模块只知道自己该知道的事情：

```
                    知道什么                         不知道什么
                    ────────                         ──────────
客户端              逻辑模型名 "auto" / "doubao-pro"  物理端点 "doubao-seed-..."
                    project_id "customer-service"     API Key
                    MCP 端点 127.0.0.1:9001           豆包 API 的 URL

server.py           逻辑名 ↔ 物理端点的映射关系       HTTP 请求怎么构造
                    project_id                        OpenAI SDK 的用法
                    路由结果和模型参数                  JSON-RPC 协议细节

router.py           所有逻辑模型名和参数               物理端点
                    谁是默认模型                       API Key
                    config.yaml 的完整内容             project_id

doubao.py           物理端点 "doubao-seed-..."         逻辑模型名 "doubao-pro"
                    API Key                           project_id
                    请求/响应格式                      模型路由规则

config.yaml         模型参数定义                       一切运行时信息
                    路由规则                           密钥、端点

.env                API Key、物理端点                  一切业务逻辑
                    Base URL                          模型参数、路由规则
```

## 如果换成 Embedding 调用，链路有什么不同？

主要差异在 ⑥⑦⑧⑨ 步骤——doubao.py 内部走了不同的路径：

```
chat_completion 的路径                    embedding 的路径
──────────────────                       ─────────────────
OpenAI SDK                               模型含 "vision"？
  ↓                                        ├─ Yes → httpx 直调
client.chat.completions.create()           │   POST /embeddings/multimodal
  ↓                                        │   input: [{"type":"text","text":"..."}]
POST /chat/completions                     │   返回: {"data": {"embedding": [...]}}
                                           │
                                           └─ No → OpenAI SDK
                                               client.embeddings.create()
                                               POST /embeddings
                                               返回: {"data": [{"embedding": [...]}]}
```

server.py 和 router.py 完全无感——它们不知道也不关心 doubao.py 内部用了哪种方式。
