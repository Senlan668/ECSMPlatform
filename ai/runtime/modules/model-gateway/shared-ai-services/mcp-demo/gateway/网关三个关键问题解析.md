逐个回答：trace_id 跨服务传递、日志存储位置、配额计量全流程。每个问题先说现状（代码实际做了什么），再说缺陷和改进方向。

## 问题一：trace_id 是如何跨服务传递的？请求挂了怎么定位？

### 直接回答：trace_id 没有跨服务传递

当前代码中，trace_id **只在网关进程内流转**，不会传给 MCP 服务。网关和 MCP 服务的日志是两套独立的体系，靠时间戳粗略对齐。

### 现状：trace_id 只在网关内部传递

```
  客户端                     网关（一个进程）                       MCP 服务（4 个独立进程）
    │                           │                                      │
    │  POST /api/tool/call      │                                      │
    │ ─────────────────────→    │                                      │
    │                           │                                      │
    │                     ① logger.py                                  │
    │                        生成 trace_id = "a1b2c3d4e5f6"           │
    │                        request.state.trace_id = "a1b2c3d4e5f6"  │
    │                           │                                      │
    │                     ② auth.py                                    │
    │                        读 trace_id，打日志:                       │
    │                        [a1b2c3d4e5f6] 认证通过                   │
    │                           │                                      │
    │                     ③ quota.py                                   │
    │                        读 trace_id，打日志:                       │
    │                        [a1b2c3d4e5f6] 配额检查通过               │
    │                           │                                      │
    │                     ④ router.py                                  │
    │                        读 trace_id，打日志:                       │
    │                        [a1b2c3d4e5f6] tool/call | ...           │
    │                           │                                      │
    │                     ⑤ mcp_client_manager.py                     │
    │                        打日志: call_tool | service=... tool=...  │
    │                        ⚠️ 注意：这里没有 trace_id 前缀！         │
    │                        ⚠️ 也没有把 trace_id 传给 MCP 服务！      │
    │                           │                                      │
    │                           │   streamablehttp_client(url)         │
    │                           │ ──────────────────────────────→      │
    │                           │                                 MCP 服务自己的日志：
    │                           │                                 没有 trace_id
    │                           │                                 只有自己记录的内容
    │                           │ ◄──────────────────────────────      │
    │                           │                                      │
    │  ◄────────────────────    │                                      │
    │  X-Trace-ID: a1b2c3d4e5f6│                                      │
    │                           │                                      │
```

### 证据：看 mcp_client_manager.py 的代码

```python
# mcp_client_manager.py 第 59-62 行
async def call_tool(self, service: str, tool: str, arguments: dict) -> Any:
    logger.info("call_tool | service=%s tool=%s", service, tool)   # ← 没有 trace_id
    async with self._session(service) as session:
        result = await session.call_tool(tool, arguments)          # ← 没有传 trace_id
        return self._parse_tool_result(result)
```

```python
# _session 方法，第 38-44 行
@asynccontextmanager
async def _session(self, service: str):
    url = self._get_url(service)
    async with streamablehttp_client(url) as (read, write, _):    # ← 没有自定义 Header
        async with ClientSession(read, write) as session:
            await session.initialize()
            yield session
```

`streamablehttp_client(url)` 调用时没有传 `headers` 参数，trace_id 不会到达 MCP 服务。

### 请求挂了怎么定位？现在能做到什么程度

```
场景：客户端调用 /api/tool/call，返回 502

客户端拿到的信息：
  HTTP 502
  X-Trace-ID: a1b2c3d4e5f6
  {"error": "MCP 服务调用失败: Connection refused",
   "hint": "请确认 llm-gateway 服务已启动",
   "trace_id": "a1b2c3d4e5f6"}

用 trace_id 去网关日志搜：
  2026-04-06 10:05:12 [gateway.access]  [a1b2c3d4e5f6] → POST /api/tool/call     ← 进来了
  2026-04-06 10:05:12 [gateway.auth]    [a1b2c3d4e5f6] 认证通过: project=...      ← 认证没问题
  2026-04-06 10:05:12 [gateway.router]  [a1b2c3d4e5f6] tool/call | service=llm... ← 路由收到了
  2026-04-06 10:05:12 [gateway.mcp]     call_tool | service=llm-gateway tool=...   ← ⚠️ 没有 trace_id
  2026-04-06 10:05:12 [gateway.router]  [a1b2c3d4e5f6] MCP 调用异常              ← 失败了
  Traceback (most recent call last):
    ... ConnectionRefusedError ...                                                  ← 异常堆栈
  2026-04-06 10:05:12 [gateway.access]  [a1b2c3d4e5f6] ← 502  15ms               ← 出去了
```

**能定位到的范围**：

| 失败位置 | 能否定位 | 怎么看出来 |
|---------|---------|----------|
| 网关认证失败 | ✅ 能 | 401 + `[trace_id] 认证失败` 日志 |
| 网关配额超限 | ✅ 能 | 429 + `[trace_id] 配额已用尽` 日志 |
| MCP 服务连不上 | ✅ 能 | 502 + `ConnectionRefused` 异常堆栈 + hint 说哪个服务 |
| MCP 服务返回业务错误 | ✅ 能 | 502 + `MCPCallError` 错误信息 |
| MCP 服务内部哪行代码报的错 | ❌ 不能 | 网关只拿到 MCP 返回的错误文本，看不到 MCP 服务的堆栈 |
| MCP 服务处理慢是哪个环节慢 | ❌ 不能 | 网关只知道总耗时，不知道 MCP 内部的时间分布 |

### 当前缺陷总结

```
断裂点 ①：mcp_client_manager.py 的日志没带 trace_id
  call_tool | service=llm-gateway tool=chat_completion     ← 和哪个请求对应？不知道

断裂点 ②：网关 → MCP 服务之间没有传 trace_id
  MCP 服务收到请求，不知道这是网关的 a1b2c3d4e5f6 请求

断裂点 ③：多个 MCP 服务各自独立日志，无法串联
  LLM 网关日志里有一条报错，Prompt 中心日志里有一条成功
  → 它们是同一个用户请求触发的吗？不知道
```

### 改进方案

**最小改动**：让 `mcp_client_manager.py` 的日志也带上 trace_id

```python
# 改法：call_tool 接收 trace_id 参数
async def call_tool(self, service: str, tool: str, arguments: dict, trace_id: str = "?"):
    logger.info("[%s] call_tool | service=%s tool=%s", trace_id, service, tool)
    async with self._session(service) as session:
        result = await session.call_tool(tool, arguments)
        return self._parse_tool_result(result)
```

**完整方案**：通过 HTTP Header 传递 trace_id 到 MCP 服务

```python
# 改法：streamablehttp_client 传自定义 Header
@asynccontextmanager
async def _session(self, service: str, trace_id: str = "?"):
    url = self._get_url(service)
    headers = {"X-Trace-ID": trace_id}
    async with streamablehttp_client(url, headers=headers) as (read, write, _):
        async with ClientSession(read, write) as session:
            await session.initialize()
            yield session

# MCP 服务端也需要改：读取 X-Trace-ID 并在日志中使用
```

**生产级方案**：接入 OpenTelemetry

```
客户端 → 网关 → MCP 服务 → 数据库/外部 API
   │        │        │           │
   └────────┴────────┴───────────┘
         所有 span 通过 trace_id 串联
         Jaeger / Zipkin 可视化完整链路
```

## 问题二：日志文件存储到哪里

### 直接回答：没有日志文件，全部输出到终端

### 证据：main.py 的日志配置

```python
# main.py 第 24-27 行
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(name)s] %(message)s",
)
```

`logging.basicConfig()` 没有传 `filename` 参数 → 默认输出到 `stderr`（终端）。

所有 MCP 服务也是一样的配置：

```
网关           main.py          logging.basicConfig(...)     → 终端
LLM 网关       server.py        logging.basicConfig(...)     → 终端
RAG 服务       server.py        logging.basicConfig(...)     → 终端
记忆服务       server.py        logging.basicConfig(...)     → 终端
Prompt 中心    server.py        logging.basicConfig(...)     → 终端
```

**全部 5 个进程都是输出到各自的终端窗口，没有写文件。**

### 实际运行时的样子

```
终端窗口 1（网关）:
  $ uvicorn gateway.main:app --port 8000
  2026-04-06 10:00:01 [gateway] HTTP 网关启动
  2026-04-06 10:00:01 [gateway]   注册服务: llm-gateway, rag-service, ...
  2026-04-06 10:05:12 [gateway.access] [a1b2c3d4e5f6] → POST /api/tool/call
  2026-04-06 10:05:12 [gateway.auth]   [a1b2c3d4e5f6] 认证通过: project=...
  2026-04-06 10:05:12 [gateway.router] [a1b2c3d4e5f6] tool/call | ...
  2026-04-06 10:05:13 [gateway.access] [a1b2c3d4e5f6] ← 200  1250ms

终端窗口 2（LLM 网关）:
  $ python -m shared.llm_gateway.server
  2026-04-06 10:00:00 [llm_gateway] LLM 网关启动 port=9001
  2026-04-06 10:05:12 [llm_gateway] chat_completion | model=doubao-pro ...
                                    ↑ 没有 trace_id，和终端 1 对不上

终端窗口 3（记忆服务）:
  ...各自独立的日志...

终端窗口 4（RAG 服务）:
  ...各自独立的日志...

终端窗口 5（Prompt 中心）:
  ...各自独立的日志...
```

### 这意味着什么

```
问题 ①：关掉终端窗口，日志就没了
  日志只存在于终端缓冲区，不会持久化

问题 ②：5 个终端窗口之间的日志无法搜索
  想找一个请求的完整链路，要在 5 个窗口之间肉眼切换

问题 ③：日志量大了终端会截断
  终端缓冲区有限，早期日志会被滚出屏幕
```

### 改进方案

**最小改动**：加 `filename` 参数写文件

```python
# main.py
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(name)s] %(message)s",
    filename="logs/gateway.log",    # ← 加这一行
)
```

**推荐改动**：同时输出到终端和文件，文件按天轮转

```python
import logging
from logging.handlers import TimedRotatingFileHandler

logger = logging.getLogger()
logger.setLevel(logging.INFO)
formatter = logging.Formatter("%(asctime)s [%(name)s] %(message)s")

# 终端输出
console = logging.StreamHandler()
console.setFormatter(formatter)
logger.addHandler(console)

# 文件输出，每天轮转，保留 30 天
file_handler = TimedRotatingFileHandler(
    "logs/gateway.log", when="midnight", backupCount=30, encoding="utf-8"
)
file_handler.setFormatter(formatter)
logger.addHandler(file_handler)
```

**生产级方案**：JSON 格式 + 日志收集

```
各服务输出 JSON 格式日志
  → Filebeat / Fluentd 收集
  → Elasticsearch 存储
  → Kibana 查询

好处：
  - 所有服务的日志集中到一处
  - 用 trace_id 一搜就能看到完整链路
  - 支持按 project_id、service、status_code 等字段过滤
```

## 问题三：Quota 是如何进行 LLM 用量统计的

### 全流程示意

用一个完整场景走一遍：客服项目调用 LLM，从进入网关到配额扣减。

```
10:05:12.000  客户端发送请求
  POST /api/tool/call
  X-API-Key: ${MCP_CUSTOMER_SERVICE_API_KEY}
  {"service": "llm-gateway", "tool": "chat_completion",
   "arguments": {"messages": [{"role":"user","content":"专业版多少钱？"}]}}
      │
      │
      ▼
10:05:12.001  ① LoggerMiddleware
      trace_id = "a1b2c3d4e5f6"
      │
      ▼
10:05:12.002  ② AuthMiddleware
      "${MCP_CUSTOMER_SERVICE_API_KEY}" → project_id = "customer-service"
      │
      ▼
10:05:12.003  ③ QuotaMiddleware ── 请求前检查（只检查，不扣）
      │
      │  读取当前状态：
      │    _limits["customer-service"] = 100000         ← config.yaml 配的
      │    _usage["customer-service"]["2026-04-06"] = 3582   ← 今天已用
      │
      │  判断：3582 < 100000 → True → 放行
      │
      │  ⚠️ 注意：这里不扣额度！
      │  ⚠️ 因为还不知道这次 LLM 调用会消耗多少 Token
      │
      ▼
10:05:12.004  ④ router.py  call_tool 端点
      │
      │  解析请求体：
      │    service = "llm-gateway"
      │    tool = "chat_completion"
      │    arguments = {"messages": [...]}
      │
      ▼
10:05:12.005  ⑤ mcp_client_manager.py
      │
      │  建立连接：streamablehttp_client("http://127.0.0.1:9001/mcp")
      │  MCP 握手：session.initialize()
      │  执行调用：session.call_tool("chat_completion", arguments)
      │
      │  ──── Streamable HTTP ────→  LLM 网关 MCP Server (:9001)
      │                               │
      │                               │  调用大模型 API（豆包/OpenAI/...）
      │                               │  大模型返回文本 + Token 用量
      │                               │
      │  ◄──── MCP 响应 ────────────  │
      │
      │  MCP 返回内容（content[0].text）：
      │  '{"content":"专业版每月 2999 元...","usage":{"prompt_tokens":28,
      │    "completion_tokens":156,"total_tokens":184},"model":"doubao-pro"}'
      │
      │  _parse_tool_result → json.loads → dict
      │
      ▼
10:05:13.200  ⑥ 回到 router.py ── 响应后判断是否扣配额
      │
      │  三个条件全部满足才扣：
      │
      │  条件 1：body.tool == "chat_completion"        ✅ 是 LLM 调用
      │  条件 2：isinstance(result, dict)              ✅ 是字典
      │  条件 3："usage" in result                     ✅ 有 usage 字段
      │
      │  提取：total_tokens = result["usage"]["total_tokens"] = 184
      │  条件 4：total_tokens > 0                      ✅
      │
      │  执行扣减：
      │  request.app.state.quota.add("customer-service", 184)
      │
      ▼
10:05:13.201  ⑦ QuotaTracker.add() 内部
      │
      │  执行前：_usage = {"customer-service": {"2026-04-06": 3582}}
      │  计算：3582 + 184 = 3766
      │  执行后：_usage = {"customer-service": {"2026-04-06": 3766}}
      │
      │  日志：quota_add | project=customer-service tokens=+184
      │
      ▼
10:05:13.202  响应返回客户端
      HTTP 200 OK
      X-Trace-ID: a1b2c3d4e5f6
      {"content":"专业版每月 2999 元...","usage":{"total_tokens":184},...}
```

### 只有 chat_completion 会扣配额

```
router.py 第 68-71 行的判断逻辑：

if body.tool == "chat_completion" and isinstance(result, dict) and "usage" in result:
    total_tokens = result["usage"].get("total_tokens", 0)
    if total_tokens > 0:
        request.app.state.quota.add(project_id, total_tokens)
```

不同 Tool 的扣费情况：

| Tool 调用 | 目标服务 | 扣配额？ | 原因 |
|-----------|---------|---------|------|
| `chat_completion` | llm-gateway | ✅ 扣 | 消耗了大模型 Token |
| `save_memory` | memory-service | ❌ 不扣 | tool 名不是 chat_completion |
| `recall_memory` | memory-service | ❌ 不扣 | tool 名不是 chat_completion |
| `search_knowledge` | rag-service | ❌ 不扣 | tool 名不是 chat_completion |
| `get_prompt` | prompt-hub | ❌ 不扣 | 这是 Prompt 接口，不走 tool/call |

### 配额内存数据结构的完整生命周期

```
=== 网关启动 ===

config.yaml 加载：
  projects:
    customer-service:
      quota:
        daily_tokens: 100000
    writing-assistant:
      quota:
        daily_tokens: 100000

QuotaTracker.__init__ 后的内部状态：
  _limits = {
      "customer-service": 100000,
      "writing-assistant": 100000,
  }
  _usage = {}                              ← 空的，刚启动没人用过


=== 第 1 次 LLM 调用（客服项目，消耗 184 tokens）===

  QuotaMiddleware.check:
    _usage.get("customer-service", {}).get("2026-04-06", 0) → 0
    0 < 100000 → 放行

  router → LLM → 成功 → quota.add("customer-service", 184)

  _usage = {
      "customer-service": {
          "2026-04-06": 184               ← 新建了项目和日期的 key
      }
  }


=== 第 2 次 LLM 调用（写作项目，消耗 312 tokens）===

  QuotaMiddleware.check:
    _usage.get("writing-assistant", {}).get("2026-04-06", 0) → 0
    0 < 100000 → 放行

  router → LLM → 成功 → quota.add("writing-assistant", 312)

  _usage = {
      "customer-service": {
          "2026-04-06": 184
      },
      "writing-assistant": {
          "2026-04-06": 312               ← 写作项目独立计量
      }
  }


=== 第 3 次 LLM 调用（客服项目，消耗 256 tokens）===

  QuotaMiddleware.check:
    _usage["customer-service"]["2026-04-06"] → 184
    184 < 100000 → 放行

  router → LLM → 成功 → quota.add("customer-service", 256)

  _usage = {
      "customer-service": {
          "2026-04-06": 440               ← 184 + 256 = 440，累加
      },
      "writing-assistant": {
          "2026-04-06": 312
      }
  }


=== 配额耗尽的请求 ===

  假设经过很多次调用，_usage 变为：
  _usage = {
      "customer-service": {
          "2026-04-06": 100023            ← 超过 100000 了
      }
  }

  下一个客服项目请求进入 QuotaMiddleware：
    _usage["customer-service"]["2026-04-06"] → 100023
    100023 < 100000 → False → 拦截！

    返回 429：
    {
        "error": "Token 配额已用尽",
        "project_id": "customer-service",
        "date": "2026-04-06",
        "used_tokens": 100023,
        "daily_limit": 100000,
        "remaining": 0
    }

    ⚠️ 此时写作项目不受影响，它有自己的计数


=== 第二天（自动重置）===

  日期从 "2026-04-06" 变为 "2026-04-07"

  QuotaMiddleware.check:
    _usage["customer-service"].get("2026-04-07", 0) → 0    ← 新日期，从 0 开始
    0 < 100000 → 放行

  _usage = {
      "customer-service": {
          "2026-04-06": 100023,           ← 昨天的数据还在，但没人查了
          "2026-04-07": 0                 ← 今天重新开始
      }
  }

  不需要定时任务清零——新日期自然从 0 开始


=== 网关重启 ===

  _usage = {}                              ← 内存清空，所有计数归零
  ⚠️ 这是内存方案的代价
```

### 配额查询接口

客户端随时可以查看自己的配额使用情况：

```
GET /api/quota/usage
X-API-Key: ${MCP_CUSTOMER_SERVICE_API_KEY}

→ QuotaTracker.get_usage("customer-service")

返回：
{
    "project_id": "customer-service",
    "date": "2026-04-06",
    "used_tokens": 440,
    "daily_limit": 100000,
    "remaining": 99560
}
```

| 字段 | 含义 | 计算方式 |
|------|------|---------|
| used_tokens | 今日已消耗 | `_usage[project_id][today]` |
| daily_limit | 每日上限 | `config.yaml` 中配置的 `daily_tokens` |
| remaining | 剩余可用 | `max(0, daily_limit - used_tokens)` |

### 当前方案的已知局限

| 局限 | 原因 | 影响 |
|------|------|------|
| 重启清零 | 内存存储，无持久化 | 重启后配额从 0 重新开始 |
| 单实例 | 内存计数器无法跨进程共享 | 多个网关实例各自计数，总量会超限 |
| 先放后扣 | 先放行请求，成功后才扣 | 并发时可能短暂超限（多个请求同时通过 check，都还没 add） |
| 只计 chat_completion | 硬编码了 tool 名判断 | 如果新增其他消耗 Token 的 Tool，不会自动纳入计量 |
| 历史数据不清理 | 旧日期的 key 一直留在内存 | 长期运行会缓慢增长（每天一个 key，影响很小） |
