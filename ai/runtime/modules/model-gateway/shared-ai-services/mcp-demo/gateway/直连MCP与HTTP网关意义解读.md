# 项目能否直连 MCP？Gateway 的意义是什么？

本文回答两个常被一起问的问题：**业务项目能不能绕过网关、直接用 MCP Client 连共享服务？** 以及 **既然能连，为什么还要做 HTTP Gateway？**

---

## 1. 项目可以直连 MCP 吗？

**可以。** MCP 是开放协议，只要网络可达、持有（或不需要）鉴权信息，任何进程都可以用 **MCP Client**（例如本仓库使用的 Python SDK + Streamable HTTP）连接各个 MCP Server：

```
直连方式（概念上）：

  项目 A 的进程
      │
      ├─ MCP Client ──→ http://localhost:9001/mcp  （LLM 网关）
      ├─ MCP Client ──→ http://localhost:9002/mcp  （RAG）
      ├─ MCP Client ──→ http://localhost:9003/mcp  （记忆）
      └─ MCP Client ──→ http://localhost:9004/mcp  （Prompt 中心）
```

技术上没有障碍：**网关不是 MCP 协议上的“唯一合法入口”**，它只是本演示工程为业务项目选定的 **接入方式**。

---

## 2. 直连时你会得到什么、失去什么？

### 2.1 能得到什么

- **少一层跳转**：请求不经过 FastAPI，延迟略低（通常不是瓶颈）。
- **调试 MCP 本身**：排查 Tool/Prompt 问题时，直连 MCP Server 更直观。
- **完全控制连接**：例如长期会话、自定义重试，都可按需在项目内实现。

### 2.2 需要自己补上的事情（网关里已经做过的）

若生产环境也坚持直连，等价能力要在 **项目侧或别处** 重复建设或分散实现：

| 能力 | 经网关（本仓库） | 项目直连 MCP |
|------|------------------|--------------|
| 统一入口 | 一个 `http://gateway:8000` | 每个项目记住 4+ 个 URL |
| 项目级身份 | `X-API-Key` → `project_id` | 需在 MCP 层自研鉴权或网络隔离 |
| 统一审计日志 / Trace | 网关中间件集中打日志 | 每个项目自己打，格式难统一 |
| Token 配额 / 限流 | `quota.py` 按项目计量 | 每个项目或每个 MCP 前重复做 |
| 对业务代码的协议形态 | 普通 `httpx` + JSON | 必须引入 MCP SDK、理解会话与传输 |

因此：**直连在“原型 / 单服务调试”很合适；在多项目、要治理和统一运维时，成本会明显上升。**

---

## 3. Gateway 的意义（为什么仍然值得要）

可以把本仓库的 **HTTP 网关**理解成三层意义上的“边界”：

### 3.1 协议与集成边界：对业务隐藏 MCP

- 业务侧只写 **REST 风格 HTTP**（如 `POST /api/tool/call`），用熟悉的 JSON 与 Header。
- **MCP 的传输、会话、多服务地址** 收敛在网关进程内，由 `mcp_client_manager` 统一维护。

这样 **客服 / 写作** 等项目不必依赖 MCP SDK，降低栈复杂度，也便于用任意语言写一个薄 HTTP 客户端对接（不限于 Python）。

### 3.2 治理边界：认证、配额、可观测性一处完成

与《[为什么用 FastAPI 网关而不是 MCP Proxy](./为什么用FastAPI网关而不是MCP Proxy.md)》中的结论一致：API Key、Trace ID、按项目 Token 配额等，在 **HTTP + FastAPI 中间件** 里实现成本低、生态成熟。若完全直连 MCP，这些能力要么进每个 MCP Server（重复），要么进每个业务项目（更重复）。

### 3.3 运维与安全边界：暴露面可控

典型部署是：

- **只对外或只对业务网暴露网关**（HTTPS、WAF、CLB 等挂在网关前）。
- **各 MCP 服务仅内网可达**，甚至不暴露公网。

直连模式下，要么每个 MCP 服务都暴露在可被项目访问的网络上并各自加固，要么仍要有一层“代理”——**那一层往往就会演化成你现在看到的网关**。

---

## 4. 和“云厂商 CLB”是不是一回事？

**不是。** CLB 主要解决 **流量分发、高可用、TLS 终结** 等基础设施问题；本网关解决的是 **应用语义与治理**（谁可以调、调哪个逻辑服务、如何转成 MCP、如何计量）。二者常 **串联使用**：CLB → 多副本 Gateway → 内网 MCP。详见日常问答中的对比即可，此处不展开。

---

## 5. 什么时候适合直连、什么时候适合走网关？

| 场景 | 更合适的做法 |
|------|----------------|
| 本地调试单个 MCP Server | 直连或使用 MCP Inspector，简单直接 |
| 教 MCP 协议、写 Server 的人 | 直连，专注协议本身 |
| 多个业务项目共享同一套能力 | **HTTP 网关（或等价接入层）** 收口更省事 |
| 要统一 API Key、配额、审计 | **网关集中实现** |
| 业务团队不想引入 MCP SDK | **HTTP 网关** |
| 极端低延迟、单项目专用部署 | 可评估直连 + 自建最小代理或网络层隔离 |

---

## 6. 小结

1. **项目可以直连 MCP**——协议允许，SDK 支持，适合开发与单点调试。  
2. **Gateway 的意义**不在于“代替 MCP”，而在于：**用 HTTP 统一接入、集中治理、缩小暴露面，并让业务代码与 MCP 细节解耦。**  
3. 是否上网关是 **架构与运维取舍**，不是技术禁令；本演示工程选择网关，是为了体现 **多项目共享 + 治理** 这一教学目标。
