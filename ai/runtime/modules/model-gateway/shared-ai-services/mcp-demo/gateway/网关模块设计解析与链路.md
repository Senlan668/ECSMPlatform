# 网关模块设计解析与链路数据示意

逐个文件拆解：这段代码为什么这么写，数据在里面怎么流。每个模块给出**设计意图**、**核心代码解读**和**链路数据走向**。

---

## 1. `main.py` — 入口：把零件装成一台网关

### 它要解决什么

6 个源文件各自只是零件（中间件、路由、MCP 客户端）。`main.py` 的职责是把这些零件按正确顺序组装到一个 FastAPI 应用上，然后一条命令启动。

### 启动时做了什么

```
config.yaml 加载
      │
      ├──► MCPClientManager(services)  →  挂到 app.state.mcp
      │     管理到 4 个 MCP 服务的连接
      │
      ├──► QuotaTracker(projects)      →  挂到 app.state.quota
      │     跟踪每个项目的 Token 用量
      │
      ├──► app.include_router(router)  →  注册 6 个 HTTP 端点
      │
      └──► 按顺序挂 3 层中间件
```

### 中间件顺序（重点）

FastAPI / Starlette 的中间件是**洋葱模型**——**最后 add 的最先执行**：

```python
# main.py 代码顺序              实际请求经过顺序
app.add_middleware(Quota  ...)   # 第 3 个 add → 最内层（第 3 个执行）
app.add_middleware(Auth   ...)   # 第 2 个 add → 中间层（第 2 个执行）
app.add_middleware(Logger ...)   # 第 1 个 add → 最外层（第 1 个执行）
```

为什么要这样排：

```
请求 → Logger（先发 trace_id，后面才能打日志）
     → Auth  （先认证拿到 project_id，配额才知道查谁）
     → Quota （拿到 project_id 后检查额度）
     → Router（一切就绪，转发到 MCP 服务）
```

### 为什么用 app.state 而不是全局变量

```
全局变量方案（不好）：                   app.state 方案（当前）：
  import mcp_manager                      request.app.state.mcp
  在 router.py 里直接引用                 路由通过 request 拿到

  问题：测试时替换困难                    好处：测试时可以注入 mock
  问题：循环引用风险                      好处：生命周期和 app 绑定
```

## 2. `logger.py` — 给每个请求发一张「身份证」

### 它要解决什么

网关并发处理多个请求，日志全混在一起看不清。需要一个**唯一标识**把同一个请求的入站、鉴权、MCP 调用、出站日志串起来。

### 核心逻辑（只有 15 行有效代码）

```
                     请求进入
                        │
                        ▼
           trace_id = uuid4().hex[:12]
           例如: "a1b2c3d4e5f6"
                        │
           ┌────────────┴────────────┐
           │                          │
    写入 request.state          打入站日志
    .trace_id = "a1b2c3d4e5f6"    [a1b2c3d4e5f6] → POST /api/tool/call
           │                          │
           └────────────┬────────────┘
                        │
                   call_next(request)
             （Auth → Quota → Router 执行）
                        │
                        ▼
                    拿到 Response
                        │
           ┌────────────┴────────────┐
           │                          │
    计算耗时 elapsed_ms          打出站日志
    = (当前时间 - start) × 1000    [a1b2c3d4e5f6] ← 200  125ms
           │                          │
           └────────────┬────────────┘
                        │
         Response.headers["X-Trace-ID"] = "a1b2c3d4e5f6"
                        │
                        ▼
                    响应返回客户端
```

### 数据流

```
输入：一个普通的 HTTP 请求（没有 trace_id）
输出：request.state.trace_id（供下游中间件和路由使用）
      响应头 X-Trace-ID（供客户端关联日志）

下游使用示例：
  auth.py:   logger.warning("[%s] 认证失败", trace_id)
  quota.py:  logger.warning("[%s] 配额已用尽", trace_id)
  router.py: logger.info("[%s] tool/call | ...", trace_id)

客户端侧：
  Response Header:  X-Trace-ID: a1b2c3d4e5f6
  → 拿这个 ID 去网关日志里搜，就能找到完整请求链路
```

### 设计选择

| 决策 | 选了什么 | 为什么 |
|------|---------|--------|
| ID 生成方式 | `uuid4().hex[:12]` | 12 位 hex = 16^12 ≈ 2.8×10^14 种，碰撞概率极低，比完整 UUID 短 |
| ID 传递方式 | `request.state` | Starlette 标准做法，中间件间共享数据的官方通道 |
| 耗时计量 | `time.monotonic()` | 不受系统时钟调整影响，比 `time.time()` 更精确 |

## 3. `auth.py` — 一把钥匙对一个项目

### 它要解决什么

多个项目共用一个网关，必须知道「这个请求是谁发的」。否则配额、日志、审计全乱了。

### 启动时：构建查找表

```python
# config.yaml 中的项目配置
projects:
  customer-service:
    api_key: "${MCP_CUSTOMER_SERVICE_API_KEY}"
  writing-assistant:
    api_key: "${MCP_WRITING_ASSISTANT_API_KEY}"
```

`AuthMiddleware.__init__` 把它翻转成查找表：

```python
# 翻转后的内部结构
_key_to_project = {
    "${MCP_CUSTOMER_SERVICE_API_KEY}": "customer-service",
    "${MCP_WRITING_ASSISTANT_API_KEY}": "writing-assistant",
}
```

为什么翻转？请求来的是 Key，要查的是 project_id，翻转后 O(1) 查找。

### 运行时：每个请求的判定

```
请求到达
  │
  ├── 路径是 /api/health、/docs、/openapi.json、/redoc？
  │     是 → project_id = "__anonymous__" → 放行
  │          （健康检查和文档不需要认证）
  │
  └── 读取 Header: X-API-Key
        │
        ├── Key = "${MCP_CUSTOMER_SERVICE_API_KEY}"
        │     → project_id = "customer-service" → 放行 ✅
        │
        ├── Key = "${MCP_WRITING_ASSISTANT_API_KEY}"
        │     → project_id = "writing-assistant" → 放行 ✅
        │
        ├── Key = "wrong-key-xxx"
        │     → 查找表里没有 → 401 ❌
        │     → {"error": "无效的 API Key", "hint": "请在请求头中设置 X-API-Key"}
        │
        └── 没有 X-API-Key 头
              → api_key = "" → 查找表里没有 → 401 ❌
```

### 数据流

```
输入：request.headers["X-API-Key"]
输出：request.state.project_id（str）

三种结果：
  1. 跳过路径  → project_id = "__anonymous__"  → 继续
  2. Key 有效  → project_id = "customer-service" 等  → 继续
  3. Key 无效  → 401 JSON 响应  → 请求到此为止，不进入 Quota 和 Router
```

### 设计选择

| 决策 | 选了什么 | 为什么 |
|------|---------|--------|
| Key 位置 | 请求头 `X-API-Key` | 不出现在 URL 中，不会被日志/浏览器记录 |
| 匿名标识 | `"__anonymous__"` | 和真实项目 ID 不会重名，下游 Quota 看到就跳过 |
| 失败行为 | 直接返回 401 | 不进入下游，无效请求零成本 |

## 4. `quota.py` — 每天每个项目一个 Token 预算

### 它要解决什么

多项目共用 LLM，不限额会导致一个项目把所有 Token 用完。需要按项目、按天限流。

### 两个组件，分工明确

```
QuotaTracker（数据层）                    QuotaMiddleware（拦截层）
  管理内存计数器                            每个请求进来时检查
  提供 check / add / get_usage              超限就拦，没超就放

  ┌─────────────────────────────┐         ┌──────────────────────────────┐
  │ _limits = {                 │         │ 请求进入                       │
  │   "customer-service": 100000│         │   │                            │
  │   "writing-assistant":100000│         │   ├─ 跳过路径？→ 放行           │
  │ }                           │         │   ├─ 匿名？→ 放行              │
  │                             │         │   └─ tracker.check(project_id) │
  │ _usage = {                  │         │       ├─ True → 放行           │
  │   "customer-service": {     │         │       └─ False → 429          │
  │     "2026-04-06": 3582      │         └──────────────────────────────┘
  │   }                         │
  │ }                           │
  └─────────────────────────────┘
```

### check 和 add 的分离设计（重点）

```
常见做法（不好）：                           当前做法：
  中间件里 check + 预扣额度                   中间件只 check（不扣）
  失败了再退还                                router 里调用成功后才 add

  问题：请求失败了也扣了额度                  好处：只对真正消耗 Token 的成功请求计量
  问题：预扣多少？请求还没开始不知道          好处：用 LLM 返回的 actual usage 计量
```

在 `router.py` 中，只有 `chat_completion` Tool 会触发扣量：

```python
# router.py call_tool 端点
if body.tool == "chat_completion" and isinstance(result, dict) and "usage" in result:
    total_tokens = result["usage"].get("total_tokens", 0)
    if total_tokens > 0:
        request.app.state.quota.add(project_id, total_tokens)
```

### 链路数据走向

```
QuotaMiddleware（请求进入时）:
  ┌──────────────────────────────────────────────────────────────────────────┐
  │ project_id = "customer-service"                                         │
  │ _limits["customer-service"] = 100000                                    │
  │ _usage["customer-service"]["2026-04-06"] = 3582                        │
  │                                                                         │
  │ check: 3582 < 100000 → True → 放行                                     │
  └──────────────────────────────────────────────────────────────────────────┘

Router（请求返回时，仅 chat_completion）:
  ┌──────────────────────────────────────────────────────────────────────────┐
  │ LLM 返回: {"usage": {"total_tokens": 184}}                              │
  │ quota.add("customer-service", 184)                                      │
  │ _usage 变为: {"customer-service": {"2026-04-06": 3766}}                 │
  └──────────────────────────────────────────────────────────────────────────┘

下一个请求再来：check 拿到的是 3766，还没到 100000 → 继续放行

如果到了 100000：
  → 429 {"error": "Token 配额已用尽", "used_tokens": 100023, "daily_limit": 100000, "remaining": 0}
```

### 配额天然自动重置

```python
today = datetime.date.today().isoformat()  # "2026-04-06"
used = self._usage[project_id].get(today, 0)
```

新的一天 `today` 变了 → `get(today, 0)` 拿到 0 → 自动重置。不需要定时任务清零。

## 5. `router.py` — 把 HTTP 请求翻译成 MCP 调用

### 它要解决什么

客户端发的是普通 HTTP/JSON，MCP 服务收的是 MCP 协议。Router 做的是**协议翻译**：

```
客户端视角（HTTP）                         MCP 服务视角
POST /api/tool/call                        session.call_tool("save_memory", {...})
{"service":"memory-service",          →
 "tool":"save_memory",
 "arguments":{...}}
```

### 6 个端点一览

```
核心端点（转发 MCP 操作）：
  POST /api/tool/call     →  MCPClientManager.call_tool()     调用工具
  POST /api/tool/list     →  MCPClientManager.list_tools()    列出工具
  POST /api/prompt/get    →  MCPClientManager.get_prompt()    获取 Prompt
  GET  /api/prompt/list   →  MCPClientManager.list_prompts()  列出 Prompt

治理端点（网关自身能力）：
  GET  /api/health        →  遍历所有服务 check_health()      健康检查
  GET  /api/quota/usage   →  QuotaTracker.get_usage()         配额查询
```

### 核心端点 `/api/tool/call` 的完整链路

```
POST /api/tool/call
Body: {"service": "memory-service", "tool": "save_memory", "arguments": {...}}
  │
  │ 解析请求
  ├── trace_id = request.state.trace_id          ← 来自 LoggerMiddleware
  ├── project_id = request.state.project_id      ← 来自 AuthMiddleware
  ├── service = body.service = "memory-service"
  ├── tool = body.tool = "save_memory"
  ├── arguments = body.arguments = {...}
  │
  │ 调用 MCP
  ├── mcp = request.app.state.mcp                ← 来自 main.py
  ├── result = await mcp.call_tool("memory-service", "save_memory", {...})
  │     │
  │     ├── 成功 → result = {"status": "ok", "id": 42, ...}
  │     │     │
  │     │     ├── tool == "chat_completion" 且有 usage？
  │     │     │     是 → quota.add(project_id, total_tokens)    ← 扣配额
  │     │     │     否 → 不扣（save_memory 不消耗 Token 配额）
  │     │     │
  │     │     └── return result     → 200 OK
  │     │
  │     ├── MCPCallError → 502 {"error": "错误信息", "trace_id": "..."}
  │     │
  │     └── 其他异常 → 502 {"error": "MCP 服务调用失败: ...",
  │                         "hint": "请确认 memory-service 服务已启动",
  │                         "trace_id": "..."}
  │
  ▼
  响应返回
```

### 错误处理设计

```
两层异常捕获：

  MCPCallError（已知错误）             Exception（未知错误）
  ├── MCP 服务返回的业务错误            ├── 连接失败、超时等
  ├── 如: "未知服务: xxx"              ├── 如: ConnectionRefused
  └── 502 + error 信息                └── 502 + error + hint + logger.exception
                                           ↑ 多了 hint 告诉调用方怎么修
                                           ↑ 多了 exception 打完整堆栈到日志
```

## 6. `mcp_client_manager.py` — 连接 MCP 服务的「翻译官」

### 它要解决什么

网关需要和 4 个 MCP Server 通信。这个文件封装了所有 MCP 连接细节，让 `router.py` 不需要知道「MCP 协议长什么样」。

### 短连接模式

每次调用都是一个完整的生命周期：

```
call_tool("memory-service", "save_memory", {...})
  │
  │ ① 查 URL
  ├── _get_url("memory-service") → "http://127.0.0.1:9003/mcp"
  │
  │ ② 建连接
  ├── streamablehttp_client("http://127.0.0.1:9003/mcp")
  │     → 得到 (read_stream, write_stream, _)
  │
  │ ③ 创建会话
  ├── ClientSession(read, write)
  │     → session.initialize()   # MCP 握手
  │
  │ ④ 执行操作
  ├── session.call_tool("save_memory", {...})
  │     → CallToolResult 对象
  │
  │ ⑤ 解析结果
  ├── _parse_tool_result(result)
  │     ├── result.isError？→ 抛出 MCPCallError
  │     ├── result.content[0].text 能 JSON 解析？→ 返回 dict
  │     └── 解析失败？→ 返回原始 text 字符串
  │
  │ ⑥ 关闭连接
  └── async with 退出 → 自动关闭 session 和 HTTP 连接
```

### 为什么每次都建新连接（不复用）

```
短连接（当前）                              长连接池（生产方案）
┌───────────────────────────────┐         ┌───────────────────────────────┐
│ 每次调用：                     │         │ 启动时：建 N 条连接           │
│   建连接 → 握手 → 操作 → 关闭  │         │ 每次调用：从池中取一条复用     │
│                               │         │                               │
│ ✅ 实现简单（30 行代码）       │         │ ❌ 需处理连接断开/重连         │
│ ✅ 无状态，不怕服务重启        │         │ ❌ 需管理池大小/超时           │
│ ❌ 每次多 ~50ms 建连开销       │         │ ✅ 延迟低                     │
│                               │         │                               │
│ 适合：演示 / 低并发            │         │ 适合：高并发生产环境           │
└───────────────────────────────┘         └───────────────────────────────┘
```

### 5 个方法的输入输出

```
┌─────────────────────────────────────────────────────────────────────────┐
│ call_tool(service, tool, arguments)                                     │
│   输入: ("memory-service", "save_memory", {"session_id": "s1", ...})   │
│   输出: {"status": "ok", "id": 42}   ← 或抛 MCPCallError              │
├─────────────────────────────────────────────────────────────────────────┤
│ get_prompt(service, prompt, arguments)                                  │
│   输入: ("prompt-hub", "customer_service_qa", {"context": "...", ...}) │
│   输出: {"messages": [{"role": "user", "content": "完整提示词..."}]}   │
├─────────────────────────────────────────────────────────────────────────┤
│ list_tools(service)                                                     │
│   输入: ("memory-service")                                              │
│   输出: {"tools": [{"name": "save_memory", "description": "..."},      │
│                    {"name": "recall_memory", "description": "..."}],   │
│          "count": 6}                                                    │
├─────────────────────────────────────────────────────────────────────────┤
│ list_prompts(service)                                                   │
│   输入: ("prompt-hub")                                                  │
│   输出: {"prompts": [{"name": "customer_service_qa",                   │
│                       "description": "...",                             │
│                       "arguments": [{"name": "context", ...}]}],       │
│          "count": 2}                                                    │
├─────────────────────────────────────────────────────────────────────────┤
│ check_health(service)                                                   │
│   输入: ("llm-gateway")                                                 │
│   正常: {"service": "llm-gateway", "status": "ok", "tools": 3}        │
│   异常: {"service": "llm-gateway", "status": "error",                  │
│          "message": "Connection refused"}                               │
│   注意: 不抛异常，永远返回 dict（方便 /api/health 聚合）               │
└─────────────────────────────────────────────────────────────────────────┘
```

### MCP 结果解析的三种走向

```
session.call_tool() 返回 CallToolResult
  │
  ├── result.isError == True
  │     → content[0].text = "Tool not found"
  │     → 抛出 MCPCallError("Tool not found")
  │
  ├── result.isError == False, text 是合法 JSON
  │     → content[0].text = '{"status":"ok","id":42}'
  │     → json.loads → 返回 dict {"status": "ok", "id": 42}
  │
  └── result.isError == False, text 不是 JSON
        → content[0].text = "操作成功"
        → json.loads 失败 → 直接返回字符串 "操作成功"
```

## 全链路：一个请求从头到尾经历了什么

以客服项目调用 LLM 为例，完整走一遍 6 个文件：

```
客户端发送:
  POST /api/tool/call
  X-API-Key: ${MCP_CUSTOMER_SERVICE_API_KEY}
  {"service": "llm-gateway", "tool": "chat_completion", "arguments": {...}}
      │
      │  ┌─────────────── main.py 组装的 FastAPI 应用 ─────────────────┐
      │  │                                                              │
      ▼  │  ① logger.py                                                │
 ─────── │  trace_id = "a1b2c3d4e5f6"                                  │
         │  request.state.trace_id = "a1b2c3d4e5f6"                    │
         │  日志: [a1b2c3d4e5f6] → POST /api/tool/call                │
         │       │                                                      │
         │       ▼                                                      │
         │  ② auth.py                                                   │
         │  X-API-Key = "${MCP_CUSTOMER_SERVICE_API_KEY}"                              │
         │  → 查表 → project_id = "customer-service"                   │
         │  request.state.project_id = "customer-service"              │
         │  日志: [a1b2c3d4e5f6] 认证通过: project=customer-service   │
         │       │                                                      │
         │       ▼                                                      │
         │  ③ quota.py                                                  │
         │  _usage["customer-service"]["2026-04-06"] = 3582            │
         │  _limits["customer-service"] = 100000                       │
         │  3582 < 100000 → 放行                                       │
         │       │                                                      │
         │       ▼                                                      │
         │  ④ router.py  /api/tool/call                                │
         │  解析: service=llm-gateway, tool=chat_completion            │
         │       │                                                      │
         │       ▼                                                      │
         │  ⑤ mcp_client_manager.py                                    │
         │  _get_url("llm-gateway") → "http://127.0.0.1:9001/mcp"    │
         │  streamablehttp_client → ClientSession → initialize         │
         │  session.call_tool("chat_completion", {...})                │
         │  ──────── HTTP ──────→  LLM 网关 MCP Server (:9001)        │
         │  ◄─────── 返回 ──────  {"content":"...","usage":{...}}      │
         │  json.loads → dict                                           │
         │  关闭连接                                                    │
         │       │                                                      │
         │       ▼                                                      │
         │  ⑥ 回到 router.py                                           │
         │  tool == "chat_completion" 且有 usage                       │
         │  → quota.add("customer-service", 184)                       │
         │  _usage 变为 {"customer-service": {"2026-04-06": 3766}}    │
         │  return result                                               │
         │       │                                                      │
         │       ▼                                                      │
         │  ③' quota.py — 响应路径（无额外操作，直接透传）              │
         │  ②' auth.py  — 响应路径（无额外操作，直接透传）              │
         │  ①' logger.py — 响应路径                                     │
         │  日志: [a1b2c3d4e5f6] ← 200  1250ms                        │
         │  Response.headers["X-Trace-ID"] = "a1b2c3d4e5f6"           │
         │                                                              │
         └──────────────────────────────────────────────────────────────┘
      │
      ▼
客户端收到:
  HTTP 200 OK
  X-Trace-ID: a1b2c3d4e5f6
  {"content": "专业版每月 2999 元...", "usage": {"total_tokens": 184}, ...}
```

### request.state 在整条链路中的传递

```
                    logger.py 写入           auth.py 写入
                         │                       │
                         ▼                       ▼
request.state = { trace_id: "a1b2c3d4e5f6", project_id: "customer-service" }
                         │                       │
                 ┌───────┴───────┐       ┌──────┴──────┐
                 │ quota.py 读取  │       │ router.py   │
                 │ auth.py 读取   │       │ 读取两个     │
                 └───────────────┘       └─────────────┘
```

`request.state` 就是中间件之间的「传话筒」——前面的中间件写入，后面的中间件和路由读取。
